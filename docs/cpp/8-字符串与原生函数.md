# C++ 字符串与原生函数

---

## 1. std::string 底层实现

### 概念与本质

std::string 是对动态字符数组的 RAII 封装，其核心是一个 `{pointer, size, capacity}` 三元组，不同标准库实现在小字符串优化策略上有本质差异。

### 底层原理：三代实现策略

#### 1.1 Eager Copy（早期实现）

最朴素的实现：每次拷贝都深拷贝整个缓冲区。

```
string a = "hello";
string b = a;  // 立即 malloc + memcpy 整个缓冲区
```

问题：频繁拷贝大字符串时性能差。早期 STL 实现（SGI STL）采用此策略。

#### 1.2 COW（Copy-On-Write，gcc 4.x libstdc++）

核心思想：多个 string 共享同一块堆内存，只在修改时才真正拷贝。

```
┌─────────────┐    ┌──────────────────────────────┐
│ string a     │───▶│ refcount=2 │ "hello\0"       │
├─────────────┤    └──────────────────────────────┘
│ string b     │───▶       ▲ 共享同一块内存
└─────────────┘
```

**引用计数机制**：

```cpp
// COW 实现的简化模型（gcc 4.x libstdc++ 风格）
struct _Rep {
    size_t _M_length;
    size_t _M_capacity;
    _Atomic_word _M_refcount;  // 引用计数，存在字符数据前面

    char* _M_data() { return reinterpret_cast<char*>(this + 1); }
};
```

- 拷贝构造：`refcount++`，O(1)
- 修改操作（`operator[]` 非 const、`push_back` 等）：检查 `refcount > 1`，若是则先深拷贝再修改
- 析构：`refcount--`，为 0 时释放内存

**C++11 为什么禁止 COW？**

```cpp
// 多线程场景下的致命问题
std::string shared = "hello world";  // 全局/共享

void thread_func() {
    std::string local = shared;  // refcount++ 需要原子操作
    local[0] = 'H';             // 触发 COW：refcount-- 再 malloc+copy
    // 即使 local 是线程私有的，refcount 的原子操作也带来缓存行竞争
}
```

C++11 标准要求 `operator[]` 不能使迭代器失效（COW 下修改会重新分配内存），且多线程下原子引用计数的 cache line bouncing 严重影响性能。GCC 5.1 起 libstdc++ 切换到 SSO。

#### 1.3 SSO（Small String Optimization，现代主流）

核心思想：短字符串直接存储在 string 对象内部（栈上），避免堆分配。

**内存布局（以 libc++ 为例，64 位平台，sizeof(string) = 24）**：

```
短字符串模式（length <= 22）：
┌───────────────────────────────────────────────┐
│ byte 0        │ bytes 1-23                     │
│ size/flag     │ 字符数据（最多22字节+\0）        │
│ [0SSSSSSS]    │ inline buffer                  │
└───────────────────────────────────────────────┘
  ▲ 最高位=0 表示短字符串模式，低7位存储 size

长字符串模式（length > 22）：
┌───────────────────────────────────────────────┐
│ cap (8B)      │ size (8B)       │ ptr (8B)     │
│ [1CCCCCCC...] │                 │ → 堆内存      │
└───────────────────────────────────────────────┘
  ▲ cap 最高位=1 表示长字符串模式
```

**libstdc++（GCC 5+）的 SSO**：sizeof(string) = 32，inline buffer 可存 15 字节。

```cpp
// libstdc++ 简化布局
struct basic_string {
    char*  _M_dataplus;   // 指向 inline buffer 或堆内存
    size_t _M_string_length;
    union {
        char   _M_local_buf[16];  // SSO buffer（15字符 + \0）
        size_t _M_allocated_capacity;
    };
};
// sizeof = 8 + 8 + 16 = 32
```

**为什么 SSO 阈值是 15 或 22？** 因为 string 对象本身已经占据 24/32 字节栈空间，利用这些空间的 padding 存储短字符串，完全"免费"——不需要额外的堆分配和 cache miss。

### 代码：验证 SSO

```cpp
#include <iostream>
#include <string>
using namespace std;

int main() {
    string s;
    // 打印 inline buffer 地址 vs 对象地址
    cout << "sizeof(string): " << sizeof(s) << endl;
    cout << "object addr:    " << (void*)&s << endl;
    cout << "data addr:      " << (void*)s.data() << endl;

    s = "short";  // SSO
    cout << "\n--- SSO mode ---" << endl;
    cout << "data addr: " << (void*)s.data() << endl;
    cout << "capacity:  " << s.capacity() << endl;

    s = "this is a long string that exceeds SSO buffer";  // 堆分配
    cout << "\n--- Heap mode ---" << endl;
    cout << "data addr: " << (void*)s.data() << endl;
    cout << "capacity:  " << s.capacity() << endl;
    return 0;
}
```

典型输出（GCC 12, x86-64）：
```
sizeof(string): 32
object addr:    0x7ffd1234abcd
data addr:      0x7ffd1234abdd  ← 在对象内部（SSO）

--- SSO mode ---
data addr: 0x7ffd1234abdd      ← 仍在栈上
capacity:  15

--- Heap mode ---
data addr: 0x5555deadbeef      ← 堆地址，远离栈
capacity:  63
```

### 对比表

| 策略 | 拷贝开销 | 线程安全 | 短字符串性能 | 采用者 |
|------|---------|---------|-------------|--------|
| Eager Copy | O(n) 每次 | 天然安全 | 差（总是堆分配） | 早期 SGI STL |
| COW | O(1) 拷贝 | 需原子操作 | 差（总是堆分配） | GCC 4.x libstdc++ |
| SSO | O(1) 短串 / O(n) 长串 | 天然安全 | 极好（零堆分配） | GCC 5+, MSVC, libc++ |

#### 1. 优缺点与权衡

**SSO（Small String Optimization）优点**：
- 短字符串零堆分配：利用 string 对象本体的 union 空间（libstdc++ 15B，libc++ 22B）存储短串，完全在栈上完成，避免 `malloc` 系统调用和堆碎片。实际业务中大量字符串（变量名、短 key、错误码）都在 SSO 阈值内
- cache 友好：短串数据和 string 元数据在同一条 cache line 内，读取 `size()` 和 `data()` 不会触发额外的 cache miss
- 线程天然安全：每个 string 对象独立拥有数据，无共享状态，不需要原子操作

**SSO 缺点/代价**：
- 对象体积增大：为了容纳 inline buffer，`sizeof(string)` 从理论最小的 24 字节（libc++）到 32 字节（libstdc++），当大量 string 对象存储在容器中但都是长字符串时，inline buffer 空间被浪费
- SSO 阈值以外仍需堆分配：超过 15/22 字节的字符串退化为常规堆分配，扩容策略通常是 2 倍，频繁拼接长字符串不 `reserve` 会产生多次 realloc

**COW（Copy-On-Write）优点**：
- 拷贝 O(1)：多个 string 共享同一堆内存，拷贝构造只需 `refcount++`，对"频繁拷贝但很少修改"的场景极为高效

**COW 缺点/代价**：
- 多线程 cache bouncing：引用计数必须用原子操作维护，即使 string 是线程私有的，`refcount` 所在 cache line 仍在多核间反复 invalidate/writeback（MESI 协议），实测性能可能比直接拷贝还差
- C++11 标准禁止：标准要求 `operator[]` 不能使迭代器失效，而 COW 下修改操作会触发深拷贝导致迭代器失效，语义矛盾。GCC 5.1 起 libstdc++ 切换到 SSO

**权衡**：现代 C++ 项目无需选择，SSO 是标准库唯一实现策略。优化重点在于：短字符串充分利用 SSO 免费；长字符串用 `reserve` 预分配避免多次扩容；只读传参用 `string_view` 避免拷贝

### 陷阱与实战经验

**reserve / resize / shrink_to_fit 的区别**：

```cpp
string s;
s.reserve(100);    // capacity >= 100, size 不变, 内容不变
s.resize(100);     // size = 100, 新增部分填充 '\0'
s.resize(100, 'x');// size = 100, 新增部分填充 'x'
s.shrink_to_fit(); // 请求释放多余容量（不保证）
```

❌ 错误：以为 reserve 会改变 size
```cpp
string s;
s.reserve(100);
s[50] = 'A';  // UB! size 仍为 0, s[50] 越界
```

✅ 正确：reserve 后用 push_back 或 append
```cpp
string s;
s.reserve(100);    // 预分配容量
s.append(50, 'A'); // size 变为 50
```

❌ 错误：COW 时代的习惯——持有 `c_str()` 指针后修改 string
```cpp
string s = "hello";
const char* p = s.c_str();
s += " world";  // 可能触发重新分配
printf("%s\n", p);  // 悬空指针！UB
```

✅ 正确：用完指针后不要再修改 string，或立即使用
```cpp
string s = "hello";
s += " world";
printf("%s\n", s.c_str());  // 安全
```

> **面试速答**
>
> `std::string` 本质是一个动态管理字符数组的 RAII 容器，解决的是 C 风格字符串手动管内存太容易泄漏和溢出的问题——分配、扩容、释放整条生命周期它全接管。现代实现里最核心的优化就是 SSO（小字符串优化）：短字符串直接存在对象本体里，libstdc++ 阈值大约 15 字节，libc++ 大约 22 字节，在阈值内完全不走堆分配。
>
> SSO 的实现是利用对象自身的 union 空间来放短串，长串才走堆分配，扩容策略通常是 2 倍。有一段历史是早期 GCC 4.x 的 COW 实现，后来被 C++11 标准明确禁止，原因是多线程下原子引用计数的 cache line bouncing 开销比省掉的拷贝还大。
>
> 有几个容易踩的坑要特别注意：`reserve` 只改 capacity 不改 size，别搞混；`c_str()` 返回的指针在 string 被修改或扩容后可能悬空；循环里频繁拼接短串要记得先 `reserve` 预分配。
>
> 和 `string_view` 比，string 拥有内存，string_view 只是非拥有视图；和 `char[]` 比，string 自动管内存但在 SSO 阈值之外会有堆分配开销。实际工程里需要拥有和修改字符串就用 `string`，只读传参用 `string_view`，性能极致的场景考虑 `reserve` 预分配或者换 `folly::fbstring`。

## 2. string_view（C++17）

### 概念与本质

`std::string_view` 是对一段连续字符序列的非拥有只读视图，本质是一个 `{const char* ptr, size_t len}` 对，不管理内存生命周期。

### 底层原理

```
string_view sv = "hello world";

┌──────────────┐
│ ptr ─────────┼───▶ "hello world\0"  （字符串字面量，静态存储区）
│ len = 11     │
└──────────────┘
sizeof(string_view) = 16   // 两个 size_t
```

零拷贝：`substr()` 返回新的 string_view（只调整 ptr 和 len），不分配内存。

```cpp
string_view sv = "hello world";
string_view sub = sv.substr(6, 5);  // sub = "world", 零拷贝
```

### 代码

```cpp
#include <iostream>
#include <string>
#include <string_view>
using namespace std;

// 优先用 string_view 做只读参数
void print_info(string_view sv) {
    cout << "content: " << sv << ", length: " << sv.size() << endl;
}

int main() {
    string s = "hello world";
    const char* cs = "c-string";

    print_info(s);       // string → string_view: 隐式转换
    print_info(cs);      // const char* → string_view: 隐式转换
    print_info("literal"); // 字面量 → string_view
    print_info(s.substr(0, 5)); // 注意：这里 s.substr 返回 string（有拷贝）

    string_view sv = s;
    print_info(sv.substr(0, 5)); // string_view::substr 是零拷贝
    return 0;
}
```

### 对比表：string vs const char* vs string_view

| 特性 | `std::string` | `const char*` | `std::string_view` |
|------|--------------|---------------|-------------------|
| 拥有内存 | 是 | 否 | 否 |
| 知道长度 | O(1) | O(n) strlen | O(1) |
| 可修改 | 是 | 否 | 否 |
| 支持 substr | 有拷贝 O(n) | 手动计算 | 零拷贝 O(1) |
| 以 \0 结尾 | 保证 | 保证 | **不保证** |
| 作为函数参数（只读） | 有拷贝开销 | 不知长度 | **最佳选择** |
| 生命周期 | 自管理 | 外部管理 | 外部管理 |

**选型决策**：
- 需要拥有和修改字符串 → `std::string`
- 函数参数只读不存储 → `std::string_view`
- C API 交互 → `const char*`
- 需要存储到成员变量/容器 → `std::string`（绝不存 string_view）

#### 2. 优缺点与权衡

**优点**：
- 零拷贝零分配：`string_view` 只是 `{ptr, len}` 对（16 字节），构造、拷贝、`substr()` 都是 O(1)，不涉及任何堆操作。作为函数参数按值传递（两个寄存器）比 `const string&` 更高效——后者遇到 `const char*` 实参会构造临时 `string` 触发堆分配
- 统一接受多种字符串类型：`string`、`const char*`、字面量都能隐式转为 `string_view`，函数只需一个重载即可处理所有场景
- `substr()` 是 O(1)：只调整指针和长度，不分配新内存，适合高频解析场景（如日志分析、协议解析）

**缺点/代价**：
- 悬空引用风险：`string_view` 不拥有底层数据，如果被引用的 `string` 析构或重新分配（`push_back`/`+=`），`string_view` 立刻悬空。这是最常见的 bug 来源，且编译器通常不会警告
- 不保证 `\0` 结尾：`string_view::substr()` 返回的子串不以 `\0` 结尾，直接传给 `printf`、`fopen` 等 C 函数是 UB，必须先转为 `string`
- 不适合存储：将 `string_view` 存入成员变量或容器中，生命周期管理极易出错，除非能保证底层数据比 `string_view` 活得更久（如字符串字面量）

**权衡**：函数参数只读场景是 `string_view` 的最佳使用场景，零开销且接口统一；需要存储字符串所有权时必须用 `string`；跨作用域传递时绝不能存 `string_view`，除非引用的是全局常量或字符串字面量

### 陷阱与实战经验

❌ 错误：string_view 引用临时 string
```cpp
string_view get_name() {
    string s = "temporary";
    return s;  // s 析构后 string_view 悬空！
}
string_view sv = get_name();  // 悬空引用，UB
```

✅ 正确：确保 string_view 引用的数据比 string_view 活得更久
```cpp
string_view get_name() {
    return "literal";  // 字符串字面量有静态生命周期，安全
}
```

❌ 错误：把 string_view 传给需要 `\0` 结尾的 C 函数
```cpp
string_view sv = "hello world";
auto sub = sv.substr(0, 5);  // sub = "hello"，但不以 \0 结尾
printf("%s\n", sub.data());  // UB! printf 需要 \0 结尾
```

✅ 正确：先转为 string
```cpp
string_view sv = "hello world";
auto sub = sv.substr(0, 5);
string tmp(sub);          // 拷贝并补 \0
printf("%s\n", tmp.c_str());
```

> **面试速答**
>
> `std::string_view` 本质就是一个 `{const char* ptr, size_t len}` 对，16 字节，对字符序列的非拥有只读视图，不管理内存生命周期。它解决的核心问题是 `const string&` 作参数会导致隐式拷贝——传字面量或 `const char*` 进去时会构造临时 string，带堆分配开销；string_view 可以同时接受 `string`、`const char*`、字面量，全程零拷贝。
>
> 它很多操作都特别便宜：`substr()` 只调整指针和长度，O(1) 没有堆分配；传参按值传（16 字节两个寄存器）比传引用还高效。
>
> 有个容易踩的坑是最常见的 bug：引用临时 `string` 会导致悬空，跨作用域保存 string_view 等于握着一个随时失效的指针。还要特别注意 string_view 不保证 `\0` 结尾，不能直接丢给 `printf`、`fopen` 这些 C 函数，必须先拷回 string 再 `.c_str()`。
>
> 和 `const string&` 比没有隐式构造开销，和 `span<char>` 比定位不同——span 用于二进制数据，string_view 专门用于文本。实际工程里函数参数只读就优先 `string_view`，需要存储所有权必须用 `string`，绝对不能跨作用域存 string_view。

## 3. 原生字符串函数实现

### 概念与本质

C 标准库字符串函数是操作以 `\0` 结尾的字节序列的底层工具，现代 glibc 实现大量使用 SIMD 指令优化，远非逐字节循环。

### 3.1 strcpy 实现

**为什么返回 `char*`？** 支持链式调用：`strlen(strcpy(dst, src))`。

```cpp
char* my_strcpy(char* dest, const char* src) {
    if (dest == NULL || src == NULL) return NULL;
    char* ret = dest;
    while ((*dest++ = *src++) != '\0');
    return ret;
}
```

**strcpy vs strncpy vs strlcpy**：

| 函数 | 终止条件 | 保证 \0 结尾 | 缓冲区溢出 | 可用性 |
|------|---------|-------------|-----------|--------|
| `strcpy(d,s)` | 遇到 \0 | 是 | **会溢出** | 标准 C |
| `strncpy(d,s,n)` | \0 或 n 字节 | **不保证** | 不会 | 标准 C |
| `strlcpy(d,s,n)` | \0 或 n 字节 | **保证** | 不会 | BSD/Linux (glibc 2.38+) |

#### 3.1.1 优缺点与权衡

**strcpy 优点**：
- 实现最简单、调用开销最小：一个循环遇到 `\0` 停止，无需传入长度参数
- 返回 `char*` 支持链式调用：如 `strlen(strcpy(dst, src))`

**strcpy 缺点/代价**：
- 缓冲区溢出的经典来源：不检查目标缓冲区大小，源串比目标长就直接写越界，是 CVE 漏洞的高频元凶

**strncpy 优点**：
- 限长拷贝不会溢出：最多写 n 字节，防止目标缓冲区越界

**strncpy 缺点/代价**：
- 不保证 `\0` 结尾：源串长度 >= n 时，目标不会追加 `\0`，后续对目标调用 `strlen`/`printf` 会越界读取。这是一个设计缺陷，必须手动补 `dest[n-1] = '\0'`
- 短串填充浪费性能：源串长度 < n 时，`strncpy` 用 `\0` 填充剩余空间直到写满 n 字节，大缓冲区场景下白白浪费 CPU 周期

**strlcpy 优点**：
- 安全且语义清晰：保证 `\0` 结尾，返回源串长度可检测截断，一个函数调用解决所有问题
- C23 纳入标准：从 BSD 扩展升级为标准函数，可移植性问题已解决

**strlcpy 缺点/代价**：
- 旧系统不可用：glibc 2.38（2023）之前不提供，需要自己实现或用其他方案

**权衡**：新代码一律用 `strlcpy` 或直接用 `std::string`；遗留 C 代码中 `strncpy` 后必须手动补 `\0`；`strcpy` 在生产代码中完全弃用

`strncpy` 的两个坑：
1. 源串长度 >= n 时，dest **不会**追加 `\0`
2. 源串长度 < n 时，会用 `\0` 填充剩余空间（性能浪费）

```cpp
char buf[8];
strncpy(buf, "hello world", 8);
// buf = {'h','e','l','l','o',' ','w','o'}  ← 没有 \0！
// 必须手动 buf[7] = '\0';
```

**安全替代：strlcpy**（C23 纳入标准）
```cpp
char buf[8];
size_t needed = strlcpy(buf, "hello world", sizeof(buf));
// buf = "hello w\0"  ← 自动截断并追加 \0
// needed = 11        ← 返回源串长度，可用于检测截断
if (needed >= sizeof(buf)) {
    // 发生了截断
}
```

> **面试速答**
>
> 字符串拷贝函数一共有三代：`strcpy` 没有长度检查，是缓冲区溢出的经典来源；`strncpy` 加了限长但设计上有坑；`strlcpy` 是 C23 标准化的，安全且语义清晰。它们要解决的根本问题就是怎么在不越界的前提下把字符串搬过去。
>
> 有两个坑要特别注意：`strncpy` 在源串长度 >= n 时不追加 `\0`，结果压根就不是合法 C 字符串，后面再用 strlen 之类的就会越界；源串 < n 时又会把剩余空间全用 `\0` 填满，白白浪费性能。`strlcpy` 就没这些问题，保证 `\0` 结尾而且返回源串长度，可以用来检测截断。另外 `strcpy` 返回 `char*` 是为了支持 `strlen(strcpy(dst, src))` 这类链式调用。
>
> 实际工程里新代码直接用 `strlcpy` 或者 `std::string`；遗留 C 代码里用了 `strncpy` 就必须手动在末尾补 `\0`；`strcpy` 永远别碰。

### 3.2 memcpy vs memmove

**概念**：
- `memcpy(dst, src, n)`：拷贝 n 字节，**不处理**内存重叠
- `memmove(dst, src, n)`：拷贝 n 字节，**正确处理**内存重叠

**为什么 memcpy 不处理重叠？** 因为处理重叠需要判断方向，多一次比较；而 90% 的使用场景不会重叠，为这 10% 牺牲 100% 的性能不值得。

**内存重叠时的正向/反向拷贝**：

```
情况1: dst < src（正向拷贝安全）
src:  [ABCDEF    ]
dst:  [  xxxxxx  ]   ← 从左到右拷贝，已拷贝的 src 区域不会被 dst 覆盖

情况2: dst > src 且有重叠（需要反向拷贝）
src:  [  ABCDEF  ]
dst:  [    xxxxxx]   ← 从右到左拷贝，否则 dst 前端会覆盖还未读取的 src
```

**手写 memmove**：

```cpp
void* my_memmove(void* dst, const void* src, size_t n) {
    if (dst == NULL || src == NULL) return NULL;
    char* d = (char*)dst;
    const char* s = (const char*)src;

    if (d <= s || d >= s + n) {
        // 无重叠或 dst 在 src 前面：正向拷贝
        while (n--) *d++ = *s++;
    } else {
        // dst 在 src 中间：反向拷贝
        d += n - 1;
        s += n - 1;
        while (n--) *d-- = *s--;
    }
    return dst;
}
```

**glibc 的真实实现优化**：

glibc 中的 memcpy/memmove 远非逐字节拷贝，根据长度分级优化：

| 长度范围 | 策略 |
|---------|------|
| 0-16 字节 | 直接用两次 load/store 覆盖（头尾各一次） |
| 16-256 字节 | SSE2 `movdqu` 128 位批量拷贝 |
| 256B-? | AVX2 `vmovdqu` 256 位批量拷贝 |
| 非常大 | `rep movsb`（ERMS 特性，CPU 内部流水线优化） |

x86 CPU 的 ERMS（Enhanced REP MOVSB）特性使得 `rep movsb` 在大块拷贝时接近甚至超过 SIMD 循环的带宽。

#### 3.2.1 优缺点与权衡

**memcpy 优点**：
- 极致性能：省掉方向判断分支，编译器和 glibc 可以放心使用最激进的 SIMD 优化路径（正向 `vmovdqu` 流水线或 `rep movsb`），90% 的不重叠场景获得最大带宽
- 编译器内建识别：`__builtin_memcpy` 可被编译器识别并在小尺寸时直接内联为 `mov` 指令，零函数调用开销

**memcpy 缺点/代价**：
- 重叠即 UB：源和目标内存重叠时行为未定义，某些 glibc 实现恰好用了反向拷贝能"正确"工作，但不可依赖——换平台或升级 glibc 随时可能出错

**memmove 优点**：
- 处理重叠保证正确：内部先判断 `dst` 和 `src` 的相对位置，重叠时选择反向拷贝，语义上是 memcpy 的安全超集

**memmove 缺点/代价**：
- 多一次方向判断：需要比较 `dst` 和 `src` 地址决定拷贝方向，虽然只是一条 `cmp` + 条件跳转，但在极端热路径的微基准测试中可测量到差异（通常 < 5%）

**权衡**：能确定源和目标不重叠时用 `memcpy`（如从网络缓冲区拷贝到应用结构体）；不确定或可能重叠时用 `memmove`（如数组元素移动、`std::copy` 内部实现）；C++ 中优先考虑 `std::copy`/`std::move`，编译器会自动选择最优底层实现

> **面试速答**
>
> `memcpy` 和 `memmove` 都是字节级内存拷贝，区别就在处不处理重叠。`memcpy` 不处理重叠，追求极致性能；`memmove` 处理重叠，保证正确性——当 `dst > src` 时内部会改成反向拷贝，避免正向拷贝覆盖还没读的源数据。
>
> 底层实现上 glibc 远不是你想象的逐字节循环：根据长度分级使用 SSE2/AVX2 做批量拷贝，大块数据用 `rep movsb`（配合 CPU 的 ERMS 特性）直接跑到接近内存带宽上限。`memcpy` 省掉了方向判断那次比较，90% 不重叠的场景更快；`memmove` 多了一次比较但换来了重叠场景下的安全。
>
> 有个容易踩的坑要特别注意：源和目标可能重叠时必须用 `memmove`，用 `memcpy` 是 UB——虽然某些实现下恰好能跑对，但这是不可依赖的，换个版本或换个长度就可能出问题。
>
> 选型很直接：确定不重叠用 `memcpy`，不确定或可能重叠用 `memmove`；C++ 里优先用 `std::copy`，编译器会根据类型和场景自动挑最优底层实现。

### 3.3 strlen 的 SIMD 优化

朴素实现 O(n)，逐字节找 `\0`：

```cpp
size_t my_strlen(const char* s) {
    const char* p = s;
    while (*p) p++;
    return p - s;
}
```

**glibc 优化思路**：一次检查 8 字节（64 位）或更多：

```
Step 1: 对齐到字边界（先逐字节处理前几个非对齐字节）
Step 2: 每次读取 8 字节（一个 uint64_t）
Step 3: 用位操作检测这 8 字节中是否包含 \0：
        - 经典 trick: ((x - 0x0101...01) & ~x & 0x8080...80) != 0
        - 原理：如果某字节为 0，减 1 后该字节最高位从 0 变 1
Step 4: 找到包含 \0 的 8 字节后，用 __builtin_ctzll 定位具体位置
```

SSE2/AVX2 版本更进一步：一次比较 16/32 字节：

```
pcmpeqb xmm0, [ptr]   ; 与全零向量比较，\0 字节变 0xFF
pmovmskb eax, xmm0    ; 提取每字节最高位到 16 位 mask
bsf eax, eax          ; 找到第一个 1 的位置 = \0 的偏移
```

> **面试速答**
>
> `strlen` 就是算 C 字符串长度，朴素实现逐字节找 `\0`，时间 O(n)。
>
> glibc 的实现比你想的骚得多：先把地址对齐到字边界，然后一次读 8 字节，用位运算 `((x - 0x0101...01) & ~x & 0x8080...80)` 检测有没有零字节——原理是零字节减 1 后最高位会从 0 变 1，非零字节不会。更激进的 SSE2/AVX2 版本一次比较 16 或 32 字节，用 `pcmpeqb` + `pmovmskb` + `bsf` 三条指令定位 `\0` 位置。所以实际性能比朴素循环快很多倍。
>
> 有个容易踩的坑要特别注意：对没有 `\0` 结尾的缓冲区调用 strlen 会一直往后读，可能 SIGSEGV 或者返回一个垃圾长度。
>
> 和 `string::size()`、`string_view::size()` 比，后两者都是 O(1) 直接读成员变量，只有 strlen 每次都要扫描。实际工程里 C++ 就尽量用 `string`/`string_view` 避开 O(n) 的 strlen；循环里反复调用 strlen 是经典性能 bug，结果应该缓存下来。

### 3.4 snprintf vs sprintf

**sprintf 是缓冲区溢出的经典来源**：

```cpp
char buf[16];
sprintf(buf, "name=%s, age=%d", "very_long_name_here", 25);
// 写入超过 16 字节 → 栈溢出！
```

**snprintf 的安全保证**：

```cpp
char buf[16];
int ret = snprintf(buf, sizeof(buf), "name=%s, age=%d", "Alice", 25);
// buf = "name=Alice, age"  ← 截断到 15 字符 + \0
// ret = 20                 ← 返回"应该写入"的长度（不含 \0）
if (ret >= (int)sizeof(buf)) {
    // 发生了截断，ret 告诉你需要多大的 buffer
    char* big = (char*)malloc(ret + 1);
    snprintf(big, ret + 1, "name=%s, age=%d", "Alice", 25);
    // ...
    free(big);
}
```

> **面试速答**
>
> `sprintf` 和 `snprintf` 都是格式化输出到字符缓冲区，区别就在限不限长。`sprintf` 不限长，是缓冲区溢出的经典来源；`snprintf` 通过限长来防止溢出。
>
> `snprintf` 的语义要记清楚：最多写 n-1 字节并保证 `\0` 结尾，返回值是"应该写入"的字符数（不含 `\0`）。这个返回值有个很实用的用法——返回值 >= n 说明发生了截断，可以据此判断缓冲区够不够；还可以做"先探测再分配"：第一次传 `size=0` 拿到所需长度，第二次传正确大小，一次分配到位。
>
> 和 C++20 的 `std::format` 比，后者类型安全、不会溢出、性能接近 `snprintf`；和 `std::ostringstream` 比，stream 安全但性能差不少。实际工程里 C++20 优先 `std::format`，C++17 及以下用 `snprintf`，生产代码里完全弃用 `sprintf`。

### 3.5 memset 的坑

**memset 是按字节设置**，不是按元素设置！

```cpp
int arr[10];
memset(arr, 0, sizeof(arr));    // 正确：每字节设 0x00 → int 值为 0
memset(arr, -1, sizeof(arr));   // 正确：每字节设 0xFF → int 值为 -1（补码）
memset(arr, 0x3f, sizeof(arr)); // 特殊用途：每个 int = 0x3f3f3f3f ≈ 1e9（竞赛常用作 INF）
```

❌ 错误：用 memset 设置 int 数组为 1
```cpp
int arr[10];
memset(arr, 1, sizeof(arr));
// 每字节设为 0x01 → 每个 int = 0x01010101 = 16843009，不是 1！
```

✅ 正确：用 std::fill 或循环
```cpp
int arr[10];
std::fill(arr, arr + 10, 1);  // 每个元素设为 1

// 或 C++11
int arr2[10] = {};  // 全部初始化为 0
for (auto& x : arr2) x = 1;
```

**memset 安全值速查**：

| 设置值 | 字节值 | int 结果 | 是否常用 |
|-------|--------|---------|---------|
| 0 | 0x00 | 0 | 是 |
| -1 | 0xFF | -1 | 是 |
| 0x3f | 0x3F | 0x3F3F3F3F ≈ 1.06e9 | 竞赛 INF |
| 0x7f | 0x7F | 0x7F7F7F7F ≈ 2.14e9 | 接近 INT_MAX |
| 1 | 0x01 | 0x01010101 = 16843009 | **不要用** |

> **面试速答**
>
> `memset` 最核心的认知只有一条：它是按**字节**设置内存的，不是按元素设置。它的用途是快速把大块内存初始化成某个字节模式，最常用就是清零。
>
> 有个容易踩的坑几乎人人都中过：`memset(arr, 1, sizeof(arr))` 会把每个字节都写成 `0x01`，所以 int 数组的每个元素变成 `0x01010101 = 16843009`，而不是 1。对 int 数组安全的值就只有三个：`0`（全零）、`-1`（全 `0xFF`，补码表示就是 -1）、`0x3f`（int 变成 `0x3f3f3f3f` 约 1e9，竞赛里常拿来做 INF，因为两个加起来也不会溢出 int）。
>
> 和 `std::fill` 比，fill 是按元素设置，类型安全，可以填任意值；`std::memset` 和 `memset` 完全一样，只是在 `<cstring>` 命名空间里。实际选型很简单：清零用 `memset` 或直接 `= {}`；要把 int 数组设成任意值就必须用 `std::fill` 或循环，千万别用 memset。

### 对比表：字符串函数总览

| 函数 | 用途 | 终止条件 | 处理重叠 | 类型限制 |
|------|------|---------|---------|---------|
| `strcpy` | 字符串拷贝 | \0 | 否 | char* |
| `strncpy` | 限长字符串拷贝 | \0 或 n | 否 | char* |
| `memcpy` | 内存拷贝 | n 字节 | **否** | void* |
| `memmove` | 安全内存拷贝 | n 字节 | **是** | void* |
| `memset` | 内存填充 | n 字节 | - | void*（按字节） |
| `strlen` | 字符串长度 | \0 | - | const char* |
| `strcmp` | 字符串比较 | \0 | - | const char* |
| `snprintf` | 格式化写入 | n 字节 | - | char* |

## 4. 柔性数组（Flexible Array Member）

### 概念与本质

柔性数组是 C99 标准中定义的一种结构体末尾的不完整数组类型，其大小在运行时由 malloc 的分配量决定，内存与结构体连续。

### 底层原理

```
普通指针方案（两次分配，内存不连续）：

  struct A { int len; char* data; };  // sizeof = 16

  ┌──────────┐     ┌──────────────┐
  │ len │ data─┼────▶│ h e l l o \0 │   ← 堆上另一块内存
  └──────────┘     └──────────────┘
  ← 堆内存1 →      ← 堆内存2 →


柔性数组方案（一次分配，内存连续）：

  struct B { int len; char data[]; };  // sizeof = 4

  ┌──────────────────────────────┐
  │ len │ h e l l o \0            │   ← 一整块连续堆内存
  └──────────────────────────────┘
```

柔性数组成员本身不占结构体空间（`sizeof(B) == 4`），`data` 的地址紧跟结构体末尾。

### 代码

```c
#include <stdlib.h>
#include <stdio.h>
#include <string.h>

// C99 标准写法
typedef struct {
    int length;
    char data[];  // 柔性数组成员，必须是最后一个成员
} FlexBuffer;

FlexBuffer* create_buffer(const char* content) {
    int len = strlen(content);
    // 一次分配：结构体 + 数据
    FlexBuffer* buf = (FlexBuffer*)malloc(sizeof(FlexBuffer) + len + 1);
    buf->length = len;
    memcpy(buf->data, content, len + 1);  // 包含 \0
    return buf;
}

int main() {
    FlexBuffer* buf = create_buffer("hello network packet");
    printf("length: %d, data: %s\n", buf->length, buf->data);
    printf("sizeof(FlexBuffer): %zu\n", sizeof(FlexBuffer));  // 4
    free(buf);  // 一次释放
    return 0;
}
```

### 柔性数组 vs VLA vs 指针成员

| 特性 | 柔性数组 `data[]` | VLA `int arr[n]` | 指针成员 `char* data` |
|------|-------------------|------------------|--------------------|
| 内存位置 | 堆（与结构体连续） | **栈** | 堆（不连续） |
| 分配次数 | 1 次 | 0 次（栈自动） | 2 次 |
| 释放次数 | 1 次 | 0 次 | 2 次（先成员后结构体） |
| sizeof | 不计入 | 运行时确定 | 计入指针大小 |
| 标准支持 | C99 | C99（C11 改为可选） | C/C++ |
| C++ 支持 | **不支持** | **不支持** | 支持 |
| cache 友好 | 好（连续内存） | 好 | 差（两次间接寻址） |

#### 4. 优缺点与权衡

**优点**：
- 一次分配一块连续内存：`malloc(sizeof(struct) + data_len)` 将 header 和变长 body 放在连续地址空间，只需一次系统调用，释放时也只需一次 `free`，减少堆碎片
- cache 友好：header 和数据紧挨着，访问 body 时大概率已经在同一条或相邻 cache line 中，避免指针成员方案的二次间接寻址导致的 cache miss
- 零额外开销：`data[]` 成员本身不占 `sizeof` 空间，没有指针的 8 字节存储开销

**缺点/代价**：
- C++ 标准不支持：柔性数组是 C99 特性，C++ 标准从未纳入。GCC 作为扩展支持，但 MSVC 和严格标准模式 (`-pedantic`) 下编译报错，跨平台代码不可用
- 只能堆分配：柔性数组的大小在编译期未知，不能放在栈上、不能作为数组元素、不能被 `new` 直接构造，使用场景受限
- 不能用于有继承/虚函数的类：柔性数组要求是结构体的最后一个成员，且结构体必须是 POD 类型，与 C++ 的面向对象特性冲突

**权衡**：C 项目中处理变长网络协议包、二进制文件格式时，柔性数组是性能最优方案（一次分配、连续内存、零间接寻址）；C++ 项目首选 `std::vector` 保证可移植性和类型安全；追求零开销的 C++ 代码可用 placement new + `reinterpret_cast<char*>(this + 1)` 模拟柔性数组语义

### 在网络协议中的实际用途

柔性数组非常适合表示变长协议包：

```c
// 典型的网络协议包头
typedef struct {
    uint16_t type;      // 消息类型
    uint16_t flags;     // 标志位
    uint32_t body_len;  // 包体长度
    char body[];        // 变长包体
} __attribute__((packed)) Packet;

// 接收网络包
void handle_packet(int fd) {
    // 先读包头，获取 body_len
    Packet header;
    read(fd, &header, sizeof(Packet));

    // 再分配完整包并读取 body
    Packet* pkt = (Packet*)malloc(sizeof(Packet) + header.body_len);
    *pkt = header;
    read(fd, pkt->body, header.body_len);

    // 处理...
    free(pkt);
}
```

### C++ 中的替代方案

C++ 标准不支持柔性数组（GCC 扩展可以，但不可移植）。替代方案：

```cpp
// 方案1：std::vector（最推荐）
struct Packet {
    uint16_t type;
    uint16_t flags;
    std::vector<char> body;  // 自动管理内存
};

// 方案2：placement new + 手动管理（零开销，需要小心）
struct Packet {
    uint16_t type;
    uint16_t flags;
    uint32_t body_len;

    char* body() { return reinterpret_cast<char*>(this + 1); }

    static Packet* create(uint32_t len) {
        void* mem = ::operator new(sizeof(Packet) + len);
        Packet* p = new (mem) Packet{};
        p->body_len = len;
        return p;
    }
    static void destroy(Packet* p) {
        p->~Packet();
        ::operator delete(p);
    }
};

// 方案3：尾部 char[1] hack（C++98 时代常用，不推荐）
struct Packet {
    uint32_t body_len;
    char body[1];  // 实际分配 sizeof(Packet) + body_len - 1
};
```

### 陷阱与实战经验

❌ 错误：在栈上使用柔性数组
```c
FlexBuffer buf;  // 编译可能通过，但 data 的空间为 0
buf.data[0] = 'x';  // 栈溢出！
```

✅ 正确：柔性数组只能配合 malloc 使用
```c
FlexBuffer* buf = malloc(sizeof(FlexBuffer) + 100);
```

❌ 错误：用 `char data[0]`（GCC 扩展，非标准）
```c
struct Bad { int len; char data[0]; };  // GCC 允许，但 C99 前非标准
```

✅ 正确：用 `char data[]`（C99 标准）
```c
struct Good { int len; char data[]; };  // C99 标准柔性数组
```

> **面试速答**
>
> 柔性数组是 C99 引入的特性，就是结构体末尾写一个不完整数组 `char data[]`，它的 `sizeof` 不计入结构体大小，实际数据紧跟结构体末尾，靠一次 `malloc(sizeof(struct) + data_len)` 分配出一整块连续内存。它解决的是变长数据（比如网络协议的包头+包体）用指针成员实现时需要两次分配、两次释放、内存还不连续的问题——柔性数组一次分配搞定，cache 也友好。
>
> 底层原理就是编译器把柔性数组成员的偏移固定在结构体末尾，malloc 完数据区直接紧跟 header，地址算起来非常规整。
>
> 有个容易踩的坑：柔性数组只能配合堆分配，不能放栈上也不能作为数组元素。还要特别注意 C++ 标准本身不支持柔性数组，GCC 是作为扩展才支持的。
>
> 和指针成员方案比，指针方案两次分配、内存不连续；和 `std::vector` 比，vector 是 C++ 标准方案但多一次间接寻址。实际工程里 C 项目的变长协议包用柔性数组，C++ 项目首选 `std::vector`，追求零开销的话可以上 placement new。

## 5. 声明与定义

### 概念与本质

声明（declaration）告诉编译器"存在这样一个实体"；定义（definition）为实体分配存储或提供实现。每个定义也是声明，但声明不一定是定义。

### 对比表

| 维度 | 声明 | 定义 |
|------|------|------|
| 作用 | 告知类型和名称 | 分配存储/实现 |
| 出现次数 | 可以多次（ODR例外） | **恰好一次** |
| 变量示例 | `extern int x;` | `int x = 10;` |
| 函数示例 | `int add(int, int);` | `int add(int a, int b) { return a+b; }` |
| 类示例 | `class Foo;`（前向声明） | `class Foo { ... };` |
| 头文件规则 | 放声明 | 不放变量定义（会重复定义） |

**例外**：以下可以在头文件中定义且不违反 ODR：
- `inline` 函数/变量（C++17）
- `constexpr` 变量
- 类定义（类内成员函数隐式 inline）
- 模板

### 陷阱与实战经验

❌ 错误：头文件中定义全局变量
```cpp
// config.h
int timeout = 30;  // 多个 .cpp include → 链接时重复定义
```

✅ 正确：声明与定义分离，或使用 inline（C++17）
```cpp
// config.h
extern int timeout;           // 声明
// 或 C++17:
inline int timeout = 30;      // inline 变量，可多次定义

// config.cpp
int timeout = 30;             // 定义（配合 extern 声明使用）
```

> **面试速答**
>
> 声明和定义的区别很直接：声明告诉编译器"存在这样一个实体"（提供类型和名称），定义则真正为它分配存储或提供实现。声明可以出现多次，定义只能有一次，这就是 ODR（One Definition Rule）。它存在的意义在于 C++ 的分离编译模型——每个 `.cpp` 独立编译时，编译器只需要通过声明知道外部符号的类型信息，链接器后面再去把声明和定义匹配起来。
>
> 有几个容易踩的坑要特别注意：头文件里放变量定义，多个 `.cpp` include 之后链接就会报重复定义；`extern int x;` 是声明，`int x = 10;` 是定义，这两个要分清楚。
>
> 另外 `class Foo;` 是前向声明，只能用来声明指针或引用，不能访问成员也不能 sizeof；`class Foo { ... };` 才是完整定义。
>
> 实际工程里规则就一条：头文件只放声明。例外是 `inline`（C++17）、`constexpr`、模板以及类内成员函数，这几类可以在头文件里定义而不违反 ODR。
