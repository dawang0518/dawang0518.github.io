# 指针与智能指针

> C++ 内存管理的核心武器：从裸指针的混乱到智能指针的秩序。

---

## 0. 全景地图

**裸指针的三大原罪**：

```
裸指针 → 忘记 delete → 内存泄漏
       → delete 后继续用 → 悬空指针（UB）
       → 两处 delete 同一块 → 双重释放（crash）
```

C++ 的解决思路：**用 RAII 把资源生命周期绑定到对象生命周期**。智能指针就是这个思路的产物。

**三种智能指针的关系**：

```
unique_ptr ── 独占所有权（零开销，默认选择）
     │
     │ 如果需要共享 ↓
shared_ptr ── 共享所有权（引用计数，有开销）
     │
     │ 打破循环引用 ↓
 weak_ptr ── 观察不拥有（不增引用计数）
```

**演进史**：`auto_ptr`（C++98，已废弃）→ `unique_ptr`（C++11）→ `shared_ptr` / `weak_ptr`（C++11，源自 Boost）。`auto_ptr` 的致命问题是拷贝时偷偷转移所有权，导致原指针悬空——这正是 `unique_ptr` 用 move 语义显式解决的。

## 1. 指针与引用基础

### 1.1 概念与本质

**指针**：存储内存地址的变量，本身占 4/8 字节（32/64 位），可以为 `nullptr`，可以重新指向。

**引用**：变量的别名，编译器层面通常也用指针实现，但语义上不是独立对象——必须初始化、不能重绑定、不能为空。

### 1.2 指针 vs 引用对比表

| 维度 | 指针 | 引用 |
|------|------|------|
| 初始化 | 可延迟，可为 `nullptr` | 必须声明时绑定，不能为空 |
| 重绑定 | 可以指向其他对象 | 初始化后不可更改 |
| 语法 | `*p` 解引用，`&x` 取地址 | 直接使用，透明 |
| sizeof | 指针本身大小（8 字节@64位） | 绑定对象的大小 |
| 多级间接 | 支持 `int**` | 不支持"引用的引用" |
| 算术 | 支持 `p++`、`p+n` | 不支持 |
| 底层实现 | 就是地址 | 编译器通常也用地址实现 |

### 1.3 传值 / 传地址 / 传引用

```cpp
void byValue(int x)    { x = 100; }       // 拷贝，不影响原变量
void byPointer(int* p)  { *p = 100; }      // 传地址，可修改原变量
void byReference(int& r) { r = 100; }      // 传引用，可修改原变量

int main() {
    int a = 0;
    byValue(a);       // a 仍然是 0
    byPointer(&a);    // a 变成 100
    a = 0;
    byReference(a);   // a 变成 100
}
```

**选型原则**：
- 小类型（int、double）→ 传值
- 大对象只读 → `const T&`
- 大对象需修改 → `T&`
- 可能为空 / 可选参数 → `T*`（或 C++17 `std::optional`）

> **面试速答**
> - **是什么/本质**：指针是存储内存地址的变量，引用是变量的别名——底层都是地址，但语义约束不同
> - **解决什么问题**：间接访问对象、避免大对象拷贝、支持多态（基类指针/引用指向派生类）
> - **底层原理**：引用编译后通常和指针生成相同的机器码（传地址），但编译器保证引用不为空、不可重绑定
> - **优缺点**：引用更安全（不可能为空），指针更灵活（可为空、可重绑定、支持算术）
> - **坑点**：引用不能重绑定，`int& r = a; r = b;` 是给 a 赋值而非让 r 指向 b；悬空引用和悬空指针一样危险
> - **选型建议**：小类型传值，大对象只读用 `const T&`，需修改用 `T&`，参数可能为空才用指针

### 1.4 优缺点与权衡

**裸指针的优点**：
- 零抽象开销：裸指针就是一个地址值，解引用只需一条 load/store 指令，没有引用计数、控制块等任何额外开销
- 最大灵活性：可以为 nullptr、可以重绑定、支持指针算术、支持多级间接（`int**`），是实现底层数据结构（链表、树、图）和与 C API 交互的基础工具
- 与硬件/OS 接口的唯一选择：系统调用（`mmap`、`read`、`write`）、硬件寄存器（MMIO）、内存映射文件等场景只能使用裸指针

**裸指针的缺点/代价**：
- 所有权语义不明确：裸指针不表达"谁负责释放"，函数签名 `void process(Widget* w)` 无法传达调用者还是被调用者负责 delete，这是内存泄漏和双重释放的根本原因
- 不安全：可以为 nullptr（解引用 UB）、可以悬空（指向已释放内存）、可以越界（数组指针算术溢出），且这三种错误都不会在编译期被检测到
- 异常不安全：`new` 后如果后续代码抛异常，裸指针指向的内存不会被自动释放，除非用 try-catch 包裹——而 RAII 智能指针自动处理这种情况

**权衡**：现代 C++ 的原则是"裸指针只用于观察（不拥有所有权）"——用 `T*` 或 `T&` 表示"借用"，用智能指针表示"拥有"。在性能关键的底层代码（内存池、自定义分配器、与 C API 交互）中裸指针不可替代，但在业务逻辑层应优先使用智能指针，将所有权语义编码进类型系统。

### 1.5 指针的常见陷阱

#### 野指针（未初始化）

```cpp
// ❌ 错误：p 未初始化，指向随机地址
int* p;
*p = 42;  // 未定义行为

// ✅ 正确：声明即初始化
int* p = nullptr;
if (p) { *p = 42; }
```

#### 悬空指针（对象已销毁）

```cpp
// ❌ 错误：返回局部变量的地址
int* danger() {
    int local = 42;
    return &local;  // local 在函数返回后销毁，指针悬空
}

// ✅ 正确：返回堆内存（或用智能指针）
std::unique_ptr<int> safe() {
    return std::make_unique<int>(42);
}
```

#### 双重释放

```cpp
// ❌ 错误：两次 delete 同一块内存
int* p = new int(42);
int* q = p;
delete p;
delete q;  // crash 或 UB

// ✅ 正确：用 unique_ptr，所有权唯一，自动释放
auto p = std::make_unique<int>(42);
// 不需要 delete，离开作用域自动释放
```

#### 内存泄漏

```cpp
// ❌ 错误：异常路径泄漏
void leak() {
    int* p = new int(42);
    doSomething();  // 如果抛异常，p 永远不会被 delete
    delete p;
}

// ✅ 正确：RAII 自动管理
void safe() {
    auto p = std::make_unique<int>(42);
    doSomething();  // 即使抛异常，析构函数也会释放
}
```

### 1.6 野指针的底层原理（虚拟内存 / 页表 / MMU）

表面上"野指针是指向无效内存的指针"，但从底层看，本质是**进程通过虚拟地址访问了未映射到物理内存、或无访问权限的地址**。

```
CPU 执行 *p = 10
      │
      ▼
MMU 翻译虚拟地址 → 查页表
      │
      ├─ 页表项不存在 → 缺页异常 → 内核转为 SIGSEGV → 段错误（崩溃）
      │
      └─ 页表项存在但地址属于其他数据 → 正常写入 → 静默篡改数据（更危险）
```

**两种底层场景**：

1. **访问未映射的虚拟地址**：堆内存释放后，对应虚拟页从进程页表中移除，CPU 通过 MMU 翻译虚拟地址时找不到页表项，触发缺页异常，内核转为 `SIGSEGV`。
2. **访问越界但已映射的地址**：数组越界但未跨越虚拟页边界（页大小通常 4KB），页表映射仍有效，CPU 能正常读写——表现为静默的数据篡改，比崩溃更难排查。

**为什么"释放后偶尔不崩溃"**：`free`/`delete` 把内存块归还给分配器（如 ptmalloc2），分配器标记为"空闲"但**未必立即解除虚拟页映射**。此时野指针访问的是空闲区内存，不触发缺页异常，但可能篡改分配器的元数据或后续分配的对象。

**栈上的野指针更危险**：函数返回后栈帧被回收（RSP 寄存器移动），但虚拟地址仍映射物理内存。访问栈野指针会篡改后续函数的栈帧数据（如返回地址），甚至被利用进行栈溢出攻击。

### 1.7 野指针检测：分阶段工具链

#### 开发阶段：静态分析

集成 `clang-tidy` 到 IDE / CI，开启关键规则：
- `bugprone-use-after-free`：检测释放后使用
- `bugprone-dangling-handle`：检测悬空引用
- 编码规范强制指针声明即初始化（`int* p = nullptr`）

#### 测试阶段：AddressSanitizer（ASAN）

编译时加 `-fsanitize=address`，比 Valgrind 快 10 倍以上。

**ASAN 的底层原理——影子内存（Shadow Memory）**：

```
用户内存（8 字节）  →  影子内存（1 字节）
标记该 8 字节是否"已分配/可访问"

编译时插桩：对所有 *p 操作插入检测逻辑
  1. 通过虚拟地址计算对应的影子内存地址
  2. 检查影子内存标记是否合法
  3. 非法 → 立即报错并打印调用栈

内存分配时 → 影子内存标记为"可访问"
内存释放时 → 影子内存标记为"非法"
```

这就是 ASAN 能精准定位"释放后使用（use-after-free）"的底层原因——释放时影子内存的标记已经翻转，后续任何访问都会被插桩代码拦截。

#### 线上阶段：core dump + gdb

线上无法使用重性能的检测工具，开启 core dump 并结合 `gdb` 分析崩溃栈。同时可以封装指针访问逻辑，在访问前加状态校验，避免直接崩溃，改为记录告警日志。

### 1.8 野指针的工程化防治

技术手段之外，流程上的管控同样重要：

1. **CI 集成 ASAN 扫描**：每次代码提交自动编译运行 ASAN 测试，避免回归
2. **CI 集成 clang-tidy**：在 MR/PR 阶段自动执行静态分析，拦截典型野指针模式
3. **代码评审**：重点检查指针初始化、释放后置空、跨线程传递逻辑
4. **系统级防护**：对核心进程释放后的内存页调用 `mprotect` 标记为 `PROT_NONE`，野指针访问立即触发 `SIGSEGV`，而非静默篡改
5. **编译器防护**：开启 `-fstack-protector`，在栈帧中插入金丝雀值（Canary），野指针篡改栈帧时函数返回前即可检测

> **面试速答**
> - **是什么/本质**：野指针是未初始化的指针（随机值），悬空指针是指向已释放内存的指针——本质都是持有无效地址
> - **解决什么问题**：识别和防治这两类指针错误，避免程序崩溃或更危险的静默数据篡改
> - **底层原理**：MMU 翻译虚拟地址时若页表项不存在则触发 SIGSEGV；但若地址恰好落在已映射的其他数据区则静默篡改——后者更危险因为不崩溃
> - **坑点**：free/delete 后指针值不变（仍持有旧地址），必须手动置 nullptr；use-after-free 在测试中可能不崩溃（内存未被复用时），上线才暴露
> - **横向对比**：防治手段分层——开发期 clang-tidy 静态分析，测试期 AddressSanitizer（影子内存追踪），线上 core dump + gdb
> - **选型建议**：最根本的解决方案是用智能指针替代裸指针，从所有权语义上杜绝悬空问题

## 2. unique_ptr 深度剖析

### 2.1 概念与本质

**一句话定义**：`unique_ptr` 是独占所有权的智能指针——同一时刻只有一个 `unique_ptr` 拥有资源，离开作用域自动释放。

**为什么需要它**：裸指针的核心问题是"谁负责 delete"说不清楚。`unique_ptr` 把所有权编码进类型系统，编译器帮你保证"恰好 delete 一次"。

### 2.2 底层实现：零开销抽象

`unique_ptr` 的核心实现极其简洁：

```cpp
// 简化的 unique_ptr 实现（核心逻辑）
template<typename T, typename Deleter = std::default_delete<T>>
class unique_ptr {
    T* ptr_;             // 唯一的数据成员（默认删除器时）
public:
    explicit unique_ptr(T* p = nullptr) : ptr_(p) {}
    ~unique_ptr() { if (ptr_) Deleter{}(ptr_); }

    // 禁止拷贝
    unique_ptr(const unique_ptr&) = delete;
    unique_ptr& operator=(const unique_ptr&) = delete;

    // 允许移动
    unique_ptr(unique_ptr&& other) noexcept : ptr_(other.ptr_) {
        other.ptr_ = nullptr;
    }
    unique_ptr& operator=(unique_ptr&& other) noexcept {
        if (this != &other) {
            if (ptr_) Deleter{}(ptr_);
            ptr_ = other.ptr_;
            other.ptr_ = nullptr;
        }
        return *this;
    }

    T& operator*() const { return *ptr_; }
    T* operator->() const { return ptr_; }
    T* get() const { return ptr_; }
};
```

**为什么和裸指针一样大？** 当使用默认删除器 `std::default_delete<T>` 时，它是空类。编译器通过**空基类优化（EBO）**，把删除器压缩到零字节。所以 `sizeof(unique_ptr<int>) == sizeof(int*) == 8`。

```cpp
// 验证：默认删除器零开销
static_assert(sizeof(std::unique_ptr<int>) == sizeof(int*));  // 通过！

// 但有状态的删除器会增加大小
auto deleter = [](int* p) { std::cout << "bye\n"; delete p; };
// 无捕获的 lambda 是空类，仍然零开销
static_assert(sizeof(std::unique_ptr<int, decltype(deleter)>) == sizeof(int*));

// 有捕获的 lambda 会增大
int count = 0;
auto stateful = [&count](int* p) { count++; delete p; };
// sizeof > sizeof(int*)，因为 lambda 携带了 count 的引用
```

### 2.3 所有权转移（std::move）

```cpp
auto p1 = std::make_unique<int>(42);
// auto p2 = p1;            // ❌ 编译错误：unique_ptr 不可拷贝
auto p2 = std::move(p1);    // ✅ 所有权转移给 p2
// 此时 p1 == nullptr，p2 拥有资源

// 函数间传递所有权
std::unique_ptr<Widget> createWidget() {
    return std::make_unique<Widget>();  // 隐式移动（RVO/NRVO）
}

void consumeWidget(std::unique_ptr<Widget> w) {
    // w 在此函数结束时自动销毁
}

auto w = createWidget();
consumeWidget(std::move(w));  // 显式转移所有权
// w 此时为 nullptr
```

### 2.4 自定义删除器

`unique_ptr` 不仅能管理 `new` 出来的内存，还能管理任何需要"获取后释放"的资源：

```cpp
// 管理 FILE*
auto fileDeleter = [](FILE* f) {
    if (f) {
        std::fclose(f);
        std::cout << "文件已关闭\n";
    }
};
std::unique_ptr<FILE, decltype(fileDeleter)> file(
    std::fopen("data.txt", "r"), fileDeleter
);
// 离开作用域自动 fclose

// 管理数组（C++11 需要特化，C++14 用 make_unique）
auto arr = std::make_unique<int[]>(10);  // 10 个 int 的数组
arr[0] = 42;  // operator[] 可用

// 管理 socket fd（Linux）
auto sockDeleter = [](int* fd) {
    if (fd && *fd >= 0) {
        ::close(*fd);
    }
    delete fd;
};
auto sock = std::unique_ptr<int, decltype(sockDeleter)>(
    new int(::socket(AF_INET, SOCK_STREAM, 0)), sockDeleter
);
```

### 2.5 实战：用 unique_ptr 实现链表

`unique_ptr` 的独占所有权天然适合表达链表的"每个节点拥有下一个节点"的语义。节点析构时自动递归释放后续节点，不需要手动遍历 delete。

```cpp
#include <iostream>
#include <memory>
using namespace std;

template<typename T>
struct Node {
    T data;
    unique_ptr<Node<T>> next;  // 每个节点独占下一个节点
};

template<typename T>
class Link {
public:
    // 头插法
    void front(const T& data) {
        auto node = make_unique<Node<T>>();
        node->data = data;
        node->next = move(head_.next);  // 新节点接管原链表
        head_.next = move(node);         // 头节点指向新节点
    }

    void print() {
        Node<T>* node = head_.next.get();  // get() 获取裸指针用于遍历，不转移所有权
        while (node) {
            cout << node->data << " ";
            node = node->next.get();
        }
        cout << endl;
    }
    // 析构时 head_ 销毁 → head_.next 销毁 → 递归释放整条链表
private:
    Node<T> head_;  // 哨兵节点
};

int main() {
    Link<int> lk;
    for (auto val : { 1, 2, 3, 4, 5 })
        lk.front(val);
    lk.print();  // 输出: 5 4 3 2 1
}
```

**关键点**：
- `next` 用 `unique_ptr` 而非裸指针，所有权链清晰，离开作用域自动释放整条链表
- 遍历时用 `get()` 获取裸指针观察，不触发所有权转移
- 插入/删除时用 `std::move` 显式转移所有权，编译器帮你保证不会出现两个指针指向同一节点

### 2.6 优缺点与权衡

**优点**：
- 真正的零开销抽象：默认删除器时 `sizeof(unique_ptr<T>) == sizeof(T*)`（空基类优化），解引用生成的汇编与裸指针完全相同，没有引用计数、控制块、原子操作的任何开销
- 所有权语义编码进类型系统：禁止拷贝、只允许移动的设计让所有权转移在编译期可见可检查，不可能出现"两个指针都认为自己拥有资源"的双重释放问题
- 异常安全：RAII 保证即使函数中途抛异常，析构函数也会被调用释放资源，彻底解决裸 new 在异常路径上的泄漏问题
- 自定义删除器通用性：不仅能管理 `new` 的内存，还能管理 FILE*（fclose）、socket fd（close）、C 库分配的内存（free）等任意资源

**缺点/代价**：
- 独占语义限制了共享场景：不能拷贝意味着多个模块不能同时持有同一个 unique_ptr，需要共享时必须升级为 shared_ptr 或用裸指针/引用"借用"
- 转移所有权需要显式 std::move：每次传递所有权都要写 `std::move(p)`，代码中 move 过多时可读性下降；且 move 后原指针变为 nullptr，继续使用是 UB（编译器不总是能警告）
- 自定义删除器影响类型：`unique_ptr<T, D1>` 和 `unique_ptr<T, D2>` 是不同类型，不能放在同一个容器中（shared_ptr 则通过类型擦除避免了这个问题）

**权衡**：unique_ptr 应该是智能指针的**默认选择**——90% 的资源管理场景是独占所有权的。只有在确实需要多个拥有者共享同一资源且无法在编译期确定谁最后释放时，才升级为 shared_ptr。在性能敏感路径上，unique_ptr 与裸指针完全等价，没有任何不用它的理由。

### 2.7 陷阱

#### make_unique vs new

```cpp
// ❌ 有异常安全问题（C++14 之前）
void bad(std::unique_ptr<A> a, std::unique_ptr<B> b);
bad(std::unique_ptr<A>(new A()), std::unique_ptr<B>(new B()));
// 编译器可能：new A() → new B() → 构造 unique_ptr<A> → 构造 unique_ptr<B>
// 如果 new B() 抛异常，new A() 的内存泄漏！

// ✅ make_unique 保证异常安全
bad(std::make_unique<A>(), std::make_unique<B>());
```

#### 不能放入标准容器的拷贝操作

```cpp
std::vector<std::unique_ptr<int>> vec;
auto p = std::make_unique<int>(42);
// vec.push_back(p);          // ❌ 编译错误：尝试拷贝
vec.push_back(std::move(p));  // ✅ 移动进去
vec.emplace_back(std::make_unique<int>(100));  // ✅ 直接构造
```

#### release() 容易泄漏

```cpp
// ❌ 危险：release 放弃所有权，返回裸指针，如果没人接手就泄漏
auto p = std::make_unique<int>(42);
p.release();  // 内存泄漏！没人 delete

// ✅ 正确用法：转交给另一个管理者
int* raw = p.release();
// ... 传给某个接管所有权的 API ...
legacyApi(raw);  // legacyApi 负责 delete
```

> **面试速答**
> - **是什么/本质**：独占所有权的智能指针，禁止拷贝只允许移动，RAII 保证离开作用域自动释放
> - **解决什么问题**：裸指针忘记 delete 导致内存泄漏，异常路径上的泄漏更难防——unique_ptr 用析构函数自动释放
> - **底层原理**：默认删除器时 sizeof 等于裸指针（空基类优化 EBO），解引用零开销——真正的零成本抽象
> - **优缺点**：优点是零开销 + 所有权语义清晰；缺点是独占语义限制了共享场景，需要转移用 std::move
> - **坑点**：自定义删除器用函数指针会增大 sizeof（多一个指针大小），用 lambda 或仿函数配合 EBO 不增大
> - **横向对比**：vs shared_ptr——unique_ptr 零开销无引用计数，shared_ptr 有控制块和原子操作开销；90% 场景 unique_ptr 够用
> - **选型建议**：优先用 make_unique 创建；只有真正需要共享所有权时才升级 shared_ptr；管理非内存资源用自定义删除器

## 3. shared_ptr 深度剖析

### 3.1 概念与本质

**一句话定义**：`shared_ptr` 是共享所有权的智能指针——多个 `shared_ptr` 可以指向同一个对象，最后一个销毁时释放资源。

**适用场景**：多个模块需要共同持有同一个资源，且无法在编译期确定谁最后使用完。比如：缓存系统、观察者模式中的被观察对象、异步回调中延长对象生命周期。

### 3.2 底层实现：控制块

`shared_ptr` 由两部分组成：

```
shared_ptr<T>
┌──────────┐
│ T* ptr_  │──────→ 实际对象（T）
│ CB* cb_  │──┐
└──────────┘  │
              ▼
        ┌─────────────────────┐
        │     控制块 (Control Block)    │
        │─────────────────────│
        │ strong_count (原子)  │  → 多少个 shared_ptr 指向对象
        │ weak_count   (原子)  │  → 多少个 weak_ptr + (strong>0 ? 1 : 0)
        │ Deleter             │  → 如何删除对象
        │ Allocator           │  → 如何释放控制块
        └─────────────────────┘
```

**sizeof(shared_ptr&lt;T>) = 2 × sizeof(void\*) = 16 字节**（64 位系统）：一个指向对象，一个指向控制块。

**生命周期规则**：
- `strong_count` 降到 0 → 调用 Deleter 销毁对象
- `weak_count` 降到 0 → 释放控制块内存

注意：即使对象已销毁，只要还有 `weak_ptr` 存在，控制块就不会释放。

### 3.3 引用计数的线程安全性

**关键区分**：

```
引用计数的操作 → 线程安全（用原子操作实现）
指向对象的操作 → 不是线程安全的！
```

```cpp
auto sp = std::make_shared<int>(42);

// ✅ 安全：不同线程拷贝/销毁 shared_ptr（只涉及引用计数）
std::thread t1([sp]() { auto local = sp; });   // 拷贝，strong_count 原子 +1
std::thread t2([sp]() { auto local = sp; });   // 拷贝，strong_count 原子 +1

// ❌ 不安全：多线程同时读写 shared_ptr 本身（指针值）
std::shared_ptr<int> global = std::make_shared<int>(0);
std::thread t3([&]() { global = std::make_shared<int>(1); });  // 写
std::thread t4([&]() { auto tmp = global; });                  // 读
// 数据竞争！需要 mutex 或 atomic<shared_ptr>（C++20）

// ❌ 不安全：多线程同时修改指向的对象
auto data = std::make_shared<int>(0);
std::thread t5([data]() { *data = 1; });  // 写
std::thread t6([data]() { *data = 2; });  // 写
// 数据竞争！需要 mutex 保护 *data
```

**底层原子操作**（以 GCC/libstdc++ 为例）：

```cpp
// shared_ptr 拷贝构造时：
__atomic_add_dispatch(&strong_count, 1);  // 原子加 1（memory_order_relaxed）

// shared_ptr 析构时：
if (__atomic_sub_dispatch(&strong_count, 1) == 1) {  // 原子减 1
    // 需要 acquire fence 确保看到所有其他线程的写入
    __atomic_thread_fence(memory_order_acquire);
    dispose();  // 调用 deleter 销毁对象
}
```

拷贝时用 `relaxed`（只需保证原子性），析构时用 `acquire`（需要看到其他线程对对象的写入）——这就是为什么引用计数操作虽然线程安全，但仍有开销。

### 3.4 make_shared vs new

```cpp
// 方式一：new（两次分配）
std::shared_ptr<Widget> sp1(new Widget());
// 分配 1：new Widget() → 堆上分配 Widget
// 分配 2：shared_ptr 内部 new 控制块

// 方式二：make_shared（一次分配）
auto sp2 = std::make_shared<Widget>();
// 一次分配：Widget + 控制块在同一块内存中
```

**内存布局对比**：

```
new Widget() + shared_ptr:
  堆块1: [Widget]
  堆块2: [strong_count | weak_count | deleter | allocator]

make_shared:
  堆块1: [strong_count | weak_count | deleter | allocator | Widget]
         └────────── 控制块 ──────────────────────┘└─ 对象 ─┘
```

**make_shared 的优势**：
1. **性能**：一次 `malloc` 代替两次，减少内存分配器开销和内存碎片
2. **异常安全**：不会出现"对象分配了但控制块还没创建"的中间状态
3. **缓存友好**：对象和控制块在连续内存中

**make_shared 的劣势**：
- 对象内存延迟释放：即使 `strong_count == 0`，只要 `weak_count > 0`，整块内存（含对象）都不能释放，因为控制块和对象在同一块里。对于大对象 + 长生命周期的 `weak_ptr`，可能浪费内存。
- 不能指定自定义删除器（要用 `std::allocate_shared`）
- 不能用 `{}` 初始化（没有 `initializer_list` 重载）

### 3.5 循环引用

这是 `shared_ptr` 最经典的陷阱：

```cpp
#include <memory>
#include <iostream>

struct B;  // 前向声明

struct A {
    std::shared_ptr<B> b_ptr;
    ~A() { std::cout << "A 析构\n"; }
};

struct B {
    std::shared_ptr<A> a_ptr;  // ❌ 循环引用！
    ~B() { std::cout << "B 析构\n"; }
};

int main() {
    auto a = std::make_shared<A>();  // a 的 strong_count = 1
    auto b = std::make_shared<B>();  // b 的 strong_count = 1

    a->b_ptr = b;  // b 的 strong_count = 2
    b->a_ptr = a;  // a 的 strong_count = 2

    // main 结束：
    // a 析构 → A 的 strong_count 从 2 降到 1（≠0，不释放）
    // b 析构 → B 的 strong_count 从 2 降到 1（≠0，不释放）
    // A 和 B 永远不会被释放 → 内存泄漏！
    // "A 析构" 和 "B 析构" 永远不会打印
}
```

**用 weak_ptr 解决**：

```cpp
struct B {
    std::weak_ptr<A> a_ptr;  // ✅ 用 weak_ptr 打破循环
    ~B() { std::cout << "B 析构\n"; }
};

int main() {
    auto a = std::make_shared<A>();  // strong_count = 1
    auto b = std::make_shared<B>();  // strong_count = 1

    a->b_ptr = b;  // b 的 strong_count = 2
    b->a_ptr = a;  // a 的 strong_count 仍然 = 1（weak_ptr 不增加 strong_count）

    // main 结束：
    // a 析构 → A 的 strong_count 从 1 降到 0 → A 被销毁 → b_ptr 析构 → B 的 strong_count 从 2 降到 1
    // b 析构 → B 的 strong_count 从 1 降到 0 → B 被销毁
    // 输出："A 析构" "B 析构"
}
```

**规则**：双向关联中，"父 → 子" 用 `shared_ptr`，"子 → 父" 用 `weak_ptr`。

### 3.6 enable_shared_from_this

**问题**：在类的成员函数里，怎么获取指向自己的 `shared_ptr`？

```cpp
// ❌ 错误：从 this 构造 shared_ptr → 创建了新的控制块 → 双重释放！
struct Bad {
    std::shared_ptr<Bad> getPtr() {
        return std::shared_ptr<Bad>(this);  // 灾难！新控制块，两次 delete
    }
};

// ✅ 正确：继承 enable_shared_from_this
struct Good : public std::enable_shared_from_this<Good> {
    std::shared_ptr<Good> getPtr() {
        return shared_from_this();  // 返回和外部相同控制块的 shared_ptr
    }
};

auto g = std::make_shared<Good>();
auto g2 = g->getPtr();  // g 和 g2 共享同一个控制块
// strong_count = 2，安全
```

**原理**：`enable_shared_from_this<T>` 内部有一个 `weak_ptr<T> weak_this_`。当第一个 `shared_ptr<T>` 被创建时（通过 `make_shared` 或 `shared_ptr` 构造函数），它会检测 T 是否继承了 `enable_shared_from_this`，如果是，就把 `weak_this_` 设为指向自己。`shared_from_this()` 就是 `weak_this_.lock()`。

**注意**：必须已经有外部的 `shared_ptr` 管理这个对象，才能调用 `shared_from_this()`。否则 `weak_this_` 为空，抛出 `std::bad_weak_ptr`。

```cpp
// ❌ 错误：在栈对象上调用
Good g;
g.getPtr();  // 抛出 bad_weak_ptr，因为没有 shared_ptr 管理 g
```

> **面试速答**
> - **是什么/本质**：CRTP 基类，让对象能在成员函数内安全获取指向自身的 shared_ptr
> - **解决什么问题**：直接 `shared_ptr<T>(this)` 会创建新控制块导致双重释放——同一个裸指针被两个控制块管理
> - **底层原理**：内部持有 `weak_ptr<T> weak_this_`，首次被 shared_ptr 托管时自动初始化；`shared_from_this()` 本质是 `weak_this_.lock()`
> - **坑点**：对象必须已被 shared_ptr 管理，在栈对象或裸 new 对象上调用 shared_from_this() 会抛 bad_weak_ptr
> - **选型建议**：异步回调中需要延长对象生命周期时使用（如 Asio 的异步操作链）；不需要 shared_from_this 就不要继承它

### 3.7 性能开销分析

| 开销来源 | 大小 / 耗时 |
|---------|------------|
| `shared_ptr` 本身 | 16 字节（2 个指针） vs 裸指针 8 字节 |
| 控制块 | 通常 32~40 字节（两个原子计数 + deleter + allocator + vtable 指针） |
| 拷贝 | 一次原子 `fetch_add`（约 5~40 ns，取决于缓存竞争） |
| 析构 | 一次原子 `fetch_sub` + 条件分支 |
| make_shared | 一次 `malloc`（vs new 的两次） |
| 解引用 | 和裸指针相同（零开销） |

**和 `unique_ptr` 对比**：`unique_ptr` 无控制块、无原子操作，解引用和裸指针完全相同。在性能敏感路径上，优先用 `unique_ptr`。

### 3.8 优缺点与权衡

**优点**：
- 自动生命周期管理：引用计数归零时自动析构对象，彻底解决"谁负责 delete"的问题，尤其适合所有权在多个模块之间动态流转的场景（缓存、异步回调、观察者模式）
- 跨模块安全：控制块中存储了 deleter，对象的析构总是在创建它的模块/DLL 的堆上执行，避免跨 DLL 的 new/delete 不匹配问题
- 类型擦除的删除器：不同删除器的 `shared_ptr<T>` 是同一类型，可以放在同一个容器中，比 unique_ptr 的模板参数删除器更灵活

**缺点/代价**：
- 原子操作开销：每次拷贝和析构涉及 `fetch_add`/`fetch_sub` 原子操作，在多核系统上需要缓存行同步（MESI 协议），单次原子操作约 5~40ns（取决于缓存竞争程度），比非原子操作慢 10~20 倍
- 内存开销：shared_ptr 本身 16 字节（2 个指针），控制块额外 32~40 字节（两个原子计数 + deleter + allocator + vtable 指针），相比裸指针 8 字节开销显著
- 循环引用导致内存泄漏：A 持有 B 的 shared_ptr，B 也持有 A 的 shared_ptr，引用计数永远不归零，这是 shared_ptr 最经典的陷阱，必须用 weak_ptr 打破
- make_shared 的延迟释放：make_shared 将对象和控制块分配在同一内存块中，即使 strong_count 归零（对象逻辑上已销毁），只要还有 weak_ptr 存在（weak_count > 0），整块内存（包括对象占用的部分）都无法释放

**权衡**：shared_ptr 的核心代价是原子操作开销和内存开销。在热路径上频繁拷贝 shared_ptr（如每次函数调用都传值）会成为性能瓶颈——优化方式是传 `const shared_ptr<T>&` 或直接传裸指针/引用（在确保生命周期被外层 shared_ptr 覆盖的前提下）。只有在真正需要共享所有权时才使用 shared_ptr，否则 unique_ptr 在各方面都更优。

### 3.9 更多陷阱

#### 用 this 构造 shared_ptr（重复上面，强调）

```cpp
// ❌ 永远不要这样做
struct Foo {
    void registerSelf(Registry& r) {
        r.add(std::shared_ptr<Foo>(this));  // 新控制块 → 双重释放
    }
};

// ✅ 用 enable_shared_from_this
struct Foo : std::enable_shared_from_this<Foo> {
    void registerSelf(Registry& r) {
        r.add(shared_from_this());
    }
};
```

#### 跨 DLL 边界

```cpp
// 在 DLL A 中 new 对象，在 DLL B 中 delete → 如果两个 DLL 用不同的堆，crash！
// shared_ptr 的解决方式：控制块里存了 deleter，deleter 在创建时绑定
// 所以 delete 总是在创建对象的 DLL 的堆上执行——前提是用 make_shared 或
// 在创建处指定 deleter
```

#### 数组（C++17 之前不直接支持）

```cpp
// C++17 之前：
std::shared_ptr<int> arr(new int[10], std::default_delete<int[]>());

// C++20：直接支持
auto arr = std::make_shared<int[]>(10);  // 10 个 int
```

> **面试速答**
> - **是什么/本质**：共享所有权的智能指针，通过控制块维护引用计数，强计数归零析构对象，弱计数归零释放控制块
> - **解决什么问题**：多个拥有者需要共同管理同一个对象的生命周期，最后一个释放者负责销毁
> - **底层原理**：引用计数增减是原子操作（拷贝用 relaxed，析构用 acquire-release）；make_shared 一次分配对象+控制块（减少一次 malloc，提升缓存局部性）
> - **优缺点**：优点是自动生命周期管理；缺点是控制块开销（两个原子计数 + 删除器 + 分配器）、原子操作比非原子慢 10-20 倍
> - **坑点**：shared_ptr 拷贝/销毁线程安全，但指向的对象不是——多线程读写同一 shared_ptr 变量或数据仍需 mutex；make_shared 下 weak_ptr 存在则对象内存无法释放
> - **横向对比**：vs unique_ptr——shared_ptr 有原子计数开销，unique_ptr 零开销；循环引用用 weak_ptr 打破（父→子 shared，子→父 weak）
> - **选型建议**：优先 make_shared 创建；确认真正需要共享所有权才用，否则 unique_ptr 更高效

## 4. weak_ptr 深度剖析

### 4.1 概念与本质

**一句话定义**：`weak_ptr` 是不增加引用计数的观察者指针——它能感知对象是否存活，但不延长对象的生命周期。

**为什么需要它**：
1. 打破 `shared_ptr` 循环引用
2. 缓存：持有资源的弱引用，资源还在就复用，被释放了就重新加载
3. 观察者模式：观察者持有被观察者的 `weak_ptr`，不影响被观察者的生命周期

### 4.2 lock() 的使用模式

`weak_ptr` 不能直接解引用，必须先 `lock()` 提升为 `shared_ptr`：

```cpp
auto sp = std::make_shared<int>(42);
std::weak_ptr<int> wp = sp;

// 使用模式：lock + 检查
if (auto locked = wp.lock()) {
    // locked 是 shared_ptr<int>，对象保证存活
    std::cout << *locked << "\n";  // 42
} else {
    // 对象已销毁
    std::cout << "对象已释放\n";
}

sp.reset();  // 销毁对象
// 此时 wp.lock() 返回空的 shared_ptr
```

### 4.3 expired() vs lock()

```cpp
// ❌ 错误：TOCTOU 竞争
if (!wp.expired()) {
    // 在这行和上一行之间，另一个线程可能已经释放了对象！
    auto sp = wp.lock();  // 可能为空
    *sp;  // UB！
}

// ✅ 正确：直接用 lock()，一步完成
if (auto sp = wp.lock()) {
    *sp;  // 安全，sp 保证非空
}
```

`expired()` 几乎只适合用于"快速检查，不需要实际使用对象"的场景，且要接受结果可能过时。

### 4.4 观察者模式中的应用

```cpp
class Subject : public std::enable_shared_from_this<Subject> {
    // 用 weak_ptr 持有观察者，不延长观察者生命周期
    std::vector<std::weak_ptr<Observer>> observers_;
public:
    void addObserver(std::shared_ptr<Observer> obs) {
        observers_.push_back(obs);
    }

    void notify() {
        // 清理已销毁的观察者，通知存活的
        auto it = observers_.begin();
        while (it != observers_.end()) {
            if (auto obs = it->lock()) {
                obs->onEvent();  // 观察者仍存活，通知它
                ++it;
            } else {
                it = observers_.erase(it);  // 观察者已销毁，清理
            }
        }
    }
};
```

### 4.5 缓存中的应用

```cpp
class TextureCache {
    std::unordered_map<std::string, std::weak_ptr<Texture>> cache_;
public:
    std::shared_ptr<Texture> getTexture(const std::string& name) {
        auto it = cache_.find(name);
        if (it != cache_.end()) {
            if (auto sp = it->second.lock()) {
                return sp;  // 缓存命中，返回已有纹理
            }
            cache_.erase(it);  // 纹理已被释放，清理缓存条目
        }
        // 缓存未命中，加载新纹理
        auto tex = std::make_shared<Texture>(loadFromDisk(name));
        cache_[name] = tex;
        return tex;
    }
};
```

### 4.6 优缺点与权衡

**优点**：
- 打破循环引用：weak_ptr 不增加 strong_count，是解决 shared_ptr 循环引用（内存泄漏）的唯一标准方案
- 安全的生存检测：通过 `lock()` 原子地检查对象是否存活并提升为 shared_ptr，比裸指针"猜测对象是否还在"安全得多——裸指针无法知道指向的对象是否已被销毁
- 缓存友好：作为缓存的值类型，资源还在就复用（lock 成功），资源被释放就重新加载（lock 返回空），不会阻止不再需要的资源被释放

**缺点/代价**：
- 使用复杂度高：不能直接解引用，每次使用都必须先 `lock()` 提升为 shared_ptr 再操作，代码比直接用 shared_ptr 或裸指针繁琐
- lock() 有原子操作开销：`lock()` 内部需要原子地读取 strong_count 并条件递增（CAS 操作），在高频调用路径上不可忽略
- 不释放控制块内存：weak_ptr 持有控制块的指针，只要还有 weak_ptr 存在，控制块就不会被释放。如果用了 make_shared（对象和控制块在同一内存块），对象已析构但内存无法归还，在大对象 + 长生命周期 weak_ptr 的场景下造成内存浪费
- expired() 的 TOCTOU 陷阱：`if (!wp.expired()) { auto sp = wp.lock(); }` 看似安全实则有竞争——检查和提升之间对象可能被另一个线程销毁

**权衡**：weak_ptr 的使用场景非常明确——循环引用、缓存、观察者模式。在这些场景中它是不可替代的。但不应该在不需要的地方使用 weak_ptr（如只需要"借用"对象时用裸指针/引用更简单高效）。性能敏感场景中，如果能通过设计保证对象生命周期覆盖观察者的使用周期，用裸指针观察比 weak_ptr 的 lock() 开销更低。

> **面试速答**
> - **是什么/本质**：不增加强引用计数的观察者指针，能感知对象是否存活但不延长其生命周期
> - **解决什么问题**：打破 shared_ptr 循环引用；缓存系统中观察资源是否还在；观察者模式中不影响被观察者生命周期
> - **底层原理**：持有指向控制块的指针，通过检查强引用计数判断对象是否存活；lock() 原子地检查计数并提升为 shared_ptr
> - **坑点**：不要用 `expired() + lock()` 两步操作——存在 TOCTOU 竞争（检查和提升之间对象可能被销毁），直接 `lock()` 一步完成
> - **横向对比**：vs 裸指针观察——裸指针无法知道对象是否已销毁，weak_ptr 通过控制块安全判断；vs shared_ptr——weak_ptr 不影响生命周期
> - **选型建议**：循环引用中子→父方向用 weak_ptr；缓存场景资源还在就复用（lock 成功），释放了就重新加载（lock 返回空）

## 5. 智能指针选型决策树

```
你需要管理堆资源吗？
├─ 否 → 裸指针 / 引用（不拥有所有权）
└─ 是 →
    需要共享所有权吗？
    ├─ 否 → unique_ptr（默认选择，零开销）
    │   ├─ 需要传递给不理解智能指针的 C API？→ .get() 或 .release()
    │   └─ 需要多态？→ unique_ptr<Base>，析构函数声明为 virtual
    └─ 是 →
        可以确定一方是"拥有者"，另一方是"观察者"吗？
        ├─ 是 → 拥有者用 shared_ptr，观察者用 weak_ptr
        └─ 否 → 都用 shared_ptr（注意不要循环引用）
```

**经验法则**：不确定时先用 `unique_ptr`。90% 的场景 `unique_ptr` 就够了。只有在真正需要共享所有权时才升级到 `shared_ptr`。

## 7. 自测问题

### Q1：为什么 `unique_ptr` 和裸指针一样大？
> [!note]- 答案
> 默认删除器 `std::default_delete<T>` 是空类（无数据成员）。编译器通过空基类优化（EBO）将其压缩为零字节。因此 `unique_ptr<T>` 只有一个 `T*` 成员，大小等于裸指针。如果使用有状态的删除器（如带捕获的 lambda），则会变大。

### Q2：`shared_ptr` 的引用计数是线程安全的，那我可以在多线程中放心使用 `shared_ptr` 吗？
> [!note]- 答案
> 不完全可以。引用计数的增减是原子操作，线程安全。但：(1) 多线程同时读写同一个 `shared_ptr` 变量（不是拷贝，是同一个）是数据竞争；(2) 多线程同时访问 `shared_ptr` 指向的对象也不是线程安全的。前者需要 mutex 或 C++20 的 `std::atomic<shared_ptr<T>>`；后者需要用 mutex 保护对象本身。

### Q3：`make_shared` 相比 `new` + `shared_ptr` 有什么优势和劣势？
> [!note]- 答案
> **优势**：(1) 一次内存分配（对象和控制块在一块内存中），性能更好；(2) 异常安全，不会出现对象分配了但控制块还没创建的中间状态。**劣势**：(1) 对象内存延迟释放——即使 `strong_count==0`，只要 `weak_count>0`，整块内存都不会释放；(2) 不支持自定义删除器；(3) 不支持花括号初始化。

### Q4：什么是循环引用？如何解决？
> [!note]- 答案
> A 持有 B 的 `shared_ptr`，B 也持有 A 的 `shared_ptr`，形成环。当外部不再持有它们时，A 和 B 的引用计数都至少为 1，永远不会降到 0，导致内存泄漏。解决方法：双向关联中，一方用 `weak_ptr`（通常是"子 → 父"方向），打破环。

### Q5：`weak_ptr::expired()` 和 `weak_ptr::lock()` 有什么区别？为什么不建议用 `expired()` + `lock()` 组合？
> [!note]- 答案
> `expired()` 只检查 `strong_count` 是否为 0，`lock()` 原子地将 `weak_ptr` 提升为 `shared_ptr`。如果先 `expired()` 再 `lock()`，两步之间另一个线程可能释放对象（TOCTOU 竞争）。正确做法是直接 `if (auto sp = wp.lock()) { ... }`，一步完成检查和提升。

### Q6：`enable_shared_from_this` 解决什么问题？原理是什么？
> [!note]- 答案
> 解决"在成员函数中获取指向 this 的 shared_ptr"的问题。如果直接 `shared_ptr<T>(this)` 会创建新控制块，导致双重释放。原理：`enable_shared_from_this` 内部存有一个 `weak_ptr<T>`，当第一个 `shared_ptr<T>` 创建时自动赋值。`shared_from_this()` 就是对这个 `weak_ptr` 调用 `lock()`。

### Q7：为什么 `auto_ptr` 被废弃，`unique_ptr` 是如何解决它的问题的？
> [!note]- 答案
> `auto_ptr` 的拷贝构造/赋值会偷偷转移所有权，导致源指针悬空。这在容器中尤其危险（比如 `sort` 可能导致部分元素变成空）。`unique_ptr` 通过 `delete` 拷贝构造/赋值、只允许 `std::move` 显式转移，把所有权转移变成编译期可检查的行为。

### Q8：`shared_ptr` 的控制块中存了哪些东西？控制块什么时候被释放？
> [!note]- 答案
> 控制块包含：`strong_count`（原子）、`weak_count`（原子）、deleter、allocator。`strong_count` 降到 0 时调用 deleter 销毁对象，但控制块还在。`weak_count` 也降到 0 时，控制块才被释放。如果用了 `make_shared`，对象和控制块在同一块内存中，所以对象的内存也要等 `weak_count` 归零才释放。

### Q9：`unique_ptr` 的自定义删除器在哪些场景下有用？使用自定义删除器会影响 `unique_ptr` 的大小吗？
> [!note]- 答案
> 场景：管理 `FILE*`（fclose）、socket fd（close）、C 库分配的内存（free）、数组等非 `new` 资源。对大小的影响：无状态删除器（空 lambda、函数对象无数据成员）通过 EBO 优化为零开销；有状态删除器（带捕获的 lambda、`std::function`）会增加 `unique_ptr` 的大小。

### Q10：`shared_ptr` 的删除器和 `unique_ptr` 的删除器在设计上有什么区别？为什么？
> [!note]- 答案
> `unique_ptr` 的删除器是模板参数（编译期绑定），不同删除器产生不同类型，零开销但不灵活。`shared_ptr` 的删除器通过类型擦除存在控制块中（运行期绑定），不影响 `shared_ptr<T>` 的类型，可以把不同删除器的 `shared_ptr<T>` 放在同一个容器中——灵活但有额外开销。设计原因：`shared_ptr` 已经有控制块了，多存一个删除器代价不大；`unique_ptr` 追求零开销，不引入间接层。

### Q11：在高并发场景下，`shared_ptr` 的原子引用计数会成为性能瓶颈吗？
> [!note]- 答案
> 会。每次拷贝和析构 `shared_ptr` 都需要原子操作（`fetch_add`/`fetch_sub`），在多核处理器上涉及缓存行同步。如果热路径上频繁拷贝 `shared_ptr`（如每次函数调用都传 `shared_ptr`），原子操作的缓存一致性协议（MESI）会成为瓶颈。优化方式：(1) 传 `const shared_ptr<T>&` 或裸指针/引用，避免不必要的拷贝；(2) 把 `shared_ptr` 降级为 `unique_ptr`（如果不需要共享）；(3) 局部用裸指针观察，确保生命周期被外层 `shared_ptr` 覆盖。

### Q12：下面代码有什么问题？
```cpp
std::shared_ptr<int> sp(new int(42));
std::shared_ptr<int> sp2(sp.get());
```
> [!note]- 答案
> `sp.get()` 返回裸指针，用它构造 `sp2` 会创建新的控制块。此时两个控制块都认为自己拥有这块内存，当 `sp` 和 `sp2` 析构时会双重释放，导致 crash 或未定义行为。正确做法：直接拷贝 `auto sp2 = sp;`。

### Q13：`unique_ptr<Base>` 能正确释放 `Derived` 对象吗？
> [!note]- 答案
> 只有当 `Base` 的析构函数声明为 `virtual` 时才能正确释放。否则 `unique_ptr<Base>` 析构时只调用 `Base::~Base()`，`Derived` 部分的资源不会被清理（未定义行为）。这和裸指针 `delete base_ptr` 的规则一样——智能指针不会绕过 C++ 的虚函数机制。
