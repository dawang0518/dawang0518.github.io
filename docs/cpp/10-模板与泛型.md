# 模板与泛型编程

> C++ 零开销抽象的核心武器：让编译器帮你写代码，类型安全 + 性能拉满。

---

## 0. 全景地图

**模板的本质**：把"类型"或"值"变成参数，让编译器在编译期根据具体类型生成代码。你写一份逻辑，编译器帮你"复印"出 N 份类型特定版本——零运行时开销。

**演进脉络**：

```
C++98  函数模板 / 类模板 / 全特化 / 偏特化
  │
C++11  可变参数模板（Variadic Templates）/ type_traits / SFINAE 成熟
  │
C++14  变量模板 / 泛型 lambda
  │
C++17  类模板参数推导 CTAD / 折叠表达式 / if constexpr
  │
C++20  Concepts —— 终结 SFINAE 天书时代
```

**核心思想**：

```
函数模板 ── 泛化算法（一份逻辑适配多种类型）
类模板   ── 泛化数据结构（vector<T>, map<K,V>）
特化     ── 对特定类型提供定制实现
SFINAE   ── 编译期条件选择（被调用者自动匹配）
Concepts ── 给模板参数加约束（可读的 SFINAE）
元编程   ── 把计算搬到编译期（零运行时开销）
```

## 1. 函数模板

### 1.1 概念与本质

函数模板不是函数，是函数的"生成蓝图"。编译器看到调用时，根据实参类型**实例化**出真正的函数。

### 1.2 底层原理

编译器对函数模板做两阶段检查（Two-phase Lookup）：
1. **定义阶段**：检查不依赖模板参数的语法（如普通变量名）
2. **实例化阶段**：代入具体类型后检查依赖模板参数的表达式

每一次用不同类型调用，编译器都生成一份独立的函数——这就是"代码膨胀"的来源。

### 1.3 模板参数三大类

| 类别 | 语法 | 示例 |
|------|------|------|
| 类型参数 | `typename T` / `class T` | `template<typename T>` |
| 非类型参数 | `int N` / `auto N`(C++17) | `template<int N>` 编译期常量 |
| 模板模板参数 | `template<typename> class C` | 容器作为参数传入 |

### 1.4 模板参数推导规则

```cpp
template<typename T>
void f(T x);          // 按值传递：忽略顶层 const 和引用

template<typename T>
void g(T& x);         // 按引用传递：保留 const，T 推导为底层类型

template<typename T>
void h(T&& x);        // 万能引用（转发引用）：完美转发的基石
```

**推导规则核心**：

| 形参类型 | 实参为左值 `int x` | 实参为右值 `42` |
|----------|-------------------|----------------|
| `T` | T=int, 拷贝 | T=int, 拷贝 |
| `T&` | T=int, 绑定左值 | 编译错误 |
| `const T&` | T=int, 绑定左值 | T=int, 绑定右值 |
| `T&&` | T=int&, 引用折叠为左值引用 | T=int, 右值引用 |

### 1.5 代码示例：手写 my_max

```cpp
// 基础版本：按值传递
template<typename T>
T my_max(T a, T b) {
    return (a > b) ? a : b;
}

// 进阶版本：完美转发 + 返回类型推导
template<typename T, typename U>
auto my_max2(T&& a, U&& b) -> decltype(true ? a : b) {
    return (a > b) ? std::forward<T>(a) : std::forward<U>(b);
}

// 非类型参数示例：编译期数组大小
template<typename T, int N>
constexpr int array_size(T (&)[N]) {
    return N;  // 编译期获取数组长度
}

int arr[] = {1, 2, 3, 4, 5};
static_assert(array_size(arr) == 5);  // 编译期断言
```

### 1.6 显式指定 vs 自动推导

```cpp
my_max(1, 2);           // 自动推导 T=int
my_max<double>(1, 2);   // 显式指定 T=double，1 和 2 隐式转换

// 什么时候必须显式指定？
template<typename R, typename T>
R convert(T x) { return static_cast<R>(x); }

convert<double>(42);    // R 无法从实参推导，必须显式指定
```

### 1.7 优缺点与权衡

| 优点 | 缺点 |
|------|------|
| 一份代码适配多种类型 | 编译速度慢（每种类型都要实例化） |
| 零运行时开销（静态分发） | 代码膨胀（每种类型一份二进制） |
| 类型安全（编译期检查） | 错误信息冗长难读 |

### 1.8 陷阱与实战

```cpp
// ❌ 两个实参类型不同，编译器不知道选哪个
my_max(1, 2.5);  // 错误：T 同时推导为 int 和 double

// ✅ 方案一：显式指定
my_max<double>(1, 2.5);

// ✅ 方案二：两个模板参数
template<typename T, typename U>
auto my_max3(T a, U b) { return (a > b) ? a : b; }
```

> **面试速答**
>
> 函数模板说白了就是函数的"蓝图"，编译器根据你传进来的实参类型，实例化出一个个具体的函数。它解决的核心问题是：避免为每种类型手写一份逻辑一模一样的代码，而且同时保持类型安全和零开销——这是它比 C 宏强的地方。
>
> 底层机制是 Two-phase Lookup（两阶段查找）加上编译期实例化：第一阶段在模板定义时解析不依赖模板参数的名字，第二阶段在实例化时解析依赖参数的名字。每种用到的类型组合都会生成一份独立的函数代码。
>
> 优点很明确：零运行时开销 + 类型安全。代价是编译慢、代码膨胀、错误信息难读。
>
> 有几个坑要特别注意：多参数类型不一致（比如一个 int 一个 double）时编译器推导不出来，得显式指定模板参数或者用两个模板参数分开；还有万能引用 `T&&` 不要当右值引用看，它在模板参数推导下会根据实参类别坍缩成左值或右值引用。
>
> 和虚函数对比一下：函数模板是静态多态（编译期搞定），虚函数是动态多态（运行时分发）。所以性能敏感场景用模板；需要运行时灵活性就用虚函数。

## 2. 类模板

### 2.1 概念与本质

类模板是类的"生成蓝图"。`vector<int>` 和 `vector<string>` 是两个完全不同的类——编译器分别实例化。

### 2.2 底层原理

**延迟实例化**：类模板的成员函数只在被调用时才实例化。这意味着：
- 如果你从不调用某个成员函数，它里面的类型错误不会被发现
- 这也是为什么 `vector<unique_ptr<int>>` 能工作——只要你不调用 `push_back(const T&)` 就行

**为什么必须在头文件中定义？**
编译器在实例化模板时需要看到完整定义。如果定义在 `.cpp` 中，其他翻译单元看不到定义，无法实例化，链接时报 `undefined reference`。

### 2.3 代码示例

```cpp
// 简单的栈实现
template<typename T, int MaxSize = 100>
class Stack {
    T data_[MaxSize];
    int top_ = -1;

public:
    void push(const T& val) {
        if (top_ >= MaxSize - 1) throw std::overflow_error("栈满");
        data_[++top_] = val;
    }

    T pop() {
        if (top_ < 0) throw std::underflow_error("栈空");
        return data_[top_--];
    }

    bool empty() const { return top_ < 0; }
    int size() const { return top_ + 1; }
};

// 使用
Stack<int> s1;          // 默认大小 100
Stack<std::string, 50> s2;  // 指定大小 50
```

### 2.4 CTAD（C++17 类模板参数推导）

```cpp
// C++17 之前
std::pair<int, double> p1(1, 2.5);
std::vector<int> v1{1, 2, 3};

// C++17 之后——自动推导
std::pair p2(1, 2.5);      // pair<int, double>
std::vector v2{1, 2, 3};   // vector<int>
std::mutex mtx;
std::lock_guard lk(mtx);   // lock_guard<mutex>

// 自定义推导指引（Deduction Guide）
template<typename T>
Stack(T) -> Stack<T>;  // 允许 Stack s(42); 推导为 Stack<int>
```

### 2.5 对比表

| 维度 | 函数模板 | 类模板 |
|------|---------|--------|
| 参数推导 | 自动推导（一直支持） | C++17 才支持 CTAD |
| 偏特化 | 不支持 | 支持 |
| 默认参数 | 支持（C++11） | 一直支持 |
| 成员实例化 | 调用时实例化 | 调用时实例化（延迟） |

### 2.6 陷阱与实战

```cpp
// ❌ 模板定义放 .cpp 中
// stack.h
template<typename T> class Stack { void push(const T&); };
// stack.cpp
template<typename T> void Stack<T>::push(const T& val) { ... }
// main.cpp
Stack<int> s; s.push(1);  // 链接报错：undefined reference

// ✅ 方案一：全部在头文件中定义
// ✅ 方案二：在 .cpp 末尾显式实例化
template class Stack<int>;
template class Stack<std::string>;
```

> **面试速答**
>
> 类模板就是类的"生成蓝图"，编译器根据模板参数给你生成一个具体的类。它存在的意义是泛化数据结构，一份代码能适配多种元素类型——STL 容器就是这个思路。
>
> 底层有一个重要机制叫延迟实例化：类模板的成员函数不是一上来就全部实例化，而是在被调用时才实例化。这会带来一个副作用——未被调用的成员函数里的错误不会被发现，等真的调用了才爆出来。
>
> 优点是代码复用 + 类型安全，代价是编译慢，而且有个硬性约束：模板定义必须放头文件里。这带来一个经典坑：如果你把类模板的成员函数实现放在 .cpp 里，链接阶段会找不到符号报错。绕开办法是要么全放头文件，要么在 .cpp 末尾显式实例化（`template class Stack<int>;`）。
>
> 和基类 + 虚函数对比一下：类模板是编译期多态，虚函数是运行时多态。选型思路很清晰——把元素类型参数化用类模板（STL 容器思路）；把行为参数化更适合用继承 + 虚函数。

## 3. 模板特化与偏特化

### 3.1 概念与本质

**全特化**：为某个具体类型提供完全定制的实现，覆盖通用模板。
**偏特化**：为满足某些条件的一类类型提供定制实现（比如"所有指针类型"）。

特化的本质是**模式匹配**——编译器根据模板参数选择最匹配的版本。

### 3.2 底层原理：匹配优先级

```
1. 非模板函数（精确匹配）     ← 最优先
2. 全特化版本
3. 偏特化版本（仅类模板）
4. 主模板（通用版本）          ← 兜底
```

对于函数模板，编译器优先选择"更特殊"的重载。函数模板不支持偏特化，但可以用重载达到类似效果。

### 3.3 代码示例

```cpp
// === 类模板全特化 ===
// 主模板
template<typename T>
struct TypeName {
    static std::string get() { return "未知类型"; }
};

// 全特化：int
template<>
struct TypeName<int> {
    static std::string get() { return "int"; }
};

// 全特化：std::string
template<>
struct TypeName<std::string> {
    static std::string get() { return "string"; }
};

// === 类模板偏特化 ===
// 偏特化：所有指针类型
template<typename T>
struct TypeName<T*> {
    static std::string get() {
        return TypeName<T>::get() + " 的指针";
    }
};

// 偏特化：所有 vector 类型
template<typename T>
struct TypeName<std::vector<T>> {
    static std::string get() {
        return "vector<" + TypeName<T>::get() + ">";
    }
};

TypeName<int>::get();              // "int"（全特化）
TypeName<int*>::get();             // "int 的指针"（偏特化）
TypeName<std::vector<int>>::get(); // "vector<int>"（偏特化）
TypeName<double>::get();           // "未知类型"（主模板）
```

### 3.4 实战：std::hash 特化

```cpp
// 为自定义类型特化 std::hash，让它能做 unordered_map 的 key
struct Point {
    int x, y;
    bool operator==(const Point& o) const = default;
};

template<>
struct std::hash<Point> {
    size_t operator()(const Point& p) const {
        // 经典的哈希组合方式
        size_t h1 = std::hash<int>{}(p.x);
        size_t h2 = std::hash<int>{}(p.y);
        return h1 ^ (h2 << 1);  // 简单异或移位
    }
};

std::unordered_map<Point, std::string> mp;
mp[{1, 2}] = "原点附近";
```

### 3.5 函数模板：重载代替偏特化

```cpp
// 函数模板不支持偏特化，用重载代替
template<typename T>
void process(T val) { std::cout << "通用版本\n"; }

template<typename T>
void process(T* val) { std::cout << "指针版本\n"; }  // 不是偏特化，是重载！

void process(int val) { std::cout << "int 版本\n"; }  // 非模板函数，优先级最高

process(42);        // "int 版本"（非模板优先）
process(3.14);      // "通用版本"
int x = 1;
process(&x);        // "指针版本"（更特殊的重载）
```

### 3.6 对比表

| 维度 | 全特化 | 偏特化 | 重载 |
|------|--------|--------|------|
| 适用对象 | 函数模板、类模板 | 仅类模板 | 仅函数 |
| 参数 | 所有参数固定 | 部分参数固定/模式匹配 | 独立函数 |
| 优先级 | 高于主模板 | 高于主模板，低于全特化 | 非模板优先于模板 |
| 语法 | `template<>` | `template<typename T>` | 普通函数声明 |

### 3.7 陷阱与实战

```cpp
// ❌ 函数模板全特化的陷阱：特化不参与重载决议
template<typename T> void f(T) { cout << "主模板\n"; }
template<typename T> void f(T*) { cout << "指针重载\n"; }

template<> void f<int*>(int*) { cout << "int* 全特化\n"; }  // 这是主模板的特化！

int* p = nullptr;
f(p);  // 输出"指针重载"——全特化被忽略了！
// 因为重载决议先选中 f(T*) 版本，再在这个版本里找特化，没找到

// ✅ 建议：函数模板优先用重载，别用全特化
```

> **面试速答**
>
> 模板特化说白了就是给某些特定类型定制一套专属实现，本质是编译期的模式匹配。为什么要搞这个？因为通用模板有时候处理不了特殊类型的需求，最经典的例子就是 `vector<bool>` 的位压缩——通用 `vector<T>` 按 T 存储，但 bool 只需要一位就能表示，特化版本改成按位存储能大幅节省空间。
>
> 底层的匹配优先级是：非模板函数 > 全特化 > 偏特化 > 主模板。编译器按这个顺序挑最合适的候选。
>
> 优点是灵活定制，缺点是增加维护复杂度，而且偏特化只能用在类模板上，函数模板不支持偏特化。
>
> 有个容易踩的大坑要特别注意：函数模板的全特化不参与重载决议，它只是对主模板"打补丁"。如果你同时有一个函数重载版本和一个全特化版本，重载决议先选中重载那个，全特化就被"吞掉"了，永远不会被调用。所以实际工程里有条结论：类模板用偏特化，函数模板干脆用重载替代特化，别用全特化，省得掉坑。
>
> 横向类比一下：偏特化像 Rust 的 trait impl 那种模式匹配；重载则更像按函数签名匹配。

## 4. 可变参数模板（Variadic Templates）

### 4.1 概念与本质

可变参数模板让模板接受任意数量、任意类型的参数。核心概念是 **Parameter Pack**（参数包）和 **Pack Expansion**（包展开）。

这是 `std::tuple`、`std::make_shared`、`std::invoke` 等现代 C++ 设施的基石。

### 4.2 底层原理

```
template<typename... Args>   // Args 是类型参数包
void f(Args... args)         // args 是函数参数包
{
    sizeof...(Args);          // 编译期获取参数个数
    // 包展开：args... 在特定上下文中展开为逗号分隔的列表
}
```

参数包**不能直接使用**，必须通过"包展开"来访问每个元素。展开方式有两种：
1. **递归展开**（C++11）：递归函数模板，每次剥离一个参数
2. **折叠表达式**（C++17）：直接对整个包应用运算符

### 4.3 代码示例：递归展开

```cpp
// 终止条件（递归基）
void print() {
    std::cout << '\n';
}

// 递归展开：每次剥离第一个参数
template<typename T, typename... Args>
void print(T first, Args... rest) {
    std::cout << first << " ";
    print(rest...);  // 递归，参数包缩小
}

print(1, "hello", 3.14, 'x');
// 展开过程：
// print(1, "hello", 3.14, 'x')  → 输出 1，调用 print("hello", 3.14, 'x')
// print("hello", 3.14, 'x')    → 输出 hello，调用 print(3.14, 'x')
// print(3.14, 'x')             → 输出 3.14，调用 print('x')
// print('x')                   → 输出 x，调用 print()
// print()                      → 输出换行，结束
```

### 4.4 代码示例：折叠表达式（C++17）

```cpp
// 一行搞定，不需要递归基
template<typename... Args>
void print(Args... args) {
    ((std::cout << args << " "), ...);  // 一元右折叠
    std::cout << '\n';
}

// 折叠表达式语法（4 种形式）
// (pack op ...)       一元右折叠：a1 op (a2 op (a3 op a4))
// (... op pack)       一元左折叠：((a1 op a2) op a3) op a4
// (pack op ... op init) 二元右折叠
// (init op ... op pack) 二元左折叠

// 实战：求和
template<typename... Args>
auto sum(Args... args) {
    return (args + ...);  // 一元右折叠
}
sum(1, 2, 3, 4);  // 10

// 实战：全部满足条件
template<typename... Args>
bool all_positive(Args... args) {
    return ((args > 0) && ...);  // 一元右折叠
}
```

### 4.5 实战：简化 tuple 实现

```cpp
// 递归继承实现 tuple 的核心思想
template<typename... Ts>
struct MyTuple {};

// 偏特化：剥离第一个类型
template<typename Head, typename... Tail>
struct MyTuple<Head, Tail...> : MyTuple<Tail...> {
    Head value;

    MyTuple(Head h, Tail... t) : MyTuple<Tail...>(t...), value(h) {}
};

// 使用
MyTuple<int, double, std::string> t(42, 3.14, "hello");
// 继承链：MyTuple<int,double,string> → MyTuple<double,string> → MyTuple<string> → MyTuple<>
```

### 4.6 包展开的上下文

```cpp
template<typename... Args>
void examples(Args... args) {
    // 1. 函数调用
    f(args...);                    // f(a1, a2, a3)
    f(process(args)...);           // f(process(a1), process(a2), process(a3))

    // 2. 初始化列表
    int arr[] = {args...};
    int arr2[] = {(args * 2)...};  // 对每个元素乘 2

    // 3. 基类列表
    // struct Derived : Bases... { };

    // 4. sizeof...
    constexpr auto n = sizeof...(Args);
}
```

### 4.7 对比表

| 维度 | 递归展开 | 折叠表达式 |
|------|---------|-----------|
| 标准 | C++11 | C++17 |
| 可读性 | 需要递归基，代码较长 | 一行搞定，简洁 |
| 编译速度 | 递归实例化多个函数 | 通常更快 |
| 灵活性 | 可以对每个元素做复杂操作 | 适合简单运算符 |
| 调试 | 递归栈深可能很深 | 扁平 |

> **面试速答**
>
> 可变参数模板就是能接受任意数量、任意类型参数的模板机制，核心概念叫"参数包"。它存在的最大价值是替代 C 语言的 `va_args`——后者完全没有类型安全，出错也不报错；可变参数模板则是编译期展开，完全类型安全。
>
> 底层机制上，参数包在编译期会被展开成具体的参数列表，每种实参组合都会生成一份独立代码。
>
> 优点是灵活强大，缺点是递归展开的方式编译比较慢。C++17 引入的折叠表达式大幅改善了可读性——以前要靠递归加递归基两个重载才能写出来的东西，现在一行 `(args + ...)` 就搞定。
>
> 有个容易踩的坑要特别注意：包展开（`...`）只在特定上下文（比如函数调用的参数列表、初始化列表）里才有效，不能乱用。另外如果你用递归方式展开，一定记得写递归基（终止条件），不然会一直实例化下去直到编译器崩溃。
>
> 选型建议：C++17 及以上优先用折叠表达式；需要对每个元素做复杂处理的场景才回去写递归展开。

## 5. SFINAE

### 5.1 概念与本质

**Substitution Failure Is Not An Error**（替换失败不是错误）。

当编译器尝试用某个类型实例化模板时，如果替换导致了无效的类型表达式，编译器不会报错，而是默默跳过这个候选，继续尝试其他候选。

这是 C++11/14 时代实现"编译期条件选择"的核心技术。

### 5.2 底层原理

```
编译器重载决议流程：
1. 收集所有候选函数（包括模板实例化）
2. 对每个模板候选，尝试用实参类型替换模板参数
3. 如果替换导致无效类型（SFINAE 错误）→ 静默丢弃该候选
4. 如果替换成功 → 加入可行候选集
5. 从可行候选集中选择最佳匹配
```

**关键区分**：SFINAE 只在"直接上下文"中生效。如果替换导致的错误发生在函数体内（不是签名中），那就是硬错误，不是 SFINAE。

### 5.3 std::enable_if 的实现和用法

```cpp
// enable_if 的实现极其简单
template<bool B, typename T = void>
struct enable_if {};  // 通用版本：没有 type 成员

template<typename T>
struct enable_if<true, T> {
    using type = T;   // 条件为 true 时才有 type 成员
};

// 用法一：返回类型 SFINAE
template<typename T>
typename std::enable_if<std::is_integral<T>::value, T>::type
safe_divide(T a, T b) {
    return b != 0 ? a / b : 0;
}

// 用法二：模板参数 SFINAE（更常用）
template<typename T,
         typename = std::enable_if_t<std::is_integral_v<T>>>
T safe_divide2(T a, T b) {
    return b != 0 ? a / b : 0;
}

safe_divide(10, 3);     // OK：int 是整数类型
safe_divide(1.0, 2.0);  // 编译错误：double 不是整数类型，SFINAE 排除后无候选
```

### 5.4 检测成员函数是否存在

```cpp
// 检测类型 T 是否有 .size() 成员函数
template<typename T, typename = void>
struct has_size : std::false_type {};

template<typename T>
struct has_size<T, std::void_t<decltype(std::declval<T>().size())>>
    : std::true_type {};

static_assert(has_size<std::vector<int>>::value);  // true
static_assert(!has_size<int>::value);               // false
```

### 5.5 void_t 技巧

```cpp
// void_t 的实现
template<typename...>
using void_t = void;  // 不管参数是啥，结果都是 void

// 它的威力在于：如果参数中的任何表达式无效，整个 void_t 就 SFINAE 失败
// 配合偏特化，实现"检测表达式是否合法"

// 检测是否可迭代
template<typename T, typename = void>
struct is_iterable : std::false_type {};

template<typename T>
struct is_iterable<T, std::void_t<
    decltype(std::begin(std::declval<T>())),
    decltype(std::end(std::declval<T>()))
>> : std::true_type {};

static_assert(is_iterable<std::vector<int>>::value);  // true
static_assert(!is_iterable<int>::value);                // false
```

### 5.6 SFINAE vs if constexpr vs Concepts

| 维度 | SFINAE | if constexpr (C++17) | Concepts (C++20) |
|------|--------|---------------------|-----------------|
| 可读性 | 差（天书） | 好（像普通 if） | 最好（声明式） |
| 错误信息 | 冗长难懂 | 一般 | 清晰指出约束不满足 |
| 使用场景 | 重载决议控制 | 函数体内分支 | 模板参数约束 |
| 标准 | C++98 起 | C++17 | C++20 |

```cpp
// 同一个需求的三种写法对比

// SFINAE 版本
template<typename T, typename = std::enable_if_t<std::is_integral_v<T>>>
void process(T val) { /* 整数处理 */ }

// if constexpr 版本
template<typename T>
void process(T val) {
    if constexpr (std::is_integral_v<T>) {
        // 整数处理
    } else {
        // 其他类型处理
    }
}

// Concepts 版本（C++20，最清晰）
template<std::integral T>
void process(T val) { /* 整数处理 */ }
```

> **面试速答**
>
> SFINAE 全称是 Substitution Failure Is Not An Error，翻译过来就是"模板参数替换失败不算错误"。当编译器尝试把一个具体类型代入模板时，如果替换的结果里出现了不合法的类型/表达式，这个候选会被静默丢弃而不是报错。为什么要搞这么绕的东西？就是为了实现编译期的条件选择——根据类型特征选择不同的模板实现。
>
> 底层机制是：在重载决议阶段，编译器对每个候选模板尝试做参数替换，替换失败的那些被悄悄踢出候选集，剩下的再按正常重载决议走。
>
> 优点是功能强大，能实现相当精巧的类型分发。缺点也非常明显：代码写出来跟天书似的，一旦出错错误信息能刷满屏幕。
>
> 有个坑要特别注意：SFINAE 只在"直接上下文"（函数签名、模板参数列表里）才生效。如果替换失败发生在函数体内部，那就是硬错误，不会被 SFINAE 救回来。
>
> 横向对比，这一路演进很清晰：SFINAE (C++11) → `if constexpr` (C++17) → Concepts (C++20)，一代比一代简化。所以选型建议也很直接：C++20 项目用 Concepts，C++17 用 `if constexpr`，只有卡在 C++11/14 的老项目才去写 SFINAE。

## 6. type_traits 库

### 6.1 概念与本质

`<type_traits>` 是标准库提供的编译期类型查询和变换工具集。本质上是一堆**模板特化 + constexpr bool**。

### 6.2 常用分类

| 类别 | 代表 | 用途 |
|------|------|------|
| 类型判断 | `is_integral`, `is_floating_point`, `is_pointer` | 判断类型属性 |
| 类型关系 | `is_same`, `is_base_of`, `is_convertible` | 判断类型间关系 |
| 类型变换 | `remove_reference`, `remove_const`, `add_pointer` | 变换类型 |
| 条件选择 | `conditional`, `enable_if` | 编译期分支 |
| 其他 | `decay`, `common_type`, `invoke_result` | 类型推导辅助 |

### 6.3 底层实现原理

```cpp
// is_same 的实现：模板特化
template<typename T, typename U>
struct is_same : std::false_type {};  // 通用版本：不同

template<typename T>
struct is_same<T, T> : std::true_type {};  // 全特化：相同

// remove_const 的实现
template<typename T>
struct remove_const { using type = T; };  // 通用版本：原样返回

template<typename T>
struct remove_const<const T> { using type = T; };  // 特化：去掉 const

// conditional 的实现
template<bool B, typename T, typename F>
struct conditional { using type = T; };

template<typename T, typename F>
struct conditional<false, T, F> { using type = F; };

// _t 和 _v 后缀是 C++14/17 的简写
// std::remove_const_t<T>  等价于  typename std::remove_const<T>::type
// std::is_same_v<T, U>    等价于  std::is_same<T, U>::value
```

### 6.4 实战：编译期类型分发

```cpp
// 根据类型选择不同的序列化策略
template<typename T>
std::string serialize(const T& val) {
    if constexpr (std::is_integral_v<T>) {
        return std::to_string(val);
    } else if constexpr (std::is_floating_point_v<T>) {
        // 浮点数保留 6 位有效数字
        char buf[32];
        snprintf(buf, sizeof(buf), "%.6g", static_cast<double>(val));
        return buf;
    } else if constexpr (std::is_same_v<T, std::string>) {
        return "\"" + val + "\"";  // 字符串加引号
    } else {
        static_assert(always_false<T>, "不支持的类型");
    }
}

// always_false 技巧：延迟 static_assert 到实例化时
template<typename> inline constexpr bool always_false = false;
```

### 6.5 decay 的重要性

```cpp
// std::decay 模拟"按值传递"时的类型退化
// 1. 去掉引用
// 2. 数组 → 指针
// 3. 函数 → 函数指针
// 4. 去掉顶层 const/volatile

std::decay_t<int&>        // int
std::decay_t<const int&>  // int
std::decay_t<int[3]>      // int*
std::decay_t<int(double)> // int(*)(double)

// 实际应用：std::make_pair 内部用 decay 确定存储类型
auto p = std::make_pair("hello", 42);
// "hello" 的类型是 const char[6]，decay 后变成 const char*
// 所以 p 的类型是 pair<const char*, int>
```

> **面试速答**
>
> `type_traits` 是标准库提供的一套编译期类型查询和变换工具，底层全靠模板特化实现。它解决的问题是让模板代码能根据类型特征做不同处理——比如判断一个类型是不是整数、是不是指针、能不能默认构造等等。
>
> 底层实现说白了就是模板特化加上 `constexpr bool`，全部在编译期算出来，运行时零开销。
>
> 优点是编译期类型计算非常强大，缺点是 C++11 原始语法比较冗长（`typename std::remove_const<T>::type` 长得劝退）。C++14 加了 `_t` 和 `_v` 后缀（`std::remove_const_t<T>`、`std::is_same_v<A, B>`）才算好看了一些。
>
> 有两个坑要特别注意：第一，`std::decay` 的行为可能和你想的不一样——它会把数组退化成指针、函数退化成函数指针、还会去掉引用和 cv 限定，`std::make_pair("hello", 42)` 之所以能工作就是因为 `decay` 把 `const char[6]` 退化成了 `const char*`。第二，`std::is_same` 会区分 cv 限定，`is_same<int, const int>` 是 `false`，别指望它帮你忽略常量性。
>
> 横向看，`type_traits` 是 SFINAE 和 Concepts 这些高级机制的基础设施。选型建议：现代 C++ 必须掌握的基础库，和 `if constexpr` 搭配食用效果最佳。

## 7. Concepts (C++20)

### 7.1 概念与本质

Concepts 是给模板参数加的**命名约束**。本质上是一个编译期求值为 bool 的表达式，但它被提升为语言特性，编译器能给出清晰的错误信息。

### 7.2 语法

```cpp
// 定义 concept
template<typename T>
concept Addable = requires(T a, T b) {
    { a + b } -> std::same_as<T>;  // a+b 必须合法，且返回类型是 T
};

// 使用方式一：requires 子句
template<typename T> requires Addable<T>
T add(T a, T b) { return a + b; }

// 使用方式二：简写（约束模板参数）
template<Addable T>
T add2(T a, T b) { return a + b; }

// 使用方式三：尾置 requires
template<typename T>
T add3(T a, T b) requires Addable<T> { return a + b; }

// 使用方式四：简写函数模板（最简洁）
auto add4(Addable auto a, Addable auto b) { return a + b; }
```

### 7.3 requires 表达式四种形式

```cpp
template<typename T>
concept MyContainer = requires(T c, typename T::value_type v) {
    // 1. 简单要求：表达式必须合法
    c.size();
    c.begin();
    c.end();

    // 2. 类型要求：类型必须存在
    typename T::value_type;
    typename T::iterator;

    // 3. 复合要求：表达式合法 + 返回类型约束 + 是否 noexcept
    { c.size() } noexcept -> std::same_as<std::size_t>;
    { c.front() } -> std::convertible_to<typename T::value_type>;

    // 4. 嵌套要求：requires 里面再 requires
    requires std::default_initializable<typename T::value_type>;
};
```

### 7.4 标准库常用 Concepts

| Concept | 含义 |
|---------|------|
| `std::same_as<T, U>` | 类型完全相同 |
| `std::derived_from<D, B>` | D 公开继承自 B |
| `std::convertible_to<From, To>` | 隐式可转换 |
| `std::integral` | 整数类型 |
| `std::floating_point` | 浮点类型 |
| `std::invocable<F, Args...>` | 可调用对象 |
| `std::ranges::range` | 可迭代范围 |
| `std::movable` / `std::copyable` | 可移动/可拷贝 |
| `std::default_initializable` | 可默认构造 |

### 7.5 实战：Concepts 版本的重载

```cpp
// 根据约束重载——编译器选择最受约束的版本
template<typename T>
void print(T val) {
    std::cout << "通用版本: " << val << "\n";
}

template<std::integral T>
void print(T val) {
    std::cout << "整数版本: " << val << "\n";
}

template<std::floating_point T>
void print(T val) {
    std::cout << "浮点版本: " << std::fixed << val << "\n";
}

print(42);    // "整数版本"
print(3.14);  // "浮点版本"
print("hi");  // "通用版本"
```

### 7.6 Concepts vs SFINAE 对比

| 维度 | SFINAE | Concepts |
|------|--------|---------|
| 可读性 | `enable_if_t<is_integral_v<T>>` | `std::integral T` |
| 错误信息 | 几十行模板替换失败 | "约束 `integral` 不满足" |
| 组合性 | `enable_if` 套 `enable_if` | `concept A = B && C` |
| 短路求值 | 手动控制 | `&&` / `||` 自动短路 |
| 约束排序 | 手动设优先级 | 编译器自动选择"更受约束"的版本 |

```cpp
// SFINAE 版本：可读性差
template<typename T,
         typename = std::enable_if_t<
             std::is_integral_v<T> && sizeof(T) >= 4>>
void heavy_compute(T val);

// Concepts 版本：清晰明了
template<typename T>
concept LargeIntegral = std::integral<T> && (sizeof(T) >= 4);

template<LargeIntegral T>
void heavy_compute(T val);
```

> **面试速答**
>
> Concepts 是 C++20 引入的给模板参数加命名约束的机制，本质上是把编译期的 bool 表达式提升成了一等语言特性。它解决的核心问题很直白：替代 SFINAE 那一套天书写法，让模板约束既可读、出错时的报错也好懂——这是 SFINAE 时代大家最痛的点。
>
> 底层机制上，编译器在重载决议的时候会检查约束，自动选择约束最严格的那个候选。
>
> 优点是可读性极好、错误信息友好，代价是需要 C++20，学习曲线也还好。
>
> 有个细节要特别注意：约束不满足时依然是编译错误，不是运行时错误——Concepts 不会把错误推迟到运行时。另外组合多个 Concepts 时要小心逻辑关系（`&&`、`||`、`requires` 嵌套等），写错了可能匹配到意想不到的版本。
>
> 横向类比，Concepts 和 Rust 的 trait bound 很像，但有个差异：Concepts 不提供默认实现，trait 可以。选型建议：C++20 项目首选 Concepts 约束模板；卡在老标准的遗留项目还是老老实实用 SFINAE。

## 8. 模板元编程基础

### 8.1 概念与本质

模板元编程（TMP）把计算从运行时搬到编译期。编译器充当"解释器"，模板实例化过程就是"执行"过程。

### 8.2 编译期计算

```cpp
// 经典：编译期阶乘
template<int N>
struct Factorial {
    static constexpr int value = N * Factorial<N - 1>::value;
};

template<>
struct Factorial<0> {
    static constexpr int value = 1;
};

static_assert(Factorial<5>::value == 120);

// 现代替代：constexpr 函数（C++14 以上更推荐）
constexpr int factorial(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i) result *= i;
    return result;
}

static_assert(factorial(5) == 120);
// 注意：constexpr 函数比模板元编程可读性好得多，优先使用
```

### 8.3 标签分发（Tag Dispatch）

```cpp
// 根据迭代器类型选择不同的 advance 实现
namespace detail {
    template<typename It>
    void advance_impl(It& it, int n, std::random_access_iterator_tag) {
        it += n;  // O(1) 随机访问
    }

    template<typename It>
    void advance_impl(It& it, int n, std::input_iterator_tag) {
        while (n-- > 0) ++it;  // O(n) 逐步前进
    }
}

template<typename It>
void my_advance(It& it, int n) {
    detail::advance_impl(it, n,
        typename std::iterator_traits<It>::iterator_category{});
}

// 编译器根据标签类型自动选择最优实现
std::vector<int> v{1, 2, 3, 4, 5};
auto it = v.begin();
my_advance(it, 3);  // 选择随机访问版本，O(1)
```

### 8.4 CRTP（Curiously Recurring Template Pattern）

```cpp
// CRTP：基类以派生类为模板参数
// 用途一：静态多态（替代虚函数，零开销）
template<typename Derived>
struct Shape {
    void draw() {
        // 编译期转为派生类调用，无虚表开销
        static_cast<Derived*>(this)->draw_impl();
    }

    double area() {
        return static_cast<Derived*>(this)->area_impl();
    }
};

struct Circle : Shape<Circle> {
    double radius;
    Circle(double r) : radius(r) {}

    void draw_impl() { std::cout << "画圆\n"; }
    double area_impl() { return 3.14159 * radius * radius; }
};

struct Rectangle : Shape<Rectangle> {
    double w, h;
    Rectangle(double w, double h) : w(w), h(h) {}

    void draw_impl() { std::cout << "画矩形\n"; }
    double area_impl() { return w * h; }
};

// 用途二：自动生成运算符
template<typename Derived>
struct Comparable {
    // 只要派生类实现 operator<，自动生成其余比较运算符
    friend bool operator>(const Derived& a, const Derived& b) { return b < a; }
    friend bool operator<=(const Derived& a, const Derived& b) { return !(b < a); }
    friend bool operator>=(const Derived& a, const Derived& b) { return !(a < b); }
};

struct Score : Comparable<Score> {
    int val;
    friend bool operator<(const Score& a, const Score& b) {
        return a.val < b.val;
    }
};
```

### 8.5 类型列表（TypeList）

```cpp
// 编译期类型列表
template<typename... Ts>
struct TypeList {};

// 获取第一个类型
template<typename List>
struct Front;

template<typename Head, typename... Tail>
struct Front<TypeList<Head, Tail...>> {
    using type = Head;
};

// 获取列表大小
template<typename List>
struct Size;

template<typename... Ts>
struct Size<TypeList<Ts...>> {
    static constexpr int value = sizeof...(Ts);
};

using MyTypes = TypeList<int, double, std::string>;
static_assert(std::is_same_v<Front<MyTypes>::type, int>);
static_assert(Size<MyTypes>::value == 3);
```

### 8.6 对比表

| 技术 | 用途 | 运行时开销 | 可读性 |
|------|------|-----------|--------|
| 模板递归计算 | 编译期数值计算 | 零 | 差（优先用 constexpr） |
| Tag Dispatch | 基于类型特征选实现 | 零 | 中等 |
| CRTP | 静态多态 / 混入功能 | 零（无虚表） | 中等 |
| TypeList | 编译期类型操作 | 零 | 差 |

> **面试速答**
>
> 模板元编程（TMP）说白了就是利用模板实例化这个机制，在编译期执行计算和类型操作。为什么要搞这个？主要是为了拿到零运行时开销的多态、编译期计算和类型级编程这几个能力。
>
> 底层机制有点反直觉：编译器递归地实例化模板，本质上就是把模板系统当成了一门图灵完备的函数式语言在用——递归实例化就是编译期的"循环"和"递归"。
>
> 优点是零运行时开销，缺点一箩筐：编译慢、可读性差、出错时错误信息能把人看懵。
>
> 有几点要特别注意：递归实例化是有深度限制的，一般编译器默认 256 到 1024 层，太深就炸。另外现代 C++ 里做编译期数值计算应该优先用 `constexpr` 函数——可读性和普通函数差不多，比递归模板强太多。
>
> 横向对比一下几种"编译期"的手段：CRTP 擅长做静态多态（不用虚表的多态），虚函数是动态多态，`constexpr` 是更可读的编译期计算。选型建议也很清晰：数值计算用 `constexpr`，静态多态用 CRTP，只有需要纯"类型操作"的场景（比如 TypeList、类型级谓词）才退回到模板元编程。

## 9. 陷阱与实战

### 9.1 模板定义放 .cpp 链接报错

```cpp
// ❌ 错误做法
// utils.h
template<typename T> T clamp(T val, T lo, T hi);

// utils.cpp
template<typename T>
T clamp(T val, T lo, T hi) { return std::max(lo, std::min(val, hi)); }

// main.cpp
int x = clamp(10, 0, 5);  // 链接错误：undefined reference

// ✅ 方案一：头文件中定义（最常用）
// utils.h
template<typename T>
T clamp(T val, T lo, T hi) { return std::max(lo, std::min(val, hi)); }

// ✅ 方案二：显式实例化（适合控制编译时间）
// utils.cpp 末尾
template int clamp<int>(int, int, int);
template double clamp<double>(double, double, double);
```

### 9.2 依赖名称的 typename 和 template

```cpp
// ❌ 编译错误：编译器不知道 T::iterator 是类型还是静态成员
template<typename T>
void foo(T& container) {
    T::iterator it = container.begin();  // 错误！
}

// ✅ 加 typename 告诉编译器这是类型
template<typename T>
void foo(T& container) {
    typename T::iterator it = container.begin();
}

// ❌ 调用依赖于模板参数的模板成员函数
template<typename T>
void bar(T& obj) {
    obj.get<int>();  // 编译器把 < 解析为小于号！
}

// ✅ 加 template 关键字
template<typename T>
void bar(T& obj) {
    obj.template get<int>();
}
```

### 9.3 模板代码膨胀

```cpp
// ❌ 每种类型都生成完整实现，二进制膨胀
template<typename T>
class BigContainer {
    // 100 个成员函数...
    void sort() { /* 大量代码 */ }
    void search() { /* 大量代码 */ }
    // ...
};

// ✅ 共性提取到非模板基类
class ContainerBase {
protected:
    void sort_impl(void* data, size_t size, size_t elem_size,
                   bool(*cmp)(const void*, const void*));
};

template<typename T>
class SmartContainer : private ContainerBase {
public:
    void sort() {
        sort_impl(data_, size_, sizeof(T),
                  [](const void* a, const void* b) {
                      return *static_cast<const T*>(a) < *static_cast<const T*>(b);
                  });
    }
};
```

### 9.4 SFINAE 错误信息天书

```cpp
// ❌ SFINAE 失败时错误信息不友好
template<typename T, typename = std::enable_if_t<std::is_integral_v<T>>>
void process(T val) { /* ... */ }

process(3.14);  // 错误信息：一堆模板替换失败的天书

// ✅ 加 static_assert 给出友好提示
template<typename T>
void process(T val) {
    static_assert(std::is_integral_v<T>,
                  "process() 只接受整数类型，你传入了非整数类型");
    // ...
}

// ✅ 最佳方案：C++20 用 Concepts
template<std::integral T>
void process(T val) { /* ... */ }
// 错误信息："约束 'integral' 不满足于 'double'"
```

### 9.5 模板与 auto 的隐式转换陷阱

```cpp
// ❌ 模板不做隐式转换
template<typename T>
T my_max(T a, T b) { return a > b ? a : b; }

my_max(1, 2.5);     // 错误：T 同时推导为 int 和 double

// 但普通函数可以隐式转换
double my_max(double a, double b) { return a > b ? a : b; }
my_max(1, 2.5);     // OK：int 隐式转换为 double
```

## 10. 综合自测问题（15题）

1. **函数模板的实例化发生在编译期还是运行期？每种类型生成几份代码？**
   > 编译期。每种不同的类型组合生成一份独立的函数代码。

2. **`T&&` 在模板中什么时候是万能引用？什么时候是右值引用？**
   > 当 T 是待推导的模板参数时是万能引用（转发引用）；当 T 已确定时（如 `vector<int>&&`）是右值引用。

3. **类模板的成员函数是在什么时候实例化的？这有什么好处？**
   > 被调用时才实例化（延迟实例化）。好处：未调用的函数即使有类型不匹配也不会报错，如 `vector<unique_ptr>` 不调用拷贝相关函数就能正常使用。

4. **为什么模板通常必须在头文件中定义？有什么替代方案？**
   > 编译器实例化模板时需要看到完整定义。替代方案：显式实例化（`template class Stack<int>;`）。

5. **函数模板支持偏特化吗？如何实现类似效果？**
   > 不支持。用函数重载代替偏特化。

6. **全特化的函数模板和普通函数重载有什么区别？优先级谁高？**
   > 非模板函数优先级最高。全特化不参与重载决议，可能被重载"吞掉"。建议函数模板用重载不用特化。

7. **解释折叠表达式 `(args + ...)` 的展开规则。如果 args 为空会怎样？**
   > 一元右折叠，展开为 `a1 + (a2 + (a3 + a4))`。空包对 `+` 是未定义行为；`&&` 空包为 true，`||` 空包为 false，`,` 空包为 void()。

8. **SFINAE 的"直接上下文"指什么？为什么函数体内的错误不是 SFINAE？**
   > 直接上下文指模板参数替换后的函数签名（返回类型、参数类型、模板参数）。函数体内的错误发生在替换成功之后，此时已选定该候选，不再是"替换失败"。

9. **`std::void_t` 只是 `using void_t = void`，为什么这么有用？**
   > 它利用 SFINAE：如果 `void_t<expr>` 中 expr 无效则整个偏特化失败，配合偏特化实现"检测表达式是否合法"。

10. **`std::decay` 做了什么？为什么 `make_pair("hello", 42)` 的类型是 `pair<const char*, int>` 而不是 `pair<const char[6], int>`？**
    > decay 去引用 + 数组退化为指针 + 函数退化为函数指针 + 去顶层 cv。字符串字面量类型 `const char[6]` 经 decay 变为 `const char*`。

11. **Concepts 相比 SFINAE 的核心优势是什么？**
    > 可读性好（命名约束）、错误信息清晰、自动约束排序、组合简洁（`&&`/`||`）。

12. **CRTP 的原理是什么？它解决什么问题？**
    > 基类以派生类为模板参数（`class Derived : Base<Derived>`），通过 `static_cast<Derived*>(this)` 在编译期调用派生类方法。解决：零开销静态多态，替代虚函数。

13. **模板代码膨胀的根本原因是什么？如何缓解？**
    > 每种类型实例化一份完整代码。缓解：将类型无关的逻辑提取到非模板基类；使用显式实例化控制实例化点。

14. **`typename T::iterator it;` 中的 `typename` 为什么不能省略？**
    > Two-phase Lookup 中，编译器不知道 `T::iterator` 是类型还是静态成员变量。`typename` 告诉编译器这是类型名。

15. **`if constexpr` 和普通 `if` 在模板中的区别是什么？**
    > `if constexpr` 在编译期求值，不满足的分支直接丢弃（不实例化）。普通 `if` 两个分支都要编译通过，即使条件永远为 false。

## 11. 选型速查表

| 场景 | 推荐技术 | 备注 |
|------|---------|------|
| 泛化一个算法 | 函数模板 | 一份逻辑适配多种类型 |
| 泛化一个数据结构 | 类模板 | `vector<T>`, `map<K,V>` |
| 特殊类型需要定制实现 | 类模板偏特化 / 函数重载 | 函数模板不用全特化 |
| 编译期条件选择（C++11） | SFINAE + enable_if | 天书但能用 |
| 编译期条件选择（C++17） | if constexpr | 函数体内分支首选 |
| 模板参数约束（C++20） | Concepts | 最佳方案 |
| 编译期数值计算 | constexpr 函数 | 比模板递归可读 |
| 静态多态（零开销） | CRTP | 替代虚函数 |
| 类型特征查询 | type_traits | 基础设施 |
| 接受任意数量参数 | 可变参数模板 + 折叠表达式 | C++17 折叠表达式最简洁 |
| 自定义类型做 hash key | 全特化 std::hash | unordered_map 必备 |
| 控制模板编译时间 | 显式实例化 | 定义在 .cpp + 显式实例化 |

### 技术演进路线图

```
模板约束：   SFINAE (C++98) → enable_if (C++11) → if constexpr (C++17) → Concepts (C++20)
编译期计算： 模板递归 (C++98) → constexpr 函数 (C++11/14) → consteval (C++20)
参数推导：   函数模板推导 (C++98) → auto (C++11) → CTAD (C++17)
可变参数：   va_args (C) → 可变参数模板 (C++11) → 折叠表达式 (C++17)
```

> **总选型建议**
> - **新项目用 C++20**：Concepts + if constexpr + constexpr 函数，告别 SFINAE 天书
> - **C++17 项目**：if constexpr + 折叠表达式 + CTAD，大幅简化模板代码
> - **C++11/14 遗留项目**：SFINAE + enable_if + type_traits，搭配 static_assert 改善错误信息
> - **性能关键路径**：CRTP 静态多态替代虚函数；模板而非继承实现策略模式
