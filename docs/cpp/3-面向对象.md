# C++ 面向对象深度笔记

> 面向对象程序设计的三大支柱：封装、继承、多态。本笔记重点下钻多态的底层实现（vtable/vptr/RTTI），补全深拷贝浅拷贝的内存原理，以及菱形继承的内存布局。

---

## 0. 全景地图

### C++ OOP 三大特性关系图

```
                    面向对象
           ┌──────────┼──────────┐
           │          │          │
         封装        继承       多态
      (数据隐藏)  (代码复用)  (接口统一)
           │          │          │
     访问控制     派生体系    ┌───┴───┐
  public/protected  单继承   静态多态  动态多态
  /private/friend   多继承  (编译期)  (运行期)
                    虚继承    │         │
                          重载/模板  虚函数/vtable
```

### 编译期多态 vs 运行期多态

| 维度 | 编译期多态（静态） | 运行期多态（动态） |
|------|-------------------|-------------------|
| 绑定时机 | 编译期确定调用地址 | 运行期通过 vtable 查找 |
| 实现手段 | 函数重载、模板、运算符重载 | 虚函数 + 继承 |
| 性能 | 零开销（内联友好） | 有间接跳转开销（阻碍内联） |
| 灵活性 | 类型必须编译期已知 | 可处理运行期才确定的类型 |
| 典型场景 | STL 算法、CRTP | 插件系统、工厂模式、回调 |

**核心认知**：C++ 同时拥有两种多态，选型原则是——能用编译期多态就不用运行期多态（性能更好），需要运行时动态分派时才用虚函数。

## 1. 封装

**一句话定义**：把数据和操作数据的方法绑定在一起，通过访问控制隐藏实现细节。

### 1.1 访问控制（public / protected / private）

| 访问权限 | 类内部 | 派生类 | 类外部 |
|---------|--------|--------|--------|
| public | 可以 | 可以 | 可以 |
| protected | 可以 | 可以 | 不可以 |
| private | 可以 | 不可以 | 不可以 |

**底层原理**：访问控制是纯编译期检查，不会生成任何运行时代码。编译器在语义分析阶段校验访问权限，通过后生成的机器码里没有任何权限信息。所以通过指针算术或 `reinterpret_cast` 可以绕过访问控制（但这是未定义行为）。

### 1.2 struct vs class 区别

| 区别 | struct | class |
|------|--------|-------|
| 默认访问权限 | public | private |
| 默认继承方式 | public 继承 | private 继承 |
| 其他 | 完全相同，都支持虚函数、继承、模板等 | 完全相同 |

**实战选型**：POD 数据聚合用 `struct`，有行为封装的用 `class`。这只是风格约定，编译器不关心。

### 1.3 友元（friend）的使用与争议

友元允许外部函数或类访问本类的 private/protected 成员，是一种**受控的封装破坏**。

```cpp
class Matrix {
    friend Matrix operator*(const Matrix& a, const Matrix& b); // 友元函数
    friend class MatrixSerializer; // 友元类
private:
    double data[4][4];
};

// 友元函数可以访问 Matrix 的 private 成员
Matrix operator*(const Matrix& a, const Matrix& b) {
    Matrix result;
    // 直接访问 a.data 和 b.data 进行矩阵乘法
    return result;
}
```

**友元的三个特性**：
1. **单向性**：A 是 B 的友元，不代表 B 是 A 的友元
2. **不可继承**：A 是 B 的友元，A 的派生类不是 B 的友元
3. **不可传递**：A 是 B 的友元，B 是 C 的友元，A 不是 C 的友元

**争议与实战**：
- 合理场景：运算符重载（`operator<<`、`operator*`）、序列化、测试框架
- 滥用信号：如果一个类有大量友元，说明接口设计有问题，应该重新审视 public API

#### 1. 优缺点与权衡

**优点**：
- 降低耦合度：调用方只依赖 public 接口，内部实现（成员变量布局、私有方法）可以随意修改而不影响外部代码，编译防火墙效果好
- 不变量维护：通过 private 隐藏数据 + public 方法做校验，保证对象始终处于合法状态（如 `vector::push_back` 内部维护 size/capacity 的一致性），比暴露裸数据安全得多
- 零运行时开销：访问控制完全是编译期检查，生成的机器码和全 public 完全一样，没有任何性能代价

**缺点/代价**：
- 接口设计成本：需要仔细划分 public/protected/private 边界，设计不好会导致频繁修改接口或滥用友元来打洞
- 调试和测试不便：private 成员无法从外部直接访问，单元测试需要通过 public 接口间接测试内部逻辑，或借助友元/`#define private public` 等 hack 手段
- 友元破坏封装的风险：友元是一种受控的封装穿透，一旦滥用（大量友元类/友元函数），封装形同虚设，且友元关系不可继承、不可传递，维护成本高

**权衡**：当类需要维护内部不变量（如资源管理、线程安全约束）时，封装带来的安全收益远大于接口设计的额外成本；对于纯数据聚合（POD）用 struct 全 public 即可，不需要过度封装

> **面试速答**
> - **是什么/本质**：通过 public/protected/private 访问控制隐藏实现细节，是编译期的约束检查，不生成任何运行时代码
> - **解决什么问题**：防止外部代码直接依赖内部实现，修改内部实现时不影响调用方，降低耦合
> - **底层原理**：编译器在符号解析阶段检查访问权限，运行时内存布局无任何差异，理论上通过指针算术可绕过（UB）；struct 和 class 唯一区别是默认访问权限不同
> - **坑点**：友元是受控的封装破坏，具有单向性、不可继承、不可传递三个特性；大量友元说明接口设计有问题
> - **横向对比**：vs Java 的 package-private——C++ 没有包级别访问控制，友元是更精确但更侵入的替代方案
> - **选型建议**：数据类/POD 用 struct（默认 public 更方便），有不变量需要维护的类用 class；友元仅限运算符重载和序列化等必要场景

## 2. 继承

**一句话定义**：派生类复用基类的数据和行为，并可以扩展或修改。

### 2.1 三种继承方式（对比表）

| 基类成员 | public 继承 | protected 继承 | private 继承 |
|---------|-------------|---------------|-------------|
| public | public | protected | private |
| protected | protected | protected | private |
| private | 不可访问 | 不可访问 | 不可访问 |

**选型**：
- `public` 继承 = "is-a" 关系（95% 的场景）
- `private` 继承 = "implemented-in-terms-of"（用组合替代更好）
- `protected` 继承 = 几乎不用

### 2.2 构造 / 析构顺序

```
构造顺序（由底向上建造）：
  1. 虚基类（如果有，按声明顺序）
  2. 直接基类（按继承列表中的声明顺序，不是初始化列表顺序）
  3. 成员变量（按类内声明顺序，不是初始化列表顺序）
  4. 派生类构造函数体

析构顺序：与构造完全相反
```

```cpp
class Base {
public:
    Base()  { std::cout << "Base()\n"; }
    ~Base() { std::cout << "~Base()\n"; }
};

class Member {
public:
    Member()  { std::cout << "Member()\n"; }
    ~Member() { std::cout << "~Member()\n"; }
};

class Derived : public Base {
    Member m;
public:
    Derived()  { std::cout << "Derived()\n"; }
    ~Derived() { std::cout << "~Derived()\n"; }
};

// 输出：Base() -> Member() -> Derived()
// 析构：~Derived() -> ~Member() -> ~Base()
```

**陷阱**：初始化列表的书写顺序不影响初始化顺序，实际顺序由声明顺序决定。

### 2.3 菱形继承问题 + 虚继承解决方案

#### 问题：普通菱形继承的内存布局

```
          Animal
         /      \
      Dog        Cat
         \      /
          Pet
```

不使用虚继承时，Pet 内部有**两份** Animal 的数据：

```
Pet 对象的内存布局（无虚继承）：
┌──────────────────────────┐
│  Dog 子对象               │
│  ┌────────────────────┐  │
│  │ Animal 副本 1       │  │  ← Dog 继承的 Animal
│  │   name, age ...    │  │
│  ├────────────────────┤  │
│  │ Dog 自身成员        │  │
│  └────────────────────┘  │
├──────────────────────────┤
│  Cat 子对象               │
│  ┌────────────────────┐  │
│  │ Animal 副本 2       │  │  ← Cat 继承的 Animal（冗余！）
│  │   name, age ...    │  │
│  ├────────────────────┤  │
│  │ Cat 自身成员        │  │
│  └────────────────────┘  │
├──────────────────────────┤
│  Pet 自身成员             │
└──────────────────────────┘
```

问题：`pet.name` 有二义性，必须写 `pet.Dog::name` 或 `pet.Cat::name`。

#### 解决方案：虚继承

```cpp
class Animal {
public:
    std::string name;
    int age;
};

class Dog : virtual public Animal {   // 虚继承
    int barkVolume;
};

class Cat : virtual public Animal {   // 虚继承
    int climbHeight;
};

class Pet : public Dog, public Cat {  // 只有一份 Animal
    int petId;
};
```

#### 虚继承的内存布局（64 位系统）

```
Pet 对象的内存布局（虚继承）：
┌──────────────────────────────┐  偏移 0
│  Dog 子对象                   │
│  ┌────────────────────────┐  │
│  │ vbptr (虚基类指针)      │  │  → 指向 Dog 的虚基类表
│  ├────────────────────────┤  │
│  │ barkVolume             │  │
│  └────────────────────────┘  │
├──────────────────────────────┤
│  Cat 子对象                   │
│  ┌────────────────────────┐  │
│  │ vbptr (虚基类指针)      │  │  → 指向 Cat 的虚基类表
│  ├────────────────────────┤  │
│  │ climbHeight            │  │
│  └────────────────────────┘  │
├──────────────────────────────┤
│  Pet 自身成员                 │
│  │ petId                  │  │
├──────────────────────────────┤
│  Animal 子对象（共享，仅一份） │  ← 放在对象末尾
│  │ name                   │  │
│  │ age                    │  │
└──────────────────────────────┘

Dog 的虚基类表 (vbtable):
┌──────────────┐
│ 0            │  ← vbptr 到 Dog 子对象起始的偏移
│ offset_to_   │  ← vbptr 到 Animal 子对象的偏移（运行时计算）
│ Animal       │
└──────────────┘
```

### 2.4 虚基类指针和虚基类表的实现

**原理**：每个虚继承的子类维护一个 vbptr（虚基类指针），指向一张虚基类表（vbtable）。表中存储的是**从 vbptr 位置到虚基类子对象的偏移量**。访问虚基类成员时，编译器生成类似如下的代码：

```cpp
// pet.name 被编译器翻译为：
// 1. 取 Dog 部分的 vbptr
// 2. 从 vbtable 读取偏移
// 3. vbptr地址 + 偏移 = Animal 子对象地址
// 4. 访问 Animal::name

// 伪代码：
char* vbptr = (char*)&pet + offsetof(Dog, vbptr);
int offset = *(int*)(*(void**)vbptr + 1);  // 表中第二项
Animal* base = (Animal*)(vbptr + offset);
base->name;  // 最终访问
```

**代价**：虚继承相比普通继承多了一次间接寻址，且虚基类子对象通常放在对象末尾，导致无法简单地用 `static_cast` 做指针调整。

#### 2. 优缺点与权衡

**继承的优缺点：**

**优点**：
- 代码复用与类型抽象：派生类自动继承基类的数据和行为，配合虚函数实现里氏替换原则（LSP），调用方可以用基类指针/引用操作所有派生类
- 编译器自动处理构造/析构链：按声明顺序逐层构造、逆序析构，不需要手动管理基类子对象的生命周期
- 接口契约强制力：纯虚函数强制派生类实现特定接口，编译期就能发现遗漏

**缺点/代价**：
- 耦合度高：派生类与基类形成编译期强耦合，基类修改（增删成员、改变虚函数签名）会导致所有派生类重新编译，深层继承链改动影响面大（脆弱基类问题）
- 认知成本：多层继承 + 多继承下，需要理解构造顺序、name lookup 规则、隐藏关系等，代码可读性急剧下降
- 组合优于继承：private 继承表示 "implemented-in-terms-of"，但几乎所有场景用组合（成员变量）都更清晰、耦合度更低

**虚继承的额外代价：**

**优点**：
- 解决菱形继承的二义性问题，虚基类子对象只保留一份，避免数据冗余和 `pet.name` 二义性

**缺点/代价**：
- 每个虚继承的子类多一个 vbptr（8 字节/64位），访问虚基类成员需要额外一次间接寻址（vbptr → vbtable → 偏移计算），性能略差
- 虚基类子对象放在对象末尾，`static_cast` 无法做正确的指针调整，必须用 `dynamic_cast`
- 构造复杂度增加：虚基类的构造由最终派生类负责（而非直接派生类），容易出错

**权衡**：继承在需要运行时多态和类型层次时不可替代，但应控制继承深度（建议不超过 3 层）；虚继承仅在确实需要菱形继承时使用，能用组合解决的不要用多继承

> **面试速答**
> - **是什么/本质**：通过 is-a 关系实现类型层次，public 继承表示接口契约，private 继承表示实现复用
> - **解决什么问题**：代码复用和类型抽象，让派生类可以被当作基类使用（里氏替换原则）
> - **底层原理**：构造顺序是虚基类→直接基类→成员变量→自身构造体，析构完全相反；菱形继承通过虚继承的 vbptr + vbtable 让虚基类子对象只保留一份（放在对象末尾）
> - **优缺点**：优点是类型多态和代码复用；缺点是深层继承链增加理解成本，虚继承额外多一个 vbptr 和一次间接寻址
> - **坑点**：初始化列表书写顺序不影响实际构造顺序（按声明顺序）；95% 场景只用 public 继承，private 继承通常组合更好
> - **横向对比**：vs 组合——优先组合而非继承，除非确实需要多态；虚继承 vs 普通继承——虚继承解决菱形问题但代价是额外指针和间接寻址
> - **选型建议**：需要多态用 public 继承，需要复用实现优先组合，只有 EBO 等特殊场景才考虑 private 继承

## 3. 多态（重点）

### 3.1 静态多态

#### 函数重载（name mangling 原理）

**一句话定义**：同一作用域内，函数名相同但参数列表不同的多个函数。

编译器通过 **name mangling（名字修饰）** 让重载函数在目标文件中有不同的符号名：

```cpp
void foo(int a);          // 编译后符号：_Z3fooi
void foo(double a);       // 编译后符号：_Z3food
void foo(int a, int b);   // 编译后符号：_Z3fooii
```

GCC 的 mangling 规则（Itanium ABI）：
- `_Z` = 前缀
- `3foo` = 函数名长度 + 函数名
- `i` = int, `d` = double, `Ss` = std::string ...

```bash
# 查看 mangled 名字
nm a.out | grep foo
# 反 mangling
c++filt _Z3fooi  # 输出：foo(int)
```

**关键点**：这就是为什么 C++ 重载只看参数不看返回值——返回值不参与 name mangling（因为调用时不一定使用返回值，无法区分）。

**`extern "C"` 的作用**：禁止 name mangling，使得 C++ 函数能被 C 代码链接。

#### 运算符重载

```cpp
class Vec2 {
    float x, y;
public:
    Vec2(float x, float y) : x(x), y(y) {}

    // 成员函数重载
    Vec2 operator+(const Vec2& rhs) const {
        return Vec2(x + rhs.x, y + rhs.y);
    }

    // 友元函数重载（支持左操作数不是本类型）
    friend Vec2 operator*(float scalar, const Vec2& v) {
        return Vec2(scalar * v.x, scalar * v.y);
    }

    // 流输出必须用友元（左操作数是 ostream）
    friend std::ostream& operator<<(std::ostream& os, const Vec2& v) {
        return os << "(" << v.x << ", " << v.y << ")";
    }
};
```

**不可重载的运算符**：`::`, `.`, `.*`, `?:`, `sizeof`——它们要么是编译期特性，要么重载会破坏基本语法语义。

#### 模板（编译期多态）

```cpp
// 函数模板——编译器为每种类型实例化一份代码
template<typename T>
T max_val(T a, T b) {
    return (a > b) ? a : b;
}

// 编译器生成：
// int max_val<int>(int, int)
// double max_val<double>(double, double)
// 各自独立的函数，零运行时开销
```

模板 vs 虚函数的选型：
- 类型在编译期已知 → 模板（零开销、可内联）
- 类型在运行期才确定（如插件加载、用户输入决定类型） → 虚函数

> **面试速答**
> - **是什么/本质**：在编译期确定调用地址的多态机制，包括函数重载和模板两种形式
> - **解决什么问题**：在不牺牲运行时性能的前提下实现泛型编程和接口统一
> - **底层原理**：函数重载靠 name mangling 区分（如 `_Z3fooi`），返回值不参与 mangling 所以不能仅靠返回值区分重载；模板为每种类型实例化独立代码，编译期展开
> - **优缺点**：优点是零运行时开销、可内联优化；缺点是模板会导致代码膨胀、编译时间增加、错误信息难读
> - **横向对比**：vs 运行期多态（虚函数）——编译期多态零开销但不能运行时切换类型，虚函数灵活但有间接调用开销
> - **选型建议**：类型在编译期可确定就用模板/重载，只有运行时才能确定类型时（如插件加载、用户输入决定类型）才用虚函数

### 3.2 动态多态（深度下钻）

#### 虚函数表（vtable）内存布局

**本质**：每个含虚函数的类在编译期生成一张虚函数表（vtable），存放在只读数据段（`.rodata`）。表中每一项是一个函数指针，指向该类实际应该调用的虚函数实现。

**单继承的 vtable 布局**：

```cpp
class Base {
public:
    virtual void func1() { std::cout << "Base::func1\n"; }
    virtual void func2() { std::cout << "Base::func2\n"; }
    virtual ~Base() {}
    int x;
};

class Derived : public Base {
public:
    void func1() override { std::cout << "Derived::func1\n"; }
    virtual void func3() { std::cout << "Derived::func3\n"; }
    int y;
};
```

```
Base 对象内存布局：                  Base 的 vtable（.rodata 段）：
┌─────────────┐                    ┌───────────────────────┐
│ vptr ────────┼──────────────────→ │ &Base::func1          │ [0]
├─────────────┤                    │ &Base::func2          │ [1]
│ x           │                    │ &Base::~Base          │ [2]
└─────────────┘                    └───────────────────────┘

Derived 对象内存布局：               Derived 的 vtable（.rodata 段）：
┌─────────────┐                    ┌───────────────────────┐
│ vptr ────────┼──────────────────→ │ &Derived::func1       │ [0] ← 覆盖了
├─────────────┤                    │ &Base::func2          │ [1] ← 继承的
│ x (Base)    │                    │ &Derived::~Derived    │ [2] ← 覆盖了
├─────────────┤                    │ &Derived::func3       │ [3] ← 新增的
│ y (Derived) │                    └───────────────────────┘
└─────────────┘
```

#### 虚函数指针（vptr）在对象中的位置

**vptr 位于对象的起始位置**（大多数编译器实现，包括 GCC 和 Clang）。这样做的好处是：通过基类指针调用虚函数时，不需要做偏移计算就能找到 vptr。

```cpp
Derived d;
// &d 的值 == d 的 vptr 的地址

// 验证 vptr 在对象头部
std::cout << sizeof(void*) << std::endl; // 8（64位系统）
void** vptr_addr = *(void***)&d;         // 取出 vptr 的值（vtable 地址）
```

**关键事实**：
- vptr 占 8 字节（64 位系统）或 4 字节（32 位系统）
- vptr 在构造函数中被设置——基类构造函数设置为基类 vtable，派生类构造函数再改写为派生类 vtable
- 这就是为什么**构造函数中调用虚函数不会表现出多态**：此时 vptr 还指向当前正在构造的类的 vtable

#### 实战：手动通过函数指针访问虚函数表

下面的代码演示了如何在运行时手动取出 vptr、从虚函数表中拿到函数地址并调用，帮助理解 vtable 的内存结构：

```cpp
#include <iostream>
using namespace std;

class A {
public:
    virtual void vfunc1() { cout << "A::vfunc1()" << endl; }
    virtual void vfunc2() { cout << "A::vfunc2()" << endl; }
private:
    int data1_, data2_;
};

class B : public A {
public:
    virtual void vfunc1() override { cout << "B::vfunc1()" << endl; }
private:
    int data3_;
};

int main() {
    B b;
    // 定义函数指针类型，用于接收从虚函数表中取出的地址
    typedef void(*Fun)(void);
    Fun pFun = nullptr;

    // 对象首地址就是 vptr 的地址，解引用得到 vtable 的起始地址
    cout << "虚函数表地址：" << (long long*)*(long long*)(&b) << endl;

    // 从 vtable[0] 取出第一个虚函数地址并调用（B 覆盖了 vfunc1）
    pFun = (Fun)*((long long*)*(long long*)(&b));
    pFun();  // 输出：B::vfunc1()

    // 从 vtable[1] 取出第二个虚函数地址并调用（B 继承了 A::vfunc2）
    pFun = (Fun)*((long long*)*(long long*)(&b) + 1);
    pFun();  // 输出：A::vfunc2()

    return 0;
}
```

> **注意**：上面使用 `long long*`（8 字节）适用于 64 位系统。32 位系统应改用 `int*`（4 字节）。这段代码依赖编译器实现细节，仅用于学习理解，生产代码中不应手动操作 vtable。

#### 3. 优缺点与权衡

**优点**：
- 开闭原则的基石：新增派生类不需要修改已有的调用方代码，只需继承并 override 虚函数，扩展性极强（插件系统、工厂模式的核心机制）
- 运行时类型切换：可以根据运行时条件（用户输入、配置文件、动态加载的 .so）选择具体实现，这是编译期多态做不到的
- 接口与实现分离：调用方只持有基类指针/引用，完全解耦具体实现类

**缺点/代价**：
- 每对象额外 8 字节 vptr 开销：对于大量小对象（如粒子系统中百万个粒子），vptr 的内存占比可能很高
- 间接调用无法内联：虚函数调用需要两次内存间接访问（取 vptr → 取 vtable 槽位），编译器无法在编译期确定调用目标，因此无法内联优化，也不利于 CPU 分支预测
- cache miss 风险：vptr 指向的 vtable 在 .rodata 段，如果多态调用分散在不同类型的对象上，vtable 可能不在 L1 cache 中，造成额外的 cache miss
- 多继承下复杂度爆炸：多张 vtable + thunk 调整 this 指针，内存布局和调试难度显著增加

**权衡**：在需要运行时动态分派的场景（插件加载、回调注册、策略模式），虚函数的灵活性收益远大于性能代价；但在性能敏感的热路径（游戏引擎 tick、HFT 交易循环），应考虑用 CRTP 静态多态或 `std::variant` + `std::visit` 替代虚函数

#### 虚函数默认参数陷阱

虚函数的**默认参数是静态绑定**的（编译期确定），而虚函数调用本身是**动态绑定**的（运行期确定）。两者的绑定时机不一致会产生反直觉的结果：

```cpp
#include <iostream>
using namespace std;

class Base {
public:
    virtual void fun(int x = 0) {
        cout << "Base::fun(), x = " << x << endl;
    }
};

class Derived : public Base {
public:
    // 派生类重新定义了默认参数 x = 10
    virtual void fun(int x = 10) {
        cout << "Derived::fun(), x = " << x << endl;
    }
};

int main() {
    Derived d;

    // 通过派生类指针调用：默认参数取 Derived 的定义（x=10）
    Derived* dp = &d;
    dp->fun();  // 输出：Derived::fun(), x = 10

    // 通过基类指针调用：函数体走 Derived（动态绑定），但默认参数取 Base 的定义（x=0）
    Base* bp = &d;
    bp->fun();  // 输出：Derived::fun(), x = 0  ← 注意这里！

    return 0;
}
```

> **原因**：默认参数不属于函数签名，不参与虚函数的动态分派。编译器在编译期根据**指针/引用的静态类型**决定使用哪个默认值。因此《Effective C++》条款37明确建议：**绝不重新定义继承而来的缺省参数值**（Never redefine function's inherited default parameters value）。

#### 虚函数调用的完整过程（从 C++ 到汇编）

```cpp
Base* p = new Derived();
p->func1();  // 这一行发生了什么？
```

**C++ 层面**：编译器看到 `p->func1()` 且 `func1` 是虚函数，不会直接 call 固定地址，而是生成间接调用。

**编译器生成的伪代码**：
```cpp
// p->func1() 被翻译为：
void* vptr   = *(void**)p;            // 步骤1：从对象头部取 vptr
void* func   = ((void**)vptr)[0];      // 步骤2：从 vtable 取第 0 项（func1 的槽位）
((void(*)(Base*))func)(p);              // 步骤3：调用函数，传 this 指针
```

**x86-64 汇编**（GCC -O0 典型输出）：
```asm
; p 在 rdi 中
mov  rax, [rdi]          ; rax = *p，即 vptr（vtable 地址）
mov  rax, [rax]          ; rax = vtable[0]，即 func1 的地址
call rax                 ; 间接调用
```

**性能影响**：
- 两次内存间接访问（可能 cache miss）
- 间接 `call` 阻碍 CPU 分支预测和编译器内联
- 这就是高性能场景（游戏引擎、HFT）倾向用 CRTP 等编译期多态替代虚函数的原因

#### 多继承下的多张虚表

```cpp
class A {
public:
    virtual void funcA() {}
    virtual ~A() {}
    int a;
};

class B {
public:
    virtual void funcB() {}
    virtual ~B() {}
    int b;
};

class C : public A, public B {
public:
    void funcA() override {}
    void funcB() override {}
    int c;
};
```

```
C 对象内存布局：                     C 的 vtable（两张表拼在一起）：

┌─────────────┐                    vtable for A 部分：
│ vptr_A ──────┼───────────────────→ ┌──────────────────────┐
├─────────────┤                     │ &C::funcA            │ [0]
│ a (A部分)    │                     │ &C::~C               │ [1]
├─────────────┤                     └──────────────────────┘
│ vptr_B ──────┼───────────────────→ vtable for B 部分：
├─────────────┤                     ┌──────────────────────┐
│ b (B部分)    │                     │ &C::funcB (thunk)    │ [0]
├─────────────┤                     │ &C::~C (thunk)       │ [1]
│ c (C自身)    │                     └──────────────────────┘
└─────────────┘

注意：通过 B* 指针调用虚函数时，需要 thunk 调整 this 指针
```

**thunk 是什么**：当通过第二基类（B）的指针调用虚函数时，`this` 指针指向的是对象中 B 子对象的位置，而不是对象起始。thunk 是编译器生成的一小段代码，负责将 `this` 指针调整回对象起始位置，然后跳转到真正的函数实现。

```asm
; thunk for C::funcB (通过 B* 调用时)
C::funcB_thunk:
    sub  rdi, 16      ; 调整 this 指针（减去 A 子对象大小）
    jmp  C::funcB     ; 跳转到真正的实现
```

> **面试速答**
> - **是什么/本质**：运行期多态的实现机制——通过虚函数表（vtable）+ 虚指针（vptr）实现间接调用，让基类指针能调用派生类的重写函数
> - **解决什么问题**：让代码依赖抽象而非具体实现，不修改调用方代码就能扩展新的派生类行为
> - **底层原理**：每个含虚函数的类有一张 vtable（函数指针数组），对象头部 8 字节存 vptr 指向该表。调用 `p->func()` 编译器生成 `mov rax,[p]; call [rax+offset]`，运行时通过 vptr 找到实际类型的函数地址
> - **优缺点**：优点是开闭原则、可扩展性强；缺点是每对象多 8 字节 vptr、间接调用无法内联、cache miss
> - **坑点**：构造/析构函数中调用虚函数不会多态（vtable 还没完全建好）；基类析构函数必须 virtual 否则 delete 派生类对象 UB；虚函数默认参数是静态绑定的
> - **横向对比**：vs 编译期多态（模板/CRTP）——虚函数灵活但有运行时开销，模板零开销但不能运行时切换类型
> - **选型建议**：需要运行时类型切换用虚函数；性能敏感的热路径考虑 CRTP 静态多态

#### 虚析构函数为什么必须要

```cpp
// ❌ 错误：基类析构函数不是虚函数
class Base {
public:
    ~Base() { std::cout << "~Base\n"; }
};

class Derived : public Base {
    int* data;
public:
    Derived() : data(new int[100]) {}
    ~Derived() { delete[] data; std::cout << "~Derived\n"; }
};

Base* p = new Derived();
delete p;  // 只调用 ~Base()，Derived::data 泄漏！

// ✅ 正确：基类析构函数声明为 virtual
class Base {
public:
    virtual ~Base() { std::cout << "~Base\n"; }
};
// delete p 时：先调 ~Derived()，再调 ~Base()，无泄漏
```

**规则**：只要一个类可能被继承且通过基类指针删除，它的析构函数就必须是 virtual。反过来，如果确定不会被继承（如用 `final` 标记），不需要虚析构（避免 vptr 开销）。

> **面试速答**
> - **是什么/本质**：将析构函数声明为 virtual，确保通过基类指针 delete 时能正确调用派生类析构函数
> - **解决什么问题**：非虚析构时 delete 基类指针只调基类析构，派生类资源泄漏——这是 UB，不仅仅是内存泄漏
> - **底层原理**：delete 先通过 vptr 在 vtable 中找到正确的析构函数地址，逐层调用派生类→基类析构；非虚析构则直接按静态类型调用
> - **坑点**：忘记虚析构是最常见的多态 bug；纯虚析构必须提供函数体（派生类析构会隐式调用基类析构）
> - **选型建议**：只要类可能被继承且通过基类指针删除就必须虚析构；确定不被继承的类标记 `final`，可省掉 vptr 的 8 字节开销

#### 纯虚函数与抽象类

```cpp
class Shape {
public:
    virtual double area() const = 0;   // 纯虚函数
    virtual ~Shape() = default;
};

// Shape s;  // ❌ 编译错误：不能实例化抽象类

class Circle : public Shape {
    double r;
public:
    Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
};
```

**纯虚函数也可以有实现**（鲜为人知）：
```cpp
class Base {
public:
    virtual void func() = 0;  // 仍然是纯虚函数，Base 仍是抽象类
};

// 提供默认实现，派生类可以显式调用
void Base::func() {
    std::cout << "Base 的默认实现\n";
}

class Derived : public Base {
public:
    void func() override {
        Base::func();  // 显式调用基类的实现
        std::cout << "Derived 的扩展\n";
    }
};
```

> **面试速答**
> - **是什么/本质**：纯虚函数用 `= 0` 声明，含纯虚函数的类是抽象类，不能实例化，强制派生类实现接口
> - **解决什么问题**：定义纯接口契约，让调用方只依赖抽象而非具体实现，是策略模式、工厂模式的基础
> - **底层原理**：vtable 中纯虚函数的槽位填 `__cxa_pure_virtual`（调用即 abort），派生类实现后覆盖该槽位
> - **坑点**：纯虚函数可以有函数体提供默认实现，派生类通过 `Base::func()` 显式调用——这点很多人不知道；纯虚析构必须提供定义
> - **横向对比**：vs Java interface——C++ 抽象类可以有成员变量和非虚函数实现，更灵活但也更复杂
> - **选型建议**：需要定义接口层时用纯虚函数 + 虚析构函数组合；如果所有函数都是纯虚的，就是纯接口类

#### RTTI 和 dynamic_cast 的底层实现

**RTTI（Run-Time Type Information）**：编译器为每个含虚函数的类生成 `type_info` 对象，存放在 vtable 的负偏移位置（vtable[-1]）。

```
vtable 的实际完整布局（GCC/Itanium ABI）：
                        ┌──────────────────────┐
vtable[-2]              │ offset_to_top         │  ← 多继承时 this 指针调整量
                        ├──────────────────────┤
vtable[-1]              │ &type_info for Class  │  ← RTTI 信息
                        ├──────────────────────┤
vtable[0] (vptr 指向)→  │ &虚函数1              │
                        │ &虚函数2              │
                        │ ...                   │
                        └──────────────────────┘
```

**`dynamic_cast` 的过程**：
```cpp
Base* p = getObject();  // 可能是 Base 也可能是 Derived
Derived* d = dynamic_cast<Derived*>(p);
```
1. 通过 `p` 的 vptr 找到 vtable
2. 从 vtable[-1] 取出 `type_info*`
3. 沿继承链遍历 `type_info`，检查目标类型是否在继承路径中
4. 如果匹配成功，用 `offset_to_top` 调整指针并返回
5. 匹配失败：指针返回 `nullptr`，引用抛出 `std::bad_cast`

**性能**：`dynamic_cast` 需要遍历继承链，时间复杂度与继承深度成正比。频繁使用说明设计有问题，应该用虚函数分派替代。

**`typeid` 运算符**：
```cpp
if (typeid(*p) == typeid(Derived)) {
    // 精确类型匹配（不考虑继承关系）
    // 比 dynamic_cast 快，但不做指针调整
}
```

**禁用 RTTI**：编译时加 `-fno-rtti` 可以节省每个多态类一个 `type_info` 的空间，但 `dynamic_cast` 和 `typeid` 将不可用。嵌入式和游戏引擎常这么做。

## 4. 拷贝语义

### 4.1 深拷贝 vs 浅拷贝

**一句话定义**：
- **浅拷贝**：按位/按成员复制，指针只复制地址值，两个对象共享同一块堆内存
- **深拷贝**：重新分配内存，复制指针指向的实际内容，两个对象各自独立

#### 完整示例 + 内存图

```cpp
class String {
    char* data_;
    size_t len_;
public:
    // 构造
    String(const char* s) : len_(strlen(s)) {
        data_ = new char[len_ + 1];
        strcpy(data_, s);
    }

    // ❌ 编译器默认生成的拷贝构造 = 浅拷贝
    // String(const String& other) : data_(other.data_), len_(other.len_) {}

    // ✅ 深拷贝构造
    String(const String& other) : len_(other.len_) {
        data_ = new char[len_ + 1];     // 分配新内存
        strcpy(data_, other.data_);      // 复制内容
    }

    // ✅ 深拷贝赋值（copy-and-swap 手法）
    String& operator=(String other) {    // 注意：参数按值传递（触发拷贝构造）
        swap(other);                     // 交换资源
        return *this;                    // other 析构时释放旧资源
    }

    void swap(String& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(len_, other.len_);
    }

    ~String() { delete[] data_; }
};
```

**浅拷贝的内存图**：
```
执行 String b = a; （浅拷贝）后：

栈                          堆
┌──────────┐
│ a.data_ ─┼────────┐
│ a.len_=5 │        │    ┌───────────┐
└──────────┘        ├──→ │ "Hello\0" │  ← 同一块内存！
┌──────────┐        │    └───────────┘
│ b.data_ ─┼────────┘
│ b.len_=5 │
└──────────┘

问题：
1. delete a 后，b.data_ 变成悬空指针
2. a 和 b 的析构函数都会 delete 同一块内存 → double free
3. 修改 a 的数据，b 也会变（意外共享）
```

**深拷贝的内存图**：
```
执行 String b = a; （深拷贝）后：

栈                          堆
┌──────────┐          ┌───────────┐
│ a.data_ ─┼────────→ │ "Hello\0" │  ← a 自己的内存
│ a.len_=5 │          └───────────┘
└──────────┘
┌──────────┐          ┌───────────┐
│ b.data_ ─┼────────→ │ "Hello\0" │  ← b 自己的内存（独立副本）
│ b.len_=5 │          └───────────┘
└──────────┘

各自独立，互不影响
```

#### 4. 优缺点与权衡

**深拷贝：**

**优点**：
- 完全独立：两个对象各自拥有独立的堆内存，修改一个不影响另一个，消除了共享状态导致的一切 bug（double free、悬空指针、数据竞争）
- 线程安全友好：独立副本天然没有数据竞争，不需要额外的同步机制

**缺点/代价**：
- 性能开销大：每次拷贝都要 `new` 分配内存 + `memcpy`/逐元素复制，对于大对象（如几 MB 的 buffer）代价很高，频繁拷贝会成为性能瓶颈
- 内存占用翻倍：每份副本都独立分配，N 份拷贝占 N 倍内存

**浅拷贝：**

**优点**：
- 极快：只复制指针值（8 字节），O(1) 操作，不涉及堆分配
- 内存高效：多个对象共享同一份数据

**缺点/代价**：
- 三大致命问题：double free（两个析构函数释放同一块内存）、悬空指针（一个对象释放后另一个仍持有旧地址）、意外数据共享（修改一个影响另一个）

**权衡**：类持有堆资源时必须实现深拷贝保证正确性；但如果对象频繁传递，优先考虑移动语义（C++11）转移所有权而非深拷贝，或用 COW（Copy-On-Write）/`shared_ptr` 延迟复制

### 4.2 拷贝构造函数

**何时被调用**：
1. 用已有对象初始化新对象：`String b = a;` 或 `String b(a);`
2. 函数参数按值传递：`void foo(String s);`
3. 函数按值返回：`String bar() { String s; return s; }`（可能被 RVO/NRVO 优化掉）

**参数为何必须是 const 引用**：
```cpp
// ✅ 正确
String(const String& other);

// ❌ 如果参数是值传递：
String(String other);  // 传参时需要拷贝 → 调用拷贝构造 → 递归调用自己 → 无限递归！

// ❌ 如果不加 const：
String(String& other);  // 无法接受临时对象（右值），如 String b = getString();
```

### 4.3 拷贝赋值运算符

**自赋值检测**：
```cpp
// 朴素实现（需要自赋值检测）
String& operator=(const String& other) {
    if (this == &other) return *this;  // 自赋值检测
    delete[] data_;                     // 释放旧资源
    len_ = other.len_;
    data_ = new char[len_ + 1];        // 分配新资源
    strcpy(data_, other.data_);
    return *this;
}
```

**copy-and-swap idiom（更优雅、异常安全）**：
```cpp
// 利用拷贝构造 + swap，天然处理自赋值，且提供强异常安全保证
String& operator=(String other) {  // 按值传递：拷贝在参数传递时完成
    swap(other);                   // 交换：noexcept
    return *this;                  // other 析构：释放旧资源
}
// 好处：如果 new 失败（拷贝构造抛异常），原对象不受影响
```

> **面试速答**
> - **是什么/本质**：浅拷贝复制指针值（共享资源），深拷贝复制指针指向的实际内容（独立资源）
> - **解决什么问题**：浅拷贝会导致 double free、悬空指针、意外数据共享三大问题，深拷贝让两个对象完全独立
> - **底层原理**：编译器默认生成的拷贝构造和赋值是逐成员复制（浅拷贝），指针成员只复制地址不复制指向的内容
> - **坑点**：管理堆资源的类必须自己实现深拷贝；拷贝赋值推荐 copy-and-swap 手法——参数按值传递触发拷贝构造，再 swap 交换，天然处理自赋值且强异常安全
> - **横向对比**：vs 移动语义——移动是"资源转移"而非复制，C++11 后应优先支持移动来避免不必要的深拷贝开销
> - **选型建议**：类持有裸指针/文件句柄等资源时必须实现深拷贝（或禁用拷贝只允许移动），用 RAII 容器管理资源则无需手写

### 4.4 Rule of Three / Rule of Five / Rule of Zero

| 规则 | 含义 |
|------|------|
| **Rule of Three** | 如果你定义了析构函数、拷贝构造、拷贝赋值中的任何一个，你应该定义全部三个 |
| **Rule of Five**（C++11） | 在 Three 基础上加上移动构造和移动赋值（共 5 个） |
| **Rule of Zero** | 优先使用 RAII 容器（`string`、`vector`、`unique_ptr`），让编译器默认生成的函数就够用，自己一个都不写 |

```cpp
// Rule of Five 完整示例
class Buffer {
    int* data_;
    size_t size_;
public:
    Buffer(size_t n) : data_(new int[n]), size_(n) {}

    ~Buffer() { delete[] data_; }                       // 1. 析构

    Buffer(const Buffer& o) : data_(new int[o.size_]),  // 2. 拷贝构造
                               size_(o.size_) {
        std::copy(o.data_, o.data_ + size_, data_);
    }

    Buffer& operator=(Buffer o) {                       // 3. 拷贝赋值（copy-and-swap）
        swap(o);
        return *this;
    }

    Buffer(Buffer&& o) noexcept                         // 4. 移动构造
        : data_(o.data_), size_(o.size_) {
        o.data_ = nullptr;
        o.size_ = 0;
    }

    Buffer& operator=(Buffer&& o) noexcept {            // 5. 移动赋值
        if (this != &o) {
            delete[] data_;
            data_ = o.data_;
            size_ = o.size_;
            o.data_ = nullptr;
            o.size_ = 0;
        }
        return *this;
    }

    void swap(Buffer& o) noexcept {
        std::swap(data_, o.data_);
        std::swap(size_, o.size_);
    }
};
```

```cpp
// Rule of Zero：用智能指针管理，一个特殊成员函数都不用写
class Buffer {
    std::unique_ptr<int[]> data_;
    size_t size_;
public:
    Buffer(size_t n) : data_(std::make_unique<int[]>(n)), size_(n) {}
    // 编译器自动生成移动构造/移动赋值
    // 拷贝被 unique_ptr 禁用（如果需要拷贝，用 shared_ptr 或显式定义）
};
```

**选型决策树**：
```
你的类管理资源（裸指针、文件句柄等）吗？
├─ 否 → Rule of Zero（什么都不写，用默认的）
└─ 是 → 能用智能指针/容器封装吗？
     ├─ 能 → Rule of Zero（让容器管理资源）
     └─ 不能 → Rule of Five（5 个全写）
```

#### 5. 优缺点与权衡

**优点**：
- 正确性保证：Rule of Five 确保拷贝、移动、析构三种场景下资源管理都是正确的，不会出现浅拷贝导致的 double free 或资源泄漏
- Rule of Zero 的优雅：用 RAII 容器（`unique_ptr`、`string`、`vector`）封装资源后，编译器默认生成的特殊成员函数就是正确的，零样板代码
- 编译器帮你兜底：一旦遵循规则，编译器隐式生成的移动操作可以自动利用成员的移动语义，性能最优

**缺点/代价**：
- 样板代码多：Rule of Five 要求手写 5 个函数（析构 + 拷贝构造 + 拷贝赋值 + 移动构造 + 移动赋值），代码量大且容易出错，尤其是 copy-and-swap、noexcept 标记等细节
- 隐式规则复杂：C++ 的隐式生成规则很绕——定义了析构函数后移动操作不再隐式生成（退化为拷贝），这个规则不直观，容易踩坑导致性能问题
- 维护负担：类新增成员变量时，5 个函数都可能需要同步修改，遗漏任一个就可能引入 bug

**权衡**：现代 C++ 应优先追求 Rule of Zero——用智能指针和标准容器管理资源，让编译器生成的默认函数自动正确；只有在封装最底层的 RAII wrapper（如自定义内存池、文件句柄封装）时才需要 Rule of Five

> **面试速答**
> - **是什么/本质**：三条规则指导何时需要手写特殊成员函数——Three（析构/拷贝构造/拷贝赋值）、Five（+移动构造/移动赋值）、Zero（一个都不写）
> - **解决什么问题**：自定义其中一个通常意味着类管理资源，不配套实现其他的会导致浅拷贝、资源泄漏等 bug
> - **底层原理**：编译器会隐式生成这些函数（逐成员操作），但一旦用户定义了析构函数，移动操作就不会被隐式生成（C++11 规则）
> - **坑点**：定义了析构忘写移动函数是最常见的性能陷阱——所有右值都会退化为拷贝
> - **横向对比**：Three 是 C++98 时代的规则，Five 是 C++11 扩展，Zero 是现代 C++ 最佳实践
> - **选型建议**：优先 Rule of Zero——用 `unique_ptr`/`string`/`vector` 等 RAII 容器管理资源，让编译器生成的函数就够用；只有封装底层资源的 RAII wrapper 才需要 Rule of Five

## 5. 重载 vs 重写 vs 隐藏

| 对比维度 | 重载（Overload） | 重写（Override） | 隐藏（Hide） |
|---------|-----------------|-----------------|-------------|
| 作用域 | 同一个类 | 基类与派生类之间 | 基类与派生类之间 |
| 函数名 | 相同 | 相同 | 相同 |
| 参数列表 | 必须不同 | 必须相同 | 可同可不同 |
| virtual | 不需要 | 基类必须是 virtual | 基类不是 virtual |
| 绑定方式 | 编译期 | 运行期 | 编译期 |
| 关键字 | 无 | override（C++11推荐） | 无 |

### 容易混淆的场景

```cpp
class Base {
public:
    virtual void foo(int x)    { std::cout << "Base::foo(int)\n"; }
    void         bar(int x)    { std::cout << "Base::bar(int)\n"; }
};

class Derived : public Base {
public:
    // 重写：参数列表完全相同，基类是 virtual
    void foo(int x) override   { std::cout << "Derived::foo(int)\n"; }

    // 隐藏（不是重写！）：参数列表不同
    void foo(double x)         { std::cout << "Derived::foo(double)\n"; }

    // 隐藏（不是重写！）：基类 bar 不是 virtual
    void bar(int x)            { std::cout << "Derived::bar(int)\n"; }
};

Derived d;
Base* p = &d;

p->foo(1);      // Derived::foo(int)  — 运行期多态，调派生类
d.foo(1);       // Derived::foo(double)! — 编译期，int 隐式转 double
                // 因为 Derived::foo(double) 隐藏了 Base::foo(int)
d.Base::foo(1); // Base::foo(int) — 显式指定基类版本

p->bar(1);      // Base::bar(int) — 非虚函数，看指针类型
d.bar(1);       // Derived::bar(int) — 看对象类型
```

**陷阱**：派生类中定义了同名函数（不论参数是否相同），会隐藏基类的所有同名函数。想同时暴露基类版本，用 `using Base::foo;`。

```cpp
class Derived : public Base {
public:
    using Base::foo;            // 把基类的 foo 拉到派生类作用域
    void foo(double x) { ... } // 现在 foo(int) 和 foo(double) 都可见
};
```

## 6. 空类的默认成员函数

编译器在需要时（隐式使用时才生成，称为"隐式声明、隐式定义"）为空类默认生成以下 6 个特殊成员函数：

```cpp
class Empty {
public:
    Empty();                                  // 1. 默认构造函数
    ~Empty();                                 // 2. 析构函数
    Empty(const Empty&);                      // 3. 拷贝构造函数
    Empty& operator=(const Empty&);           // 4. 拷贝赋值运算符
    Empty(Empty&&) noexcept;                  // 5. 移动构造函数（C++11）
    Empty& operator=(Empty&&) noexcept;       // 6. 移动赋值运算符（C++11）
};
```

**关键注意**：
- `sizeof(Empty)` == 1（不是 0，为了保证每个对象有唯一地址）
- 如果你定义了任何构造函数，编译器不再生成默认构造函数
- 如果你定义了拷贝操作，编译器不再隐式生成移动操作
- 如果你定义了析构函数，编译器仍会生成拷贝操作（但 C++11 起这是 deprecated 行为）
- 用 `= default` 显式要求编译器生成，`= delete` 显式禁止

```cpp
class NonCopyable {
public:
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;            // 禁止拷贝
    NonCopyable& operator=(const NonCopyable&) = delete; // 禁止拷贝赋值
    NonCopyable(NonCopyable&&) = default;                // 允许移动
    NonCopyable& operator=(NonCopyable&&) = default;     // 允许移动赋值
};
```

## 7. 运算符重载

### 7.1 概念与本质

运算符重载是让自定义类型支持运算符语法的机制，**本质是特殊的函数调用**：`a + b` 被编译器转换为 `a.operator+(b)` 或 `operator+(a, b)`。

### 7.2 重载规则

**可重载的运算符**：`+` `-` `*` `/` `%` `^` `&` `|` `~` `!` `=` `<` `>` `+=` `-=` `*=` `/=` `%=` `^=` `&=` `|=` `<<` `>>` `>>=` `<<=` `==` `!=` `<=` `>=` `&&` `||` `++` `--` `,` `->*` `->` `()` `[]` `new` `delete`

**不可重载**：`::` `.` `.*` `?:` `sizeof` `typeid`

**必须是成员函数**：`=`、`()`、`[]`、`->`

**建议非成员（友元）**：`+`、`-`、`*`、`==`、`<<` — 保持操作数对称性（左操作数不必是自身类型）

### 7.3 关键运算符实现

#### operator=（赋值运算符）

```cpp
class Buffer {
    int* data_;
    size_t size_;
public:
    // Copy-and-Swap 惯用法：同时解决自赋值和异常安全
    Buffer& operator=(Buffer other) {  // 注意：按值传入（触发拷贝/移动）
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
        return *this;  // other 析构时释放旧资源
    }
};
```

#### operator++（前置 vs 后置）

```cpp
class Counter {
    int val_;
public:
    Counter& operator++() {        // 前置 ++c：返回引用，高效
        ++val_;
        return *this;
    }
    Counter operator++(int) {      // 后置 c++：返回旧值副本，多一次拷贝
        Counter old = *this;
        ++val_;
        return old;
    }
};
// 结论：优先用前置 ++，尤其是迭代器
```

#### operator&lt;&lt;（流输出，必须友元）

```cpp
class Point {
    double x_, y_;
public:
    // 必须是非成员（左操作数是 ostream，不是 Point）
    friend std::ostream& operator<<(std::ostream& os, const Point& p) {
        return os << "(" << p.x_ << ", " << p.y_ << ")";
    }
};
```

#### operator()（仿函数/函数对象）

```cpp
struct Comparator {
    bool reverse;
    bool operator()(int a, int b) const {  // 可携带状态，可被 STL 算法内联
        return reverse ? a > b : a < b;
    }
};
std::sort(v.begin(), v.end(), Comparator{true});
```

#### 类型转换运算符

```cpp
class SmartBool {
    int val_;
public:
    explicit operator bool() const { return val_ != 0; }  // explicit 防止意外转换
};
SmartBool sb{42};
if (sb) { /* OK */ }          // explicit 允许在条件语境中使用
// int x = sb;                // ❌ 编译错误：explicit 禁止隐式转换
```

#### operator&lt;=>（C++20 三路比较）

```cpp
#include <compare>
struct Version {
    int major, minor, patch;
    auto operator<=>(const Version&) const = default;  // 自动生成 <, <=, >, >=, ==, !=
};
```

### 7.4 成员 vs 非成员对比表

| 维度 | 成员函数重载 | 非成员（友元）重载 |
|------|------------|-------------------|
| 左操作数 | 必须是自身类型 | 任意类型 |
| 对称性 | 不对称（`obj + 1` 可以，`1 + obj` 不行） | 对称（两个方向都行） |
| 访问权限 | 直接访问 private | 需要 friend 声明 |
| 必须成员 | `=` `()` `[]` `->` | — |
| 建议非成员 | — | `+` `-` `*` `==` `!=` `<<` `>>` |

### 7.5 优缺点与权衡

**优点**：
- 语法自然直观，`a + b` 比 `a.add(b)` 可读性好
- 配合 STL 算法（如 `sort` 需要 `operator<`、`unordered_map` 需要 `operator==`）

**缺点/代价**：
- 滥用导致代码难懂（如重载 `^` 表示幂运算，违反直觉）
- 运算符优先级和结合性无法改变，可能和自定义语义冲突

**权衡**：只在运算符的语义与内置类型一致时才重载（如数学向量的 `+`）；不确定时用命名函数。

### 7.6 陷阱与实战

**❌ 重载 `&&` `||` 导致短路求值失效**
```cpp
// ❌ 内置 && 有短路：左边 false 就不算右边
// 重载后变成普通函数调用，两边都会求值！
MyBool operator&&(const MyBool& a, const MyBool& b);
```
**✅ 不要重载逻辑运算符 `&&` `||` `,`**

**❌ 隐式类型转换运算符导致意外转换**
```cpp
class Foo {
public:
    operator int() const { return 42; }  // 隐式转换
};
Foo f;
double d = f + 1.5;  // Foo → int → double，可能非预期
```
**✅ 类型转换运算符加 `explicit`**

> **面试速答**
> - **是什么/本质**：让自定义类型支持运算符语法，编译器将 `a+b` 转换为 `a.operator+(b)` 函数调用
> - **解决什么问题**：让自定义类型的使用方式和内置类型一样自然，配合 STL 算法
> - **底层原理**：编译器查找成员函数和非成员函数两个候选集，按重载决议规则选择最佳匹配
> - **坑点**：重载 `&&`/`||` 丧失短路求值；隐式类型转换运算符导致意外转换要加 `explicit`；`operator=` 必须处理自赋值
> - **横向对比**：`=` `()` `[]` `->` 必须是成员；`+` `==` `<<` 建议非成员保持对称性
> - **选型建议**：只在语义和内置类型一致时重载；不确定时用命名函数

## 8. 函数指针与回调

### 8.1 概念与本质

函数指针存储函数在代码段中的地址，是 C/C++ 实现回调和策略模式的基础机制。C++ 扩展出了仿函数、Lambda、`std::function` 四种可调用对象。

### 8.2 四种可调用对象

#### 函数指针

```cpp
int add(int a, int b) { return a + b; }
int (*fp)(int, int) = add;   // 函数名即代码段地址
int result = fp(1, 2);       // 通过指针调用
```

#### 函数对象（仿函数）

```cpp
struct Adder {
    int base;
    int operator()(int x) const { return base + x; }  // 重载 () 运算符
};
Adder add5{5};
int result = add5(3);  // 8，可携带状态，编译器可内联
```

#### Lambda 表达式

```cpp
int base = 5;
auto add5 = [base](int x) { return base + x; };  // 捕获状态的匿名函数对象
int result = add5(3);  // 8
```

#### std::function（类型擦除包装器）

```cpp
#include <functional>
std::function<int(int)> fn;
fn = [](int x) { return x * 2; };    // 包装 Lambda
fn = Adder{10};                        // 包装仿函数
// 统一接口，但有堆分配 + 间接调用开销
```

### 8.3 成员函数指针

```cpp
class Foo {
public:
    int bar(int x) { return x + 1; }
};

// 成员函数指针语法——比普通函数指针多一个类限定
int (Foo::*mfp)(int) = &Foo::bar;

Foo obj;
int result = (obj.*mfp)(42);  // 通过对象调用

// 为什么成员函数指针可能是 16 字节？
// 多继承/虚继承时，需要额外存储 this 指针偏移量
```

**现代替代方案**：
```cpp
// std::bind（C++11，不推荐）
auto fn = std::bind(&Foo::bar, &obj, std::placeholders::_1);

// Lambda 捕获（推荐）
auto fn = [&obj](int x) { return obj.bar(x); };

// std::mem_fn
auto fn = std::mem_fn(&Foo::bar);
fn(obj, 42);
```

### 8.4 对比表

| 维度 | 函数指针 | 仿函数 | Lambda | std::function |
|------|---------|--------|--------|---------------|
| 携带状态 | ❌ 无 | ✅ 成员变量 | ✅ 捕获列表 | ✅ 包装任意可调用 |
| 内联优化 | ❌ 间接调用 | ✅ 编译器可内联 | ✅ 编译器可内联 | ❌ 类型擦除阻止内联 |
| 类型 | 具体函数签名 | 每个类是独立类型 | 每个 Lambda 独立类型 | 统一 `std::function<>` |
| 开销 | 1 个指针 | 对象大小 | 对象大小 | 堆分配 + 虚调用 |
| C 兼容 | ✅ | ❌ | ✅（无捕获时） | ❌ |
| 使用场景 | C 回调接口 | STL 算法谓词 | 就地短回调 | 存储异构回调 |

### 8.5 优缺点与权衡

**优点**：
- 函数指针零开销，C ABI 兼容
- 仿函数/Lambda 可内联，性能最佳
- std::function 接口统一，可存储任意可调用对象

**缺点/代价**：
- 函数指针不能携带状态，不能内联
- Lambda 类型匿名，不能跨翻译单元传递（除非用 std::function 包装）
- std::function 有堆分配开销（小对象可能 SBO 优化）、虚调用阻止内联

**权衡**：模板参数接受可调用对象是零开销方案（`template<typename F> void forEach(F fn)`），但增加编译时间和代码膨胀；std::function 是运行时多态方案，灵活但有开销。

### 8.6 选型决策树

```
需要 C 回调兼容吗？
├─ 是 → 函数指针（或无捕获 Lambda）
└─ 否 → 需要存储到容器/成员变量吗？
         ├─ 是 → std::function（运行时多态）
         └─ 否 → 模板参数（编译期多态，零开销）
                  └─ 就地使用 → Lambda
                  └─ 需要复用 → 仿函数
```

### 8.7 陷阱与实战

**❌ std::function 在热路径上使用导致性能下降**
```cpp
// ❌ 每次调用有虚函数开销，无法内联
void process(const std::vector<int>& v, std::function<bool(int)> pred) {
    for (int x : v) if (pred(x)) { /* ... */ }
}
```
**✅ 用模板参数零开销**
```cpp
template<typename Pred>
void process(const std::vector<int>& v, Pred pred) {
    for (int x : v) if (pred(x)) { /* ... */ }  // pred 可内联
}
```

> **面试速答**
> - **是什么/本质**：C++ 有四种可调用对象——函数指针（存储代码地址）、仿函数（重载 `operator()`）、Lambda（匿名仿函数）、std::function（类型擦除包装器）
> - **解决什么问题**：实现回调、策略模式、算法定制化，让行为可以作为参数传递
> - **底层原理**：函数指针就是代码段地址；Lambda 编译器生成匿名类；std::function 用类型擦除（虚函数或函数指针 + void* 数据）包装任意可调用对象
> - **优缺点**：函数指针零开销但无状态；Lambda/仿函数可内联性能最佳；std::function 统一但有堆分配和虚调用开销
> - **坑点**：成员函数指针可能 16 字节（多继承偏移）；std::function 热路径性能差；std::bind 已被 Lambda 取代
> - **横向对比**：模板参数是编译期多态（零开销但代码膨胀），std::function 是运行时多态（灵活但有开销）
> - **选型建议**：C 接口用函数指针；STL 算法/模板用 Lambda；需要存储回调用 std::function；热路径用模板参数

## 9. 自测问题

### Q1: vptr 在什么时候被设置？构造函数中调用虚函数会怎样？

> [!note]- 答案
> vptr 在每个构造函数的隐式初始化阶段被设置。基类构造函数将 vptr 指向基类 vtable，派生类构造函数再将 vptr 改写为派生类 vtable。因此在基类构造函数中调用虚函数，调用的是基类版本，不会表现出多态。析构函数中同理——vptr 在每层析构开始时被改回当前类的 vtable。

### Q2: 为什么 C++ 的函数重载不考虑返回值类型？

> [!note]- 答案
> 因为调用函数时可以忽略返回值（`foo();` 不使用返回值），此时编译器无法仅凭返回值类型区分该调用哪个重载版本。Name mangling 也不包含返回值类型。

### Q3: dynamic_cast 和 static_cast 有什么区别？什么时候用哪个？

> [!note]- 答案
> `static_cast` 在编译期做类型转换，不检查运行时类型，如果转换错误会产生未定义行为。`dynamic_cast` 通过 RTTI（访问 vtable[-1] 的 type_info）在运行时检查类型安全性，转换失败指针返回 nullptr、引用抛 bad_cast。向下转型（基类→派生类）且不确定类型时用 dynamic_cast，确定类型时用 static_cast（更快）。

### Q4: 菱形继承中，虚基类的构造函数由谁负责调用？

> [!note]- 答案
> 由最终派生类（most derived class）负责调用虚基类的构造函数，中间类对虚基类构造函数的调用会被忽略。这就是为什么虚继承时，最终派生类的初始化列表中必须显式调用虚基类的构造函数（如果虚基类没有默认构造函数）。

### Q5: 多继承时，将派生类指针赋给第二个基类的指针，指针值会变吗？

> [!note]- 答案
> 会变。编译器会自动调整指针值，使其指向对象中第二基类子对象的起始位置。例如 `C` 继承 `A` 和 `B`，`B* pb = &c;` 时 `pb` 的值比 `&c` 大了一个 `A` 子对象的大小。这种调整在编译期完成（对于 static_cast）。

### Q6: 浅拷贝会导致什么具体问题？至少说三个。

> [!note]- 答案
> 1. **double free**：两个对象的析构函数都会 delete 同一块内存。2. **悬空指针**：一个对象析构后，另一个对象的指针变成野指针。3. **意外共享**：修改一个对象的数据，另一个对象的数据也跟着变（违背值语义）。4. **内存泄漏**：赋值操作中如果不先释放左侧对象的旧资源，旧资源就泄漏了。

### Q7: 拷贝赋值运算符的参数能用值传递吗？这和 copy-and-swap 有什么关系？

> [!note]- 答案
> 拷贝赋值的参数可以用值传递，这正是 copy-and-swap idiom 的实现方式：`T& operator=(T other)`。值传递时拷贝构造已经创建了一份副本，然后 swap 交换资源，旧资源随 `other` 析构而释放。好处：天然处理自赋值、强异常安全保证、代码简洁。注意：拷贝构造函数的参数不能用值传递（否则无限递归）。

### Q8: 什么是 thunk？多继承时为什么需要它？

> [!note]- 答案
> thunk 是编译器自动生成的一小段代码（通常几条指令），用于在多继承场景下调整 this 指针。当通过第二个（或更后面的）基类指针调用虚函数时，this 指针指向的是该基类子对象的位置，而不是完整对象的起始。thunk 先将 this 减去一个偏移量，调整回对象起始，然后跳转到真正的虚函数实现。

### Q9: `extern "C"` 做了什么？为什么 C++ 需要它？

> [!note]- 答案
> `extern "C"` 告诉编译器不要对函数名做 name mangling，而是使用 C 风格的简单符号名。这样 C 代码的链接器就能找到这个函数。C++ 需要 name mangling 来支持函数重载（同名函数生成不同符号），但 C 没有 name mangling，所以跨语言调用时必须用 `extern "C"` 消除 mangling。

### Q10: 虚函数表存放在内存的什么位置？为什么？

> [!note]- 答案
> vtable 存放在 `.rodata`（只读数据段）。因为 vtable 的内容在编译期就完全确定了，运行时不会修改。放在只读段可以：1. 防止被意外或恶意修改（安全性）；2. 多个进程加载同一个共享库时，只读页面可以共享（节省内存）。

### Q11: Rule of Zero 是什么？为什么现代 C++ 推荐它？

> [!note]- 答案
> Rule of Zero：如果一个类的资源管理可以完全交给成员（如 `unique_ptr`、`string`、`vector`），就不需要自定义任何特殊成员函数（析构、拷贝构造、拷贝赋值、移动构造、移动赋值），让编译器默认生成即可。推荐它是因为：1. 减少样板代码；2. 消除忘记实现某个函数的风险；3. 编译器默认生成的函数通常是正确且高效的。

### Q12: 为什么构造函数不能是虚函数？

> [!note]- 答案
> 虚函数调用依赖 vptr，而 vptr 是在构造函数中被初始化的。构造函数执行之前，对象的内存空间还没有 vptr，虚函数机制无法工作。此外，构造函数的职责是创建对象，调用时对象类型必须已知（编译器需要知道要分配多少内存、初始化哪些成员），不存在"根据运行时类型决定如何构造"的语义。如果需要类似"虚构造"的效果，用工厂模式或 clone 虚函数。

### Q13: 隐藏（hide）和重写（override）最容易搞混的场景是什么？

> [!note]- 答案
> 派生类定义了一个和基类虚函数**同名但参数不同**的函数。这不是重写（参数不同），而是隐藏——它会把基类的所有同名函数都隐藏掉。通过基类指针调用时，走的还是虚函数分派（调基类版本）；通过派生类对象直接调用时，只能看到派生类版本。解决办法是在派生类中加 `using Base::foo;`。

### Q14: 在虚继承中，`sizeof` 一个派生类对象会比普通继承大吗？为什么？

> [!note]- 答案
> 通常会更大。虚继承的每个直接虚继承子类需要额外存储一个 vbptr（虚基类指针，8 字节/64位），用来在运行时查找虚基类子对象的偏移。同时虚基类子对象被移到对象末尾单独存放，中间可能有对齐填充。相比普通继承（虚基类数据直接内嵌在子类中，无需 vbptr），空间开销更大。这是用空间和间接寻址换取菱形继承二义性消除的代价。

### Q15: 如果一个类既有虚函数又虚继承了基类，对象中 vptr 和 vbptr 是什么关系？

> [!note]- 答案
> 它们是两个独立的指针，指向不同的表。vptr 指向虚函数表（vtable），用于虚函数分派；vbptr 指向虚基类表（vbtable），用于定位虚基类子对象的偏移。一个对象中可能同时存在 vptr 和 vbptr（分别占一个指针大小）。在某些编译器实现中（如 MSVC），它们是分开的；在其他实现中（如 GCC/Itanium ABI），虚基类偏移信息可能合并到 vtable 中（通过 vcall offset），此时不需要单独的 vbptr。

### 虚函数限制速查

| 能否是虚函数？ | 答案 | 原因 |
|--------------|------|------|
