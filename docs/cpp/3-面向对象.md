## 1. 封装

**一句话定义**：把数据和操作数据的方法绑定在一起，通过访问控制隐藏实现细节。

### 1.1 访问控制（public / protected / private）

**一句话定义**：三个关键字划定"谁能碰这个成员"的边界，是 C++ 封装的唯一手段。

#### 为什么需要访问控制？—— 从 C 的痛点说起

C 语言的 struct 所有字段都是公开的，任何代码都能直接改任何字段：

```c
struct Vector {
    int* data;
    int size;
    int capacity;
};

// 项目里 200 个地方都在直接操作这三个字段
vec.size = 999;       // 谁都能改，capacity 才 10 → 越界崩溃
vec.data = NULL;      // 忘了 free 旧内存 → 内存泄漏
```

**核心痛点**：当 200 个地方都能修改 `size`，出了 bug 你要在 200 个地方排查。更致命的是 `size` 和 `capacity` 之间有**不变量**（`size <= capacity`），但 struct 无法保证，任何人都能打破它。

C++ 的访问控制就是为了解决这个问题：**缩小能修改数据的代码范围，让不变量有人守护**。

```cpp
class Vector {
public:
    void push_back(int val) {       // 唯一的修改入口，内部保证 size <= capacity
        if (size_ == capacity_) grow();
        data_[size_++] = val;
    }
    int size() const { return size_; }  // 只读接口
private:
    int* data_;      // 外部碰不到 → 出 bug 只需查 Vector 内部
    int size_;
    int capacity_;
};
```

现在 `size_` 是 private，能修改它的代码只有 `Vector` 自己的成员函数。200 个调用点变成了 5~6 个成员函数，排查范围缩小了 30 倍。

#### 为什么是三级？两级不够吗？

如果只有 public 和 private 两级，会出现一个尴尬：

```
                 public（谁都能访问）
                    ↑
         ────── 巨大的鸿沟 ──────
                    ↓
                 private（只有自己能访问）
```

派生类和外部调用方被同等对待。但继承是一种**特殊关系**——派生类 IS-A 基类，它比外部代码更"亲近"，往往需要访问基类的实现细节来扩展行为，但又不应该把这些细节暴露给所有人：

```cpp
class TcpConnection {
public:
    void send(const std::string& data);    // 外部调用
protected:
    int fd_;                                // 派生类需要 fd 来做特化（如 SSL 包装）
    void onError(int code);                 // 派生类可重写错误处理
private:
    char recv_buf_[4096];                   // 内部缓冲区，连派生类都不该碰
    void parseProtocol();                   // 纯内部实现
};

class SslConnection : public TcpConnection {
    void handshake() {
        SSL_set_fd(ssl_, fd_);   // ✅ 需要访问 fd_，protected 刚好
        // recv_buf_;             // ❌ private，改不了也不该改
    }
};
```

**三级的设计逻辑**：

| 级别 | 信任圈 | 设计意图 |
|------|--------|---------|
| public | 全世界 | 对外承诺的契约，改了就是 breaking change |
| protected | 自己 + 子类 | 给继承者的扩展点，外部不可见 |
| private | 仅自己 | 纯实现细节，连子类都不依赖 |

**如果没有 protected**：要么把 `fd_` 设为 public（过度暴露），要么设为 private 再加 getter（多一层间接，而且 getter 又变成 public 了，和直接 public 没区别）。protected 恰好填了"信任子类但不信任外部"这个缝隙。

> Java 还多了一级 package-private（同包可访问），C++ 没有包概念，所以用 friend 做更精确的点对点授权。

#### 访问权限矩阵

| 访问权限 | 类自身成员函数 | 友元 | 派生类成员函数 | 类外部代码 |
|---------|-------------|------|-------------|-----------|
| public | ✅ | ✅ | ✅ | ✅ |
| protected | ✅ | ✅ | ✅（只能通过 this） | ❌ |
| private | ✅ | ✅ | ❌ | ❌ |

注意 protected 那个"只能通过 this"——这是最常踩的坑，下面马上讲。

#### 代码：三种权限的实际效果

```cpp
class Base {
public:
    int pub = 1;
protected:
    int prot = 2;
private:
    int priv = 3;
};

class Derived : public Base {
    void foo() {
        pub  = 10;  // ✅ public 成员，随便访问
        prot = 20;  // ✅ protected 成员，派生类通过 this 访问
        // priv = 30; // ❌ 编译错误：private 成员不可访问
    }

    void bar(Base& other) {
        // other.prot = 20; // ❌ 编译错误！不能通过别人的对象访问 protected
        // 只能访问自己继承来的 prot，不能摸别人的
    }
};

void external() {
    Base b;
    b.pub = 10;   // ✅
    // b.prot = 20; // ❌
    // b.priv = 30; // ❌
}
```

#### protected 的"只能通过 this"规则（高频踩坑）

派生类能访问 protected 成员，但**只能通过自己的对象（this 或本类型的实例）**，不能通过基类引用/指针访问：

```cpp
class Base {
protected:
    int x = 42;
};

class Derived : public Base {
    void good() {
        x = 1;              // ✅ 通过 this 访问
        Derived d;
        d.x = 2;            // ✅ 同类型对象也行
    }

    void bad(Base& b) {
        // b.x = 3;         // ❌ 编译错误！通过基类引用访问 protected
    }
};
```

**为什么有这个限制？** 因为 `b` 可能指向另一个不相关的派生类对象（比如 `DerivedB`），让 `Derived` 修改 `DerivedB` 的 protected 成员会破坏封装。标准要求 protected 访问必须通过"当前类或其派生类"的类型。

#### 底层原理

访问控制是**纯编译期检查**，不会生成任何运行时代码。编译器在语义分析阶段校验访问权限，通过后生成的机器码里没有任何权限信息——public 对象和 private 对象在内存布局、访问指令上完全相同。

```
源码                    编译器语义检查              机器码
obj.priv = 1;  ──→  检查调用者是否有权限  ──→  mov [rbp+offset], 1
                     ↓ 无权限 → 编译报错         （和 public 生成的指令一模一样）
```

这意味着：通过指针算术或 `reinterpret_cast` 理论上可以绕过访问控制（但这是 UB）。

#### 选型决策：什么时候用哪个

```
这个成员是对外接口吗？
├─ 是 → public
└─ 否 → 派生类需要访问吗？
         ├─ 是 → protected（但优先考虑提供 protected 方法而非暴露数据）
         └─ 否 → private（默认选择）
```

**经验法则**：数据成员尽量 private，通过 protected 方法暴露给派生类。直接把数据成员设为 protected 等于放弃了在基类中维护不变量的能力。

#### 陷阱

| 陷阱 | 说明 |
|------|------|
| protected 数据成员 | 派生类可以随意修改，基类无法校验不变量。改用 protected getter/setter |
| `#define private public` 来测试 | 违反 ODR（One Definition Rule），是 UB。用友元或接口测试 |
| 以为 private 能"安全保护"数据 | 只是编译期检查，`memcpy`、指针偏移都能读到，不是安全机制 |

#### 同一套关键字的第二个战场：继承修饰词

public/protected/private 除了修饰成员，还出现在继承声明里。很多人搞混这两个位置，其实逻辑完全不同：

- **修饰成员**：控制"谁能访问这个成员"
- **修饰继承**：控制"基类成员在派生类中的最高可见度"

继承修饰词相当于一个**天花板**，基类成员的可见度不能超过这个天花板：

```
class Derived : public Base     → 基类成员保持原样（public 还是 public）
class Derived : protected Base  → 基类 public 成员降级为 protected
class Derived : private Base    → 基类所有成员降级为 private
```

完整的权限变化：

| 基类成员原始权限 | public 继承后 | protected 继承后 | private 继承后 |
|---------------|-------------|-----------------|---------------|
| public | public | protected | private |
| protected | protected | protected | private |
| private | 不可访问 | 不可访问 | 不可访问 |

> private 成员永远不可访问——天花板再高也没用，基类 private 对派生类始终关闭。

**为什么需要继承修饰词？**

关键在于**控制派生类的外部用户能看到多少基类的接口**：

```cpp
class Engine {
public:
    void start();
    void stop();
    int rpm() const;
};

// public 继承 = IS-A：Car 是一种 Engine？语义上不对，但外部能直接调 car.start()
class Car_Bad : public Engine {};

// private 继承 = 用 Engine 实现 Car，但不暴露 Engine 的接口
class Car : private Engine {
public:
    void drive() {
        start();          // ✅ 内部可以用 Engine 的功能
        // 但外部不能 car.start()，因为 start() 被降级为 private
    }
};

Car c;
// c.start();  // ❌ 编译错误！private 继承把 start() 藏起来了
// c.rpm();    // ❌ 同理
c.drive();     // ✅ 只暴露 Car 自己定义的接口
```

**三种继承的语义**：

| 继承方式 | 语义 | 使用频率 | 说明 |
|---------|------|---------|------|
| public | IS-A（是一种） | 95% | 外部通过基类指针/引用使用派生类，多态的基础 |
| private | implemented-in-terms-of（借助...实现） | 4% | 复用基类实现但不暴露接口，通常**组合更好** |
| protected | 给孙子类留扩展点 | <1% | 几乎不用，想不出合理场景就别用 |

**private 继承 vs 组合**：

```cpp
// private 继承
class Timer : private Thread { ... };

// 组合（通常更好）
class Timer {
    Thread thread_;   // 组合：Timer 有一个 Thread
    ...
};
```

两者功能等价，但组合耦合更低、更灵活（可以换实现、可以持有多个）。只有两种场景 private 继承优于组合：
1. 需要 override 基类的虚函数
2. 需要访问基类的 protected 成员

**一句话总结**：修饰成员 = 控制"谁能进门"，修饰继承 = 控制"进了门之后能往外传多少"。

### 1.2 struct vs class 区别

| 区别 | struct | class |
|------|--------|-------|
| 默认访问权限 | public | private |
| 默认继承方式 | public 继承 | private 继承 |
| 其他 | 完全相同，都支持虚函数、继承、模板等 | 完全相同 |

**实战选型**：POD 数据聚合用 `struct`，有行为封装的用 `class`。这只是风格约定，编译器不关心。

### 1.3 友元（friend）的使用与争议

#### 为什么需要 friend？—— 封装和语法的矛盾

有了 public/protected/private 之后，封装看似完美。但实际写代码会遇到一个尴尬：**有些操作逻辑上属于这个类，语法上却不能是成员函数**。

最经典的例子是 `operator<<`：

```cpp
class Point {
    double x_, y_;   // private，封装得很好
public:
    Point(double x, double y) : x_(x), y_(y) {}

    // ❌ 不可能把 operator<< 做成成员函数
    // 因为左操作数是 ostream，不是 Point
    // 成员函数的调用方式是 point << cout，语义反了
    // std::ostream& operator<<(std::ostream& os) { ... }  // point << cout ???
};
```

`cout << point` 要求左操作数是 `ostream`，所以 `operator<<` 必须是**非成员函数**。但非成员函数又没法访问 private 成员。这就产生了矛盾：

**不用 friend 的三种替代方案，全都有问题**：

```cpp
// 方案 1：把 x_, y_ 设为 public
// ❌ 放弃封装，200 个地方都能改你的坐标

// 方案 2：加 getter
class Point {
    double x_, y_;
public:
    double x() const { return x_; }
    double y() const { return y_; }
};
std::ostream& operator<<(std::ostream& os, const Point& p) {
    return os << "(" << p.x() << ", " << p.y() << ")";  // 能用，但...
}
// ⚠️ 简单类型可以凑合，但复杂类型（矩阵、图结构）要暴露大量 getter，
// 等于把 private 数据通过 getter 全部公开了，封装变成了摆设

// 方案 3：提供一个 print 成员函数
class Point {
    double x_, y_;
public:
    void print(std::ostream& os) const { os << "(" << x_ << ", " << y_ << ")"; }
};
std::ostream& operator<<(std::ostream& os, const Point& p) {
    p.print(os);  // 绕了一圈
    return os;
}
// ⚠️ 能用，但每个类都要写一个 print，而且格式化逻辑被锁死在类内部
```

**friend 是最干净的解决方案**：

```cpp
class Point {
    double x_, y_;
    friend std::ostream& operator<<(std::ostream& os, const Point& p);
public:
    Point(double x, double y) : x_(x), y_(y) {}
};

std::ostream& operator<<(std::ostream& os, const Point& p) {
    return os << "(" << p.x_ << ", " << p.y_ << ")";
    // 直接访问 private 成员，不需要 getter，不需要中间函数
}
```

**friend 的本质**：不是"破坏封装"，而是**扩大封装的边界**——把这个函数视为类的逻辑组成部分，只是语法上写在类外面。

#### 四个真实场景：什么时候非用 friend 不可

**场景 1：二元运算符重载（最常见）**

```cpp
class BigInt {
    std::vector<uint32_t> digits_;
    friend BigInt operator+(const BigInt& a, const BigInt& b);
    friend bool operator==(const BigInt& a, const BigInt& b);
};

// 如果不用 friend，要么暴露 digits_（破坏封装），
// 要么写 a.getDigits()（多此一举）
BigInt operator+(const BigInt& a, const BigInt& b) {
    BigInt result;
    // 直接操作 a.digits_ 和 b.digits_ 做加法
    return result;
}
```

**场景 2：紧耦合的协作类**

```cpp
class TreeNode {
    int key_;
    TreeNode* left_;
    TreeNode* right_;
    friend class Tree;   // Tree 需要直接操作节点的指针做旋转、重平衡
};

class Tree {
    TreeNode* root_;
public:
    void insert(int key) {
        // 直接操作 node->left_, node->right_
        // 如果要通过 getter/setter，红黑树旋转的代码可读性会变成灾难
    }
};
```

TreeNode 的指针是内部实现细节，不应该暴露给外部，但 Tree 作为管理者必须直接操作。这不是"破坏封装"，而是 Tree 和 TreeNode **本来就是一个整体**。

**场景 3：工厂函数需要调用 private 构造函数**

```cpp
class Connection {
    Connection(int fd) : fd_(fd) {}   // private 构造函数：不允许外部随便创建
    int fd_;
    friend Connection createConnection(const std::string& addr);
};

// 只有工厂函数能创建 Connection，保证连接的创建流程可控
Connection createConnection(const std::string& addr) {
    int fd = ::socket(AF_INET, SOCK_STREAM, 0);
    // ... connect 逻辑、错误处理
    return Connection(fd);   // 调用 private 构造函数
}
```

**场景 4：单元测试访问 private 成员**

```cpp
class Parser {
    std::vector<Token> tokenize(const std::string& input);  // private 内部步骤
    ASTNode* parse(const std::vector<Token>& tokens);       // private 内部步骤
public:
    ASTNode* compile(const std::string& input);  // public 接口
    friend class ParserTest;  // 测试类需要单独测 tokenize 和 parse
};
```

#### friend 的三个限制（防止滥用的安全阀）

| 限制 | 含义 | 为什么这么设计 |
|------|------|--------------|
| **单向性** | A 声明 B 是友元，B 不能反过来访问 A | 授权是主动的，不是互相的 |
| **不可继承** | A 是 B 的友元，A 的派生类不是 | 你信任张三，不代表信任张三的儿子 |
| **不可传递** | A 是 B 的友元，B 是 C 的友元，A 不是 C 的友元 | 朋友的朋友不是你的朋友 |

这三个限制的设计意图很明确：**friend 是精确的点对点授权，不会扩散**。

#### 优缺点

| 优点 | 缺点 |
|------|------|
| 解决"逻辑上属于类、语法上不能是成员"的矛盾 | 增加耦合——友元函数/类与被友元的类的实现细节绑死 |
| 比暴露 getter 更干净，不扩大 public 接口 | 被友元的类修改 private 成员布局时，所有友元都要跟着改 |
| 精确授权（只授权给特定函数/类） | 无法在运行时动态授权，编译期写死 |

#### 什么时候该用 / 不该用

```
需要从类外部访问 private 成员？
├─ 能通过 public 接口完成吗？ → 用 public 接口，不要 friend
├─ 加 getter 就行？ → 看情况：
│   ├─ 只有 1-2 个简单字段 → getter 够了
│   └─ 需要暴露大量内部细节 → getter 等于假封装，用 friend 更诚实
├─ 运算符重载（左操作数不是本类）？ → friend，标准做法
├─ 两个类是一个逻辑整体（Tree/Node）？ → friend class
└─ 一个类有 5 个以上 friend？ → 接口设计有问题，重新审视
```

> **面试速答**
>
> 封装说白了就是用 public/protected/private 这三个关键字控制谁能访问什么，核心目的是把实现细节藏起来、让外部代码只依赖你的接口。这样一来修改内部实现就不会波及调用方，耦合自然就下来了。它是纯编译期的约束，不生成任何运行时代码。
>
> 底层看，编译器在符号解析阶段检查访问权限，运行时的内存布局完全没有差异——理论上你甚至能通过指针算术绕过（当然是 UB）。struct 和 class 在 C++ 里唯一的区别就是默认访问权限不同，其他完全一样。
>
> 有个点要特别注意：友元是一种"受控的封装破坏"，它有单向性、不可继承、不可传递这三个特性。实际工程里如果你发现一个类有一大堆友元，那基本就是接口设计出问题了。和 Java 的 package-private 相比，C++ 没有包级别的访问控制，友元算是更精确但也更侵入的替代方案。
>
> 选型上，纯数据聚合类（POD）用 struct 比较方便，毕竟默认 public；有不变量需要维护的类就用 class。友元只在运算符重载、序列化这种确实绕不过去的场景才用。

## 2. 继承

**一句话定义**：派生类复用基类的数据和行为，并可以扩展或修改。

### 2.1 三种继承方式

> 权限变化表和选型分析见 [[#同一套关键字的第二个战场：继承修饰词|1.1 继承修饰词]]，此处不重复。

要点回顾：public 继承 = IS-A（95%），private 继承 = 借助实现（组合通常更好），protected 继承几乎不用。

### 2.2 构造 / 析构顺序

```
构造顺序（由底向上建造）：
  1. 虚基类（如果有，按声明顺序）
  2. 直接基类（按继承列表中的声明顺序，不是初始化列表顺序）
  3. 成员变量（按类内声明顺序，不是初始化列表顺序）
  4. 派生类构造函数体

析构顺序：与构造完全相反
```

```cpp
class Base {
public:
    Base()  { std::cout << "Base()\n"; }
    ~Base() { std::cout << "~Base()\n"; }
};

class Member {
public:
    Member()  { std::cout << "Member()\n"; }
    ~Member() { std::cout << "~Member()\n"; }
};

class Derived : public Base {
    Member m;
public:
    Derived()  { std::cout << "Derived()\n"; }
    ~Derived() { std::cout << "~Derived()\n"; }
};

// 输出：Base() -> Member() -> Derived()
// 析构：~Derived() -> ~Member() -> ~Base()
```

**陷阱**：初始化列表的书写顺序不影响初始化顺序，实际顺序由声明顺序决定。

### 2.3 菱形继承问题 + 虚继承解决方案

#### 问题：普通菱形继承的内存布局

```
          Animal
         /      \
      Dog        Cat
         \      /
          Pet
```

不使用虚继承时，Pet 内部有**两份** Animal 的数据：

```
Pet 对象的内存布局（无虚继承）：
┌──────────────────────────┐
│  Dog 子对象               │
│  ┌────────────────────┐  │
│  │ Animal 副本 1       │  │  ← Dog 继承的 Animal
│  │   name, age ...    │  │
│  ├────────────────────┤  │
│  │ Dog 自身成员        │  │
│  └────────────────────┘  │
├──────────────────────────┤
│  Cat 子对象               │
│  ┌────────────────────┐  │
│  │ Animal 副本 2       │  │  ← Cat 继承的 Animal（冗余！）
│  │   name, age ...    │  │
│  ├────────────────────┤  │
│  │ Cat 自身成员        │  │
│  └────────────────────┘  │
├──────────────────────────┤
│  Pet 自身成员             │
└──────────────────────────┘
```

问题：`pet.name` 有二义性，必须写 `pet.Dog::name` 或 `pet.Cat::name`。

#### 解决方案：虚继承

```cpp
class Animal {
public:
    std::string name;
    int age;
};

class Dog : virtual public Animal {   // 虚继承
    int barkVolume;
};

class Cat : virtual public Animal {   // 虚继承
    int climbHeight;
};

class Pet : public Dog, public Cat {  // 只有一份 Animal
    int petId;
};
```

#### 虚继承的内存布局（64 位系统）

```
Pet 对象的内存布局（虚继承）：
┌──────────────────────────────┐  偏移 0
│  Dog 子对象                   │
│  ┌────────────────────────┐  │
│  │ vbptr (虚基类指针)      │  │  → 指向 Dog 的虚基类表
│  ├────────────────────────┤  │
│  │ barkVolume             │  │
│  └────────────────────────┘  │
├──────────────────────────────┤
│  Cat 子对象                   │
│  ┌────────────────────────┐  │
│  │ vbptr (虚基类指针)      │  │  → 指向 Cat 的虚基类表
│  ├────────────────────────┤  │
│  │ climbHeight            │  │
│  └────────────────────────┘  │
├──────────────────────────────┤
│  Pet 自身成员                 │
│  │ petId                  │  │
├──────────────────────────────┤
│  Animal 子对象（共享，仅一份） │  ← 放在对象末尾
│  │ name                   │  │
│  │ age                    │  │
└──────────────────────────────┘

Dog 的虚基类表 (vbtable):
┌──────────────┐
│ 0            │  ← vbptr 到 Dog 子对象起始的偏移
│ offset_to_   │  ← vbptr 到 Animal 子对象的偏移（运行时计算）
│ Animal       │
└──────────────┘
```

### 2.4 虚基类指针和虚基类表的实现

**原理**：每个虚继承的子类维护一个 vbptr（虚基类指针），指向一张虚基类表（vbtable）。表中存储的是**从 vbptr 位置到虚基类子对象的偏移量**。访问虚基类成员时，编译器生成类似如下的代码：

```cpp
// pet.name 被编译器翻译为：
// 1. 取 Dog 部分的 vbptr
// 2. 从 vbtable 读取偏移
// 3. vbptr地址 + 偏移 = Animal 子对象地址
// 4. 访问 Animal::name

// 伪代码：
char* vbptr = (char*)&pet + offsetof(Dog, vbptr);
int offset = *(int*)(*(void**)vbptr + 1);  // 表中第二项
Animal* base = (Animal*)(vbptr + offset);
base->name;  // 最终访问
```

**代价**：虚继承相比普通继承多了一次间接寻址，且虚基类子对象通常放在对象末尾，导致无法简单地用 `static_cast` 做指针调整。

#### 2. 优缺点与权衡

**继承的优缺点：**

**优点**：
- 代码复用与类型抽象：派生类自动继承基类的数据和行为，配合虚函数实现里氏替换原则（LSP），调用方可以用基类指针/引用操作所有派生类
- 编译器自动处理构造/析构链：按声明顺序逐层构造、逆序析构，不需要手动管理基类子对象的生命周期
- 接口契约强制力：纯虚函数强制派生类实现特定接口，编译期就能发现遗漏

**缺点/代价**：
- 耦合度高：派生类与基类形成编译期强耦合，基类修改（增删成员、改变虚函数签名）会导致所有派生类重新编译，深层继承链改动影响面大（脆弱基类问题）
- 认知成本：多层继承 + 多继承下，需要理解构造顺序、name lookup 规则、隐藏关系等，代码可读性急剧下降
- 组合优于继承：private 继承表示 "implemented-in-terms-of"，但几乎所有场景用组合（成员变量）都更清晰、耦合度更低

**虚继承的额外代价：**

**优点**：
- 解决菱形继承的二义性问题，虚基类子对象只保留一份，避免数据冗余和 `pet.name` 二义性

**缺点/代价**：
- 每个虚继承的子类多一个 vbptr（8 字节/64位），访问虚基类成员需要额外一次间接寻址（vbptr → vbtable → 偏移计算），性能略差
- 虚基类子对象放在对象末尾，`static_cast` 无法做正确的指针调整，必须用 `dynamic_cast`
- 构造复杂度增加：虚基类的构造由最终派生类负责（而非直接派生类），容易出错

**权衡**：继承在需要运行时多态和类型层次时不可替代，但应控制继承深度（建议不超过 3 层）；虚继承仅在确实需要菱形继承时使用，能用组合解决的不要用多继承

### 2.5 组合优于继承（Composition over Inheritance）

"组合优于继承"是 GoF 设计模式的第一原则，但不代表"禁止继承"。关键在于理解两者的语义差别：

```cpp
// 继承 = IS-A：Dog 是一种 Animal
class Dog : public Animal { ... };

// 组合 = HAS-A：Car 有一个 Engine
class Car {
    Engine engine_;    // 组合：Car 持有一个 Engine
    ...
};
```

**为什么组合通常更好？** 看一个实际例子：

```cpp
// ❌ 用继承：Stack 继承 vector
class Stack : public std::vector<int> {
public:
    void push(int x) { push_back(x); }
    int pop() {
        int top = back();
        pop_back();
        return top;
    }
};

Stack s;
s.push(1);
s.push(2);
s.insert(s.begin(), 999);  // 💥 绕过了 Stack 的接口！
                             // 因为 public 继承暴露了 vector 的所有方法
                             // Stack 的"后进先出"不变量被破坏了
```

```cpp
// ✅ 用组合：Stack 内部持有 vector
class Stack {
    std::vector<int> data_;   // 组合：外部看不到 vector
public:
    void push(int x) { data_.push_back(x); }
    int pop() {
        int top = data_.back();
        data_.pop_back();
        return top;
    }
    // 外部无法调 data_.insert()，不变量安全
};
```

**对比表**：

| 维度 | 继承 | 组合 |
|------|------|------|
| 关系语义 | IS-A（是一种） | HAS-A（有一个） |
| 耦合度 | 高（派生类依赖基类实现细节） | 低（只依赖接口） |
| 灵活性 | 编译期绑定，不能换基类 | 运行时可以换实现（依赖注入） |
| 封装性 | 暴露基类的 protected 甚至 public 接口 | 完全隐藏内部实现 |
| 多态 | 支持（虚函数） | 不直接支持（可通过接口+组合实现） |

**选型决策**：

```
需要运行时多态（基类指针/引用统一处理）？
├─ 是 → public 继承（唯一正确选择）
└─ 否 → 是纯粹的代码复用？
         ├─ 是 → 组合（几乎总是更好）
         └─ 需要 override 基类虚函数或访问 protected？
              ├─ 是 → private 继承（罕见）
              └─ 否 → 组合
```

> **面试速答**
>
> 继承的本质是通过 is-a 关系建立类型层次。public 继承表达的是接口契约——派生类可以被当作基类使用，也就是里氏替换原则；private 继承则是实现复用的另一种形式。为什么要搞这个？一方面是代码复用，另一方面更重要的是类型抽象，让上层代码只依赖基类接口。
>
> 构造顺序有明确规则：虚基类 → 直接基类 → 成员变量 → 自身构造体，析构完全反过来。菱形继承靠虚继承解决，机制是编译器加一个 vbptr 指向 vbtable，让虚基类子对象在整个派生体系里只保留一份（而且放在对象末尾）。代价是每个虚继承多一个 vbptr，访问虚基类成员还要多一次间接寻址。
>
> 有个容易踩的坑：初始化列表里写的顺序不影响实际构造顺序，实际是按成员声明顺序来的，这点新手经常搞混。还有，95% 场景其实只会用 public 继承，很多人想用 private 继承复用实现的时候，用组合往往是更好的选择。
>
> 选型上记一条口诀：需要多态用 public 继承，需要复用实现优先组合，只有 EBO 这种特殊场景才考虑 private 继承。继承深度最好控制在 3 层以内，能用组合解决的就不要硬上多继承。

## 3. 多态（重点）

### 编译期多态 vs 运行期多态

| 维度 | 编译期多态（静态） | 运行期多态（动态） |
|------|-------------------|-------------------|
| 绑定时机 | 编译期确定调用地址 | 运行期通过 vtable 查找 |
| 实现手段 | 函数重载、模板、运算符重载 | 虚函数 + 继承 |
| 性能 | 零开销（内联友好） | 有间接跳转开销（阻碍内联） |
| 灵活性 | 类型必须编译期已知 | 可处理运行期才确定的类型 |
| 典型场景 | STL 算法、CRTP | 插件系统、工厂模式、回调 |

**核心认知**：C++ 同时拥有两种多态，选型原则是——能用编译期多态就不用运行期多态（性能更好），需要运行时动态分派时才用虚函数。

### 3.1 静态多态

#### 函数重载（name mangling 原理）

**一句话定义**：同一作用域内，函数名相同但参数列表不同的多个函数。

编译器通过 **name mangling（名字修饰）** 让重载函数在目标文件中有不同的符号名：

```cpp
void foo(int a);          // 编译后符号：_Z3fooi
void foo(double a);       // 编译后符号：_Z3food
void foo(int a, int b);   // 编译后符号：_Z3fooii
```

GCC 的 mangling 规则（Itanium ABI）：
- `_Z` = 前缀
- `3foo` = 函数名长度 + 函数名
- `i` = int, `d` = double, `Ss` = std::string ...

```bash
# 查看 mangled 名字
nm a.out | grep foo
# 反 mangling
c++filt _Z3fooi  # 输出：foo(int)
```

**关键点**：这就是为什么 C++ 重载只看参数不看返回值——返回值不参与 name mangling（因为调用时不一定使用返回值，无法区分）。

**`extern "C"` 的作用**：禁止 name mangling，使得 C++ 函数能被 C 代码链接。

#### 运算符重载

```cpp
class Vec2 {
    float x, y;
public:
    Vec2(float x, float y) : x(x), y(y) {}

    // 成员函数重载
    Vec2 operator+(const Vec2& rhs) const {
        return Vec2(x + rhs.x, y + rhs.y);
    }

    // 友元函数重载（支持左操作数不是本类型）
    friend Vec2 operator*(float scalar, const Vec2& v) {
        return Vec2(scalar * v.x, scalar * v.y);
    }

    // 流输出必须用友元（左操作数是 ostream）
    friend std::ostream& operator<<(std::ostream& os, const Vec2& v) {
        return os << "(" << v.x << ", " << v.y << ")";
    }
};
```

**不可重载的运算符**：`::`, `.`, `.*`, `?:`, `sizeof`——它们要么是编译期特性，要么重载会破坏基本语法语义。

#### 模板（编译期多态）

```cpp
// 函数模板——编译器为每种类型实例化一份代码
template<typename T>
T max_val(T a, T b) {
    return (a > b) ? a : b;
}

// 编译器生成：
// int max_val<int>(int, int)
// double max_val<double>(double, double)
// 各自独立的函数，零运行时开销
```

模板 vs 虚函数的选型：
- 类型在编译期已知 → 模板（零开销、可内联）
- 类型在运行期才确定（如插件加载、用户输入决定类型） → 虚函数

> **面试速答**
>
> 静态多态就是在编译期就把调用地址定死的多态机制，C++ 里主要有函数重载和模板两种形式。它存在的意义是在不牺牲运行时性能的前提下也能做泛型和接口统一——零开销、还能被编译器进一步内联优化。
>
> 底层实现上，函数重载靠的是 name mangling：编译器把参数类型编码进符号名，比如 `foo(int)` 变成 `_Z3fooi`。要注意返回值不参与 mangling，所以不能只靠返回值类型区分重载。模板则是为每个用到的类型实例化出一份独立的代码，全都在编译期展开。
>
> 优点很明显：零运行时开销，还能内联。缺点也很现实：模板会导致代码膨胀、编译时间变长，错误信息一出来一屏幕读起来想骂人。
>
> 和运行期多态（虚函数）比，静态多态胜在零开销，但代价是类型必须编译期就确定。所以选型很直白：类型在编译期能确定就用模板/重载；只有插件加载、用户输入这种运行时才知道类型的场景，才用虚函数。

### 3.2 动态多态（深度下钻）

#### 一个完整的例子：从写代码到 CPU 执行，多态是怎么发生的

先看一段最简单的多态代码，然后一步一步拆解底层：

```cpp
#include <iostream>
using namespace std;

class Animal {
public:
    virtual void speak() { cout << "..." << endl; }
    virtual ~Animal() = default;
};

class Dog : public Animal {
public:
    void speak() override { cout << "汪汪！" << endl; }
};

class Cat : public Animal {
public:
    void speak() override { cout << "喵喵！" << endl; }
};

int main() {
    Animal* a = new Dog();   // 基类指针指向派生类对象
    a->speak();              // 输出 "汪汪！" —— 多态！
    delete a;

    a = new Cat();
    a->speak();              // 输出 "喵喵！" —— 多态！
    delete a;
}
```

问题来了：编译器看到 `a->speak()` 时，`a` 的类型是 `Animal*`，它怎么知道该调 `Dog::speak` 还是 `Cat::speak`？

答案是：**它不知道，也不需要知道。它生成了一段"运行时去查表"的代码。**

---

**第 1 步：编译器为每个含虚函数的类生成一张 vtable**

编译时，编译器扫描到 Animal、Dog、Cat 三个类都有虚函数，为每个类生成一张虚函数表（vtable），放在可执行文件的**只读数据段**（`.rodata`）：

```
Animal 的 vtable:                Dog 的 vtable:                Cat 的 vtable:
┌──────────────────┐            ┌──────────────────┐          ┌──────────────────┐
│ &Animal::speak   │ [0]        │ &Dog::speak      │ [0]      │ &Cat::speak      │ [0]
│ &Animal::~Animal │ [1]        │ &Dog::~Dog       │ [1]      │ &Cat::~Cat       │ [1]
└──────────────────┘            └──────────────────┘          └──────────────────┘
```

注意 Dog 的 vtable 中，`[0]` 号槽位**不是** `&Animal::speak`，而是 `&Dog::speak`——因为 Dog 重写了 `speak()`。这就是多态的关键：**每个类的 vtable 里，同一个槽位放的是自己的实现**。

---

**第 2 步：`new Dog()` 时，构造函数把 vptr 塞进对象头部**

`new Dog()` 做了两件事：
1. 在堆上分配内存（`malloc` 或 `operator new`）
2. 调用构造函数

构造函数执行时，**编译器偷偷插入一行代码**，把 Dog 的 vtable 地址写入对象的头 8 字节（这个隐藏字段叫 vptr）：

```
堆内存中的 Dog 对象：
┌──────────────────────────────┐
│ vptr = &Dog::vtable ─────────┼──→ Dog 的 vtable（在 .rodata 段）
├──────────────────────────────┤
│ （Animal 的数据成员，如果有） │
├──────────────────────────────┤
│ （Dog 的数据成员，如果有）    │
└──────────────────────────────┘
          ↑
          a 指向这里
```

构造函数里实际发生的事（伪代码）：
```cpp
Dog::Dog() {
    // 编译器插入的隐藏代码：
    this->vptr = &Dog_vtable;   // ← 这行你看不到，但它存在

    // 你写的构造函数体（这个例子里是空的）
}
```

---

**第 3 步：`a->speak()` 调用时，运行时查表**

编译器看到 `a->speak()`，知道 `speak` 是虚函数，**不生成直接调用**，而是生成"查表间接调用"：

```
编译器生成的逻辑（伪代码）：

a->speak()
    ↓ 翻译为
step 1: vptr = *(a)              // 从对象头部取出 vptr（Dog 的 vtable 地址）
step 2: func = vptr[0]           // speak 在 vtable 中的偏移是 [0]，取出函数地址
step 3: func(a)                  // 用 a 作为 this 指针，调用这个函数
```

对应的 x86-64 汇编（3 条指令）：

```asm
mov  rax, [rdi]        ; rdi = a（this指针），rax = *a = vptr
mov  rax, [rax]        ; rax = vptr[0] = Dog::speak 的地址
call rax               ; 间接调用 Dog::speak
```

**这就是全部了。** 多态的魔法就是这 3 条指令：取 vptr → 查 vtable → 间接调用。

---

**第 4 步：如果换成 `a = new Cat()`，同样的代码走不同路径**

```
new Cat() 时，对象的 vptr 指向 Cat 的 vtable：

┌──────────────────────────────┐
│ vptr = &Cat::vtable ─────────┼──→ Cat 的 vtable
└──────────────────────────────┘      [0] = &Cat::speak
          ↑                           [1] = &Cat::~Cat
          a 指向这里

a->speak() 还是那 3 条指令：
  mov rax, [rdi]     → 取出 Cat 的 vtable 地址
  mov rax, [rax]     → 取出 Cat::speak 的地址
  call rax           → 调用 Cat::speak → 输出 "喵喵！"
```

**同样的机器码，因为 vptr 不同，走到了不同的函数**——这就是"运行时多态"的含义。

---

**完整流程图**：

```
源码: a->speak()

编译期（编译器做的事）：
  1. 识别 speak 是虚函数 → 不生成直接 call
  2. 查出 speak 在 vtable 中的偏移量（index 0）
  3. 生成 3 条间接调用指令

运行时（CPU 做的事）：
  1. 读对象头部 → 拿到 vptr → 知道是哪个类的 vtable
  2. 在 vtable 中按偏移取出函数地址
  3. 跳转执行
  ┌──────────────┐
  │ a 指向 Dog？ │ → vptr → Dog::vtable → Dog::speak → "汪汪！"
  │ a 指向 Cat？ │ → vptr → Cat::vtable → Cat::speak → "喵喵！"
  └──────────────┘
```

> **和"不用多态"对比**：如果不用虚函数，你只能写 `if (type == DOG) dog_speak(); else if (type == CAT) cat_speak();`——每加一种动物就要改这个 if-else。虚函数把这个 if-else 变成了一次表查找，新增动物只需新增一个类，调用方完全不用改。

#### 虚函数表（vtable）内存布局

**本质**：每个含虚函数的类在编译期生成一张虚函数表（vtable），存放在只读数据段（`.rodata`）。表中每一项是一个函数指针，指向该类实际应该调用的虚函数实现。

**单继承的 vtable 布局**：

```cpp
class Base {
public:
    virtual void func1() { std::cout << "Base::func1\n"; }
    virtual void func2() { std::cout << "Base::func2\n"; }
    virtual ~Base() {}
    int x;
};

class Derived : public Base {
public:
    void func1() override { std::cout << "Derived::func1\n"; }
    virtual void func3() { std::cout << "Derived::func3\n"; }
    int y;
};
```

```
Base 对象内存布局：                  Base 的 vtable（.rodata 段）：
┌─────────────┐                    ┌───────────────────────┐
│ vptr ────────┼──────────────────→ │ &Base::func1          │ [0]
├─────────────┤                    │ &Base::func2          │ [1]
│ x           │                    │ &Base::~Base          │ [2]
└─────────────┘                    └───────────────────────┘

Derived 对象内存布局：               Derived 的 vtable（.rodata 段）：
┌─────────────┐                    ┌───────────────────────┐
│ vptr ────────┼──────────────────→ │ &Derived::func1       │ [0] ← 覆盖了
├─────────────┤                    │ &Base::func2          │ [1] ← 继承的
│ x (Base)    │                    │ &Derived::~Derived    │ [2] ← 覆盖了
├─────────────┤                    │ &Derived::func3       │ [3] ← 新增的
│ y (Derived) │                    └───────────────────────┘
└─────────────┘
```

#### 虚函数指针（vptr）在对象中的位置

**vptr 位于对象的起始位置**（大多数编译器实现，包括 GCC 和 Clang）。这样做的好处是：通过基类指针调用虚函数时，不需要做偏移计算就能找到 vptr。

```cpp
Derived d;
// &d 的值 == d 的 vptr 的地址

// 验证 vptr 在对象头部
std::cout << sizeof(void*) << std::endl; // 8（64位系统）
void** vptr_addr = *(void***)&d;         // 取出 vptr 的值（vtable 地址）
```

**关键事实**：
- vptr 占 8 字节（64 位系统）或 4 字节（32 位系统）
- vptr 在构造函数中被设置，析构函数中被回退（下面详细讲）

#### 实战：手动通过函数指针访问虚函数表

下面的代码演示了如何在运行时手动取出 vptr、从虚函数表中拿到函数地址并调用，帮助理解 vtable 的内存结构：

```cpp
#include <iostream>
using namespace std;

class A {
public:
    virtual void vfunc1() { cout << "A::vfunc1()" << endl; }
    virtual void vfunc2() { cout << "A::vfunc2()" << endl; }
private:
    int data1_, data2_;
};

class B : public A {
public:
    virtual void vfunc1() override { cout << "B::vfunc1()" << endl; }
private:
    int data3_;
};

int main() {
    B b;
    // 定义函数指针类型，用于接收从虚函数表中取出的地址
    typedef void(*Fun)(void);
    Fun pFun = nullptr;

    // 对象首地址就是 vptr 的地址，解引用得到 vtable 的起始地址
    cout << "虚函数表地址：" << (long long*)*(long long*)(&b) << endl;

    // 从 vtable[0] 取出第一个虚函数地址并调用（B 覆盖了 vfunc1）
    pFun = (Fun)*((long long*)*(long long*)(&b));
    pFun();  // 输出：B::vfunc1()

    // 从 vtable[1] 取出第二个虚函数地址并调用（B 继承了 A::vfunc2）
    pFun = (Fun)*((long long*)*(long long*)(&b) + 1);
    pFun();  // 输出：A::vfunc2()

    return 0;
}
```

> **注意**：上面使用 `long long*`（8 字节）适用于 64 位系统。32 位系统应改用 `int*`（4 字节）。这段代码依赖编译器实现细节，仅用于学习理解，生产代码中不应手动操作 vtable。

#### 3. 优缺点与权衡

**优点**：
- 开闭原则的基石：新增派生类不需要修改已有的调用方代码，只需继承并 override 虚函数，扩展性极强（插件系统、工厂模式的核心机制）
- 运行时类型切换：可以根据运行时条件（用户输入、配置文件、动态加载的 .so）选择具体实现，这是编译期多态做不到的
- 接口与实现分离：调用方只持有基类指针/引用，完全解耦具体实现类

**缺点/代价**：
- 每对象额外 8 字节 vptr 开销：对于大量小对象（如粒子系统中百万个粒子），vptr 的内存占比可能很高
- 间接调用无法内联：虚函数调用需要两次内存间接访问（取 vptr → 取 vtable 槽位），编译器无法在编译期确定调用目标，因此无法内联优化，也不利于 CPU 分支预测
- cache miss 风险：vptr 指向的 vtable 在 .rodata 段，如果多态调用分散在不同类型的对象上，vtable 可能不在 L1 cache 中，造成额外的 cache miss
- 多继承下复杂度爆炸：多张 vtable + thunk 调整 this 指针，内存布局和调试难度显著增加

**权衡**：在需要运行时动态分派的场景（插件加载、回调注册、策略模式），虚函数的灵活性收益远大于性能代价；但在性能敏感的热路径（游戏引擎 tick、HFT 交易循环），应考虑用 CRTP 静态多态或 `std::variant` + `std::visit` 替代虚函数

#### 虚函数默认参数陷阱

虚函数的**默认参数是静态绑定**的（编译期确定），而虚函数调用本身是**动态绑定**的（运行期确定）。两者的绑定时机不一致会产生反直觉的结果：

```cpp
#include <iostream>
using namespace std;

class Base {
public:
    virtual void fun(int x = 0) {
        cout << "Base::fun(), x = " << x << endl;
    }
};

class Derived : public Base {
public:
    // 派生类重新定义了默认参数 x = 10
    virtual void fun(int x = 10) {
        cout << "Derived::fun(), x = " << x << endl;
    }
};

int main() {
    Derived d;

    // 通过派生类指针调用：默认参数取 Derived 的定义（x=10）
    Derived* dp = &d;
    dp->fun();  // 输出：Derived::fun(), x = 10

    // 通过基类指针调用：函数体走 Derived（动态绑定），但默认参数取 Base 的定义（x=0）
    Base* bp = &d;
    bp->fun();  // 输出：Derived::fun(), x = 0  ← 注意这里！

    return 0;
}
```

> **原因**：默认参数不属于函数签名，不参与虚函数的动态分派。编译器在编译期根据**指针/引用的静态类型**决定使用哪个默认值。因此《Effective C++》条款37明确建议：**绝不重新定义继承而来的缺省参数值**（Never redefine function's inherited default parameters value）。

#### 虚函数调用的完整过程（从 C++ 到汇编）

```cpp
Base* p = new Derived();
p->func1();  // 这一行发生了什么？
```

**C++ 层面**：编译器看到 `p->func1()` 且 `func1` 是虚函数，不会直接 call 固定地址，而是生成间接调用。

**编译器生成的伪代码**：
```cpp
// p->func1() 被翻译为：
void* vptr   = *(void**)p;            // 步骤1：从对象头部取 vptr
void* func   = ((void**)vptr)[0];      // 步骤2：从 vtable 取第 0 项（func1 的槽位）
((void(*)(Base*))func)(p);              // 步骤3：调用函数，传 this 指针
```

**x86-64 汇编**（GCC -O0 典型输出）：
```asm
; p 在 rdi 中
mov  rax, [rdi]          ; rax = *p，即 vptr（vtable 地址）
mov  rax, [rax]          ; rax = vtable[0]，即 func1 的地址
call rax                 ; 间接调用
```

**性能影响**：
- 两次内存间接访问（可能 cache miss）
- 间接 `call` 阻碍 CPU 分支预测和编译器内联
- 这就是高性能场景（游戏引擎、HFT）倾向用 CRTP 等编译期多态替代虚函数的原因

#### 多继承下的多张虚表

```cpp
class A {
public:
    virtual void funcA() {}
    virtual ~A() {}
    int a;
};

class B {
public:
    virtual void funcB() {}
    virtual ~B() {}
    int b;
};

class C : public A, public B {
public:
    void funcA() override {}
    void funcB() override {}
    int c;
};
```

```
C 对象内存布局：                     C 的 vtable（两张表拼在一起）：

┌─────────────┐                    vtable for A 部分：
│ vptr_A ──────┼───────────────────→ ┌──────────────────────┐
├─────────────┤                     │ &C::funcA            │ [0]
│ a (A部分)    │                     │ &C::~C               │ [1]
├─────────────┤                     └──────────────────────┘
│ vptr_B ──────┼───────────────────→ vtable for B 部分：
├─────────────┤                     ┌──────────────────────┐
│ b (B部分)    │                     │ &C::funcB (thunk)    │ [0]
├─────────────┤                     │ &C::~C (thunk)       │ [1]
│ c (C自身)    │                     └──────────────────────┘
└─────────────┘

注意：通过 B* 指针调用虚函数时，需要 thunk 调整 this 指针
```

**thunk 是什么**：当通过第二基类（B）的指针调用虚函数时，`this` 指针指向的是对象中 B 子对象的位置，而不是对象起始。thunk 是编译器生成的一小段代码，负责将 `this` 指针调整回对象起始位置，然后跳转到真正的函数实现。

```asm
; thunk for C::funcB (通过 B* 调用时)
C::funcB_thunk:
    sub  rdi, 16      ; 调整 this 指针（减去 A 子对象大小）
    jmp  C::funcB     ; 跳转到真正的实现
```

> **面试速答**
>
> 虚函数是 C++ 实现运行期多态的核心机制，底层靠虚函数表（vtable）加虚指针（vptr）做间接调用，让基类指针能调用到派生类重写的函数。为什么要有这东西？就是为了让代码依赖抽象而不是具体实现，不改调用方就能扩展新派生类的行为，也就是开闭原则。
>
> 底层其实不复杂：每个含虚函数的类在编译期生成一张 vtable，本质是函数指针数组，放在只读数据段。对象头部那 8 字节就是 vptr，指向自己类型的 vtable。调用 `p->func()` 时编译器生成的代码大致是 `mov rax,[p]; call [rax+offset]`——先从对象里取出 vptr，再按固定偏移取出函数地址跳过去。运行时就靠这个 vptr 定位到实际类型的函数。
>
> 代价也在这：每个对象多 8 字节 vptr，间接调用无法内联，还可能 cache miss。
>
> 有几个坑要特别注意。第一，构造和析构函数里调用虚函数不会触发多态，因为那时 vtable 还没建好或已拆掉，调的是当前类的版本。第二，基类析构必须 `virtual`，不然通过基类指针 delete 派生类对象就是 UB。第三，虚函数的默认参数是静态绑定的，跟着指针的静态类型走，很反直觉。
>
> 选型上，和编译期多态（模板/CRTP）比，虚函数胜在灵活、能运行时切换类型，代价就是那点运行时开销。需要运行时类型切换就大胆用虚函数；性能敏感的热路径可以考虑 CRTP 换成静态多态。

#### 虚析构函数为什么必须要

```cpp
// ❌ 错误：基类析构函数不是虚函数
class Base {
public:
    ~Base() { std::cout << "~Base\n"; }
};

class Derived : public Base {
    int* data;
public:
    Derived() : data(new int[100]) {}
    ~Derived() { delete[] data; std::cout << "~Derived\n"; }
};

Base* p = new Derived();
delete p;  // 只调用 ~Base()，Derived::data 泄漏！

// ✅ 正确：基类析构函数声明为 virtual
class Base {
public:
    virtual ~Base() { std::cout << "~Base\n"; }
};
// delete p 时：先调 ~Derived()，再调 ~Base()，无泄漏
```

**规则**：只要一个类可能被继承且通过基类指针删除，它的析构函数就必须是 virtual。反过来，如果确定不会被继承（如用 `final` 标记），不需要虚析构（避免 vptr 开销）。

> **面试速答**
>
> 虚析构就是把析构函数标成 `virtual`，保证通过基类指针 `delete` 时能正确调到派生类的析构函数。为什么必须搞这个？因为非虚析构的情况下，`delete` 基类指针只会调基类析构，派生类那部分资源就漏了——而且这在标准里是 UB，不只是内存泄漏那么简单。
>
> 底层机制是：有虚析构时，`delete` 先通过 vptr 在 vtable 里找到对应的析构函数地址，然后逐层调用派生类 → 基类析构；没有虚析构就直接按静态类型定死调用。
>
> 实际工程里，忘记虚析构是最常见的多态 bug 之一。还有个细节要特别注意：纯虚析构必须提供函数体，因为派生类的析构会隐式调用基类析构，没函数体链接就过不去。
>
> 选型很简单：只要类可能被继承、且有人会通过基类指针 delete，析构就必须 `virtual`。反过来，如果确定不会被继承，用 `final` 标掉可以省掉那 8 字节 vptr 的开销。

#### 纯虚函数与抽象类

```cpp
class Shape {
public:
    virtual double area() const = 0;   // 纯虚函数
    virtual ~Shape() = default;
};

// Shape s;  // ❌ 编译错误：不能实例化抽象类

class Circle : public Shape {
    double r;
public:
    Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
};
```

**纯虚函数也可以有实现**（鲜为人知）：
```cpp
class Base {
public:
    virtual void func() = 0;  // 仍然是纯虚函数，Base 仍是抽象类
};

// 提供默认实现，派生类可以显式调用
void Base::func() {
    std::cout << "Base 的默认实现\n";
}

class Derived : public Base {
public:
    void func() override {
        Base::func();  // 显式调用基类的实现
        std::cout << "Derived 的扩展\n";
    }
};
```

> **面试速答**
>
> 纯虚函数就是用 `= 0` 声明的虚函数，含有纯虚函数的类就是抽象类，不能直接实例化，编译器会强制派生类把这些函数实现掉。为什么要搞这东西？就是为了定义纯接口契约，让调用方只依赖抽象，不依赖具体实现，策略模式、工厂模式这些都是建立在它之上的。
>
> 底层实现挺直白：vtable 里对应纯虚函数的槽位填的是 `__cxa_pure_virtual`，真要调进去就直接 abort。派生类把函数实现后，这个槽位就被覆盖成真实地址。
>
> 有个冷知识很多人不知道：纯虚函数其实可以有函数体，用来提供默认实现，派生类通过 `Base::func()` 显式调用即可。还有个要特别注意的点——纯虚析构必须提供定义，否则链接会过不去。
>
> 和 Java interface 比，C++ 抽象类更强大：可以有成员变量，也可以有非虚函数实现，灵活但也更复杂。选型上，需要定义接口层就用纯虚函数 + 虚析构的组合；如果所有函数都是纯虚的，那就是纯接口类。

#### RTTI 和 dynamic_cast 的底层实现

**RTTI（Run-Time Type Information）**：编译器为每个含虚函数的类生成 `type_info` 对象，存放在 vtable 的负偏移位置（vtable[-1]）。

```
vtable 的实际完整布局（GCC/Itanium ABI）：
                        ┌──────────────────────┐
vtable[-2]              │ offset_to_top         │  ← 多继承时 this 指针调整量
                        ├──────────────────────┤
vtable[-1]              │ &type_info for Class  │  ← RTTI 信息
                        ├──────────────────────┤
vtable[0] (vptr 指向)→  │ &虚函数1              │
                        │ &虚函数2              │
                        │ ...                   │
                        └──────────────────────┘
```

**`dynamic_cast` 的过程**：
```cpp
Base* p = getObject();  // 可能是 Base 也可能是 Derived
Derived* d = dynamic_cast<Derived*>(p);
```
1. 通过 `p` 的 vptr 找到 vtable
2. 从 vtable[-1] 取出 `type_info*`
3. 沿继承链遍历 `type_info`，检查目标类型是否在继承路径中
4. 如果匹配成功，用 `offset_to_top` 调整指针并返回
5. 匹配失败：指针返回 `nullptr`，引用抛出 `std::bad_cast`

**性能**：`dynamic_cast` 需要遍历继承链，时间复杂度与继承深度成正比。频繁使用说明设计有问题，应该用虚函数分派替代。

**`typeid` 运算符**：
```cpp
if (typeid(*p) == typeid(Derived)) {
    // 精确类型匹配（不考虑继承关系）
    // 比 dynamic_cast 快，但不做指针调整
}
```

**禁用 RTTI**：编译时加 `-fno-rtti` 可以节省每个多态类一个 `type_info` 的空间，但 `dynamic_cast` 和 `typeid` 将不可用。嵌入式和游戏引擎常这么做。

**什么时候该用 / 不该用 dynamic_cast**：

```
需要向下转型（基类 → 派生类）？
├─ 能用虚函数分派解决吗？ → 用虚函数，不要 dynamic_cast
├─ 类型在编译期已知？ → static_cast（更快，零 RTTI 开销）
├─ 运行时才知道类型，且需要安全检查？ → dynamic_cast
└─ 频繁调用（循环内/热路径）？ → 设计有问题，重构为虚函数分派
```

#### 对象切片（Object Slicing）

按值传递派生类给基类参数时，派生类特有的部分被"切掉"，只保留基类部分。这是多态最常见的陷阱之一：

```cpp
class Animal {
public:
    virtual void speak() { cout << "..." << endl; }
    string name;
};

class Dog : public Animal {
public:
    void speak() override { cout << "汪汪！" << endl; }
    int tricks = 5;   // Dog 特有成员
};

// ❌ 按值传递 → 切片
void makeSpeak(Animal a) {    // 注意：不是引用，是值！
    a.speak();                 // 调的是 Animal::speak，不是 Dog::speak
}

Dog dog;
dog.name = "旺财";
makeSpeak(dog);   // 输出 "..." 而不是 "汪汪！"
                   // dog 的 tricks 成员被切掉了
                   // vptr 也被替换成了 Animal 的 vtable
```

**为什么会切片？** 按值传递时，编译器调用 `Animal` 的拷贝构造函数，只复制 `Animal` 部分的数据。新对象的类型就是 `Animal`，vptr 指向 `Animal` 的 vtable，派生类的成员和行为全部丢失。

```
原始 Dog 对象：                     切片后的 Animal 对象：
┌───────────────────┐              ┌───────────────────┐
│ vptr → Dog_vtable │              │ vptr → Animal_vtable │ ← 被改了！
│ name = "旺财"     │  ──拷贝→     │ name = "旺财"         │
│ tricks = 5        │              └───────────────────────┘
└───────────────────┘                tricks 没了！
```

**三种切片场景**：

```cpp
// 场景 1：函数参数按值传递
void process(Animal a);     // ❌ 切片
void process(Animal& a);    // ✅ 引用，保留多态
void process(Animal* a);    // ✅ 指针，保留多态

// 场景 2：容器存值类型
vector<Animal> animals;
animals.push_back(Dog());   // ❌ 切片！Dog 变成 Animal
vector<unique_ptr<Animal>> animals;  // ✅ 存指针

// 场景 3：赋值
Animal a;
Dog d;
a = d;   // ❌ 切片赋值，a 只有 Animal 部分
```

**防御方法**：将基类的拷贝构造设为 `protected` 或 `= delete`，从编译期阻止切片：

```cpp
class Animal {
public:
    virtual void speak() = 0;
    virtual ~Animal() = default;
protected:
    Animal(const Animal&) = default;  // 外部不能按值拷贝，但派生类可以
};
```

#### 协变返回类型（Covariant Return Types）

虚函数重写时，派生类可以将返回类型从"基类指针/引用"改为"派生类指针/引用"。这是标准允许的特例：

```cpp
class Animal {
public:
    virtual Animal* clone() const {
        return new Animal(*this);
    }
    virtual ~Animal() = default;
};

class Dog : public Animal {
public:
    // 返回类型从 Animal* 改为 Dog*，这是合法的协变返回
    Dog* clone() const override {
        return new Dog(*this);
    }
};

Dog d;
Dog* d2 = d.clone();     // ✅ 直接拿到 Dog*，不需要 dynamic_cast

Animal* a = &d;
Animal* a2 = a->clone(); // ✅ 通过基类指针调用，返回 Animal*（实际是 Dog*）
```

**为什么允许这个？** 因为 `Dog*` 可以隐式转换为 `Animal*`，所以用基类指针调用时仍然类型安全。好处是通过派生类指针调用时，能直接拿到派生类类型，避免向下转型。

**规则**：返回类型必须是类的指针或引用，且派生类的返回类型必须是基类返回类型的派生类。值类型不支持协变。

#### NVI 模式（Non-Virtual Interface）

public 非虚函数调用 private 虚函数，把"接口"和"可定制点"分开：

```cpp
// 传统做法：public virtual，接口和实现混在一起
class Logger {
public:
    virtual void log(const string& msg) {   // 既是接口又是扩展点
        cout << msg << endl;
    }
};

// NVI 做法：public 非虚（接口）+ private 虚（扩展点）
class Logger {
public:
    // 非虚：控制调用流程，保证前后不变量
    void log(const string& msg) {
        if (msg.empty()) return;       // 前置检查（所有子类都遵守）
        doLog(formatMsg(msg));          // 调用可定制的部分
        ++logCount_;                    // 后置操作（所有子类都遵守）
    }

private:
    virtual void doLog(const string& msg) {   // 子类只定制这个
        cout << msg << endl;
    }
    string formatMsg(const string& msg) { return "[INFO] " + msg; }
    int logCount_ = 0;
};

class FileLogger : public Logger {
private:
    void doLog(const string& msg) override {  // 只改实现，不改接口
        file_ << msg << endl;
    }
    ofstream file_{"app.log"};
};
```

**为什么 NVI 比 public virtual 好？**

| 维度 | public virtual | NVI（public 非虚 + private 虚） |
|------|---------------|-------------------------------|
| 前后不变量 | 子类可以跳过，忘了调 `base::log()` 就丢了 | 不可能跳过，非虚函数里写死了 |
| 接口稳定性 | 子类可以改变函数签名（隐藏） | 接口由非虚函数锁定 |
| 扩展性 | 以后想加日志计数？所有子类都要改 | 只改基类的非虚函数 |

**口诀**：公开的不虚，虚的不公开。

## 4. 拷贝语义

### 4.1 深拷贝 vs 浅拷贝

**一句话定义**：
- **浅拷贝**：按位/按成员复制，指针只复制地址值，两个对象共享同一块堆内存
- **深拷贝**：重新分配内存，复制指针指向的实际内容，两个对象各自独立

**为什么编译器默认做浅拷贝？** 编译器生成的拷贝构造/拷贝赋值只做"逐成员拷贝"——对基本类型（int、double）直接复制值，对指针也只复制地址值（不知道指针指向的内存有多大、是不是你申请的、该不该复制）。编译器没有足够的信息判断一个指针成员是"拥有"这块内存还是"借用"的，所以只能做最简单的按位复制，把"是否需要深拷贝"的决定权留给程序员。

#### 完整示例 + 内存图

```cpp
class String {
    char* data_;
    size_t len_;
public:
    // 构造
    String(const char* s) : len_(strlen(s)) {
        data_ = new char[len_ + 1];
        strcpy(data_, s);
    }

    // ❌ 编译器默认生成的拷贝构造 = 浅拷贝
    // String(const String& other) : data_(other.data_), len_(other.len_) {}

    // ✅ 深拷贝构造
    String(const String& other) : len_(other.len_) {
        data_ = new char[len_ + 1];     // 分配新内存
        strcpy(data_, other.data_);      // 复制内容
    }

    // ✅ 深拷贝赋值（copy-and-swap 手法）
    String& operator=(String other) {    // 注意：参数按值传递（触发拷贝构造）
        swap(other);                     // 交换资源
        return *this;                    // other 析构时释放旧资源
    }

    void swap(String& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(len_, other.len_);
    }

    ~String() { delete[] data_; }
};
```

**浅拷贝的内存图**：
```
执行 String b = a; （浅拷贝）后：

栈                          堆
┌──────────┐
│ a.data_ ─┼────────┐
│ a.len_=5 │        │    ┌───────────┐
└──────────┘        ├──→ │ "Hello\0" │  ← 同一块内存！
┌──────────┐        │    └───────────┘
│ b.data_ ─┼────────┘
│ b.len_=5 │
└──────────┘

问题：
1. delete a 后，b.data_ 变成悬空指针
2. a 和 b 的析构函数都会 delete 同一块内存 → double free
3. 修改 a 的数据，b 也会变（意外共享）
```

**深拷贝的内存图**：
```
执行 String b = a; （深拷贝）后：

栈                          堆
┌──────────┐          ┌───────────┐
│ a.data_ ─┼────────→ │ "Hello\0" │  ← a 自己的内存
│ a.len_=5 │          └───────────┘
└──────────┘
┌──────────┐          ┌───────────┐
│ b.data_ ─┼────────→ │ "Hello\0" │  ← b 自己的内存（独立副本）
│ b.len_=5 │          └───────────┘
└──────────┘

各自独立，互不影响
```

#### 4. 优缺点与权衡

**深拷贝：**

**优点**：
- 完全独立：两个对象各自拥有独立的堆内存，修改一个不影响另一个，消除了共享状态导致的一切 bug（double free、悬空指针、数据竞争）
- 线程安全友好：独立副本天然没有数据竞争，不需要额外的同步机制

**缺点/代价**：
- 性能开销大：每次拷贝都要 `new` 分配内存 + `memcpy`/逐元素复制，对于大对象（如几 MB 的 buffer）代价很高，频繁拷贝会成为性能瓶颈
- 内存占用翻倍：每份副本都独立分配，N 份拷贝占 N 倍内存

**浅拷贝：**

**优点**：
- 极快：只复制指针值（8 字节），O(1) 操作，不涉及堆分配
- 内存高效：多个对象共享同一份数据

**缺点/代价**：
- 三大致命问题：double free（两个析构函数释放同一块内存）、悬空指针（一个对象释放后另一个仍持有旧地址）、意外数据共享（修改一个影响另一个）

**权衡**：类持有堆资源时必须实现深拷贝保证正确性；但如果对象频繁传递，优先考虑移动语义（C++11）转移所有权而非深拷贝，或用 COW（Copy-On-Write）/`shared_ptr` 延迟复制

### 4.2 拷贝构造函数

**何时被调用**：
1. 用已有对象初始化新对象：`String b = a;` 或 `String b(a);`
2. 函数参数按值传递：`void foo(String s);`
3. 函数按值返回：`String bar() { String s; return s; }`（可能被 RVO/NRVO 优化掉）

**参数为何必须是 const 引用**：
```cpp
// ✅ 正确
String(const String& other);

// ❌ 如果参数是值传递：
String(String other);  // 传参时需要拷贝 → 调用拷贝构造 → 递归调用自己 → 无限递归！

// ❌ 如果不加 const：
String(String& other);  // 无法接受临时对象（右值），如 String b = getString();
```

### 4.3 拷贝赋值运算符

**自赋值检测**：
```cpp
// 朴素实现（需要自赋值检测）
String& operator=(const String& other) {
    if (this == &other) return *this;  // 自赋值检测
    delete[] data_;                     // 释放旧资源
    len_ = other.len_;
    data_ = new char[len_ + 1];        // 分配新资源
    strcpy(data_, other.data_);
    return *this;
}
```

**copy-and-swap idiom（更优雅、异常安全）**：
```cpp
// 利用拷贝构造 + swap，天然处理自赋值，且提供强异常安全保证
String& operator=(String other) {  // 按值传递：拷贝在参数传递时完成
    swap(other);                   // 交换：noexcept
    return *this;                  // other 析构：释放旧资源
}
// 好处：如果 new 失败（拷贝构造抛异常），原对象不受影响
```

> **面试速答**
>
> 浅拷贝就是复制指针本身的值（两个对象共享同一块资源），深拷贝是把指针指向的实际内容也复制一份（两个对象互相独立）。为什么这事很关键？因为浅拷贝会引出三大问题：double free、悬空指针、还有意外的数据共享；而深拷贝让两个对象完全独立，不会互相影响。
>
> 底层机制是：编译器默认生成的拷贝构造和拷贝赋值是逐成员复制，对指针成员来说就是地址复制，内容不管。所以只要类里有裸指针/文件句柄这种资源，就必须自己实现深拷贝。
>
> 有个手法要特别掌握——copy-and-swap：拷贝赋值的参数按值传递，触发一次拷贝构造，然后和 `*this` 做 swap，原资源在函数返回时随临时对象析构。这个写法天然处理自赋值、而且是强异常安全（如果 new 失败，原对象完好无损）。
>
> 和移动语义对比一下，移动是"资源转移"而不是复制，C++11 之后应该优先支持移动，能避免大量不必要的深拷贝开销。
>
> 选型上：类持有裸指针这类资源时必须实现深拷贝，或者干脆禁用拷贝只允许移动；如果你把资源都交给 `unique_ptr`/`string`/`vector` 这样的 RAII 容器管，编译器生成的默认函数就够用，不用手写。

### 4.4 Rule of Three / Rule of Five / Rule of Zero

| 规则 | 含义 |
|------|------|
| **Rule of Three** | 如果你定义了析构函数、拷贝构造、拷贝赋值中的任何一个，你应该定义全部三个 |
| **Rule of Five**（C++11） | 在 Three 基础上加上移动构造和移动赋值（共 5 个） |
| **Rule of Zero** | 优先使用 RAII 容器（`string`、`vector`、`unique_ptr`），让编译器默认生成的函数就够用，自己一个都不写 |

```cpp
// Rule of Five 完整示例
class Buffer {
    int* data_;
    size_t size_;
public:
    Buffer(size_t n) : data_(new int[n]), size_(n) {}

    ~Buffer() { delete[] data_; }                       // 1. 析构

    Buffer(const Buffer& o) : data_(new int[o.size_]),  // 2. 拷贝构造
                               size_(o.size_) {
        std::copy(o.data_, o.data_ + size_, data_);
    }

    Buffer& operator=(Buffer o) {                       // 3. 拷贝赋值（copy-and-swap）
        swap(o);
        return *this;
    }

    Buffer(Buffer&& o) noexcept                         // 4. 移动构造
        : data_(o.data_), size_(o.size_) {
        o.data_ = nullptr;
        o.size_ = 0;
    }

    Buffer& operator=(Buffer&& o) noexcept {            // 5. 移动赋值
        if (this != &o) {
            delete[] data_;
            data_ = o.data_;
            size_ = o.size_;
            o.data_ = nullptr;
            o.size_ = 0;
        }
        return *this;
    }

    void swap(Buffer& o) noexcept {
        std::swap(data_, o.data_);
        std::swap(size_, o.size_);
    }
};
```

```cpp
// Rule of Zero：用智能指针管理，一个特殊成员函数都不用写
class Buffer {
    std::unique_ptr<int[]> data_;
    size_t size_;
public:
    Buffer(size_t n) : data_(std::make_unique<int[]>(n)), size_(n) {}
    // 编译器自动生成移动构造/移动赋值
    // 拷贝被 unique_ptr 禁用（如果需要拷贝，用 shared_ptr 或显式定义）
};
```

**选型决策树**：
```
你的类管理资源（裸指针、文件句柄等）吗？
├─ 否 → Rule of Zero（什么都不写，用默认的）
└─ 是 → 能用智能指针/容器封装吗？
     ├─ 能 → Rule of Zero（让容器管理资源）
     └─ 不能 → Rule of Five（5 个全写）
```

#### 5. 优缺点与权衡

**优点**：
- 正确性保证：Rule of Five 确保拷贝、移动、析构三种场景下资源管理都是正确的，不会出现浅拷贝导致的 double free 或资源泄漏
- Rule of Zero 的优雅：用 RAII 容器（`unique_ptr`、`string`、`vector`）封装资源后，编译器默认生成的特殊成员函数就是正确的，零样板代码
- 编译器帮你兜底：一旦遵循规则，编译器隐式生成的移动操作可以自动利用成员的移动语义，性能最优

**缺点/代价**：
- 样板代码多：Rule of Five 要求手写 5 个函数（析构 + 拷贝构造 + 拷贝赋值 + 移动构造 + 移动赋值），代码量大且容易出错，尤其是 copy-and-swap、noexcept 标记等细节
- 隐式规则复杂：C++ 的隐式生成规则很绕——定义了析构函数后移动操作不再隐式生成（退化为拷贝），这个规则不直观，容易踩坑导致性能问题
- 维护负担：类新增成员变量时，5 个函数都可能需要同步修改，遗漏任一个就可能引入 bug

**权衡**：现代 C++ 应优先追求 Rule of Zero——用智能指针和标准容器管理资源，让编译器生成的默认函数自动正确；只有在封装最底层的 RAII wrapper（如自定义内存池、文件句柄封装）时才需要 Rule of Five

> **面试速答**
>
> Rule of Three/Five/Zero 是三条指导你何时需要手写特殊成员函数的经验法则。Three 说的是：析构、拷贝构造、拷贝赋值这三个函数，如果你写了一个就应该全写，因为一个类会自定义析构通常意味着在管资源，三个不配套就容易出浅拷贝、资源泄漏这些 bug。Five 是 C++11 的扩展，在 Three 的基础上加上移动构造和移动赋值，共五个。Zero 是现代 C++ 的最佳实践——一个都不手写。
>
> 底层机制上，编译器默认会隐式生成这些函数（逐成员操作）。但有条规则很绕：一旦用户自己定义了析构函数，移动操作就不再被隐式生成了，这是 C++11 的规矩。
>
> 有个最常见的性能坑要特别注意：定义了析构却忘写移动函数，结果就是所有右值都退化成拷贝，性能直接打回 C++98。维护负担也是个问题——类新增成员变量时，这 5 个函数都可能要同步改，遗漏一个就是 bug。
>
> 选型上现代 C++ 应当优先 Rule of Zero：资源都交给 `unique_ptr`/`string`/`vector` 这类 RAII 容器，编译器生成的默认函数就是对的。只有在封装最底层的 RAII wrapper（比如自定义内存池、文件句柄）的时候，才真的需要走 Rule of Five 老老实实把五个都写全。

## 5. 重载 vs 重写 vs 隐藏

| 对比维度 | 重载（Overload） | 重写（Override） | 隐藏（Hide） |
|---------|-----------------|-----------------|-------------|
| 作用域 | 同一个类 | 基类与派生类之间 | 基类与派生类之间 |
| 函数名 | 相同 | 相同 | 相同 |
| 参数列表 | 必须不同 | 必须相同 | 可同可不同 |
| virtual | 不需要 | 基类必须是 virtual | 基类不是 virtual |
| 绑定方式 | 编译期 | 运行期 | 编译期 |
| 关键字 | 无 | override（C++11推荐） | 无 |

### 容易混淆的场景

```cpp
class Base {
public:
    virtual void foo(int x)    { std::cout << "Base::foo(int)\n"; }
    void         bar(int x)    { std::cout << "Base::bar(int)\n"; }
};

class Derived : public Base {
public:
    // 重写：参数列表完全相同，基类是 virtual
    void foo(int x) override   { std::cout << "Derived::foo(int)\n"; }

    // 隐藏（不是重写！）：参数列表不同
    void foo(double x)         { std::cout << "Derived::foo(double)\n"; }

    // 隐藏（不是重写！）：基类 bar 不是 virtual
    void bar(int x)            { std::cout << "Derived::bar(int)\n"; }
};

Derived d;
Base* p = &d;

p->foo(1);      // Derived::foo(int)  — 运行期多态，调派生类
d.foo(1);       // Derived::foo(double)! — 编译期，int 隐式转 double
                // 因为 Derived::foo(double) 隐藏了 Base::foo(int)
d.Base::foo(1); // Base::foo(int) — 显式指定基类版本

p->bar(1);      // Base::bar(int) — 非虚函数，看指针类型
d.bar(1);       // Derived::bar(int) — 看对象类型
```

**陷阱**：派生类中定义了同名函数（不论参数是否相同），会隐藏基类的所有同名函数。想同时暴露基类版本，用 `using Base::foo;`。

```cpp
class Derived : public Base {
public:
    using Base::foo;            // 把基类的 foo 拉到派生类作用域
    void foo(double x) { ... } // 现在 foo(int) 和 foo(double) 都可见
};
```

> **面试速答**
>
> 重载、重写、隐藏这三个概念长得像但本质完全不同。重载是同一个类里函数名相同但参数不同，编译期根据参数类型选调用目标，底层靠 name mangling 把参数编码进符号名来区分。重写是派生类覆盖基类的虚函数，函数名和参数必须一模一样，运行期通过 vtable 查表实现动态分派。隐藏是派生类定义了和基类同名的函数（不管参数是否相同），如果基类那个不是 virtual，就不是重写而是隐藏——编译期按静态类型决定调谁。
>
> 最容易踩的坑就是隐藏：派生类只要定义了同名函数，基类的所有同名重载全部被隐藏。比如基类有 `virtual foo(int)` 和 `foo(double)`，派生类只重写了 `foo(int)`，结果 `foo(double)` 也被隐藏了，通过派生类对象调 `foo(3.14)` 会走到 `foo(int)` 而不是基类的 `foo(double)`。解决办法是在派生类里加 `using Base::foo;` 把基类版本拉进来。
>
> C++11 的 `override` 关键字就是为了防止你以为自己在重写，实际上参数写错了变成隐藏。加了 `override` 编译器会检查基类是否真的有这个虚函数，写错就报错。

## 6. 空类的默认成员函数

编译器在需要时（隐式使用时才生成，称为"隐式声明、隐式定义"）为空类默认生成以下 6 个特殊成员函数：

```cpp
class Empty {
public:
    Empty();                                  // 1. 默认构造函数
    ~Empty();                                 // 2. 析构函数
    Empty(const Empty&);                      // 3. 拷贝构造函数
    Empty& operator=(const Empty&);           // 4. 拷贝赋值运算符
    Empty(Empty&&) noexcept;                  // 5. 移动构造函数（C++11）
    Empty& operator=(Empty&&) noexcept;       // 6. 移动赋值运算符（C++11）
};
```

**关键注意**：
- `sizeof(Empty)` == 1（不是 0，为了保证每个对象有唯一地址）
- 如果你定义了任何构造函数，编译器不再生成默认构造函数
- 如果你定义了拷贝操作，编译器不再隐式生成移动操作
- 如果你定义了析构函数，编译器仍会生成拷贝操作（但 C++11 起这是 deprecated 行为）
- 用 `= default` 显式要求编译器生成，`= delete` 显式禁止

```cpp
class NonCopyable {
public:
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;            // 禁止拷贝
    NonCopyable& operator=(const NonCopyable&) = delete; // 禁止拷贝赋值
    NonCopyable(NonCopyable&&) = default;                // 允许移动
    NonCopyable& operator=(NonCopyable&&) = default;     // 允许移动赋值
};
```

### 委托构造函数（C++11）

构造函数可以在初始化列表中调用同一个类的另一个构造函数，减少重复代码：

```cpp
class Connection {
    string host_;
    int port_;
    int timeout_;
public:
    // 主构造函数：所有逻辑集中在这里
    Connection(const string& host, int port, int timeout)
        : host_(host), port_(port), timeout_(timeout) {
        // 复杂的初始化逻辑（建立连接、校验参数等）
    }

    // 委托给主构造函数
    Connection(const string& host, int port)
        : Connection(host, port, 30) {}     // 默认超时 30s

    Connection(const string& host)
        : Connection(host, 8080) {}          // 默认端口 8080

    // ❌ 不能同时委托和初始化其他成员
    // Connection(const string& host)
    //     : Connection(host, 8080), timeout_(60) {}  // 编译错误！
};
```

**C++11 之前的替代方案**：写一个 private `init()` 函数，每个构造函数都调它——但 `init()` 在构造函数体内执行，成员已经被默认初始化了一次，有额外开销。委托构造直接在初始化阶段完成，没有这个问题。

> **面试速答**
>
> 空类编译器会默认生成 6 个特殊成员函数：默认构造、析构、拷贝构造、拷贝赋值、移动构造、移动赋值。但它们是按需生成的——只有代码里用到了才会真正生成，叫"隐式声明、隐式定义"。
>
> 关键规则是：你手动定义了任何构造函数，默认构造就不再生成；定义了拷贝操作，移动操作就不再生成；定义了析构函数，移动操作也不再生成（C++11 起拷贝操作虽然还生成但已经是 deprecated 行为了）。用 `= default` 可以显式让编译器生成，`= delete` 可以显式禁止。
>
> `sizeof(Empty)` 是 1 不是 0，因为 C++ 标准要求每个对象有唯一地址。但作为基类时，编译器可以做空基类优化（EBO），不占空间。
>
> C++11 还加了委托构造函数——一个构造函数可以在初始化列表里调同类的另一个构造函数，避免重复初始化逻辑。注意不能同时委托又初始化别的成员。

## 7. 运算符重载

### 7.1 概念与本质

运算符重载是让自定义类型支持运算符语法的机制，**本质是特殊的函数调用**：`a + b` 被编译器转换为 `a.operator+(b)` 或 `operator+(a, b)`。

### 7.2 重载规则

**可重载的运算符**：`+` `-` `*` `/` `%` `^` `&` `|` `~` `!` `=` `<` `>` `+=` `-=` `*=` `/=` `%=` `^=` `&=` `|=` `<<` `>>` `>>=` `<<=` `==` `!=` `<=` `>=` `&&` `||` `++` `--` `,` `->*` `->` `()` `[]` `new` `delete`

**不可重载**：`::` `.` `.*` `?:` `sizeof` `typeid`

**必须是成员函数**：`=`、`()`、`[]`、`->`

**建议非成员（友元）**：`+`、`-`、`*`、`==`、`<<` — 保持操作数对称性（左操作数不必是自身类型）

### 7.3 关键运算符实现

#### operator=（赋值运算符）

```cpp
class Buffer {
    int* data_;
    size_t size_;
public:
    // Copy-and-Swap 惯用法：同时解决自赋值和异常安全
    Buffer& operator=(Buffer other) {  // 注意：按值传入（触发拷贝/移动）
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
        return *this;  // other 析构时释放旧资源
    }
};
```

#### operator++（前置 vs 后置）

```cpp
class Counter {
    int val_;
public:
    Counter& operator++() {        // 前置 ++c：返回引用，高效
        ++val_;
        return *this;
    }
    Counter operator++(int) {      // 后置 c++：返回旧值副本，多一次拷贝
        Counter old = *this;
        ++val_;
        return old;
    }
};
// 结论：优先用前置 ++，尤其是迭代器
```

#### operator<<（流输出，必须友元）

```cpp
class Point {
    double x_, y_;
public:
    // 必须是非成员（左操作数是 ostream，不是 Point）
    friend std::ostream& operator<<(std::ostream& os, const Point& p) {
        return os << "(" << p.x_ << ", " << p.y_ << ")";
    }
};
```

#### operator()（仿函数/函数对象）

```cpp
struct Comparator {
    bool reverse;
    bool operator()(int a, int b) const {  // 可携带状态，可被 STL 算法内联
        return reverse ? a > b : a < b;
    }
};
std::sort(v.begin(), v.end(), Comparator{true});
```

#### 类型转换运算符

```cpp
class SmartBool {
    int val_;
public:
    explicit operator bool() const { return val_ != 0; }  // explicit 防止意外转换
};
SmartBool sb{42};
if (sb) { /* OK */ }          // explicit 允许在条件语境中使用
// int x = sb;                // ❌ 编译错误：explicit 禁止隐式转换
```

#### operator<=>（C++20 三路比较）

```cpp
#include <compare>
struct Version {
    int major, minor, patch;
    auto operator<=>(const Version&) const = default;  // 自动生成 <, <=, >, >=, ==, !=
};
```

### 7.4 成员 vs 非成员对比表

| 维度 | 成员函数重载 | 非成员（友元）重载 |
|------|------------|-------------------|
| 左操作数 | 必须是自身类型 | 任意类型 |
| 对称性 | 不对称（`obj + 1` 可以，`1 + obj` 不行） | 对称（两个方向都行） |
| 访问权限 | 直接访问 private | 需要 friend 声明 |
| 必须成员 | `=` `()` `[]` `->` | — |
| 建议非成员 | — | `+` `-` `*` `==` `!=` `<<` `>>` |

### 7.5 优缺点与权衡

**优点**：
- 语法自然直观，`a + b` 比 `a.add(b)` 可读性好
- 配合 STL 算法（如 `sort` 需要 `operator<`、`unordered_map` 需要 `operator==`）

**缺点/代价**：
- 滥用导致代码难懂（如重载 `^` 表示幂运算，违反直觉）
- 运算符优先级和结合性无法改变，可能和自定义语义冲突

**权衡**：只在运算符的语义与内置类型一致时才重载（如数学向量的 `+`）；不确定时用命名函数。

### 7.6 陷阱与实战

**❌ 重载 `&&` `||` 导致短路求值失效**
```cpp
// ❌ 内置 && 有短路：左边 false 就不算右边
// 重载后变成普通函数调用，两边都会求值！
MyBool operator&&(const MyBool& a, const MyBool& b);
```
**✅ 不要重载逻辑运算符 `&&` `||` `,`**

**❌ 隐式类型转换运算符导致意外转换**
```cpp
class Foo {
public:
    operator int() const { return 42; }  // 隐式转换
};
Foo f;
double d = f + 1.5;  // Foo → int → double，可能非预期
```
**✅ 类型转换运算符加 `explicit`**

> **面试速答**
>
> 运算符重载说白了就是让自定义类型也能用 `+`、`==`、`<<` 这种运算符语法。编译器会把 `a + b` 翻译成 `a.operator+(b)` 这样的函数调用。搞这个的目的是让自定义类型用起来和内置类型一样自然，也方便和 STL 算法配合。
>
> 底层机制是：编译器查找成员函数和非成员函数两个候选集，按重载决议规则选最佳匹配。
>
> 这里有几个经典坑要特别注意。第一，重载 `&&` 和 `||` 会丧失短路求值——因为重载后变成普通函数调用，参数都会先求值，这点很反直觉，所以一般不建议重载这两个。第二，类型转换运算符会导致意外的隐式转换，最好加 `explicit` 限制。第三，`operator=` 必须处理自赋值的情况。
>
> 横向对比下几个运算符的位置选择：`=`、`()`、`[]`、`->` 这四个必须是成员函数；`+`、`==`、`<<` 建议写成非成员函数，好处是能保持左右对称性（比如 `1 + obj` 也能工作）。
>
> 选型建议很保守：只在语义和内置类型一致的时候重载；语义不清晰或者容易引起歧义时，老老实实用命名函数，可读性更好。

## 8. 函数指针与回调

### 8.1 概念与本质

函数指针存储函数在代码段中的地址，是 C/C++ 实现回调和策略模式的基础机制。C++ 扩展出了仿函数、Lambda、`std::function` 四种可调用对象。

### 8.2 四种可调用对象

#### 函数指针

```cpp
int add(int a, int b) { return a + b; }
int (*fp)(int, int) = add;   // 函数名即代码段地址
int result = fp(1, 2);       // 通过指针调用
```

#### 函数对象（仿函数）

```cpp
struct Adder {
    int base;
    int operator()(int x) const { return base + x; }  // 重载 () 运算符
};
Adder add5{5};
int result = add5(3);  // 8，可携带状态，编译器可内联
```

#### Lambda 表达式

```cpp
int base = 5;
auto add5 = [base](int x) { return base + x; };  // 捕获状态的匿名函数对象
int result = add5(3);  // 8
```

#### std::function（类型擦除包装器）

```cpp
#include <functional>
std::function<int(int)> fn;
fn = [](int x) { return x * 2; };    // 包装 Lambda
fn = Adder{10};                        // 包装仿函数
// 统一接口，但有堆分配 + 间接调用开销
```

### 8.3 成员函数指针

```cpp
class Foo {
public:
    int bar(int x) { return x + 1; }
};

// 成员函数指针语法——比普通函数指针多一个类限定
int (Foo::*mfp)(int) = &Foo::bar;

Foo obj;
int result = (obj.*mfp)(42);  // 通过对象调用

// 为什么成员函数指针可能是 16 字节？
// 多继承/虚继承时，需要额外存储 this 指针偏移量
```

**现代替代方案**：
```cpp
// std::bind（C++11，不推荐）
auto fn = std::bind(&Foo::bar, &obj, std::placeholders::_1);

// Lambda 捕获（推荐）
auto fn = [&obj](int x) { return obj.bar(x); };

// std::mem_fn
auto fn = std::mem_fn(&Foo::bar);
fn(obj, 42);
```

### 8.4 对比表

| 维度 | 函数指针 | 仿函数 | Lambda | std::function |
|------|---------|--------|--------|---------------|
| 携带状态 | ❌ 无 | ✅ 成员变量 | ✅ 捕获列表 | ✅ 包装任意可调用 |
| 内联优化 | ❌ 间接调用 | ✅ 编译器可内联 | ✅ 编译器可内联 | ❌ 类型擦除阻止内联 |
| 类型 | 具体函数签名 | 每个类是独立类型 | 每个 Lambda 独立类型 | 统一 `std::function<>` |
| 开销 | 1 个指针 | 对象大小 | 对象大小 | 堆分配 + 虚调用 |
| C 兼容 | ✅ | ❌ | ✅（无捕获时） | ❌ |
| 使用场景 | C 回调接口 | STL 算法谓词 | 就地短回调 | 存储异构回调 |

### 8.5 优缺点与权衡

**优点**：
- 函数指针零开销，C ABI 兼容
- 仿函数/Lambda 可内联，性能最佳
- std::function 接口统一，可存储任意可调用对象

**缺点/代价**：
- 函数指针不能携带状态，不能内联
- Lambda 类型匿名，不能跨翻译单元传递（除非用 std::function 包装）
- std::function 有堆分配开销（小对象可能 SBO 优化）、虚调用阻止内联

**权衡**：模板参数接受可调用对象是零开销方案（`template<typename F> void forEach(F fn)`），但增加编译时间和代码膨胀；std::function 是运行时多态方案，灵活但有开销。

### 8.6 选型决策树

```
需要 C 回调兼容吗？
├─ 是 → 函数指针（或无捕获 Lambda）
└─ 否 → 需要存储到容器/成员变量吗？
         ├─ 是 → std::function（运行时多态）
         └─ 否 → 模板参数（编译期多态，零开销）
                  └─ 就地使用 → Lambda
                  └─ 需要复用 → 仿函数
```

### 8.7 陷阱与实战

**❌ std::function 在热路径上使用导致性能下降**
```cpp
// ❌ 每次调用有虚函数开销，无法内联
void process(const std::vector<int>& v, std::function<bool(int)> pred) {
    for (int x : v) if (pred(x)) { /* ... */ }
}
```
**✅ 用模板参数零开销**
```cpp
template<typename Pred>
void process(const std::vector<int>& v, Pred pred) {
    for (int x : v) if (pred(x)) { /* ... */ }  // pred 可内联
}
```

> **面试速答**
>
> C++ 里能当"行为作为参数"用的东西一共四种：函数指针（存的就是代码段地址）、仿函数（重载 `operator()` 的类）、Lambda（语法糖包装的匿名仿函数）、还有 `std::function`（类型擦除的通用包装器）。它们解决的问题都是同一类：实现回调、策略模式、算法定制化。
>
> 底层机制差异很大。函数指针就是一个代码地址，没法带状态。Lambda 编译器会为每个 lambda 表达式生成一个匿名类，捕获的变量就是类的成员，所以能带状态、还能被内联，性能最佳。`std::function` 走的是类型擦除——内部用虚函数或函数指针 + void* 数据的方式包装任意可调用对象，能把不同类型的 callable 放进同一个容器，代价是可能有堆分配（小对象优化能缓解）和虚调用开销。
>
> 有几个坑要特别注意：成员函数指针在多继承情况下可能是 16 字节，因为要存 this 偏移；`std::function` 在热路径上性能是真的差，能不用就不用；`std::bind` 基本已经被 Lambda 完全取代，新代码别再写了。
>
> 横向对比下：模板参数是编译期多态，零开销但会代码膨胀；`std::function` 是运行时多态，灵活但有开销。选型口诀：和 C 接口交互用函数指针；传给 STL 算法或自己写的模板用 Lambda；需要把回调存起来用 `std::function`；性能敏感的热路径改用模板参数。

## 9. 自测问题

### Q1: vptr 在什么时候被设置？构造函数中调用虚函数会怎样？

> [!note]- 答案
> vptr 在每个构造函数的隐式初始化阶段被设置。基类构造函数将 vptr 指向基类 vtable，派生类构造函数再将 vptr 改写为派生类 vtable。因此在基类构造函数中调用虚函数，调用的是基类版本，不会表现出多态。析构函数中同理——vptr 在每层析构开始时被改回当前类的 vtable。
>
> **详细拆解：**
>
> 先看一个反直觉的例子：
>
> ```cpp
> class Animal {
> public:
>     Animal() {
>         cout << "Animal构造" << endl;
>         speak();   // 基类构造函数里调虚函数
>     }
>     virtual void speak() { cout << "..." << endl; }
>     virtual ~Animal() {}
> };
>
> class Dog : public Animal {
> public:
>     Dog() {
>         cout << "Dog构造" << endl;
>         speak();   // 派生类构造函数里调虚函数
>     }
>     void speak() override { cout << "汪汪！" << endl; }
> };
>
> Dog dog;
> // 输出：
> // Animal构造
> // ...          ← 不是 "汪汪！"，调的是 Animal::speak
> // Dog构造
> // 汪汪！       ← 这里才调 Dog::speak
> ```
>
> 为什么在 Animal 构造函数里 `speak()` 调的不是 Dog 版本？因为 vptr 在每一层构造函数中会被改写：
>
> ```
> Dog dog; 触发的完整过程：
>
> 第 1 步：分配内存
> ┌──────────────────────┐
> │ vptr = ???（未初始化） │
> │ Animal 的数据         │
> │ Dog 的数据            │
> └──────────────────────┘
>
> 第 2 步：进入 Animal 的构造函数
>   编译器插入隐藏代码：this->vptr = &Animal_vtable  ← 指向 Animal 的表
> ┌──────────────────────────────┐
> │ vptr = &Animal_vtable ───────┼──→ [0] = &Animal::speak
> │ Animal 的数据                 │
> │ Dog 的数据（还没初始化）       │
> └──────────────────────────────┘
>   执行 Animal() 函数体：
>     speak() → vptr 指向 Animal_vtable → 查表 → Animal::speak → 输出 "..."
>
> 第 3 步：进入 Dog 的构造函数
>   编译器插入隐藏代码：this->vptr = &Dog_vtable    ← 覆盖为 Dog 的表
> ┌──────────────────────────────┐
> │ vptr = &Dog_vtable ──────────┼──→ [0] = &Dog::speak
> │ Animal 的数据                 │
> │ Dog 的数据                    │
> └──────────────────────────────┘
>   执行 Dog() 函数体：
>     speak() → vptr 指向 Dog_vtable → 查表 → Dog::speak → 输出 "汪汪！"
> ```
>
> **vptr 被写了两次**：Animal 构造时写一次，Dog 构造时覆盖一次。
>
> **为什么要这么设计？** 因为基类构造函数执行时，**派生类还不存在**——Dog 的数据成员还没初始化，Dog 的构造函数还没跑。如果此时调 `Dog::speak()`，而 `speak()` 里用了 Dog 的成员变量，就会访问未初始化的内存。所以 C++ 的规则是：**在 Animal 构造函数里，这个对象"还只是一个 Animal"**，vptr 指向 Animal 的 vtable，虚函数只走 Animal 的版本。
>
> **析构时完全反过来**——vptr 从派生类逐层降级回基类：
>
> ```
> dog 析构时：
>
> 第 1 步：进入 Dog 的析构函数
>   编译器插入：this->vptr = &Dog_vtable（其实已经是了）
>   执行 ~Dog() 函数体
>   Dog 的成员被销毁
>
> 第 2 步：进入 Animal 的析构函数
>   编译器插入：this->vptr = &Animal_vtable  ← 改回 Animal 的表
>   执行 ~Animal() 函数体
>   如果这里调 speak() → 调的是 Animal::speak，不是 Dog::speak
>   因为 Dog 的部分已经销毁了，调 Dog::speak 可能访问已释放的内存
> ```
>
> **一句话总结**：vptr 在构造时从基类到派生类**逐层升级**，析构时从派生类到基类**逐层降级**。在每一层，虚函数调用只走到当前层。这就是为什么**构造函数和析构函数里不要依赖多态**。

### Q2: 为什么 C++ 的函数重载不考虑返回值类型？

> [!note]- 答案
> 因为调用函数时可以忽略返回值（`foo();` 不使用返回值），此时编译器无法仅凭返回值类型区分该调用哪个重载版本。Name mangling 也不包含返回值类型。

### Q3: dynamic_cast 和 static_cast 有什么区别？什么时候用哪个？

> [!note]- 答案
> `static_cast` 在编译期做类型转换，不检查运行时类型，如果转换错误会产生未定义行为。`dynamic_cast` 通过 RTTI（访问 vtable[-1] 的 type_info）在运行时检查类型安全性，转换失败指针返回 nullptr、引用抛 bad_cast。向下转型（基类→派生类）且不确定类型时用 dynamic_cast，确定类型时用 static_cast（更快）。

### Q4: 菱形继承中，虚基类的构造函数由谁负责调用？

> [!note]- 答案
> 由最终派生类（most derived class）负责调用虚基类的构造函数，中间类对虚基类构造函数的调用会被忽略。这就是为什么虚继承时，最终派生类的初始化列表中必须显式调用虚基类的构造函数（如果虚基类没有默认构造函数）。

### Q5: 多继承时，将派生类指针赋给第二个基类的指针，指针值会变吗？

> [!note]- 答案
> 会变。编译器会自动调整指针值，使其指向对象中第二基类子对象的起始位置。例如 `C` 继承 `A` 和 `B`，`B* pb = &c;` 时 `pb` 的值比 `&c` 大了一个 `A` 子对象的大小。这种调整在编译期完成（对于 static_cast）。

### Q6: 浅拷贝会导致什么具体问题？至少说三个。

> [!note]- 答案
> 1. **double free**：两个对象的析构函数都会 delete 同一块内存。2. **悬空指针**：一个对象析构后，另一个对象的指针变成野指针。3. **意外共享**：修改一个对象的数据，另一个对象的数据也跟着变（违背值语义）。4. **内存泄漏**：赋值操作中如果不先释放左侧对象的旧资源，旧资源就泄漏了。

### Q7: 拷贝赋值运算符的参数能用值传递吗？这和 copy-and-swap 有什么关系？

> [!note]- 答案
> 拷贝赋值的参数可以用值传递，这正是 copy-and-swap idiom 的实现方式：`T& operator=(T other)`。值传递时拷贝构造已经创建了一份副本，然后 swap 交换资源，旧资源随 `other` 析构而释放。好处：天然处理自赋值、强异常安全保证、代码简洁。注意：拷贝构造函数的参数不能用值传递（否则无限递归）。

### Q8: 什么是 thunk？多继承时为什么需要它？

> [!note]- 答案
> thunk 是编译器自动生成的一小段代码（通常几条指令），用于在多继承场景下调整 this 指针。当通过第二个（或更后面的）基类指针调用虚函数时，this 指针指向的是该基类子对象的位置，而不是完整对象的起始。thunk 先将 this 减去一个偏移量，调整回对象起始，然后跳转到真正的虚函数实现。

### Q9: `extern "C"` 做了什么？为什么 C++ 需要它？

> [!note]- 答案
> `extern "C"` 告诉编译器不要对函数名做 name mangling，而是使用 C 风格的简单符号名。这样 C 代码的链接器就能找到这个函数。C++ 需要 name mangling 来支持函数重载（同名函数生成不同符号），但 C 没有 name mangling，所以跨语言调用时必须用 `extern "C"` 消除 mangling。

### Q10: 虚函数表存放在内存的什么位置？为什么？

> [!note]- 答案
> vtable 存放在 `.rodata`（只读数据段）。因为 vtable 的内容在编译期就完全确定了，运行时不会修改。放在只读段可以：1. 防止被意外或恶意修改（安全性）；2. 多个进程加载同一个共享库时，只读页面可以共享（节省内存）。

### Q11: Rule of Zero 是什么？为什么现代 C++ 推荐它？

> [!note]- 答案
> Rule of Zero：如果一个类的资源管理可以完全交给成员（如 `unique_ptr`、`string`、`vector`），就不需要自定义任何特殊成员函数（析构、拷贝构造、拷贝赋值、移动构造、移动赋值），让编译器默认生成即可。推荐它是因为：1. 减少样板代码；2. 消除忘记实现某个函数的风险；3. 编译器默认生成的函数通常是正确且高效的。

### Q12: 为什么构造函数不能是虚函数？

> [!note]- 答案
> 虚函数调用依赖 vptr，而 vptr 是在构造函数中被初始化的。构造函数执行之前，对象的内存空间还没有 vptr，虚函数机制无法工作。此外，构造函数的职责是创建对象，调用时对象类型必须已知（编译器需要知道要分配多少内存、初始化哪些成员），不存在"根据运行时类型决定如何构造"的语义。如果需要类似"虚构造"的效果，用工厂模式或 clone 虚函数。

### Q13: 隐藏（hide）和重写（override）最容易搞混的场景是什么？

> [!note]- 答案
> 派生类定义了一个和基类虚函数**同名但参数不同**的函数。这不是重写（参数不同），而是隐藏——它会把基类的所有同名函数都隐藏掉。通过基类指针调用时，走的还是虚函数分派（调基类版本）；通过派生类对象直接调用时，只能看到派生类版本。解决办法是在派生类中加 `using Base::foo;`。

### Q14: 在虚继承中，`sizeof` 一个派生类对象会比普通继承大吗？为什么？

> [!note]- 答案
> 通常会更大。虚继承的每个直接虚继承子类需要额外存储一个 vbptr（虚基类指针，8 字节/64位），用来在运行时查找虚基类子对象的偏移。同时虚基类子对象被移到对象末尾单独存放，中间可能有对齐填充。相比普通继承（虚基类数据直接内嵌在子类中，无需 vbptr），空间开销更大。这是用空间和间接寻址换取菱形继承二义性消除的代价。

### Q15: 如果一个类既有虚函数又虚继承了基类，对象中 vptr 和 vbptr 是什么关系？

> [!note]- 答案
> 它们是两个独立的指针，指向不同的表。vptr 指向虚函数表（vtable），用于虚函数分派；vbptr 指向虚基类表（vbtable），用于定位虚基类子对象的偏移。一个对象中可能同时存在 vptr 和 vbptr（分别占一个指针大小）。在某些编译器实现中（如 MSVC），它们是分开的；在其他实现中（如 GCC/Itanium ABI），虚基类偏移信息可能合并到 vtable 中（通过 vcall offset），此时不需要单独的 vbptr。

### 虚函数限制速查

| 能否是虚函数？ | 答案 | 原因 |
|--------------|------|------|
| 普通函数（非成员） | 不能 | 虚函数必须属于类 |
| 构造函数 | 不能 | vptr 尚未初始化 |
| 静态成员函数 | 不能 | 没有 this 指针，无法通过 vptr 分派 |
| 内联函数 | 可以声明为 virtual，但多态调用时不会内联 | 编译器只在非多态调用时才内联 |
| 析构函数 | 可以且推荐 | 基类指针 delete 派生类对象时必须 |
| 模板成员函数 | 不能 | 模板实例化数量编译期未知，无法确定 vtable 大小（见下方解释） |

**为什么模板成员函数不能是虚函数？**

vtable 在编译期就要确定大小（多少个槽位）。但模板函数的实例化数量取决于调用方用了多少种类型，编译器无法在定义类的时候知道：

```cpp
class Base {
public:
    template<typename T>
    virtual void process(T val);  // ❌ 编译错误
    // 如果允许，vtable 需要多少项？
    // process<int>、process<double>、process<string>...
    // 编译器：我怎么知道将来有多少种 T？
};
```

每个翻译单元（.cpp 文件）可能用不同的 `T` 实例化，链接器无法把所有翻译单元的实例化汇总成一张大小固定的 vtable。这和虚函数的 vtable 机制从根本上冲突。
