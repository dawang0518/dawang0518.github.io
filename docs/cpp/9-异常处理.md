# C++ 异常处理

> 一句话定位：异常处理 = 把"检测错误"和"处理错误"在调用链上彻底分离，让正常逻辑不被错误处理代码淹没。

---

## 0. 全景地图

### C++ 错误处理的演进

```
C 时代                     C++ 时代                        现代 C++
───────────────────────    ──────────────────────────      ─────────────────────────
errno (全局变量)            try / catch / throw             noexcept (C++11)
setjmp / longjmp           异常规范 throw(T...)            noexcept(expr) 条件规范
返回值约定 (-1/NULL)        std::exception 层次体系         std::exception_ptr (C++11)
                           dynamic exception spec (废弃)    std::expected<T,E> (C++23)
```

### 三种错误处理范式定位

| 维度 | 错误码/errno | C++ 异常 | std::expected (C++23) |
|------|-------------|---------|----------------------|
| **错误传播** | 手动逐层检查返回值 | 自动沿调用栈展开 | 手动但类型安全 |
| **正常路径开销** | 每次都要检查 | 零开销（table-based） | 和错误码相当 |
| **异常路径开销** | 无额外开销 | 较大（查表+栈展开） | 无额外开销 |
| **强制处理** | 不强制，容易忽略 | 不处理就终止程序 | [[nodiscard]] 可提醒 |
| **跨边界** | 安全（ABI 稳定） | 不安全（跨 DLL/语言困难） | 安全 |
| **适用场景** | 性能敏感/底层库/C 接口 | 罕见的、不可忽略的错误 | 预期内的可恢复错误 |

> **面试速答**
> - **是什么/本质**：C++ 异常处理是一种将错误检测点和错误处理点解耦的机制，通过栈展开自动传播错误
> - **解决什么问题**：解决 C 语言中返回值传递错误层层检查、容易遗漏的问题
> - **底层原理**：现代编译器用 table-based（零开销）模型，正常路径不插入任何额外指令，异常路径通过查找异常表跳转
> - **优缺点**：优——正常路径零开销、错误不会被忽略、代码清晰；缺——异常路径慢、二进制体积增大、不能跨 ABI 边界
> - **坑点**：不同编译器的异常 ABI 不兼容，跨 DLL 传异常是未定义行为
> - **横向对比**：异常适合罕见错误，错误码适合高频/底层，std::expected 适合预期内可恢复错误
> - **选型建议**：应用层用异常，性能关键路径用错误码，C++23 项目优先考虑 std::expected

## 1. 异常机制基础

### 1.1 try / catch / throw 语法

```cpp
#include <iostream>
#include <stdexcept>
#include <string>

double divide(double a, double b) {
    if (b == 0.0) {
        throw std::invalid_argument("除数不能为零");  // 抛出异常对象
    }
    return a / b;
}

int main() {
    try {
        double result = divide(10.0, 0.0);
        std::cout << "结果: " << result << std::endl;
    }
    catch (const std::invalid_argument& e) {   // 精确匹配
        std::cerr << "参数错误: " << e.what() << std::endl;
    }
    catch (const std::exception& e) {          // 基类兜底
        std::cerr << "异常: " << e.what() << std::endl;
    }
    catch (...) {                               // 万能兜底
        std::cerr << "未知异常" << std::endl;
    }
    return 0;
}
```

### 1.2 异常匹配规则

catch 块按**声明顺序**从上到下匹配，匹配规则如下：

| 优先级 | 匹配方式 | 示例 |
|--------|---------|------|
| 1 | 精确类型匹配 | `catch(std::invalid_argument&)` |
| 2 | 基类引用匹配 | `catch(std::exception&)` 可捕获所有派生类 |
| 3 | 指针的 void* 匹配 | `catch(void*)` 可匹配任意指针类型异常 |
| 4 | `catch(...)` 万能兜底 | 匹配任何类型 |

**关键规则**：派生类 catch 必须写在基类 catch 之前，否则永远匹配不到。

### 1.3 重新抛出（rethrow）

```cpp
void process() {
    try {
        riskyOperation();
    }
    catch (const std::exception& e) {
        log(e.what());   // 记录日志
        throw;           // 重新抛出原始异常（保持多态类型）
        // throw e;      // ❌ 这会切片！只抛出 std::exception 部分
    }
}
```

`throw;` 和 `throw e;` 的区别：

| 写法 | 行为 | 是否保留派生类信息 |
|------|------|------------------|
| `throw;` | 重新抛出原始异常对象 | 是 |
| `throw e;` | 拷贝构造一个新的 std::exception 对象 | 否（对象切片） |

### 1.4 标准异常层次

```
std::exception
├── std::logic_error              // 程序逻辑错误（可预防）
│   ├── std::invalid_argument     // 无效参数
│   ├── std::domain_error         // 数学域错误
│   ├── std::length_error         // 超出长度限制
│   └── std::out_of_range         // 下标越界
├── std::runtime_error            // 运行时错误（不可预防）
│   ├── std::range_error          // 值超出范围
│   ├── std::overflow_error       // 算术溢出
│   └── std::underflow_error      // 算术下溢
├── std::bad_alloc                // new 失败
├── std::bad_cast                 // dynamic_cast 引用失败
├── std::bad_typeid               // typeid 作用于空指针
└── std::bad_exception            // unexpected handler 中使用
```

**设计哲学**：`logic_error` 是程序 bug（应该修代码），`runtime_error` 是外部环境导致的（应该捕获处理）。

> **面试速答**
> - **是什么/本质**：try/catch/throw 是 C++ 的结构化异常处理语法，throw 创建异常对象并触发栈展开，catch 按类型匹配并处理
> - **解决什么问题**：将错误检测（throw）和错误处理（catch）在代码结构上分离
> - **底层原理**：throw 调用 __cxa_allocate_exception 分配异常对象，然后 __cxa_throw 启动栈展开流程
> - **优缺点**：优——代码结构清晰，错误不会被忽略；缺——catch 顺序敏感，throw e 有切片陷阱
> - **坑点**：catch 基类写在前面会屏蔽派生类；throw e 会切片丢失派生信息，应该用 throw;
> - **横向对比**：logic_error 是可预防的 bug，runtime_error 是不可预防的外部错误
> - **选型建议**：自定义异常继承 std::runtime_error，用 const 引用捕获，派生类 catch 写在前面

## 2. 底层原理：栈展开（Stack Unwinding）

### 2.1 从 throw 到 catch 的完整流程

```
throw MyException("error")
    │
    ▼
① __cxa_allocate_exception()   // 在异常专用堆上分配内存
    │
    ▼
② 在分配的内存上构造异常对象
    │
    ▼
③ __cxa_throw()                // 启动栈展开
    │
    ▼
④ _Unwind_RaiseException()    // libunwind 开始遍历栈帧
    │
    ├─── 第一遍：搜索阶段（Search Phase）
    │    遍历调用栈，对每个栈帧调用 personality routine
    │    查找匹配的 catch 块（只查找，不执行析构）
    │
    ├─── 找到匹配 → 进入第二遍
    │    找不到 → 调用 std::terminate()
    │
    └─── 第二遍：清理阶段（Cleanup Phase）
         从 throw 点到 catch 点，逐帧执行：
         ① 调用当前帧中所有局部对象的析构函数（逆序）
         ② personality routine 读取 LSDA 确定清理动作
         ③ 跳到下一帧继续
         到达 catch 帧后，跳入 catch 代码块执行
```

### 2.2 零开销异常模型（Table-Based / Itanium ABI）

这是 GCC/Clang 在 x86-64 上的默认模型。

**核心思想**：编译器在编译时生成异常表（`.gcc_except_table`），记录每个函数中哪些代码区域有什么 catch 处理器和清理动作。正常执行时完全不碰这些表，只有抛异常时才查表。

```
正常路径：
  函数调用 → 执行代码 → 返回
  零额外指令，零额外分支，零性能损失

异常路径：
  throw → 分配异常对象 → 遍历 .eh_frame 找栈帧信息
        → 查 .gcc_except_table 找 landing pad
        → 逐帧析构局部对象 → 跳到 catch 处理器
```

### 2.3 异常表结构

```
.gcc_except_table (LSDA - Language Specific Data Area)
├── Call Site Table     // 调用点表：代码区间 → landing pad 映射
│   ├── [起始偏移, 长度] → landing_pad_1, action_1
│   ├── [起始偏移, 长度] → landing_pad_2, action_2
│   └── ...
├── Action Table        // 动作表：catch 类型过滤器列表
│   ├── type_filter_1 → 下一个动作
│   └── type_filter_2 → 0 (结束)
└── Type Table          // 类型表：typeinfo 指针数组
    ├── &typeid(MyException)
    └── &typeid(std::exception)

.eh_frame                // 栈帧展开信息（DWARF CFI）
├── CIE (Common Info Entry)    // 公共信息
└── FDE (Frame Desc Entry)     // 每个函数一条，描述如何恢复寄存器/栈指针
```

### 2.4 Table-Based vs SJLJ 模型对比

| 维度 | Table-Based（零开销） | SJLJ（setjmp/longjmp） |
|------|---------------------|----------------------|
| **正常路径开销** | 零（不插入任何代码） | 每个 try 块调用 setjmp（开销大） |
| **异常路径开销** | 较大（查表+遍历栈帧） | 较小（直接 longjmp） |
| **二进制体积** | 增大（需要异常表段） | 较小 |
| **平台支持** | x86/x86-64/ARM64 主流平台 | 老平台/某些嵌入式 |
| **当前状态** | GCC/Clang 默认 | 基本已弃用 |

**为什么零开销模型胜出**：异常应该是"异常"情况，正常路径执行频率远高于异常路径。Table-Based 模型把开销完全转移到罕见的异常路径上，符合"为常见情况优化"的工程原则。

> **面试速答**
> - **是什么/本质**：栈展开是异常抛出后，运行时从 throw 点到 catch 点逐帧销毁局部对象的过程
> - **解决什么问题**：保证异常传播路径上所有 RAII 资源被正确释放，防止资源泄漏
> - **底层原理**：两阶段展开——搜索阶段找 catch、清理阶段逐帧析构；依赖 .eh_frame（DWARF CFI）和 .gcc_except_table（LSDA）
> - **优缺点**：Table-Based 正常路径零开销，但增大二进制体积 10-30%；异常路径慢（微秒级）
> - **坑点**：如果析构函数在栈展开期间抛异常，程序直接 std::terminate
> - **横向对比**：Table-Based（正常路径零开销，异常路径慢）vs SJLJ（正常路径有开销，异常路径快）
> - **选型建议**：现代平台一律用 Table-Based，不需要手动选择

## 3. 异常安全

### 3.1 三级异常安全保证

| 保证级别 | 含义 | 典型手段 | 标准库示例 |
|---------|------|---------|-----------|
| **不抛出保证（Nothrow）** | 操作绝不抛异常 | noexcept、swap、析构 | `vector::clear()`, `swap()` |
| **强保证（Strong）** | 异常后状态回滚到操作前 | copy-and-swap、事务语义 | `vector::push_back()`（有时） |
| **基本保证（Basic）** | 异常后对象有效但状态不确定 | RAII + 不泄漏资源 | 大多数标准库操作 |

### 3.2 copy-and-swap 实现强保证

```cpp
#include <algorithm>  // std::swap
#include <cstring>

class String {
    char* data_;
    size_t size_;

public:
    // 构造函数：可能抛异常（new 可能失败）
    String(const char* str = "")
        : size_(std::strlen(str))
        , data_(new char[size_ + 1])  // 可能抛 bad_alloc
    {
        std::memcpy(data_, str, size_ + 1);
    }

    // 拷贝构造：可能抛异常
    String(const String& other)
        : size_(other.size_)
        , data_(new char[size_ + 1])  // 可能抛 bad_alloc
    {
        std::memcpy(data_, other.data_, size_ + 1);
    }

    // 析构：不抛异常
    ~String() noexcept {
        delete[] data_;
    }

    // swap：不抛异常（只交换指针和整数）
    friend void swap(String& a, String& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    // copy-and-swap 赋值运算符 —— 提供强保证
    String& operator=(String other) {  // 注意：按值传参（拷贝发生在参数传递时）
        swap(*this, other);            // noexcept 交换
        return *this;                  // other 析构释放旧资源
    }
    // 如果拷贝构造失败（new 抛异常），*this 完全不变 → 强保证
    // 如果拷贝成功，swap 不会失败 → 要么全部成功，要么状态不变
};
```

**强保证的关键**：先在"旁边"准备好新状态（拷贝），然后用 noexcept 操作（swap）一步切换。任何失败都发生在准备阶段，不影响原对象。

### 3.3 标准库容器的异常安全

| 操作 | 保证级别 | 说明 |
|------|---------|------|
| `vector::push_back()` | 强保证 | 扩容失败则状态不变 |
| `vector::insert()` | 基本保证 | 元素可能已经移动 |
| `vector::clear()` | 不抛出 | 只调用析构函数（必须 noexcept） |
| `map::insert()` | 强保证 | 节点分配失败则不修改 |
| `swap()` | 不抛出 | 交换内部指针 |

> **面试速答**
> - **是什么/本质**：异常安全是对函数在异常发生时行为的承诺，分为基本/强/不抛出三级
> - **解决什么问题**：保证异常发生后程序状态可控，资源不泄漏
> - **底层原理**：基本保证靠 RAII，强保证靠 copy-and-swap（先复制再交换），不抛出保证靠 noexcept
> - **优缺点**：强保证安全但有拷贝开销；基本保证开销小但调用者需要处理不确定状态
> - **坑点**：以为 push_back 总是强保证——当移动构造不是 noexcept 时会退化为拷贝
> - **横向对比**：基本保证（最低要求）→ 强保证（事务语义）→ 不抛出（最强承诺）
> - **选型建议**：所有代码至少满足基本保证；关键操作用 copy-and-swap 实现强保证；析构/swap/移动操作必须 noexcept

## 4. noexcept

### 4.1 两种用法：规范 vs 运算符

```cpp
// ① noexcept 规范（specifier）：声明函数不会抛异常
void func() noexcept;           // 等价于 noexcept(true)
void func2() noexcept(false);   // 可能抛异常（和不写 noexcept 类似）

// ② noexcept 运算符（operator）：编译期检查表达式是否声明为 noexcept
constexpr bool is_ne = noexcept(func());    // true
constexpr bool is_ne2 = noexcept(func2());  // false

// ③ 条件 noexcept：根据模板参数的特性决定
template<typename T>
void wrapper(T& a, T& b) noexcept(noexcept(a.swap(b))) {
    a.swap(b);  // 当且仅当 T::swap 是 noexcept 时，wrapper 也是 noexcept
}
```

### 4.2 noexcept 与移动语义的关系

**核心问题：vector 扩容时为什么要求 noexcept 移动构造？**

```cpp
// vector 扩容过程：
// 1. 分配更大的新内存
// 2. 把旧元素"搬"到新内存
// 3. 释放旧内存

// 如果用移动构造搬元素，搬到一半抛异常了：
//   - 旧内存的一部分元素已经被移走（处于 moved-from 状态）
//   - 新内存的一部分元素已经构造好
//   - 无法恢复原始状态！→ 违反强保证

// 所以 std::vector 的策略：
//   - 移动构造是 noexcept → 用移动（高效）
//   - 移动构造不是 noexcept → 退化为拷贝（安全但慢）
```

```cpp
#include <vector>
#include <iostream>

class Good {
public:
    Good() = default;
    Good(Good&&) noexcept {                // ✅ noexcept 移动
        std::cout << "移动构造" << std::endl;
    }
    Good(const Good&) {
        std::cout << "拷贝构造" << std::endl;
    }
};

class Bad {
public:
    Bad() = default;
    Bad(Bad&&) {                            // ❌ 没有 noexcept
        std::cout << "移动构造" << std::endl;
    }
    Bad(const Bad&) {
        std::cout << "拷贝构造" << std::endl;
    }
};

int main() {
    std::vector<Good> v1;
    v1.reserve(1);
    v1.emplace_back();
    v1.emplace_back();   // 扩容 → 调用移动构造 ✅

    std::vector<Bad> v2;
    v2.reserve(1);
    v2.emplace_back();
    v2.emplace_back();   // 扩容 → 退化为拷贝构造 ❌ 性能损失
    return 0;
}
```

### 4.3 违反 noexcept 的后果

```cpp
void dangerous() noexcept {
    throw std::runtime_error("oops");  // 声明了 noexcept 却抛异常
}

int main() {
    try {
        dangerous();      // 不会被 catch 捕获！
    }
    catch (...) {
        // 永远到不了这里
    }
    return 0;
}
// 程序直接调用 std::terminate() → 默认调用 std::abort()
// 栈不会展开，局部对象的析构函数不保证被调用（实现定义）
```

### 4.4 哪些函数应该加 noexcept

| 函数类型 | 是否加 noexcept | 原因 |
|---------|----------------|------|
| 析构函数 | **必须**（默认隐式 noexcept） | 栈展开时析构抛异常 → terminate |
| 移动构造/移动赋值 | **强烈建议** | vector 扩容需要 noexcept 移动 |
| swap 函数 | **强烈建议** | copy-and-swap 依赖 noexcept swap |
| 内存释放/关闭资源 | **应该** | 失败了也没什么能做的 |
| 普通业务函数 | **谨慎** | 将来可能需要抛异常，加了就不能去掉 |

> **面试速答**
> - **是什么/本质**：noexcept 是 C++11 引入的异常规范，声明函数不会抛出异常
> - **解决什么问题**：让编译器和标准库能利用"不抛异常"这个信息做优化（特别是移动语义）
> - **底层原理**：编译器可以省略异常处理相关的栈帧信息；违反 noexcept 直接 terminate
> - **优缺点**：优——启用移动优化、减小二进制体积；缺——一旦声明就不能再抛异常，接口约束永久化
> - **坑点**：违反 noexcept 不是编译错误而是运行时 terminate；移动构造忘加 noexcept 导致 vector 退化为拷贝
> - **横向对比**：noexcept 替代了旧的 throw() 规范（C++17 废弃 throw()，C++20 移除）
> - **选型建议**：析构/移动/swap 必加；叶子函数确定不抛可以加；接口函数谨慎添加

## 5. 异常 vs 错误码

### 5.1 全面对比

| 维度 | 异常 | 错误码 | std::expected (C++23) |
|------|-----|--------|----------------------|
| **语法负担** | 低（正常路径干净） | 高（每次调用都要检查） | 中等（需要解包） |
| **正常路径性能** | 零开销 | 每次检查有分支 | 和错误码相当 |
| **异常路径性能** | 慢（微秒级） | 快（只是返回值） | 快 |
| **能否忽略** | 不能（会 terminate） | 能（容易忘记检查） | 难（[[nodiscard]]） |
| **多层传播** | 自动（栈展开） | 手动逐层转发 | 手动（monadic 操作简化） |
| **二进制体积** | 增大 10-30% | 无影响 | 无影响 |
| **调试难度** | 较高（栈展开后信息丢失） | 低（逐行跟踪） | 低 |
| **跨 ABI 边界** | 危险 | 安全 | 安全 |
| **RTTI 依赖** | 是 | 否 | 否 |

### 5.2 Google/LLVM 禁用异常的原因

Google C++ Style Guide 明确禁用异常，主要原因：

1. **历史包袱**：大量遗留代码不是异常安全的，启用异常会导致资源泄漏
2. **二进制体积**：异常表增大二进制约 10-30%，对大规模部署是显著成本
3. **可预测性**：异常路径的执行流程不直观，增加代码审查和调试难度
4. **跨团队一致性**：几千人的代码库，异常安全很难全局保证

**替代方案**：
- `absl::Status` / `absl::StatusOr<T>`：类似 std::expected 的错误处理
- 返回值 + `ABSL_CHECK` 宏（致命错误直接 abort）

### 5.3 std::expected (C++23)

```cpp
#include <expected>
#include <string>
#include <iostream>

enum class ParseError {
    InvalidFormat,
    OutOfRange,
    Empty
};

// 返回值要么是 int（成功），要么是 ParseError（失败）
std::expected<int, ParseError> parse_int(const std::string& s) {
    if (s.empty()) {
        return std::unexpected(ParseError::Empty);
    }
    try {
        int val = std::stoi(s);
        return val;  // 隐式转换为 expected
    }
    catch (...) {
        return std::unexpected(ParseError::InvalidFormat);
    }
}

int main() {
    auto result = parse_int("42");
    if (result) {
        std::cout << "解析成功: " << *result << std::endl;
    } else {
        std::cout << "解析失败" << std::endl;
    }

    // monadic 操作（链式调用）
    auto doubled = parse_int("21")
        .transform([](int v) { return v * 2; })        // 成功时变换
        .or_else([](ParseError e) -> std::expected<int, ParseError> {
            std::cerr << "错误" << std::endl;
            return std::unexpected(e);
        });
    return 0;
}
```

> **面试速答**
> - **是什么/本质**：三种 C++ 错误处理范式各有适用场景，不存在银弹
> - **解决什么问题**：不同场景对性能、安全性、可读性的权衡需求不同
> - **底层原理**：异常靠栈展开自动传播；错误码靠返回值手动传播；expected 是类型安全的联合体
> - **优缺点**：异常代码干净但慢/大；错误码快但啰嗦；expected 是折中但需要 C++23
> - **坑点**：Google 禁用异常不代表异常不好，是因为历史代码不异常安全
> - **横向对比**：Rust 的 Result&lt;T,E> ≈ std::expected；Go 的 error 接口 ≈ 错误码；Java 的 checked exception ≈ 强制处理的异常
> - **选型建议**：新项目优先 std::expected；遗留项目遵循已有约定；性能关键路径避免异常

## 6. 陷阱与实战

### 6.1 构造函数中抛异常导致内存泄漏

```cpp
// ❌ 错误示范：裸指针 + 构造函数异常
class BadResource {
    int* data1_;
    int* data2_;
public:
    BadResource()
        : data1_(new int[100])    // 成功
        , data2_(new int[200])    // 如果这里抛 bad_alloc...
    {
        // data1_ 泄漏！因为对象没构造完成，析构函数不会被调用
    }
    ~BadResource() {
        delete[] data1_;   // 不会执行
        delete[] data2_;
    }
};

// ✅ 正确做法：用智能指针
class GoodResource {
    std::unique_ptr<int[]> data1_;
    std::unique_ptr<int[]> data2_;
public:
    GoodResource()
        : data1_(std::make_unique<int[]>(100))    // 成功
        , data2_(std::make_unique<int[]>(200))    // 如果这里抛异常...
    {
        // data1_ 作为已构造的成员，其析构函数会被调用 → 不泄漏
    }
    // 析构函数自动释放，不需要手写
};
```

**原理**：C++ 保证——已经构造完成的成员变量，即使整个对象的构造函数抛异常，那些成员的析构函数仍然会被调用。但对象本身的析构函数不会被调用。

### 6.2 析构函数中抛异常

```cpp
// ❌ 致命错误：析构中抛异常
class Dangerous {
    ~Dangerous() {
        throw std::runtime_error("析构失败");
        // 如果在栈展开过程中执行到这里 → 同时存在两个活跃异常
        // → 直接调用 std::terminate() → 程序崩溃
    }
};

// ✅ 正确做法：析构中捕获所有异常
class Safe {
    ~Safe() noexcept {  // 析构函数默认就是 noexcept
        try {
            closeResource();  // 可能失败的清理操作
        }
        catch (const std::exception& e) {
            // 记录日志，但绝不让异常逃出析构函数
            std::cerr << "清理失败: " << e.what() << std::endl;
        }
    }
};
```

### 6.3 catch(...) 吞掉所有异常

```cpp
// ❌ 吞掉异常，掩盖 bug
void bad_handler() {
    try {
        doWork();
    }
    catch (...) {
        // 什么都不做 —— 最危险的代码之一
    }
}

// ✅ 至少记录，然后决定是否重新抛出
void good_handler() {
    try {
        doWork();
    }
    catch (const std::exception& e) {
        spdlog::error("doWork 失败: {}", e.what());
        throw;   // 重新抛出，让上层决定如何处理
    }
    catch (...) {
        spdlog::error("doWork 抛出未知异常");
        throw;
    }
}
```

### 6.4 用异常做流程控制

```cpp
// ❌ 滥用异常做流程控制（性能极差）
bool exists_bad(const std::vector<int>& v, int target) {
    try {
        for (size_t i = 0; ; ++i) {
            if (v.at(i) == target) return true;  // at() 越界抛异常来结束循环
        }
    }
    catch (const std::out_of_range&) {
        return false;   // 用异常控制循环退出 —— 荒谬
    }
}

// ✅ 异常只用于真正的异常情况
bool exists_good(const std::vector<int>& v, int target) {
    return std::find(v.begin(), v.end(), target) != v.end();
}
```

### 6.5 异常与多线程

```cpp
#include <thread>
#include <exception>
#include <iostream>

// 异常不能跨线程传播！子线程的异常不会自动传到主线程
// 必须用 std::exception_ptr 手动搬运

std::exception_ptr g_eptr = nullptr;

void worker() {
    try {
        throw std::runtime_error("子线程出错");
    }
    catch (...) {
        g_eptr = std::current_exception();  // 捕获异常指针
    }
}

int main() {
    std::thread t(worker);
    t.join();

    if (g_eptr) {
        try {
            std::rethrow_exception(g_eptr);  // 在主线程重新抛出
        }
        catch (const std::exception& e) {
            std::cout << "从子线程捕获: " << e.what() << std::endl;
        }
    }
    return 0;
}
```

**std::async 更优雅**：`std::future::get()` 会自动重新抛出子任务中的异常。

```cpp
#include <future>

int main() {
    auto fut = std::async(std::launch::async, []() -> int {
        throw std::runtime_error("async 任务失败");
        return 42;
    });

    try {
        int val = fut.get();  // 自动重新抛出子线程中的异常
    }
    catch (const std::exception& e) {
        std::cout << "捕获: " << e.what() << std::endl;
    }
    return 0;
}
```

### 6.6 陷阱速查表

| 陷阱 | 后果 | 正确做法 |
|------|------|---------|
| 构造函数中裸 new 后抛异常 | 内存泄漏 | 用智能指针管理资源 |
| 析构函数抛异常 | std::terminate | 析构中 try-catch，绝不让异常逃出 |
| catch(...) 空处理 | 掩盖 bug | 至少记录日志，考虑重新抛出 |
| 异常做流程控制 | 性能灾难 | 异常只用于异常情况 |
| throw e 代替 throw; | 对象切片 | 重新抛出用 throw; |
| 忘给移动构造加 noexcept | vector 退化为拷贝 | 移动操作必加 noexcept |
| 跨线程抛异常 | 未捕获 → terminate | 用 exception_ptr 或 std::async |
| catch 基类写在派生类前面 | 派生类 catch 永远不执行 | 派生类 catch 写前面 |

> **面试速答**
> - **是什么/本质**：异常处理中的常见陷阱，大多涉及资源管理和异常安全
> - **解决什么问题**：避免异常相关的内存泄漏、程序崩溃、性能问题
> - **底层原理**：构造函数异常时，已构造成员会析构但对象析构函数不调用；栈展开期间再抛异常触发 terminate
> - **优缺点**：遵循 RAII 能避免大部分陷阱，但需要团队纪律
> - **坑点**：最隐蔽的是"移动构造忘加 noexcept"——不报错，只是默默变慢
> - **横向对比**：Java/C# 有 finally 块做清理，C++ 靠 RAII 析构函数，更自动但对析构要求更严
> - **选型建议**：一条原则覆盖 80% 陷阱：所有资源都用 RAII 管理，永远不写裸 new/delete

## 7. 综合自测问题

### 基础理解

**Q1**：throw 和 throw e 有什么区别？什么时候用哪个？
> `throw;` 重新抛出原始异常对象，保留多态类型信息。`throw e;` 会拷贝构造一个新对象，发生对象切片。在 catch 中重新抛出必须用 `throw;`。

**Q2**：catch 块的匹配顺序是什么？派生类和基类 catch 的先后顺序为什么重要？
> 按声明顺序从上到下匹配。派生类必须在基类前面，否则会被基类 catch 截获，派生类 catch 永远执行不到。

**Q3**：标准异常层次中 logic_error 和 runtime_error 的区别是什么？
> logic_error 表示程序逻辑 bug（可以通过修改代码预防），runtime_error 表示运行时环境导致的错误（无法在编译期或代码审查时预防）。

### 底层原理

**Q4**：解释 Table-Based 异常模型为什么叫"零开销"？
> 正常执行路径不插入任何额外指令（不需要 setjmp/注册 handler）。异常处理信息存储在只读的异常表中，只有抛异常时才查表。"零开销"指的是正常路径零开销，异常路径其实代价更大。

**Q5**：栈展开过程中，如果某个析构函数又抛出异常会怎样？
> 同时存在两个活跃异常，C++ 标准规定调用 `std::terminate()`，程序终止。所以析构函数必须是 noexcept 的。

**Q6**：异常表（.gcc_except_table）包含哪些信息？
> 包含调用点表（代码区间到 landing pad 的映射）、动作表（catch 类型过滤器）、类型表（typeinfo 指针）。personality routine 查询这些表来决定是否匹配 catch 和执行清理。

### 异常安全

**Q7**：什么是 copy-and-swap？它如何实现强异常保证？
> 赋值运算符按值传参（触发拷贝构造），然后用 noexcept 的 swap 交换内容。如果拷贝失败（异常），原对象完全不变；如果拷贝成功，swap 不会失败，保证要么全部完成要么不变。

**Q8**：为什么 vector 扩容时，如果元素的移动构造不是 noexcept，就退化为拷贝？
> 移动操作搬到一半如果抛异常，已经搬走的元素处于 moved-from 状态，无法恢复原始 vector。用拷贝的话，异常发生时原 vector 不变，可以安全回退。

### noexcept

**Q9**：noexcept 声明了但函数内部抛异常会怎样？
> 运行时直接调用 `std::terminate()`，不会进入 catch 块。注意编译器通常不会对此报错（不是编译期错误）。

**Q10**：条件 noexcept `noexcept(noexcept(expr))` 中两个 noexcept 分别是什么角色？
> 外层 noexcept 是规范（specifier），声明函数是否 noexcept。内层 noexcept 是运算符（operator），在编译期计算 expr 是否声明为 noexcept，返回 bool。

### 实战与选型

**Q11**：Google 为什么在 C++ 代码中禁用异常？
> 主要原因：大量遗留代码不是异常安全的；异常表增大二进制体积；跨团队难以保证异常安全一致性；异常路径调试困难。替代方案是 absl::Status/StatusOr。

**Q12**：std::expected 和 std::optional 有什么区别？
> `std::optional<T>` 只知道有没有值，不知道为什么没有。`std::expected<T,E>` 没有值时还携带错误信息 E。expected 适合错误处理，optional 适合"可能没有"的语义。

**Q13**：异常不能跨线程传播，如何把子线程的异常传到主线程？
> 两种方式：① 手动用 `std::current_exception()` 捕获 `exception_ptr`，主线程 `std::rethrow_exception()` 重新抛出。② 用 `std::async` + `std::future::get()`，自动重新抛出。

**Q14**：什么情况下应该用异常？什么情况下用错误码？
> 异常适合：罕见的、不可忽略的错误，构造函数/运算符重载（没有返回值可用）。错误码适合：高频可能失败的操作、性能关键路径、跨 ABI 边界、需要确定性执行时间的场景。

**Q15**：为什么 C++ 析构函数默认就是 noexcept 的？
> C++11 起析构函数隐式声明为 noexcept。因为栈展开期间会调用析构函数，如果析构抛异常就是 terminate。默认 noexcept 是让语言默认行为符合最安全的实践。

## 8. 选型速查表

```
需要处理错误？
│
├─ 错误极其罕见？（如内存分配失败、文件格式损坏）
│  └─→ 使用 C++ 异常
│
├─ 错误是常见的、预期内的？（如用户输入错误、网络超时）
│  ├─ C++23 可用？ → std::expected<T, E>
│  └─ C++23 不可用？ → 错误码 / 自定义 Result<T, E>
│
├─ 性能关键路径？（如游戏主循环、高频交易）
│  └─→ 错误码（零开销，确定性执行时间）
│
├─ 跨 DLL / 跨语言边界？
│  └─→ 错误码（ABI 安全）
│
├─ 构造函数/运算符重载中的错误？
│  └─→ 异常（没有返回值可用）
│
└─ 遗留代码库（非异常安全）？
   └─→ 遵循项目现有约定，通常是错误码
```

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 应用层业务逻辑 | 异常 | 代码清晰，自动传播 |
| 库的公共 API | 异常 + noexcept 标注 | 用户期望异常，noexcept 标注帮助优化 |
| 性能关键路径 | 错误码 | 零开销，确定性 |
| 跨 ABI/跨语言 | 错误码 | 异常不能跨 ABI |
| C++23 新项目 | std::expected | 类型安全，不依赖 RTTI |
| 构造函数报错 | 异常 | 唯一合理的选择 |
| 析构/清理函数 | 不抛出 + 日志 | 析构必须 noexcept |
| Google/LLVM 风格 | absl::Status | 项目规范要求 |
