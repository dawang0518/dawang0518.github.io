# C++ 设计模式与惯用法

> 一句话定位：C++ 惯用法不是 GoF 设计模式的翻译，而是利用 RAII、模板、移动语义等语言特性演化出的编程范式——理解它们才算真正写"C++ 风格"的代码。

---

### C++ 惯用法 vs GoF 设计模式

```
GoF 设计模式                              C++ 惯用法
────────────                              ──────────
语言无关的 OOP 套路                        与 C++ 语言特性深度绑定
重点：接口抽象、组合复用                    重点：资源管理、编译期计算、零开销抽象
```

## 1. RAII（Resource Acquisition Is Initialization）

### 1.1 概念与本质

RAII 的核心思想：**资源的生命周期绑定到对象的生命周期**。

- 构造函数中获取资源（打开文件、申请内存、加锁）
- 析构函数中释放资源（关闭文件、释放内存、解锁）
- 利用 C++ 的**栈展开机制**，即使发生异常也能保证析构函数被调用

### 1.2 底层原理

```
函数调用              异常抛出
┌───────┐            ┌───────┐
│ 构造A  │            │ 构造A  │
│ 构造B  │            │ 构造B  │
│ 构造C  │            │ 构造C  │
│ ...    │            │ throw! │ ← 异常点
│ 析构C  │            │ 析构C  │ ← 栈展开：逆序析构
│ 析构B  │            │ 析构B  │
│ 析构A  │            │ 析构A  │
└───────┘            └───────┘
```

编译器保证：无论正常返回还是异常退出，**已构造的局部对象**都会被逆序析构。这是 RAII 成立的语言层面保证。

### 1.3 代码示例

**文件句柄封装**：

```cpp
class FileGuard {
public:
    explicit FileGuard(const char* path, const char* mode)
        : fp_(fopen(path, mode)) {
        if (!fp_) throw std::runtime_error("打开文件失败");
    }

    ~FileGuard() {
        if (fp_) fclose(fp_);  // 析构时自动关闭
    }

    // 禁止拷贝，资源独占
    FileGuard(const FileGuard&) = delete;
    FileGuard& operator=(const FileGuard&) = delete;

    // 允许移动
    FileGuard(FileGuard&& other) noexcept : fp_(other.fp_) {
        other.fp_ = nullptr;
    }

    FILE* get() const { return fp_; }

private:
    FILE* fp_;
};

void process() {
    FileGuard file("data.txt", "r");  // 构造时打开
    // ... 使用 file.get() 读取数据 ...
    // 无论正常返回还是异常，file 的析构函数都会关闭文件
}
```

**互斥锁封装（lock_guard 简化实现）**：

```cpp
class LockGuard {
public:
    explicit LockGuard(std::mutex& mtx) : mtx_(mtx) {
        mtx_.lock();    // 构造时加锁
    }

    ~LockGuard() {
        mtx_.unlock();  // 析构时解锁
    }

    LockGuard(const LockGuard&) = delete;
    LockGuard& operator=(const LockGuard&) = delete;

private:
    std::mutex& mtx_;  // 持有引用，不拥有 mutex 本身
};
```

### 1.4 对比表

| 资源管理方式 | 优点 | 缺点 | 典型语言 |
|------------|------|------|---------|
| **RAII** | 确定性释放、异常安全、零运行时开销 | 需要手写/使用封装类 | C++ |
| **GC（垃圾回收）** | 程序员无需关心释放 | 不确定何时释放、不适合非内存资源、GC 暂停 | Java/Go/C# |
| **手动管理** | 完全控制 | 容易忘记释放、异常不安全、双重释放 | C |

### 1.5 优缺点与权衡

**优点**：
- 资源释放时机确定（离开作用域即释放）
- 异常安全（栈展开保证析构）
- 组合使用时天然支持多资源管理

**缺点**：
- 每种资源都需要一个封装类（标准库已提供大量：unique_ptr、lock_guard、fstream 等）
- 循环引用场景需要 weak_ptr 打破

### 1.6 陷阱与实战

```cpp
// ❌ 临时对象立即析构！锁在这一行结束就释放了
if (std::lock_guard<std::mutex>(mtx), shared_data > 0) { ... }

// ❌ 更隐蔽的错误：没有给 lock_guard 命名
{
    std::lock_guard<std::mutex>(mtx);  // 这是声明了一个叫 mtx 的临时对象！
    // 锁已经释放，下面的操作不受保护
    shared_data++;
}

// ✅ 正确：给 RAII 对象命名
{
    std::lock_guard<std::mutex> lock(mtx);  // lock 的生命周期覆盖整个块
    shared_data++;
}
```

> **面试速答**
>
> RAII 的本质就一句话：把资源的生命周期绑定到对象的生命周期，构造时获取资源、析构时释放资源。它解决的核心问题是资源泄漏，尤其是异常路径上的泄漏——手动 close/free 在异常抛出时基本管不到，但栈上对象的析构函数一定会被调用。
>
> 它的底层依赖是 C++ 的栈展开机制：当异常传播时，已经构造的局部对象会按逆序析构，资源自然而然就释放了。这是 C++ 比 GC 语言更适合管理非内存资源（文件句柄、锁、socket）的关键——GC 是不确定的，而 RAII 是确定性释放。
>
> 有几个容易踩的坑：RAII 临时对象如果没命名，会在当前表达式结束就立即析构（比如 `std::lock_guard<std::mutex>(mtx);` 根本不锁，锁完立刻解），必须给它一个名字；另外构造函数如果抛异常，析构函数不会被调用（因为对象还没完成构造），所以构造里要自己管好已分配的部分。
>
> 和其他方案对比：GC 不确定、不适合非内存资源，手动管理不安全。所以 C++ 里一切资源管理都应该优先走 RAII，能用标准库现成的封装（unique_ptr/lock_guard/fstream）就别自己造。

## 2. CRTP（Curiously Recurring Template Pattern）

### 2.1 概念与本质

派生类将自己作为模板参数传给基类，实现**编译期多态**。

```cpp
template <typename Derived>
class Base {
    // 基类可以通过 static_cast<Derived*>(this) 调用派生类方法
};

class Derived : public Base<Derived> {
    // Derived 继承 Base<Derived>
};
```

### 2.2 底层原理

```
虚函数多态（运行时）                    CRTP（编译期）
┌──────────┐                          ┌──────────────────┐
│ 对象      │                          │ 对象              │
│ ┌──────┐ │                          │ （无虚表指针）      │
│ │ vptr │─┼─→ vtable → func()        │                  │
│ └──────┘ │                          │                  │
│ 成员数据  │                          │ 成员数据           │
└──────────┘                          └──────────────────┘
间接调用，有 vtable 开销                直接调用，编译器内联优化
```

关键机制：基类中 `static_cast<Derived*>(this)` 在编译期确定类型，编译器直接生成对派生类方法的调用，无虚表查找。

### 2.3 代码示例

**静态多态接口**：

```cpp
template <typename Derived>
class Shape {
public:
    // 对外接口：非虚函数
    double area() const {
        // 编译期向下转型，直接调用派生类实现
        return static_cast<const Derived*>(this)->area_impl();
    }

    void draw() const {
        static_cast<const Derived*>(this)->draw_impl();
    }
};

class Circle : public Shape<Circle> {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}

    double area_impl() const { return 3.14159 * radius_ * radius_; }
    void draw_impl() const { std::cout << "画圆, 半径=" << radius_ << "\n"; }
};

class Rectangle : public Shape<Rectangle> {
    double w_, h_;
public:
    Rectangle(double w, double h) : w_(w), h_(h) {}

    double area_impl() const { return w_ * h_; }
    void draw_impl() const { std::cout << "画矩形 " << w_ << "x" << h_ << "\n"; }
};

// 编译期多态：模板函数
template <typename T>
void print_area(const Shape<T>& shape) {
    std::cout << "面积: " << shape.area() << "\n";
}
```

**enable_shared_from_this 的原理**：

```cpp
// 标准库 enable_shared_from_this 是 CRTP 的经典应用
// 简化版原理展示
template <typename T>
class enable_shared_from_this {
    weak_ptr<T> weak_this_;  // shared_ptr 构造时注入

public:
    shared_ptr<T> shared_from_this() {
        return shared_ptr<T>(weak_this_);  // 从 weak_ptr 构造
    }

protected:
    ~enable_shared_from_this() = default;
};

// 使用方式
class Session : public enable_shared_from_this<Session> {
public:
    void start() {
        // 在回调中安全地传递自身的 shared_ptr
        auto self = shared_from_this();
        async_read(socket_, buffer_, [self](auto ec, auto bytes) {
            self->on_read(ec, bytes);
        });
    }
};
```

### 2.4 对比表

| 维度 | 虚函数多态 | CRTP 编译期多态 |
|------|----------|----------------|
| **调度时机** | 运行时（vtable 查找） | 编译期（模板实例化） |
| **性能开销** | 间接调用 + 可能的 cache miss | 零开销，可内联 |
| **灵活性** | 可通过基类指针/引用存储异构对象 | 每个派生类产生不同基类类型，无法统一容器 |
| **代码膨胀** | 无 | 每个派生类实例化一份基类代码 |
| **典型场景** | 运行时需要多态 | 性能关键路径的静态多态 |

### 2.5 陷阱与实战

```cpp
// ❌ 派生类传错模板参数
class Dog : public Base<Cat> { };  // 编译不报错，但 static_cast 行为未定义

// ✅ C++23 deducing this 可以替代部分 CRTP 场景
// C++23 之前用 CRTP 实现链式调用
template <typename Derived>
class Builder {
public:
    Derived& set_name(const std::string& name) {
        name_ = name;
        return static_cast<Derived&>(*this);
    }
protected:
    std::string name_;
};
```

> **面试速答**
>
> CRTP（奇异递归模板模式）本质就是派生类把自己当作模板参数传给基类，靠模板在编译期实现多态。它的动机是想要虚函数带来的"代码复用"，但不想要虚函数的运行时开销（查虚表、无法内联）。
>
> 实现上，基类在需要调用派生类方法时，通过 `static_cast<Derived*>(this)` 把自己转成派生类指针，由于派生类类型在编译期就确定，编译器可以直接内联调用，彻底消除虚表和虚调用。
>
> 优点是零虚表开销、可内联；缺点是代码会膨胀（每个派生都实例化一份基类）、而且没法运行时多态（不能用基类指针统一存 Derived1、Derived2）。
>
> 有个容易踩的坑：派生类如果把模板参数传错（比如 `class B : public CRTPBase<A>`），编译器不会报错，但行为完全未定义，调试起来很恶心。
>
> 和虚函数对比：虚函数运行时灵活、能做异构容器，CRTP 是编译期高效。选型很直接——性能敏感的静态多态（比如表达式模板、数值计算库）用 CRTP；需要异构容器或运行时扩展的用虚函数。

## 3. Pimpl（Pointer to Implementation）

### 3.1 概念与本质

用**前向声明 + 指针**隐藏实现细节，头文件只暴露接口，实现类在 .cpp 中定义。这被称为**编译防火墙**。

### 3.2 底层原理

```
传统方式：                              Pimpl 方式：
Widget.h                               Widget.h
┌─────────────────┐                    ┌─────────────────┐
│ #include <vector>│ ← 暴露依赖         │ class Impl;      │ ← 前向声明
│ #include <string>│                    │ unique_ptr<Impl> │
│ 所有私有成员      │                    │ 公有接口声明       │
└─────────────────┘                    └─────────────────┘
修改私有成员 → 所有包含者重新编译         修改 Impl → 只需重编 Widget.cpp
```

### 3.3 代码示例

**Widget.h**：

```cpp
#pragma once
#include <memory>
#include <string>

class Widget {
public:
    Widget(const std::string& name);
    ~Widget();  // 必须声明，在 .cpp 中定义！

    // 移动操作也必须在 .cpp 中定义
    Widget(Widget&& other) noexcept;
    Widget& operator=(Widget&& other) noexcept;

    void doWork();
    std::string name() const;

private:
    struct Impl;                    // 前向声明
    std::unique_ptr<Impl> pImpl_;   // 指向实现的指针
};
```

**Widget.cpp**：

```cpp
#include "Widget.h"
#include <vector>
#include <iostream>

// Impl 的完整定义，只在 .cpp 中可见
struct Widget::Impl {
    std::string name;
    std::vector<int> data;
    // 可以随意添加成员，不影响头文件

    Impl(const std::string& n) : name(n) {}

    void doWorkImpl() {
        std::cout << name << " 正在工作, 数据量: " << data.size() << "\n";
    }
};

// 构造函数：创建 Impl
Widget::Widget(const std::string& name)
    : pImpl_(std::make_unique<Impl>(name)) {}

// 析构函数必须在 .cpp 中定义！
// 因为此处编译器能看到 Impl 的完整定义，才能正确生成 unique_ptr 的删除器
Widget::~Widget() = default;

Widget::Widget(Widget&& other) noexcept = default;
Widget& Widget::operator=(Widget&& other) noexcept = default;

void Widget::doWork() { pImpl_->doWorkImpl(); }
std::string Widget::name() const { return pImpl_->name; }
```

### 3.4 优缺点与权衡

| 优点 | 缺点 |
|------|------|
| 修改实现不触发头文件依赖者重编译 | 每个对象多一次堆分配 |
| 隐藏私有成员，保护 ABI 稳定性 | 每次调用多一次指针间接访问 |
| 减少头文件的 #include 传播 | 代码量增加（转发函数） |
| 适合动态库的接口稳定 | 不能内联实现函数 |

### 3.5 陷阱与实战

```cpp
// ❌ 忘记在 .cpp 中定义析构函数
class Bad {
    struct Impl;
    std::unique_ptr<Impl> p;
    // 编译器在头文件生成默认析构函数
    // 此时 Impl 是不完整类型 → unique_ptr 无法调用 delete → 编译错误
};

// ❌ 在头文件中 inline 定义析构函数，同样不行
class Bad2 {
    struct Impl;
    std::unique_ptr<Impl> p;
    ~Bad2() = default;  // 在头文件中，Impl 不完整 → 错误
};

// ✅ 析构函数声明在头文件，定义在 .cpp 中
// Widget::~Widget() = default;  // 在 Widget.cpp 中
```

> **面试速答**
>
> Pimpl 就是"指向实现的指针"——头文件里只放一个前向声明的 Impl 类和一个指针，真正的数据成员和私有函数全搬到 .cpp 里。它被称为编译防火墙，主要解决三个问题：减少编译依赖（头文件不再包含实现细节所需的头）、保护 ABI 稳定（改实现不改头就不影响二进制兼容）、隐藏私有实现。
>
> 原理上依赖了"指针大小固定"这件事：头文件只要知道 Impl 是个类就能声明指针，不需要完整定义；真正的 Impl 类在 .cpp 里定义。这样头文件变化少，下游用户重编译代价小。
>
> 优点是编译隔离好、ABI 稳定；缺点也明显——多一次堆分配、每次访问都多一层间接调用、成员函数无法内联。
>
> 有个容易踩的坑要特别注意：析构函数必须显式定义在 .cpp 里，不能用默认的 inline 析构——因为 unique_ptr 在销毁时需要 Impl 的完整类型才能正确 delete，头文件里只有前向声明，编译会挂。
>
> 选型上，大型项目的公共接口、动态库 API 用 Pimpl 很值；内部实现类、模板类、性能敏感的小对象就别上了。

## 4. NVI（Non-Virtual Interface）

### 4.1 概念与本质

**公有接口是非虚的，虚函数是私有（或 protected）的**。基类控制调用流程（前置检查、后置处理），派生类只负责核心逻辑。

这本质上是**模板方法模式**在 C++ 中的最佳实践表达。

### 4.2 底层原理

```
NVI 调用链:
                    基类::public_method()
                    ┌──────────────────┐
                    │ 1. 前置检查/日志   │  ← 基类控制
                    │ 2. do_method()    │  ← 虚函数，派生类实现
                    │ 3. 后置处理/统计   │  ← 基类控制
                    └──────────────────┘

直接虚函数:
                    派生类::method() override
                    ┌──────────────────┐
                    │ 一切由派生类决定    │  ← 无法统一控制流程
                    └──────────────────┘
```

### 4.3 代码示例

```cpp
class Logger {
public:
    // 公有非虚接口：控制完整流程
    void log(const std::string& msg) {
        // 前置处理：添加时间戳
        auto now = std::chrono::system_clock::now();
        std::string timestamped = format_time(now) + " " + msg;

        // 核心操作：交给派生类
        do_log(timestamped);

        // 后置处理：统计日志计数
        ++log_count_;
    }

    virtual ~Logger() = default;
    size_t log_count() const { return log_count_; }

private:
    // 私有虚函数：派生类实现核心逻辑
    virtual void do_log(const std::string& msg) = 0;

    size_t log_count_ = 0;

    static std::string format_time(auto tp) {
        // 时间格式化逻辑...
        return "[timestamp]";
    }
};

class FileLogger : public Logger {
private:
    std::ofstream file_;

    void do_log(const std::string& msg) override {
        file_ << msg << "\n";  // 只负责写文件
    }

public:
    explicit FileLogger(const std::string& path) : file_(path) {}
};

class ConsoleLogger : public Logger {
private:
    void do_log(const std::string& msg) override {
        std::cout << msg << std::endl;  // 只负责输出到控制台
    }
};
```

### 4.4 对比表

| 维度 | NVI 模式 | 直接公有虚函数 |
|------|---------|-------------|
| **流程控制** | 基类统一管理前置/后置 | 每个派生类自行处理 |
| **扩展性** | 新增前置处理不影响派生类 | 所有派生类都得改 |
| **接口稳定** | 公有接口不变，内部可重构 | 虚函数即接口，改动影响所有 |
| **灵活性** | 派生类只能改核心逻辑 | 派生类完全自由 |
| **复杂度** | 多一层转发 | 直接简单 |

### 4.5 陷阱与实战

```cpp
// ❌ 虚函数设为 public，派生类可以不调用基类版本导致流程不一致
class Bad {
public:
    virtual void process() {
        validate();  // 派生类可能忘记调用基类版本
        // ...
    }
};

// ✅ NVI：基类强制流程
class Good {
public:
    void process() {
        validate();
        do_process();  // 派生类只实现这一步
        cleanup();
    }
private:
    virtual void do_process() = 0;
};
```

> **面试速答**
>
> NVI（非虚接口）的核心思路是：公有接口不是虚函数，虚函数放在 private 或 protected。基类的 public 非虚函数负责统一的前置/后置处理，中间再调用 private 虚函数让派生类定制。它解决的是"派生类可能破坏调用契约"的问题——比如你希望每次调用前后都加日志/加锁，如果虚函数是 public，派生类 override 后这些事就绕过去了。
>
> 它其实就是模板方法模式在 C++ 里的实现，巧妙利用了一个容易被误解的事实：C++ 里"访问控制"和"虚函数覆盖"是正交的——派生类完全可以 override 基类的 private 虚函数，因为 override 只需要通过基类的 vtable 进行查找，不需要在派生类里直接调用基类版本。
>
> 优点是流程统一、接口稳定；缺点是派生类自由度受限、多一层转发。
>
> 和其他方案对比：直接暴露虚函数自由但不可控、CRTP 是编译期版本。选型上，框架性代码、需要强制生命周期/调用流程一致的地方用 NVI；简单的继承体系直接用虚函数就行，别过度设计。

## 5. 单例模式

### 5.1 概念与本质

保证一个类只有一个实例，并提供全局访问点。在 C++ 中，关键问题是**线程安全**和**初始化顺序**。

### 5.2 Meyers' Singleton（推荐方式）

```cpp
class Singleton {
public:
    // C++11 保证：局部 static 变量的初始化是线程安全的
    static Singleton& instance() {
        static Singleton inst;  // magic static
        return inst;
    }

    void doSomething() { /* ... */ }

    // 禁止拷贝和移动
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

private:
    Singleton() { /* 初始化 */ }
    ~Singleton() { /* 清理 */ }
};
```

### 5.3 底层原理：magic statics

C++11 标准 [stmt.dcl] 规定：局部 static 变量的初始化在首次执行到该语句时发生，且**如果多个线程同时到达，只有一个线程执行初始化，其他线程阻塞等待**。

GCC/Clang 在底层使用 `__cxa_guard_acquire` / `__cxa_guard_release` 实现：

```
线程 A 调用 instance()        线程 B 调用 instance()
────────────────              ────────────────
__cxa_guard_acquire(&guard)
  → 成功，执行构造             __cxa_guard_acquire(&guard)
  Singleton()                   → 阻塞，等待线程 A
__cxa_guard_release(&guard)
  → 标记完成                    → 返回，跳过构造
return inst;                  return inst;
```

### 5.4 为什么不用全局变量：SIOF

```cpp
// a.cpp
Logger logger;  // 全局变量

// b.cpp
extern Logger logger;
Config config(logger);  // config 依赖 logger

// 问题：不同翻译单元的全局变量初始化顺序未定义！
// 如果 config 先于 logger 构造 → 使用未初始化对象 → UB
```

这就是 **SIOF（Static Initialization Order Fiasco）**。Meyers' Singleton 解决了这个问题：首次调用时才初始化，顺序由调用顺序决定。

### 5.5 DCLP 为什么在 C++11 之前不安全

```cpp
// 经典双重检查锁定（C++11 之前的写法）
Singleton* Singleton::instance() {
    if (!ptr) {                    // 第一次检查（无锁）
        std::lock_guard<std::mutex> lock(mtx);
        if (!ptr) {                // 第二次检查（有锁）
            ptr = new Singleton;   // ← 问题在这里！
        }
    }
    return ptr;
}
// C++11 之前：编译器/CPU 可能重排 new 操作：
// 1. 分配内存 → 2. 赋值给 ptr → 3. 调用构造函数
// 线程 B 在步骤 2 后看到 ptr 非空，但对象未构造完成 → UB

// C++11 之后：用 std::atomic + memory_order 可以安全实现
// 但直接用 Meyers' Singleton 更简单
```

### 5.6 为什么尽量避免单例

| 问题 | 说明 |
|------|------|
| **全局状态** | 隐式依赖，代码难以推理 |
| **测试困难** | 无法 mock，状态在测试间泄漏 |
| **生命周期不可控** | 析构顺序依赖程序退出流程 |
| **并发竞争** | 多线程访问单例内部状态仍需加锁 |
| **违背单一职责** | 类既管理自身生命周期又实现业务逻辑 |

**替代方案**：依赖注入——把单例对象作为参数传入，而不是在函数内部直接调用 `::instance()`。

> **面试速答**
>
> 单例模式就是保证一个类只有一个实例，并提供全局访问点，常用于日志、配置、连接池这类"全局唯一资源"的管理。
>
> 现代 C++ 里最推荐的写法是 Meyers' Singleton——在函数内用 `static` 局部变量实现。它利用了 C++11 引入的 magic statics 特性，底层由 `__cxa_guard_acquire` 保证线程安全的延迟初始化，不用自己写锁。
>
> 优点是全局唯一+延迟初始化，代码极简；缺点也不少——全局状态、单元测试困难（很难 mock）、生命周期不可控。
>
> 有几个坑要特别注意：SIOF（Static Initialization Order Fiasco，不同翻译单元的静态对象初始化顺序不确定）；C++11 之前流行的 DCLP（双检锁）写法在弱内存模型下其实是不安全的，必须等到 C++11 有了原子变量和内存序才能正确写出来；还有析构顺序问题——多个单例互相依赖时，谁先析构说不准。
>
> 和其他方案对比：Meyers' Singleton 简洁且 C++11 后安全，DCLP 复杂且老标准下不安全，直接用全局变量还会撞上 SIOF。选型建议是——真的必须用再选 Meyers' Singleton，能用依赖注入就优先依赖注入，避免全局状态对测试的污染。

## 6. 工厂模式

### 6.1 概念与本质

将对象的创建逻辑与使用逻辑分离。C++ 中的关键特点：返回 `unique_ptr`，利用多态返回派生类对象。

### 6.2 三种工厂对比

| 类型 | 简单工厂 | 工厂方法 | 抽象工厂 |
|------|---------|---------|---------|
| **结构** | 一个函数，switch/if 选择 | 每个产品一个工厂子类 | 一组相关产品的工厂接口 |
| **扩展方式** | 修改工厂函数（违反开闭） | 新增工厂子类 | 新增工厂子类 |
| **适用场景** | 产品种类少且稳定 | 产品种类会扩展 | 产品族（如 UI 主题） |
| **C++ 实现** | 函数 + enum | 虚函数 | 虚函数组 |

### 6.3 代码示例

**简单工厂 + unique_ptr**：

```cpp
enum class ShapeType { Circle, Rectangle, Triangle };

class Shape {
public:
    virtual ~Shape() = default;
    virtual void draw() const = 0;
};

class Circle : public Shape {
public:
    void draw() const override { std::cout << "画圆\n"; }
};

class Rectangle : public Shape {
public:
    void draw() const override { std::cout << "画矩形\n"; }
};

// 简单工厂函数
std::unique_ptr<Shape> createShape(ShapeType type) {
    switch (type) {
        case ShapeType::Circle:    return std::make_unique<Circle>();
        case ShapeType::Rectangle: return std::make_unique<Rectangle>();
        default: return nullptr;
    }
}
```

**注册表工厂（开闭原则）**：

```cpp
class ShapeFactory {
    // 类型名 → 创建函数 的注册表
    using Creator = std::function<std::unique_ptr<Shape>()>;
    std::unordered_map<std::string, Creator> registry_;

public:
    // 注册新类型
    void registerShape(const std::string& name, Creator creator) {
        registry_[name] = std::move(creator);
    }

    // 按名字创建对象
    std::unique_ptr<Shape> create(const std::string& name) const {
        auto it = registry_.find(name);
        if (it == registry_.end()) return nullptr;
        return it->second();
    }
};

// 使用：无需修改工厂代码即可扩展
ShapeFactory factory;
factory.registerShape("circle", []{ return std::make_unique<Circle>(); });
factory.registerShape("rect", []{ return std::make_unique<Rectangle>(); });

auto shape = factory.create("circle");
shape->draw();
```

### 6.4 陷阱与实战

```cpp
// ❌ 工厂返回 raw pointer，调用者容易忘记 delete
Shape* createShape(int type);

// ✅ 返回 unique_ptr，所有权语义清晰
std::unique_ptr<Shape> createShape(int type);

// ❌ 注册表工厂中 std::function 有堆分配开销
// ✅ 对于无状态 lambda，开销可忽略；高频场景可用函数指针
```

> **面试速答**
>
> 工厂模式说白了就是把"对象创建"这件事封装起来，让使用者不直接 new 具体类型，而是通过工厂拿到基类指针。它解决的是"客户代码和具体类型解耦"的问题——新增一种产品不用到处改调用点。
>
> 底层原理靠的是多态——工厂返回基类指针，实际指向派生类对象。在 C++ 里要用 `unique_ptr<Base>` 明确所有权归属，避免裸指针带来的所有权模糊。
>
> 优点是解耦、可扩展；缺点是增加了类数量，简单场景会有过度设计的嫌疑。
>
> 有两个容易踩的坑：一是返回裸指针会让所有权不清，用户不知道谁该 delete；二是注册表模式里 key 拼写错误要等运行时才暴露，编译期查不出来。
>
> 三种形态各有取舍：简单工厂（一个 switch-case）写得快但新增产品要改工厂代码；工厂方法（基类定义创建接口、子类实现）可扩展但类变多；注册表（运行时注册 creator）最灵活、插件化友好。选型上，产品种类固定用简单工厂；需要运行时可扩展、想做插件化用注册表模式。

## 7. 观察者模式

### 7.1 概念与本质

定义对象间一对多的依赖关系：当被观察者状态变化时，自动通知所有观察者。C++ 中的关键：用 `std::function` + `weak_ptr` 实现安全回调。

### 7.2 代码示例

**std::function + vector 实现**：

```cpp
template <typename... Args>
class Signal {
public:
    using SlotId = size_t;
    using Slot = std::function<void(Args...)>;

    // 连接观察者，返回 ID 用于断开
    SlotId connect(Slot slot) {
        SlotId id = next_id_++;
        slots_.emplace_back(id, std::move(slot));
        return id;
    }

    // 断开连接
    void disconnect(SlotId id) {
        slots_.erase(
            std::remove_if(slots_.begin(), slots_.end(),
                [id](const auto& p) { return p.first == id; }),
            slots_.end());
    }

    // 触发通知
    void emit(Args... args) const {
        for (const auto& [id, slot] : slots_) {
            slot(args...);
        }
    }

private:
    std::vector<std::pair<SlotId, Slot>> slots_;
    SlotId next_id_ = 0;
};

// 使用
Signal<int, const std::string&> on_message;
auto id = on_message.connect([](int code, const std::string& msg) {
    std::cout << "收到消息 [" << code << "]: " << msg << "\n";
});
on_message.emit(200, "成功");
on_message.disconnect(id);
```

**weak_ptr 安全观察者（避免悬空回调）**：

```cpp
class EventBus {
public:
    using Callback = std::function<void(const std::string&)>;

    // 观察者通过 weak_ptr 注册，自动感知生命周期
    void subscribe(std::weak_ptr<void> weak_owner, Callback cb) {
        observers_.emplace_back(std::move(weak_owner), std::move(cb));
    }

    void publish(const std::string& event) {
        // 通知前检查观察者是否存活
        observers_.erase(
            std::remove_if(observers_.begin(), observers_.end(),
                [&event](auto& entry) {
                    if (auto locked = entry.first.lock()) {
                        entry.second(event);  // 观察者存活，调用
                        return false;
                    }
                    return true;  // 观察者已销毁，清除
                }),
            observers_.end());
    }

private:
    std::vector<std::pair<std::weak_ptr<void>, Callback>> observers_;
};
```

### 7.3 信号槽机制

| 实现 | 特点 |
|------|------|
| **自己实现** | 轻量，如上面的 Signal 类 |
| **Qt 信号槽** | 需要 MOC 预处理，支持跨线程、自动断开 |
| **Boost.Signals2** | 头文件库，线程安全，支持返回值收集 |

### 7.4 陷阱与实战

```cpp
// ❌ 观察者析构后，被观察者仍持有其裸指针 → 悬空指针
class Observer { /* ... */ };
subject.attach(&observer);  // 裸指针
// observer 析构后，subject 仍会调用 → UB

// ✅ 用 weak_ptr 或在析构中主动断开
// 方案1：weak_ptr 自动检测
// 方案2：RAII 风格的连接对象
class Connection {
    Signal<>* signal_;
    Signal<>::SlotId id_;
public:
    ~Connection() { signal_->disconnect(id_); }  // 析构自动断开
};
```

> **面试速答**
>
> 观察者模式是一种一对多的依赖关系：主题状态变化时，自动通知所有订阅它的观察者。它解决的核心问题是"解耦事件源和事件处理者"——发布者不用知道谁在听，只负责广播。
>
> 实现上很直接，维护一个回调列表（比如 `vector<function<void(Event)>>`），事件发生时遍历调用一遍。
>
> 优点是松耦合、可动态增删订阅；缺点是通知顺序不确定、一旦链路变长调试非常困难。
>
> 有两个坑要特别注意：第一是观察者生命周期问题——如果观察者先销毁而回调还在列表里，调用时就是悬空回调，会 crash（用 weak_ptr 配合可以缓解）；第二是通知过程中如果有观察者反过来增删订阅，会导致迭代器失效，常见做法是遍历前先拷贝一份列表，或者用"标记删除"延迟清理。
>
> 实现选型：简单场景手写 `function + vector` 就够了、轻量；Qt 的信号槽支持跨线程分发（会经过事件循环）；Boost.Signals2 线程安全、支持自动断连。选型上，简单场景手写就够，需要线程安全或跨线程再上 Qt / Boost。

## 8. Copy-and-Swap Idiom

### 8.1 概念与本质

通过**值传递参数 + swap 交换**实现异常安全的赋值运算符。一个手法同时解决**自赋值**和**异常安全**两个问题。

### 8.2 底层原理

```
传统赋值（异常不安全）                     Copy-and-Swap
operator=(const T& rhs)                  operator=(T rhs)  // 值传递已经拷贝
{                                        {
    delete[] data_;        // 1.释放旧    swap(*this, rhs);  // 交换
    data_ = new[rhs.sz_]; // 2.分配新    }  // rhs 析构，释放原来 this 的资源
    // ↑ 如果 new 抛异常，data_ 已释放
    // 对象处于不一致状态！
    copy(rhs);             // 3.拷贝
}

关键：拷贝在参数传递时完成（可能抛异常）
     swap 是 noexcept 的
     → 要么完全成功，要么完全不变
```

### 8.3 代码示例

```cpp
class Buffer {
    int* data_;
    size_t size_;

public:
    Buffer(size_t size = 0)
        : data_(size ? new int[size]{} : nullptr), size_(size) {}

    // 拷贝构造
    Buffer(const Buffer& other)
        : data_(other.size_ ? new int[other.size_] : nullptr),
          size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 移动构造
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    ~Buffer() { delete[] data_; }

    // 非成员 swap（ADL 可找到）
    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    // 赋值运算符：值传递 + swap
    // 传值时：左值触发拷贝构造，右值触发移动构造
    Buffer& operator=(Buffer rhs) {  // 值传递！
        swap(*this, rhs);            // noexcept 交换
        return *this;
    }                                // rhs 析构，释放旧资源

    size_t size() const { return size_; }
};
```

### 8.4 为什么能同时处理自赋值和异常安全

| 场景 | 分析 |
|------|------|
| **自赋值** `a = a` | rhs 是 a 的拷贝，swap 后 rhs 析构释放原数据，a 持有拷贝数据，完全安全 |
| **异常安全** | 如果拷贝构造抛异常（参数传递时），this 完全未修改；swap 是 noexcept，不会抛异常 |
| **移动赋值** `a = std::move(b)` | rhs 通过移动构造生成，swap 后 rhs 析构释放 a 的旧资源，高效 |

> **面试速答**
>
> Copy-and-Swap 是一种写赋值运算符的惯用法：参数用**值传递**（在传参时完成拷贝），然后和当前对象 `swap`（swap 必须是 noexcept 的），函数返回时旧资源随着临时参数析构被释放。它一石三鸟地解决了自赋值安全、异常安全，还统一了拷贝赋值和移动赋值。
>
> 原理上很巧妙：拷贝这个"可能抛异常"的动作被放到参数传递阶段（在函数体执行前），一旦拷贝失败，异常直接在调用点抛出，当前对象完全不变，自然满足强异常保证；swap 是 noexcept 的指针级操作，不会抛。所以结果要么全成功、要么全不变，不会出现"改了一半"的中间状态。
>
> 优点是代码简洁、异常安全强保证；缺点是自赋值时会多做一次不必要的拷贝（不过自赋值本来就很罕见）。
>
> 有个硬性要求要特别注意：swap 必须是 noexcept 的，否则整套保证就失效了。如果一个类根本不需要深拷贝（比如都是值类型成员），也不需要这个手法，编译器默认生成的赋值就够用。
>
> 和传统赋值对比：传统赋值（先清理再拷贝）只能给基本异常保证，清理后新拷贝失败就留下了残缺对象；Copy-and-Swap 给出强异常保证。选型建议：管理资源的类（有自定义析构）都应该考虑用它，这也是 Rule of Five 的优雅实现方式。

## 9. Tag Dispatch

### 9.1 概念与本质

利用**类型标签**（空结构体）在编译期选择函数重载，实现编译期条件分发。

### 9.2 底层原理

```
运行时分发：if/switch → 运行时判断，有分支预测开销
Tag Dispatch：编译器根据类型标签选择重载 → 编译期确定，零运行时开销
SFINAE：enable_if 在模板参数推导阶段选择 → 编译期，但错误信息难读
if constexpr：C++17 编译期 if → 编译期，最直观
```

### 9.3 代码示例

```cpp
// 标准库 iterator_tag 就是 Tag Dispatch 的经典应用
namespace detail {

// 对随机访问迭代器：直接跳转
template <typename Iter>
void advance_impl(Iter& it, int n, std::random_access_iterator_tag) {
    it += n;  // O(1)
}

// 对双向迭代器：逐步移动
template <typename Iter>
void advance_impl(Iter& it, int n, std::bidirectional_iterator_tag) {
    if (n > 0) while (n--) ++it;
    else        while (n++) --it;  // O(n)
}

// 对前向迭代器：只能前进
template <typename Iter>
void advance_impl(Iter& it, int n, std::forward_iterator_tag) {
    while (n--) ++it;  // O(n)，只能前进
}

} // namespace detail

template <typename Iter>
void my_advance(Iter& it, int n) {
    // 编译器根据 iterator_tag 类型选择正确的重载
    detail::advance_impl(it,  n,
        typename std::iterator_traits<Iter>::iterator_category{});
}
```

### 9.4 三种编译期分发对比

| 方式 | 语法 | 优点 | 缺点 | C++ 版本 |
|------|------|------|------|---------|
| **Tag Dispatch** | 空结构体参数重载 | 直观、错误信息清晰 | 需要定义 tag 类型 | C++98 |
| **SFINAE / enable_if** | `enable_if_t<condition>` | 不需要 tag | 语法复杂、错误信息难懂 | C++11 |
| **if constexpr** | `if constexpr (cond)` | 最简洁直观 | 所有分支必须语法合法 | C++17 |
| **concepts** | `requires` 约束 | 错误信息最好 | 需要 C++20 | C++20 |

```cpp
// 同一个功能的三种写法对比

// 1. Tag Dispatch
template <typename T>
void process(T val, std::true_type /*is_integral*/) { /* 整数处理 */ }
template <typename T>
void process(T val, std::false_type /*is_integral*/) { /* 浮点处理 */ }
template <typename T>
void process(T val) {
    process(val, std::is_integral<T>{});
}

// 2. SFINAE
template <typename T>
std::enable_if_t<std::is_integral_v<T>> process(T val) { /* 整数 */ }
template <typename T>
std::enable_if_t<!std::is_integral_v<T>> process(T val) { /* 浮点 */ }

// 3. if constexpr（C++17，推荐）
template <typename T>
void process(T val) {
    if constexpr (std::is_integral_v<T>) { /* 整数处理 */ }
    else { /* 浮点处理 */ }
}
```

> **面试速答**
>
> Tag Dispatch（标签分派）的核心思路是用空结构体类型当作"标签"参数，利用函数重载在编译期选择不同的实现版本。它解决的典型问题是"根据类型特征选择算法"——最经典的例子是 STL 里 `std::advance` 根据迭代器类别（random_access_iterator_tag / bidirectional_iterator_tag / input_iterator_tag）选择 O(1) 还是 O(n) 的实现。
>
> 原理上就是利用 C++ 的函数重载解析：编译器在编译期根据 tag 的类型匹配到对应的重载版本，tag 参数只参与类型匹配、不产生任何运行时代码。
>
> 优点是零运行时开销、错误信息比 SFINAE 清晰得多；缺点是需要额外定义一堆 tag 类型。
>
> 有个细节要特别注意：tag 对象本身虽然是空的，但作为独立对象仍占 1 字节（C++ 规定空类最小 1 字节）。作为函数参数时被优化掉不影响，但如果你把 tag 作为成员存储就要小心了。
>
> 和其他编译期分派手段对比：SFINAE 更灵活但可读性差、错误信息惨不忍睹；`if constexpr` 最简洁但要 C++17 起；concepts 最干净但要 C++20。选型建议：C++17 及以上优先 `if constexpr`，C++20 用 concepts，低版本或者标准库风格的代码用 Tag Dispatch。

## 10. 陷阱与实战总结

### 常见陷阱速查表

| 陷阱 | 错误写法 | 正确写法 | 根本原因 |
|------|---------|---------|---------|
| RAII 临时对象立即析构 | `std::lock_guard<std::mutex>(mtx);` | `std::lock_guard<std::mutex> lock(mtx);` | 未命名的临时对象在语句结束时析构 |
| Pimpl 析构函数位置 | 头文件中 `~Widget() = default;` | .cpp 中 `Widget::~Widget() = default;` | unique_ptr 需要完整类型才能 delete |
| 单例 SIOF | 全局变量依赖另一个全局变量 | Meyers' Singleton | 翻译单元间全局变量初始化顺序未定义 |
| CRTP 传错类型 | `class Dog : public Base<Cat>` | `class Dog : public Base<Dog>` | static_cast 到错误类型是 UB |
| 观察者悬空指针 | 裸指针注册回调 | weak_ptr 或 RAII Connection | 观察者先于被观察者析构 |
| Copy-and-Swap 漏写 swap | 只写了 operator= | 同时提供 friend swap | swap 必须 noexcept 且交换所有成员 |
| 工厂返回裸指针 | `return new Derived;` | `return make_unique<Derived>();` | 裸指针所有权不清晰，易泄漏 |

### 异常安全级别速查

| 级别 | 含义 | 典型实现 |
|------|------|---------|
| **无保证** | 异常后状态未知 | 手动 new/delete 无保护 |
| **基本保证** | 异常后对象有效但状态不确定 | 大多数标准库操作 |
| **强保证** | 异常后状态回滚到操作前 | Copy-and-Swap |
| **不抛异常** | 操作保证不抛 | swap、移动构造、析构 |

### 设计模式反模式

```cpp
// ❌ 过度使用单例：全局状态导致代码难以测试
class DatabaseManager : public Singleton<DatabaseManager> { ... };
// → 改为依赖注入：构造函数接收 Database& 参数

// ❌ 过度使用工厂：只有一个产品也写工厂
auto p = ShapeFactory::create("circle");
// → 直接 auto p = std::make_unique<Circle>();

// ❌ 过度使用观察者：简单的两个对象通信也搞信号槽
// → 直接函数调用或回调
```

## 11. 综合自测问题

1. **RAII 的核心原理是什么？为什么说它是 C++ 内存管理的基石？**
   提示：构造获取、析构释放、栈展开保证

2. **`std::lock_guard<std::mutex>(mtx);` 这行代码有什么问题？**
   提示：临时对象立即析构，锁在这一行结束就释放了

3. **CRTP 和虚函数多态的本质区别是什么？各适用什么场景？**
   提示：编译期 vs 运行时，能否用基类指针统一存储

4. **Pimpl 模式中，为什么析构函数必须在 .cpp 中定义？**
   提示：unique_ptr 的删除器需要完整类型

5. **NVI 模式的虚函数为什么要设为 private？派生类还能 override 吗？**
   提示：C++ 中访问控制和虚函数覆盖是正交的

6. **Meyers' Singleton 的线程安全是如何保证的？底层用了什么机制？**
   提示：C++11 magic statics, `__cxa_guard_acquire`

7. **什么是 SIOF？Meyers' Singleton 如何解决它？**
   提示：不同翻译单元全局变量初始化顺序未定义

8. **C++11 之前的 DCLP 为什么不安全？**
   提示：指令重排，ptr 赋值可能先于构造函数调用

9. **Copy-and-Swap 如何同时解决自赋值和异常安全？**
   提示：拷贝在参数传递时完成，swap 是 noexcept

10. **Tag Dispatch、SFINAE、if constexpr 三种编译期分发的优劣？**
    提示：可读性、错误信息、C++ 版本要求

11. **观察者模式中如何避免悬空回调？**
    提示：weak_ptr 检测生命周期，或 RAII Connection 自动断开

12. **注册表工厂相比简单工厂的优势和劣势？**
    提示：开闭原则 vs 运行时错误

13. **为什么尽量避免使用单例模式？有什么替代方案？**
    提示：全局状态、测试困难、依赖注入替代

14. **RAII 对象在构造函数中抛异常，析构函数会被调用吗？为什么？**
    提示：构造未完成，对象不存在，析构不会被调用

15. **enable_shared_from_this 是如何利用 CRTP 实现的？**
    提示：基类持有 weak_ptr，shared_ptr 构造时注入

## 12. 惯用法选型速查表

| 场景 | 推荐惯用法/模式 | 理由 |
|------|----------------|------|
| 管理文件/锁/内存等资源 | **RAII** | 确定性释放，异常安全 |
| 性能关键的静态多态 | **CRTP** | 零虚表开销，可内联 |
| 需要运行时多态的异构容器 | **虚函数** | 统一基类指针存储 |
| 大型项目减少编译时间 | **Pimpl** | 编译防火墙 |
| 动态库保持 ABI 稳定 | **Pimpl** | 头文件不暴露实现 |
| 框架层强制调用流程 | **NVI** | 基类控制前置/后置处理 |
| 全局唯一资源（日志/配置） | **Meyers' Singleton** | 线程安全+延迟初始化 |
| 按类型名创建对象 | **注册表工厂** | 运行时可扩展 |
| 事件通知/回调 | **观察者（Signal）** | 松耦合 |
| 自定义赋值运算符 | **Copy-and-Swap** | 强异常安全 |
| 编译期按类型特征分发 | **if constexpr**（C++17）或 **Tag Dispatch** | 零运行时开销 |

### 决策流程图

```
需要创建对象？
├── 类型编译期已知 → 直接 make_unique
└── 类型运行时决定 → 工厂模式
    ├── 种类固定 → 简单工厂
    └── 需要扩展 → 注册表工厂

需要多态？
├── 运行时多态 + 异构容器 → 虚函数
├── 编译期多态 + 性能优先 → CRTP
└── 统一流程控制 → NVI

需要管理资源？
└── 一律 RAII（unique_ptr/lock_guard/自定义封装）

需要全局唯一？
├── 真的需要 → Meyers' Singleton
└── 可以避免 → 依赖注入（优先）

需要事件通知？
├── 简单场景 → std::function 回调
└── 多观察者 → Signal + weak_ptr

需要安全赋值？
└── 自定义资源类 → Copy-and-Swap
```

> **全章总结**：C++ 惯用法的本质是将语言特性（RAII、模板、移动语义）转化为可复用的编程范式。它们不是 GoF 模式的翻译，而是 C++ 程序员解决资源安全、编译效率、零开销抽象等实际工程问题的结晶。掌握这些惯用法，才能写出真正"C++ 风格"的代码。
