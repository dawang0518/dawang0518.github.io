### inline 关键字

#### 1. 概念与本质
`inline`是对编译器的**内联建议**，请求将函数调用处替换为函数体，消除调用开销；现代C++中更重要的语义是**允许多翻译单元定义同名函数而不违反ODR（One Definition Rule）**。

#### 2. 底层原理

##### 函数调用的开销（汇编层面）
普通函数调用在x86-64上产生如下开销：
```asm
; 调用 int add(int a, int b)
mov  edi, 1          ; 第一个参数放入 edi
mov  esi, 2          ; 第二个参数放入 esi
call _Z3addii        ; 压入返回地址，跳转到函数
; --- 函数内部 ---
push rbp             ; 保存调用者栈帧
mov  rbp, rsp        ; 建立新栈帧
; ... 函数体 ...
pop  rbp             ; 恢复栈帧
ret                  ; 弹出返回地址，跳回调用点
```
对于一个只做 `return a + b;` 的函数，函数体只需1条 `lea eax, [edi+esi]`，但调用/返回开销（call/ret/push/pop）可能比函数体本身还大。

##### inline 展开后
```asm
; 内联后，直接在调用处生成函数体
mov  edi, 1
mov  esi, 2
lea  eax, [edi+esi]  ; 直接计算，无call/ret开销
```

##### 现代编译器的内联决策
编译器使用**成本模型**决定是否内联，核心因素：
- 函数体大小（指令条数）
- 调用频率（热点函数更可能被内联）
- 调用上下文（能否带来常量传播/死代码消除等后续优化）

GCC/Clang 通过 `-finline-limit=N` 控制内联阈值，`__attribute__((always_inline))` 可强制内联。

##### ODR 语义（C++17 inline 变量）
```cpp
// header.h 中定义 inline 变量，多个 .cpp 包含不会链接报错
inline int global_config = 42;  // C++17 inline 变量
```
链接器看到多个翻译单元中的同名 inline 符号时，选择其中一份，丢弃其余（COMDAT 机制）。

#### 3. 代码示例

```cpp
// 内联函数定义（必须在头文件中，或定义处加 inline）
inline int square(int x) {
    return x * x;
}

int main() {
    int a = square(5);  // 编译器可能替换为: int a = 5 * 5; → int a = 25;
    return a;
}
```

#### 4. 对比表：inline vs 宏

| 对比项       | inline 函数                    | 宏 (`#define`)                 |
|--------------|-------------------------------|-------------------------------|
| 展开阶段     | 编译期                        | 预处理期（文本替换）          |
| 类型检查     | 有，遵循函数规则              | 无，纯文本替换               |
| 作用域       | 遵循C++作用域规则             | 全局替换，无作用域           |
| 调试         | 可设断点                      | 无法调试                     |
| 副作用安全   | 参数只求值一次                | 参数可能被多次求值           |
| 展开可控性   | 编译器可忽略                  | 必定展开                     |
| ODR 语义     | 允许多定义（链接器选一份）    | 无此概念                     |

宏的副作用陷阱：
```cpp
#define MAX(a, b) ((a) > (b) ? (a) : (b))
int x = 1, y = 2;
int z = MAX(x++, y++);  // x++ 或 y++ 可能被求值两次！

// inline 版本无此问题
inline int safe_max(int a, int b) { return a > b ? a : b; }
```

#### 5. 优缺点与权衡

**优点**：
- 消除函数调用开销：省去 call/ret/push/pop 指令，对于函数体只有 1~2 条指令的短小函数，调用开销可能占总执行时间的 50% 以上，内联后性能提升显著
- 触发后续优化链：内联后编译器能看到调用处的上下文，进而做常量传播、死代码消除、循环不变量外提等优化，这些优化在函数边界处无法进行
- ODR 豁免：允许在头文件中定义函数/变量，多个翻译单元包含不会链接报错，简化了 header-only 库的实现

**缺点/代价**：
- 代码膨胀（code bloat）：同一个函数在 N 个调用点各展开一份，生成的二进制体积线性增长。二进制膨胀的真正代价不是磁盘空间，而是 **icache（指令缓存）压力**——展开后的代码分散在不同地址，icache miss 率上升，流水线停顿增多，反而降低性能
- 编译时间增加：头文件中的 inline 函数每个包含它的翻译单元都要编译一遍，修改后所有依赖文件都需重编译
- 调试困难：内联展开后源码行号与机器码不再一一对应，单步调试可能跳来跳去，`-Og` 优化级别下编译器会减少内联以改善调试体验

**权衡**：函数体 <= 5 行且在热路径上被频繁调用时，内联收益明显大于代码膨胀代价。大函数（>= 20 行）即使被频繁调用也不应强制内联，因为 icache miss 的性能损失远大于省下的 call/ret 开销。现代编译器（GCC -O2/Clang -O2）的内联决策已经非常成熟，绝大多数场景下信任编译器即可，只在 profile 数据明确指出瓶颈时才手动干预。

#### 6. 陷阱与实战经验

**❌ 错误：在头文件中定义非 inline 函数**
```cpp
// utils.h
int helper() { return 42; }  // 多个 .cpp 包含 → 链接时 “multiple definition” 错误
```

**✅ 正确：加 inline 或放到 .cpp 中**
```cpp
// utils.h
inline int helper() { return 42; }  // inline 允许多翻译单元定义
```

**❌ 错误：对大函数强制 inline，导致代码膨胀**
```cpp
// 100行的函数被10个地方调用 → 生成约1000行重复代码 → icache 压力增大，性能反而下降
__attribute__((always_inline)) void huge_function() { /* 100行 */ }
```

**✅ 正确：信任编译器，只对关键热路径小函数用 inline**

#### 7. 特殊场景：inline 与 virtual
- 虚函数通过指针/引用调用时走 vtable，**运行期才知道调用哪个版本**，无法内联。
- 通过具体对象调用虚函数，编译器能确定版本，可能内联。

```cpp
class Base {
public:
    inline virtual void who() { std::cout << “Base\n”; }
};
class Derived : public Base {
public:
    void who() override { std::cout << “Derived\n”; }
};

int main() {
    Derived d;
    d.who();        // 编译期确定版本 → 可能内联
    Base* p = &d;
    p->who();       // 运行期 vtable 查找 → 不可内联
}
```

#### 8. 权衡决策

```
需要函数？
  ├─ 函数体 ≤ 5行 且 调用频繁？
  │    ├─ 是 → 定义在头文件，加 inline
  │    └─ 否 → 定义在 .cpp，不加 inline
  ├─ 需要跨翻译单元共享全局变量？
  │    └─ C++17: inline 变量
  └─ 需要宏的”文本粘贴”能力？
       └─ 考虑 constexpr 函数 或 模板，而非宏
```

> **面试速答**
>
> inline 说白了是给编译器的一个内联建议，希望它在调用处直接展开函数体，把 push/pop/call/ret 这套调用开销省掉。对于短小函数，这些开销可能比函数体本身还大，内联就是为这个场景存在的。不过 C++17 之后 inline 还有个更重要的语义是 ODR 豁免——允许你在头文件里直接定义函数或变量，而不会违反链接规则报 multiple definition。
>
> 底层上，编译器根据成本模型自主决策是否真的内联——函数体大小、调用频率、能不能触发常量传播都会影响判断。链接器那边则靠 COMDAT 段把多份相同定义合并，只保留一份。内联成功能带来零调用开销，还能顺带触发常量传播和死代码消除这些后续优化；代价是代码膨胀，icache 压力会变大。
>
> 要特别注意 inline 只是建议不是命令，编译器可以不理你。另外跟宏比一下——inline 有类型检查、有作用域、能调试，宏是无脑文本替换，参数还可能被多次求值，现代 C++ 里能用 inline 就别用宏。
>
> 实际工程里，短小的热点函数可以加 inline，复杂函数不要强制内联（编译器比你聪明），头文件里定义函数或变量必须加 inline 否则链接会挂。

### const 关键字

#### 1. 概念与本质
`const`声明一个**编译期只读约束**：被修饰的对象在其生命周期内不可通过该名字/路径修改。

#### 2. 底层原理

##### const 与 .rodata 段
全局/静态 `const` 变量在编译后通常被放入 ELF 的 `.rodata`（read-only data）段。操作系统将 `.rodata` 段映射到只读内存页，写入会触发 **SIGSEGV** (/sɪɡ ˈseɡvi/ 中文谐音：**西格 赛格维**)：

```cpp
// 全局 const → 编译后进入 .rodata 段
const int MAGIC = 0xDEADBEEF;

int main() {
    // 用 const_cast 去掉 const 后写入 → 运行时段错误（SIGSEGV）
    // 因为 MAGIC 所在内存页是只读的
    int* p = const_cast<int*>(&MAGIC);
    *p = 42;  // 未定义行为！全局const在.rodata中，写入触发段错误
}
```

用 `objdump` 验证：
```bash
g++ -c const_demo.cpp -o const_demo.o
objdump -t const_demo.o | grep MAGIC
# 输出: 0000000000000000 r MAGIC   ← 'r' 表示 .rodata 段
```

> **注意**：局部 `const` 变量在栈上，不在 `.rodata` 中。`const_cast` 去 const 后修改局部 const 变量是未定义行为（UB），但不一定段错误，可能”静默成功”——这更危险。

##### const 折叠（Constant Folding）
编译器遇到 `const` 变量时，可能在编译期直接用其值替换所有引用（类似宏替换），称为 const 折叠：

```cpp
const int N = 100;
int arr[N];  // 编译器将 N 折叠为 100，等价于 int arr[100]
```

这就是为什么 C++ 中 `const int` 可以用作数组大小，而 C 中不行（C 的 `const` 不参与常量折叠）。

#### 3. 用法速查

##### 修饰变量/指针/引用
```cpp
const int num = 10;        // 只读变量
const int* p1 = &num;     // 指向 const 的指针（指向内容不可改，指针可改）
int* const p2 = &a;       // const 指针（指针不可改，指向内容可改）
const int* const p3 = &a; // 双 const
const int& ref = num;     // const 引用，可绑定临时值
```

**口诀**：`const` 在 `*` 左边修饰指向内容，在 `*` 右边修饰指针本身。

##### 修饰成员函数
```cpp
class Foo {
    int x_ = 0;
public:
    int getX() const { return x_; }   // const 成员函数：this 变为 const Foo*
    void setX(int v) { x_ = v; }
    
    // const 重载：根据对象是否 const 调用不同版本
    int& at(int i)       { return data_[i]; }
    const int& at(int i) const { return data_[i]; }
private:
    int data_[10]{};
};
```

> `const` 成员函数中，`this` 的类型是 `const Foo*`，所以不能修改任何非 `mutable` 成员。

##### 修饰函数参数和返回值
```cpp
void func(const std::string& s);  // 避免拷贝，禁止修改
const std::string& getName() const; // 返回 const 引用
```

#### 4. 对比表：const vs constexpr

| 对比项         | `const`                          | `constexpr`                        |
|----------------|----------------------------------|------------------------------------|
| 求值时机       | 可运行期初始化                   | **必须**编译期求值                 |
| 可用作模板参数 | 仅当值为编译期常量时             | 始终可以                           |
| 可用作数组大小 | C++中可以（const折叠）           | 始终可以                           |
| 修饰函数       | 仅修饰成员函数（表示不改状态）   | 表示函数**可在编译期求值**         |
| 语义           | “我承诺不修改”                   | “编译器必须在编译期算出来”         |
| 适用场景       | 接口约束、参数保护               | 编译期常量、模板元编程             |

**决策树**：
```
需要编译期常量（模板参数、数组大小、case标签）？
  ├─ 是 → constexpr
  └─ 否 → 运行期才知道值？
       ├─ 是 → const（如 const int n = getUserInput();）
       └─ 否 → 值编译期可知 → 优先 constexpr
```

#### 5. 优缺点与权衡

**优点**：
- 编译器优化：const 变量可被编译器做常量折叠（constant folding），直接将变量替换为字面值，省去内存访问；全局 const 放入 .rodata 段，操作系统提供硬件级写保护
- 接口契约：`const T&` 参数既避免大对象拷贝又承诺不修改，让调用者放心传入。const 成员函数明确标记"不改变对象状态"，是 const 对象唯一能调用的成员函数
- 线程安全基础：C++ 标准规定 const 成员函数应该是线程安全的（不修改共享状态），这是标准库容器和算法并发安全的基石

**缺点/代价**：
- const 正确性的传染性：一旦底层函数的参数标了 const，整条调用链上所有相关函数都必须标 const，否则编译不过。在大型遗留代码库中补 const 可能需要修改数十个函数签名，工程成本高
- mutable 的例外增加复杂度：缓存、互斥锁等场景需要在 const 成员函数中修改成员，必须用 mutable 修饰，破坏了"const = 不可变"的简单心智模型
- const_cast 滥用风险：与 C 库或遗留 API 交互时常需 const_cast 去掉 const，如果误对真正的 const 对象去 const 后写入则是 UB（全局 const 在 .rodata 中，写入直接 SIGSEGV）

**权衡**：const 正确性的工程代价主要在初始标注阶段，一旦建立起来后续维护成本很低，且编译器优化收益和代码安全性收益是持续的。新项目应从第一天就贯彻 const 正确性；遗留项目可以从公共 API 边界逐步向内推进，而非一次性全量改造。

#### 6. 陷阱与实战经验

**❌ 错误：const_cast 去 const 后修改原本为 const 的对象**
```cpp
const int x = 42;
int* p = const_cast<int*>(&x);
*p = 100;  // 未定义行为！编译器可能已将 x 折叠为 42
std::cout << x;  // 可能输出 42（const 折叠），也可能输出 100
```

**✅ 正确用法：const_cast 用于去掉接口层面多余的 const**
```cpp
// 底层数据本身非 const，但 API 返回了 const 指针
void legacy_api(const char* s);
char buf[] = “hello”;
legacy_api(buf);  // OK，buf 本身非 const
```

**❌ 错误：忘记 const 成员函数，导致 const 对象无法调用**
```cpp
class Config {
public:
    std::string getName() { return name_; }  // 非 const！
};
void print(const Config& cfg) {
    cfg.getName();  // 编译错误：const 对象不能调用非 const 成员函数
}
```

**✅ 正确：所有不修改状态的成员函数都应加 const**
```cpp
std::string getName() const { return name_; }
```

#### 6. 代码总结速查

```cpp
// --- 变量与指针 ---
const int a = 10;                // 只读变量
const int* p1 = &a;             // 指向 const 的指针
int* const p2 = &b;             // const 指针
const int* const p3 = &a;       // 双 const

// --- 引用 ---
const int& ref = a;              // const 引用
void func(const std::string& s); // 参数：避免拷贝 + 禁止修改

// --- 类 ---
class A {
    const int id;                // const 成员，必须用初始化列表
    A(int i) : id(i) {}
    int getId() const;           // const 成员函数
    int getId();                 // 非 const 重载
};
const A obj(1);                  // const 对象，只能调用 const 成员函数
```

> **面试速答**
>
> const 本质上是一个编译期的只读约束，告诉编译器"这个值不会被改"，编译器据此做 const 折叠之类的优化。为什么要搞这个？一来防止意外修改，二来支持编译期优化，三来在接口层面表达契约——比如 const 引用参数就是对调用者承诺"我不会动你的数据"。
>
> 底层实现上，全局 const 变量会被放到 .rodata 只读段，这是硬件级的写保护，真要往里写会直接触发 SIGSEGV。局部 const 就在栈上，硬件不会保护，但编译器会做 const 折叠，直接把用到的地方替换成字面值。好处是安全加优化都有了，代价是 const 正确性需要贯穿整个调用链，一处漏标就会传染开来。
>
> 有几个容易踩的坑：用 const_cast 去掉 const 后再改全局 const 是 UB，因为 .rodata 写入会崩；const 成员函数内部想改成员变量得配合 mutable 关键字。另外要分清 const 和 constexpr——const 只是运行时只读，constexpr 是编译期求值保证，而且 constexpr 变量本身就隐含 const。
>
> 实际工程里的选型：能用 constexpr 就不用 const；函数参数尽量用 const& 避免拷贝；成员函数只要不改状态就标上 const。

### static 关键字

#### 1. 概念与本质
`static`有三重语义：**延长生命周期**（局部静态变量）、**限制链接可见性**（文件作用域静态）、**类级别共享**（静态成员）。核心是控制存储期（storage duration）和链接性（linkage）。

#### 2. 底层原理

##### 存储位置：.bss 和 .data 段
```
ELF 可执行文件布局：
┌─────────────────┐
│ .text（代码段）   │
├─────────────────┤
│ .rodata（只读数据）│
├─────────────────┤
│ .data（已初始化静态数据）│  ← static int x = 42; 在这里
├─────────────────┤
│ .bss（未初始化静态数据） │  ← static int y; 在这里（零初始化）
├─────────────────┤
│ 堆（heap）        │
│       ↓          │
│       ↑          │
│ 栈（stack）       │
└─────────────────┘
```
- 已初始化的 static 变量 → `.data` 段
- 未初始化/零初始化的 static 变量 → `.bss` 段（不占文件空间，加载时零填充）

##### ELF 符号可见性
`static` 修饰的全局变量/函数在符号表中标记为 **LOCAL**（`t`/`d`），链接器不会导出：

```bash
# 查看符号类型
nm -C file1.o
# T external_func   ← 大写 T = 全局符号（external linkage）
# t static_func     ← 小写 t = 局部符号（internal linkage）
# D external_var    ← 大写 D = 全局数据
# d static_var      ← 小写 d = 局部数据
```

##### C++11 Magic Statics（局部静态变量的线程安全）
C++11 标准保证局部 static 变量的初始化是**线程安全**的。编译器会插入类似双重检查锁的机制：

```cpp
Singleton& getInstance() {
    static Singleton instance;  // C++11 保证线程安全初始化
    return instance;
}
```

编译器生成的伪代码（GCC 使用 `__cxa_guard_acquire`）：
```cpp
// 编译器生成的等价逻辑
static Singleton instance;
static guard_t guard;  // 守护变量

Singleton& getInstance() {
    if (!guard.initialized) {           // 第一次检查（无锁快路径）
        __cxa_guard_acquire(&guard);    // 获取锁
        if (!guard.initialized) {       // 第二次检查
            new (&instance) Singleton();
            guard.initialized = true;
        }
        __cxa_guard_release(&guard);    // 释放锁
    }
    return instance;
}
```

> **重要**：C++11 之前，局部 static 初始化**不是线程安全的**，多线程可能导致重复构造。

#### 3. 代码示例

##### 局部静态变量
```cpp
#include <iostream>

int counter() {
    static int count = 0;  // 只初始化一次，生命周期贯穿整个程序
    return ++count;
}

int main() {
    std::cout << counter() << “\n”;  // 1
    std::cout << counter() << “\n”;  // 2
    std::cout << counter() << “\n”;  // 3
}
```

##### 类静态成员
```cpp
class Connection {
    static int total_count_;    // 声明（类内）
public:
    Connection()  { ++total_count_; }
    ~Connection() { --total_count_; }
    static int getCount() { return total_count_; }  // 静态成员函数
};

int Connection::total_count_ = 0;  // 定义（类外，必须！）
```

#### 4. static 的内部链接属性（跨文件符号管理）

`static`修饰的全局变量和函数具有**内部链接（internal linkage）**属性，即该符号仅在当前编译单元（.cpp文件）中可见，其他文件即使用`extern`声明也无法访问。这是`static`在大型项目中防止符号冲突的关键机制。

**实战验证：多文件编译中的符号可见性**

```cpp
// common.h —— 声明外部链接的符号
#pragma once
void external_func();          // 外部链接函数声明
extern int external_var;       // 外部链接变量声明
```

```cpp
// file1.cpp —— 定义内部链接和外部链接的符号
#include “common.h”

// static 修饰 → 内部链接，仅 file1.cpp 可见
static void static_func() {}
static int static_var = 20;

// 外部链接函数，可以在内部使用 static 符号
void external_func() {
    static_func();    // 同文件内可正常调用
    static_var++;     // 同文件内可正常访问
}

int external_var = 10;  // 外部链接变量
```

```cpp
// file2.cpp —— 尝试跨文件访问 static 符号
#include “common.h”

extern void static_func();  // 即使用 extern 声明，也无法访问

int main() {
    external_func();   // ✅ 外部链接函数，正常调用
    static_func();     // ❌ 链接时报错：undefined reference
    return external_var; // ✅ 外部链接变量，正常访问
}
```

**用 `nm` 命令查看符号表验证**：
```bash
g++ -c file1.cpp -o file1.o
nm file1.o
```
```text
0000000000000000 T __Z13external_funcv  # T = 外部链接函数（全局可见）
0000000000000012 t __Z10static_funcv    # t = 内部链接函数（仅本文件可见）
0000000000000004 D _external_var        # D = 外部链接变量（全局可见）
0000000000000008 d _static_var          # d = 内部链接变量（仅本文件可见）
```

> **关键区分**：符号表中大写字母（T/D）表示外部链接，小写字母（t/d）表示内部链接。链接器只能解析外部链接符号，`static`修饰的符号对其他编译单元完全不可见。

#### 5. 对比表：static vs 匿名 namespace

| 对比项             | `static`（文件作用域）         | 匿名 `namespace`              |
|--------------------|-------------------------------|-------------------------------|
| 适用目标           | 变量、函数                    | 变量、函数、类型、模板        |
| C 兼容             | 是                            | 否（C++ only）                |
| 可修饰类/模板      | 否                            | 是                            |
| C++ 推荐           | 不推荐用于隐藏符号            | **推荐**                      |
| 链接属性           | internal linkage              | internal linkage              |

**决策树**：
```
需要限制符号仅本文件可见？
  ├─ C 代码 → static
  └─ C++ 代码
       ├─ 仅变量/函数 → 匿名 namespace（推荐）或 static
       └─ 包含类型/模板 → 匿名 namespace（static 做不到）
```

#### 6. 优缺点与权衡

**优点**：
- 生命周期控制：局部 static 变量跨函数调用存活，适合实现惰性初始化（如 Meyers 单例），且 C++11 保证线程安全初始化
- 符号隔离：文件级 static（或匿名 namespace）将符号限制在当前编译单元，避免全局命名空间污染和符号冲突，是大型项目模块化的基本手段
- 共享状态：类 static 成员属于类而非实例，天然适合统计实例计数、全局配置等场景，不需要外部全局变量

**缺点/代价**：
- 全局状态导致测试困难：static 变量的状态在测试用例之间不会自动重置，导致测试之间互相影响、顺序依赖。单元测试中 mock 全局 static 对象非常困难，通常需要依赖注入重构
- 线程安全代价：类 static 成员变量被所有实例共享，多线程并发访问需要额外加锁（mutex），而这个锁的竞争程度与实例数量和访问频率正相关
- 初始化顺序未定义（Static Initialization Order Fiasco）：不同编译单元的全局/静态变量初始化顺序由链接器决定，未被标准定义。如果 A.cpp 的全局 static 依赖 B.cpp 的全局 static，可能在 B 初始化前就被访问，导致 UB
- 析构顺序问题：全局 static 的析构顺序与构造顺序相反，但跨编译单元的析构顺序仍然不可预测，析构时访问已析构的其他 static 对象是常见的隐蔽 bug

**权衡**：局部 static 用于单例和惰性初始化非常合适（C++11 线程安全保证），但全局 static 状态应尽量避免——它让代码变成隐式有状态的，破坏可测试性和可推理性。如果必须用全局状态，优先用 Construct On First Use 惯用法（局部 static）避免初始化顺序问题，并用依赖注入让测试代码可以替换全局状态。

#### 7. 陷阱与实战经验

**❌ 错误：头文件中定义 static 全局变量**
```cpp
// config.h
static int g_timeout = 30;  // 每个包含此头文件的 .cpp 都有独立副本！
// a.cpp 改 g_timeout 不会影响 b.cpp 中的 g_timeout
```

**✅ 正确：头文件中用 extern 声明，.cpp 中定义**
```cpp
// config.h
extern int g_timeout;
// config.cpp
int g_timeout = 30;
```

**❌ 错误：C++11 之前依赖局部 static 做单例（线程不安全）**
```cpp
// C++03 下多线程调用可能构造两次
Singleton& getInstance() {
    static Singleton s;  // C++03：非线程安全
    return s;
}
```

**✅ 正确：C++11 及以上，magic statics 已保证线程安全**

> **面试速答**
>
> static 是一个关键字带三重语义的典型——局部 static 控制存储期让变量跨调用存活，文件 static 控制链接性避免符号污染全局命名空间，类 static 控制归属让成员属于类而不是某个对象提供不依赖实例的共享状态。每种场景解决的问题都不一样。
>
> 底层上局部和全局 static 存储在 .data 段（已初始化）或 .bss 段（零初始化）。C++11 之后局部 static 的初始化是线程安全的，这就是 magic statics 特性，底层用 `__cxa_guard_acquire` 实现了双重检查锁，保证多线程并发进入时只初始化一次。
>
> 有个经典大坑叫 Static Initialization Order Fiasco——不同编译单元之间的全局 static 变量初始化顺序是未定义的，跨文件依赖全局 static 很容易踩雷，这时候要用局部 static 或 Construct On First Use 惯用法绕过去。还有个对比要知道：文件级隐藏符号现在推荐用匿名 namespace 而不是 static，因为前者还能隐藏类和模板，static 做不到。
>
> 选型上单例用 Meyers 单例（局部 static），C++11 之后天然线程安全；想隐藏内部实现用匿名 namespace；类级共享数据用类 static 成员。

### volatile 关键字

#### 1. 概念与本质
`volatile`告诉编译器：**这个变量可能被当前代码流之外的因素修改，禁止对其读写做优化**。每次访问必须真正读写内存。

#### 2. 底层原理

##### 编译器优化与 volatile 的对抗
无 volatile 时，编译器可能做以下优化：
1. **寄存器缓存**：将变量值缓存到寄存器，后续读取不再访问内存
2. **死存储消除**：连续多次写同一变量，只保留最后一次
3. **指令重排**：将不相关的读写重新排序以提升流水线效率

```cpp
// 无 volatile：编译器可能优化为只读一次
int* status = (int*)0xFF00;
while (*status == 0) {}   // 可能被优化为 if (*status == 0) while(1) {}

// 有 volatile：每次循环都真正读内存
volatile int* status = (volatile int*)0xFF00;
while (*status == 0) {}   // 每次循环都产生一条 load 指令
```

用 GCC 查看汇编差异：
```bash
# 无 volatile
g++ -O2 -S test.cpp
# 循环体中没有 load 指令，status 的值被缓存在寄存器

# 有 volatile
# 循环体中每次都有 mov (%rax), %edx 从内存读取
```

##### 编译器屏障 vs 内存屏障（关键区分）

| 概念         | 编译器屏障（Compiler Barrier）    | 内存屏障（Memory Barrier/Fence）  |
|--------------|----------------------------------|----------------------------------|
| 作用层面     | **编译器**指令重排               | **CPU** 指令重排 + 缓存一致性    |
| 实现方式     | `volatile` / `asm volatile("":::"memory")` | `std::atomic_thread_fence` / CPU 指令（mfence, dmb）|
| 解决的问题   | 编译器把读写优化掉/重排          | CPU 乱序执行 + store buffer 导致其他核看到旧值 |

**volatile 只是编译器屏障，不是内存屏障！** 在多核 CPU 上，即使编译器不重排，CPU 也可能乱序执行，导致其他核心看不到最新值。

###### 核心：代码到执行经历两次重排

要理解这俩屏障的区别，必须先明白一件事——**源代码到实际执行其实经过了两层重排**：

```
C++ 源代码
    ↓  ← 编译器可能重排（编译时）
机器指令
    ↓  ← CPU 可能重排（运行时）
实际执行
```

**编译器屏障**只管第一层，**内存屏障**管两层。

###### 第一层：编译器乱序

你写：
```cpp
int x = 1;    // ①
int y = 2;    // ②
```

编译器一看：这俩互相不依赖，把 ② 先执行可能寄存器调度更优，于是生成的机器码可能是：
```asm
mov DWORD PTR [y], 2      ; ② 先
mov DWORD PTR [x], 1      ; ① 后
```

**编译器认为**：单线程语义下谁先谁后读回来结果一样，我有权重排。

###### 第二层：CPU 乱序

就算机器码顺序是对的，CPU 执行时有个东西叫 **store buffer**——写操作不会立刻进 cache，而是先进 store buffer 排队。不同地址的 store 进入 cache 的先后，可能跟指令顺序不一样。**从其他 CPU 核心看过来，就像顺序被打乱了**。

这不是编译器的问题，是**硬件的问题**——编译器完全按顺序生成了指令，CPU 自己给乱了。

###### 两种屏障分别管哪一层

**编译器屏障**：
```cpp
asm volatile("" ::: "memory");                       // GCC 内嵌汇编
std::atomic_signal_fence(std::memory_order_seq_cst); // C++11 标准写法
```
- 告诉编译器"这行前后的内存读写不能跨过来"
- **不生成任何机器指令**——汇编里看不到它
- **CPU 运行时该怎么乱序还是怎么乱序**

**内存屏障**：
```cpp
std::atomic_thread_fence(std::memory_order_seq_cst); // C++ 标准
asm volatile("mfence" ::: "memory");                 // x86 直接写
```
- 既阻止编译器重排
- **又生成真实的 CPU 屏障指令**（x86 是 `mfence`/`lfence`/`sfence`，ARM 是 `dmb`/`dsb`）
- CPU 遇到这条指令时会 flush store buffer、等待前面的内存操作完成

所以 **内存屏障 = 编译器屏障 + CPU 指令**。

###### 典型出 bug 的例子

```cpp
int data = 0;
int ready = 0;

// 线程 A
data = 42;        // ①
ready = 1;        // ②

// 线程 B
while (!ready);                // ③ 等 ready 变成 1
printf("%d\n", data);          // ④ 期望打印 42
```

直觉上线程 B 看到 ready = 1 时 data 应该是 42，但实际上**可能打印 0**。两种原因：

**情况 1：编译器调换了 ① ② 顺序**
编译器觉得这俩互不依赖，生成的机器码可能是先写 ready 再写 data。线程 B 跳出循环时，data 还是 0。

**情况 2：编译器没调，CPU 调了**
机器码顺序正确，但 store buffer 里 `data = 42` 和 `ready = 1` 进入 cache 的先后不保证。如果 `ready = 1` 先被其他核心看到，B 就提前跳出，读到 data = 0。

三种解法对比：

```cpp
// ❌ 只加编译器屏障——不够
data = 42;
asm volatile("" ::: "memory");   // 只挡编译器
ready = 1;
// 防住情况 1，挡不住情况 2；在 x86 上碰巧能跑，到 ARM 必炸

// ✅ 加完整内存屏障——可靠
data = 42;
std::atomic_thread_fence(std::memory_order_release);
ready = 1;

// ✅ 直接用 atomic——推荐
std::atomic<int> ready{0};
data = 42;
ready.store(1, std::memory_order_release);
```

###### 什么时候只用编译器屏障就够

**单线程内的场景**——比如信号处理：
```cpp
volatile sig_atomic_t flag = 0;
void handler(int) { flag = 1; }   // 同一 CPU 同一线程被中断

int main() {
    while (!flag) {
        asm volatile("" ::: "memory");   // 够了
    }
}
```
信号处理器是**同一个 CPU 上的同一线程**被中断执行的，没有跨核心 store buffer 问题，只要阻止编译器把 `while (!flag)` 优化成 `while (!cached_flag)` 就行。

还有一个场景是**性能测量防止重排**：
```cpp
auto start = now();
asm volatile("" ::: "memory");   // 别把 work() 跨到 start 前面
work();
asm volatile("" ::: "memory");   // 别把 work() 跨到 end 后面
auto end = now();
```
单线程测量，不需要管 CPU，只要挡住编译器别把 work 重排到计时区间外即可。

###### x86 vs ARM：为什么移植常炸

| 平台 | 内存模型 | 需要内存屏障吗 |
|------|---------|---------------|
| **x86 / x64** | 强模型（TSO）| 很多场景不需要 mfence |
| **ARM / ARM64** | 弱模型 | 几乎每个共享状态都需要 dmb |
| **PowerPC** | 最弱 | 到处都要 sync |

x86 的 CPU 乱序规则非常克制：load 不会重排到前面的 load 之前、store 不会重排到前面的 store 之前，只有 **StoreLoad**（读被重排到写之前）会发生。所以 x86 上只用编译器屏障的代码**碰巧**经常能跑。

ARM 几乎所有类型的乱序都会发生——**同一份代码 x86 跑通不代表正确，到 ARM 可能立刻炸**。这就是"x86 是程序员的安全网"那个梗的由来，很多隐藏的数据竞争被强内存模型掩盖了。

###### 四种屏障方向 & C++ 内存序对应

| 屏障类型 | 阻止什么 | 直觉 |
|---------|---------|------|
| **LoadLoad** | load 重排到 load 之前 | 读的顺序不变 |
| **LoadStore** | store 重排到 load 之前 | 读完才写 |
| **StoreStore** | store 重排到 store 之前 | 写的顺序不变 |
| **StoreLoad** | load 重排到 store 之前 | 最贵，x86 也需要 mfence |

C++11 的内存序对应关系：

| 内存序 | 包含的屏障 |
|--------|----------|
| `relaxed` | 无 |
| `acquire` | LoadLoad + LoadStore（load 侧） |
| `release` | LoadStore + StoreStore（store 侧） |
| `acq_rel` | acquire + release |
| `seq_cst` | 全部（含 StoreLoad） |

这也解释了为什么 `seq_cst` 最贵——必须插入 StoreLoad 屏障，x86 上对应慢指令 `mfence`。

###### 类比：工地比喻

- **编译器** = 工头，拿到施工图（源代码），会调整任务顺序提升效率
- **CPU** = 工人，即便工头按顺序下命令，工人也可能并发执行、完成顺序打乱
- **编译器屏障** = 告诉工头"这里别调顺序"——但工人还是可能乱序完工
- **内存屏障** = 除了告诉工头，还让工头喊一声"停，所有工人同步一下"

**一句话记忆**：编译器屏障只管源码→机器码这一层，内存屏障既管源码→机器码又管机器码→实际执行。多线程共享数据一定要内存屏障，单线程优化欺骗用编译器屏障就够。

##### 为什么 volatile 不能替代 atomic
```
线程A 写入 volatile int x = 1;
  ↓
  写入 store buffer（CPU 流水线优化）
  ↓
  store buffer 还没刷到 L1 cache
  ↓
线程B（运行在另一个核心）读 x → 可能读到旧值 0
```
`std::atomic` 会插入适当的内存屏障，确保写入对其他核心可见。

#### 3. 代码示例

##### 硬件寄存器访问（嵌入式经典场景）
```cpp
// 内存映射的硬件状态寄存器
volatile uint32_t* const STATUS_REG = (volatile uint32_t*)0x40021000;
volatile uint32_t* const DATA_REG   = (volatile uint32_t*)0x40021004;

void wait_and_read() {
    // 等待硬件就绪（每次循环必须真正读寄存器）
    while (!(*STATUS_REG & 0x01)) {}
    uint32_t data = *DATA_REG;  // 读取数据寄存器
}
```

##### 信号处理中的 volatile
```cpp
#include <signal.h>
#include <stdio.h>

volatile sig_atomic_t got_signal = 0;  // sig_atomic_t 保证单次读写原子性

void handler(int sig) {
    got_signal = 1;  // 信号处理函数中修改
}

int main() {
    signal(SIGINT, handler);
    while (!got_signal) {
        // 没有 volatile，编译器可能把 got_signal 缓存到寄存器
        // 导致 Ctrl+C 后循环不退出
    }
    printf(“收到信号，退出\n”);
}
```

#### 4. 对比表：volatile vs atomic

| 对比项               | `volatile`                      | `std::atomic<T>`                |
|----------------------|--------------------------------|--------------------------------|
| 禁止编译器优化       | 是                             | 是                             |
| 禁止 CPU 重排        | **否**                         | **是**（根据 memory_order）    |
| 保证原子性           | **否**（i++ 不是原子操作）     | **是**（load/store/RMW 原子）  |
| 多核可见性           | **不保证**                     | **保证**                       |
| 适用场景             | 硬件寄存器、信号处理、调试     | 多线程共享变量                 |
| 性能                 | 无额外开销（只影响编译器）     | 有内存屏障开销                 |

**决策规则**：
```
变量被多线程共享？
  ├─ 是 → std::atomic 或 mutex（永远不要用 volatile）
  └─ 否 → 被硬件/信号处理/调试器修改？
       ├─ 是 → volatile
       └─ 否 → 什么都不用
```

#### 5. 优缺点与权衡

**优点**：
- 零运行时开销：volatile 只影响编译器的代码生成策略（禁止寄存器缓存、死存储消除、指令重排），不插入任何额外的 CPU 指令（如内存屏障），对性能几乎无影响
- 精确控制硬件访问：嵌入式/驱动开发中，MMIO 寄存器的每一次读写都有副作用（如触发硬件状态转移），volatile 保证编译器不会优化掉任何一次读写
- 信号处理安全：`volatile sig_atomic_t` 是 POSIX 标准中信号处理函数与主程序通信的唯一安全方式（不涉及多线程，只涉及异步信号中断）

**缺点/代价**：
- 不保证原子性：`volatile int x; x++;` 仍然是 load-add-store 三步操作，多线程并发执行会产生数据竞争
- 不保证多核可见性：volatile 不插入 CPU 内存屏障，写入可能停留在 store buffer 中，其他核心看到的仍是旧值。在 x86 上由于强内存模型（TSO）某些场景碰巧可见，但在 ARM/RISC-V 等弱内存模型架构上几乎必然出问题
- 阻碍编译器优化：volatile 变量的每次读写都必须访问内存，编译器无法将其缓存到寄存器或消除冗余访问，对性能敏感的循环内频繁使用会明显降低性能

**权衡**：volatile 的正确使用场景非常窄——只有硬件寄存器（MMIO）和信号处理两种。多线程同步必须用 `std::atomic`（提供原子性 + 内存屏障 + 多核可见性保证），即使看到某些遗留代码用 volatile 做多线程"能跑"，那也是因为 x86 的强内存模型掩盖了 bug，移植到 ARM 上必然出问题。选型规则：多线程用 atomic，硬件/信号用 volatile，两者不可互相替代。

#### 6. 陷阱与实战经验

**❌ 错误：用 volatile 做多线程同步**
```cpp
volatile bool ready = false;
volatile int data = 0;

// 线程A
data = 42;
ready = true;

// 线程B
while (!ready) {}
assert(data == 42);  // 可能失败！CPU 可能重排 data 和 ready 的写入顺序
```

**✅ 正确：用 std::atomic**
```cpp
std::atomic<bool> ready{false};
int data = 0;

// 线程A
data = 42;
ready.store(true, std::memory_order_release);  // release 屏障

// 线程B
while (!ready.load(std::memory_order_acquire)) {}  // acquire 屏障
assert(data == 42);  // 保证成功
```

**❌ 错误：认为 volatile 的 i++ 是原子操作**
```cpp
volatile int counter = 0;
// 两个线程同时执行：
counter++;  // 不是原子操作！分解为 load → add → store 三步
// 最终 counter 可能小于预期值
```

**✅ 正确：用 atomic**
```cpp
std::atomic<int> counter{0};
counter.fetch_add(1, std::memory_order_relaxed);  // 原子递增
```

> **面试速答**
>
> volatile 本质是一个编译器屏障，强制每次访问都真正读写内存，禁止编译器做寄存器缓存、死存储消除、指令重排这些优化。为什么要搞这个？因为像 MMIO 硬件寄存器这种地址的值随时可能被外部改变（硬件、DMA、中断都可能动它），编译器不知道这些，如果不加 volatile 它会自作聪明把读写优化掉。
>
> 要特别注意 volatile 只约束编译器生成的指令顺序，不会插入 CPU 内存屏障——多核系统上 CPU 乱序执行和 store buffer 的延迟它完全管不了。所以 volatile 既不保证原子性也不保证多核可见性，这是新手最容易搞混的地方。
>
> 跟 std::atomic 对比一下就清楚了：atomic 既有编译器屏障又有 CPU 内存屏障，还保证原子操作；volatile 只管编译器不管 CPU。所以多线程共享变量必须用 std::atomic 或 mutex，绝对不能用 volatile 当同步原语。
>
> 实际使用：MMIO 硬件寄存器和信号处理（volatile sig_atomic_t）才用 volatile；多线程同步一律用 std::atomic 或 mutex。

**Q7**: 两个线程分别执行 `data=42; ready=true;` 和 `while(!ready){} assert(data==42);`，把 ready 声明为 `volatile bool` 能保证 assert 成功吗？为什么？

### assert 断言

#### 1. 概念与本质
`assert(expr)` 是一个**调试期契约检查宏**：表达式为 false 时终止程序并输出诊断信息；定义 `NDEBUG` 后被编译器完全消除，零运行时开销。

#### 2. 底层原理

`assert` 的典型实现（glibc）：
```cpp
// <cassert> 内部展开
#ifdef NDEBUG
  #define assert(expr) ((void)0)  // Release 模式：完全消除
#else
  #define assert(expr) \
    ((expr) ? (void)0 : __assert_fail(#expr, __FILE__, __LINE__, __func__))
#endif
```
- `#expr`：将表达式转为字符串，输出到 stderr
- `__assert_fail`：调用 `abort()` → 发送 `SIGABRT` → 生成 core dump（如果启用）

#### 3. 代码示例

```cpp
#include <cassert>

double divide(double a, double b) {
    assert(b != 0 && “除数不能为零”);  // 字符串技巧：失败时显示提示信息
    return a / b;
}

int main() {
    divide(10.0, 2.0);  // OK
    // divide(10.0, 0.0);  // 触发断言，输出：
    // Assertion failed: b != 0 && “除数不能为零”, file main.cpp, line 4
}
```

C++11 `static_assert`（编译期断言）：
```cpp
static_assert(sizeof(int) == 4, “需要32位int”);  // 编译期检查
static_assert(sizeof(void*) == 8);  // C++17 可省略消息
```

#### 4. 优缺点与权衡

**优点**：
- Debug 模式零成本检测：assert 在 Debug 构建中捕获"不应该发生"的逻辑错误，触发时输出文件名、行号、表达式文本和函数名，定位问题极其方便
- Release 模式零开销：定义 NDEBUG 后 assert 展开为 `((void)0)`，完全被编译器消除，不产生任何运行时指令
- 自文档化：`assert(ptr != nullptr && "指针不能为空")` 既是运行时检查又是代码意图的文档，比注释更可靠（注释不会过期，但 assert 会在不满足时立即暴露）

**缺点/代价**：
- Release 模式被完全消除的风险：assert 中的表达式在 Release 构建中不会被执行，如果表达式有副作用（如函数调用），程序行为在 Debug 和 Release 之间不一致，这是最常见的 assert 陷阱
- 不适合用户输入校验：assert 在 Release 模式下消失，如果用它校验外部输入（网络数据、用户输入、文件内容），Release 版本就失去了校验，安全漏洞由此产生
- 失败行为过于激烈：assert 失败直接调用 `abort()` 终止进程，在服务端程序中可能导致整个服务崩溃，无法优雅降级

**权衡**：assert 适合检查"程序员的逻辑错误"（前置条件、不变量、后置条件），不适合检查"运行时的外部输入"。对于需要在 Release 模式下也生效的检查，应使用 if + 异常/错误码。C++20 的 contract 属性（`[[expects]]`/`[[ensures]]`）是 assert 的现代替代方案，允许在 Release 模式下选择性保留检查。

#### 5. 陷阱与实战经验

**❌ 错误：assert 中放有副作用的表达式**
```cpp
assert(important_init() == true);  // NDEBUG 后 important_init() 不会被调用！
```

**✅ 正确：副作用表达式独立执行**
```cpp
bool ok = important_init();
assert(ok);
```

**❌ 错误：用 assert 做输入校验（用户输入不可信）**
```cpp
assert(user_input > 0);  // Release 模式下被消除，非法输入不会被检查
```

**✅ 正确：用 if + 异常/错误码处理用户输入**
```cpp
if (user_input <= 0) throw std::invalid_argument(“输入必须为正数”);
```

> **原则**：`assert` 检查”程序员的逻辑错误”（不应该发生的情况），`if` 检查”运行时的外部输入”（可能发生的情况）。

> **面试速答**
>
> assert 是调试期的契约检查宏，static_assert 是 C++11 引入的编译期断言——一个运行时检查，一个编译时检查。assert 专门用来捕获"不应该发生"的程序员逻辑错误，static_assert 则是在编译期就把类型大小、对齐这类约束卡住，越早发现问题越好。
>
> 底层上 assert 表达式为 false 时会调用 `abort()` 终止程序并生成 core dump，定义 NDEBUG 之后整个宏展开为空，运行时零开销。static_assert 不通过直接编译报错，根本走不到运行期。
>
> 有个很容易踩的坑：assert 的表达式里绝对不能有副作用，比如函数调用。因为 Release 编译定义了 NDEBUG 之后 assert 整个消失，副作用也跟着消失，代码行为就变了。另外用户输入校验要用 if + 异常，不能用 assert——用户输入是预期内可能发生的情况，不是逻辑错误。
>
> 选型上程序员的逻辑错误用 assert，编译期能确定的约束用 static_assert，外部输入校验用 if + 异常或错误码。C++20 的 contract 属性（[[expects]]）是 assert 的未来替代方案，目前还没普及。

### sizeof 运算符

#### 1. 概念与本质
`sizeof`是一个**编译期运算符**（不是函数），返回类型或表达式占用的字节数，类型为 `size_t`。

#### 2. 底层原理

##### 编译期求值
`sizeof` 的操作数**不会被求值**，编译器只分析其类型：
```cpp
int i = 0;
sizeof(i++);   // i 不会自增！编译器只看 i 的类型是 int → 返回 4
sizeof(1/0);   // 不会除零错误！编译器只看表达式类型是 int → 返回 4
```

编译后 `sizeof` 直接变成一个常量，没有运行时开销：
```cpp
// 源码
int n = sizeof(double);
// 等价于编译后
int n = 8;  // 编译器直接替换为常量
```

##### VLA 的运行时 sizeof（C99，非标准C++）
C99 的变长数组（VLA）是 `sizeof` 唯一可能在运行时求值的情况：
```c
// C99 代码（非标准 C++）
void func(int n) {
    int arr[n];          // VLA
    size_t s = sizeof(arr);  // 运行时计算：n * sizeof(int)
}
```
> C++ 标准不支持 VLA，但 GCC 作为扩展支持。在标准 C++ 中应使用 `std::vector`。

##### struct 的内存对齐与 sizeof
```cpp
struct A {
    char  c;   // 1 字节 + 3 字节填充
    int   i;   // 4 字节
    char  c2;  // 1 字节 + 3 字节填充
};
// sizeof(A) = 12（不是 6！因为内存对齐）

struct B {
    char  c;   // 1 字节
    char  c2;  // 1 字节 + 2 字节填充
    int   i;   // 4 字节
};
// sizeof(B) = 8（重排字段可减少填充）
```

对齐规则：每个成员的偏移量必须是其对齐值（通常等于自身大小）的倍数，struct 总大小必须是最大对齐值的倍数。

#### 3. 代码示例

```cpp
#include <cstdio>
#include <cstring>

int SizeOf(char pString[]) {
    return sizeof(pString);  // 数组退化为指针 → 返回指针大小（8字节/64位系统）
}

int main() {
    // sizeof 各种类型
    printf("char=%zu, int=%zu, double=%zu, void*=%zu\n",
           sizeof(char), sizeof(int), sizeof(double), sizeof(void*));
    // 典型输出: 1, 4, 8, 8

    char arr[100] = "hello";
    printf("sizeof(arr)=%zu\n", sizeof(arr));     // 100（数组总大小）
    printf("strlen(arr)=%zu\n", strlen(arr));      // 5（字符串长度）
    printf("SizeOf(arr)=%d\n", SizeOf(arr));       // 8（退化为指针）

    // 空类的大小
    struct Empty {};
    printf("sizeof(Empty)=%zu\n", sizeof(Empty));  // 1（C++规定非零，保证每个对象有唯一地址）
}
```

#### 4. 对比表：sizeof vs strlen

| 对比项     | `sizeof`                         | `strlen`                         |
|------------|----------------------------------|----------------------------------|
| 本质       | 编译期运算符                     | 运行时函数（`<cstring>`）        |
| 求值时机   | 编译期（VLA 除外）               | 运行时遍历到 `\0`               |
| 对数组     | 返回数组总字节数                 | 返回字符串长度（不含 `\0`）      |
| 对指针     | 返回指针大小（4/8字节）          | 从指针地址遍历到 `\0`           |
| 包含 `\0`  | 包含                             | 不包含                           |

#### 5. 陷阱与实战经验

**❌ 错误：对函数参数中的数组用 sizeof**
```cpp
void process(int arr[]) {
    int n = sizeof(arr) / sizeof(arr[0]);  // 错误！arr 已退化为指针
    // n = sizeof(int*) / sizeof(int) = 2（64位）或 1（32位），而非数组长度
}
```

**✅ 正确：通过模板或额外参数传递大小**
```cpp
template<size_t N>
void process(int (&arr)[N]) {
    // N 就是数组大小
}
// 或者
void process(int* arr, size_t n);
```

> **面试速答**
>
> sizeof 其实是一个编译期运算符，不是函数。它的操作数不会被求值，编译器根据类型信息在编译期直接算出大小，最后替换为常量。所以 `sizeof(i++)` 里的 i 不会自增——表达式根本不执行，只是拿来分析类型。
>
> 对 struct 来说 sizeof 的结果要遵循对齐规则：每个成员的偏移量必须是其对齐值的倍数，struct 总大小必须是最大对齐值的倍数，中间会塞 padding 字节。所以成员按大小从大到小排列通常能减少 padding。
>
> 有个容易踩的坑：数组作为函数参数时会退化成指针，在函数里 sizeof 得到的是指针大小（64 位下 8 字节）而不是数组大小，这导致常见的 `sizeof(arr)/sizeof(arr[0])` 套路在函数参数上直接失效。用 `std::size` 或模板函数拿数组长度比这种除法更安全。
>
> 跟 alignof 分清楚：sizeof 返回类型占用字节数，alignof 返回对齐要求，两者都是编译期运算符。

#### 6. 自测问题

**Q13**: `sizeof` 的操作数表达式会被求值吗？`int i=0; sizeof(i++)` 执行后 i 的值是多少？
> 提示：sizeof 是编译期运算符。

**Q14**: 空类 `struct Empty {};` 的 sizeof 是多少？为什么不是 0？如果继承自 Empty 呢（EBO）？
> 提示：C++ 要求每个对象有唯一地址；空基类优化。

### extern “C”

#### 1. 概念与本质
`extern “C”` 指定一段代码使用 **C 语言的链接规范（linkage specification）**：不做 C++ 名称修饰（name mangling），使 C 和 C++ 代码能互相调用。

#### 2. 底层原理：Name Mangling

##### 什么是 Name Mangling
C++ 支持函数重载、命名空间、模板，编译器必须为每个重载生成唯一符号名。方法是将参数类型、命名空间等信息编码进符号名：

```cpp
// C++ 源码
namespace math {
    int add(int a, int b);
    double add(double a, double b);
}
```

```bash
# 用 nm 查看编译后的符号（GCC/Itanium ABI）
nm -C math.o    # -C 显示 demangle 后的名字
# _ZN4math3addEii    → math::add(int, int)
# _ZN4math3addEdd    → math::add(double, double)

nm math.o       # 不加 -C 显示 mangled name
# _ZN4math3addEii
# _ZN4math3addEdd
```

Itanium ABI mangling 规则（GCC/Clang 遵循）：
```
_Z               → mangled name 前缀
N4math3addE      → 嵌套名: namespace math（长度4）+ 函数 add（长度3）
ii               → 两个 int 参数
dd               → 两个 double 参数
```

##### C 语言没有 mangling
C 编译器生成的符号名就是函数名本身（可能加下划线前缀）：
```bash
# C 编译器生成的符号
nm c_math.o
# add    ← 就是原始函数名，无类型编码
```

##### extern “C” 的作用
告诉 C++ 编译器：**按 C 的方式生成符号名，不做 mangling**。

```cpp
extern “C” int add(int a, int b);
// 生成的符号名: add（而不是 _Z3addii）
```

##### 实战验证
```bash
# 写一个测试文件
cat > test_mangling.cpp << 'EOF'
int cpp_func(int x) { return x; }
extern “C” int c_func(int x) { return x; }
EOF

g++ -c test_mangling.cpp -o test_mangling.o
nm test_mangling.o
# _Z8cpp_funci    ← C++ mangled name
# c_func           ← C linkage，无 mangling
```

#### 3. 代码示例

##### C++ 调用 C 库
```cpp
// 方法一：extern “C” 块包裹头文件
extern “C” {
    #include “c_library.h”
}

// 方法二：C 头文件自带 __cplusplus 保护（推荐）
// c_library.h
#pragma once
#ifdef __cplusplus
extern “C” {
#endif

int c_add(int a, int b);
void c_init(void);

#ifdef __cplusplus
}
#endif
```

##### C 调用 C++ 函数
```cpp
// cpp_api.h —— C/C++ 通用头文件
#pragma once
#ifdef __cplusplus
extern “C” {
#endif

// 对 C 暴露的接口（不能用类、模板、重载、默认参数）
int compute(int x, int y);

#ifdef __cplusplus
}
#endif

// cpp_api.cpp
#include “cpp_api.h”
#include <algorithm>  // 内部可用 C++ 特性

extern “C” int compute(int x, int y) {
    return std::max(x, y);  // 函数体内可以用 C++
}
```

#### 4. 优缺点与权衡

**优点**：
- ABI 兼容：使 C++ 代码能调用 C 库（如 libc、SQLite、OpenSSL 等），也使 C 代码能调用 C++ 导出的函数，是混合语言编程的基石
- 跨编译器兼容：C 的 ABI 是事实标准（函数名就是符号名），不同编译器（GCC/Clang/MSVC）编译的 C 代码可以互相链接；C++ 的 name mangling 规则则因编译器而异
- 插件/FFI 基础：动态库（.so/.dll）导出函数通常用 `extern “C”` 避免 mangling，`dlsym` 按原始函数名查找符号，这是插件系统和 FFI（Python ctypes、Rust FFI 等）的基础

**缺点/代价**：
- 不能使用函数重载：C 没有 mangling，同名函数只能有一个，必须用不同函数名（如 `process_int`/`process_double`）手动区分
- 不能使用模板：模板实例化依赖 mangling 将类型参数编码进符号名，extern “C” 环境下无法使用
- 不能使用异常：C 没有异常机制，如果 extern “C” 函数内部抛出 C++ 异常且未被捕获，行为未定义（通常导致 `std::terminate`）
- 不能使用类的非静态成员函数：成员函数需要隐式的 this 指针，C ABI 无法表达

**权衡**：extern “C” 的代价本质上是放弃 C++ 的类型系统优势（重载、模板、异常）来换取 ABI 兼容性。最佳实践是：对外暴露的接口层用 extern “C”（保持 C ABI 兼容），内部实现自由使用 C++ 特性。接口层做好异常到错误码的转换（`try { ... } catch (...) { return ERROR_CODE; }`），绝不让异常穿越 extern “C” 边界。

#### 5. 陷阱与实战经验

**❌ 错误：extern “C” 函数使用 C++ 特性（函数重载）**
```cpp
extern “C” {
    int process(int x);
    int process(double x);  // 编译错误！C linkage 不支持重载
}
```

**✅ 正确：用不同函数名区分**
```cpp
extern “C” {
    int process_int(int x);
    int process_double(double x);
}
```

**❌ 错误：忘记 extern “C” 导致链接失败**
```
// 报错信息：
undefined reference to `add(int, int)'
// 实际符号表中只有 _add（C编译），C++ 在找 _Z3addii
```

**✅ 正确：始终用 `#ifdef __cplusplus` 包裹 C 头文件**

#### 5. ABI 限制
`extern “C”` 函数不能使用以下 C++ 特性：
- 函数重载（无 mangling，同名冲突）
- 默认参数（C 不支持）
- 类的非静态成员函数（需要 this 指针，ABI 不兼容）
- 模板（需要实例化后的 mangled name）
- 异常规范（C 无异常机制）

> **面试速答**
>
> extern "C" 其实就是告诉 C++ 编译器"这段代码按 C 链接规范来"，核心效果是不做 name mangling。为什么要搞这个？因为 C++ 为了支持重载会把参数类型编码进符号名，比如 `add(int,int)` 被 mangling 成 `_Z3addii`，而 C 直接用原始函数名 `add`。混合编译的时候，C++ 这边找 `_Z3addii`，C 那边生成的是 `add`，符号对不上链接就失败了。
>
> 加了 extern "C" 之后，C++ 编译器会生成跟 C 编译器一模一样的符号名，链接器直接按名字匹配上。要分清楚 extern 和 extern "C"——extern（不带 "C"）只是声明外部链接，不改变 mangling 规则；extern "C" 既声明外部链接又切换到 C ABI。
>
> 有个代价要知道：extern "C" 函数不能用 C++ 的重载、模板、默认参数、类的非静态成员函数、异常规范——这些要么依赖 mangling 要么是 C 不支持的 ABI。
>
> 实际工程里 C 头文件用 `#ifdef __cplusplus extern "C" { #endif` 包裹是标准做法，让头文件同时兼容 C 和 C++ 编译器；纯 C++ 项目不需要 extern "C"。

### typedef 与 #define

#### 1. 概念与本质
- `typedef`：在编译期为已有类型创建**别名**，遵循作用域规则，类型安全。
- `#define`：预处理期**文本替换**，无作用域、无类型检查。

#### 2. 对比表

| 对比项       | `typedef`                        | `#define`                        |
|--------------|----------------------------------|----------------------------------|
| 处理阶段     | 编译期                           | 预处理期                         |
| 类型检查     | 有                               | 无                               |
| 作用域       | 遵循块作用域/命名空间            | 从定义处到文件末尾（或 `#undef`）|
| 指针类型     | 正确处理                         | 易出错（见下方陷阱）            |
| 调试         | 调试器能识别                     | 替换后原始名字消失              |

#### 3. 陷阱

**❌ 错误：用 #define 定义指针类型**
```cpp
#define PINT int*
PINT a, b;  // 展开为 int* a, b; → a 是指针，b 是 int！
```

**✅ 正确：用 typedef**
```cpp
typedef int* PINT;
PINT a, b;  // a 和 b 都是 int*
```

> **现代C++推荐**：用 `using` 代替 `typedef`，语法更清晰：
> ```cpp
> using PINT = int*;
> using Callback = void(*)(int, int);  // 比 typedef void(*Callback)(int,int) 易读
> ```

> **面试速答**
>
> typedef 和 #define 看起来都能"起别名"但本质完全不同——typedef 是编译期的类型别名，编译器理解类型语义；#define 是预处理期的纯文本替换，只是字符串粘贴。主要用途都是简化复杂类型声明，比如函数指针、嵌套模板，让代码更可读可维护。
>
> 有个经典的坑能说明两者区别：`#define PINT int*` 之后写 `PINT a, b`，预处理器展开成 `int* a, b`，结果 b 变成 int 而不是指针，因为 * 只绑定到 a 上；用 typedef 定义 PINT 就不会有这个问题，因为它懂类型语义，a 和 b 都会是指针。
>
> 跟 using 对比：C++11 的 using 语法更清晰，`using Vec = std::vector<int>` 这种形式一眼就能看懂，而且 using 支持模板别名（alias template），typedef 根本做不到。
>
> 现代 C++ 优先用 using 替代 typedef；#define 留给条件编译和头文件保护，不要用来定义类型。

### auto 与 decltype

#### 1. 概念与本质
- `auto`：让编译器从**初始化表达式**推导变量类型，会**剥离**顶层 const 和引用。
- `decltype`：从表达式推导类型，**完整保留** const、引用、volatile 等修饰符。

#### 2. 底层原理：类型推导规则

##### auto 的推导规则（类似模板参数推导）
```cpp
const int ci = 42;
const int& cr = ci;

auto a = ci;    // int（剥离顶层 const）
auto b = cr;    // int（剥离引用，再剥离顶层 const）
auto& c = ci;   // const int&（auto& 保留 const）
auto& d = cr;   // const int&（保留引用和 const）
const auto e = ci;  // const int（手动加 const）
```

**auto 推导规则总结**：
| 声明形式       | 推导规则                                    |
|----------------|---------------------------------------------|
| `auto x = expr`  | 剥离引用，剥离顶层 const/volatile          |
| `auto& x = expr` | 保留引用目标的 const/volatile              |
| `auto&& x = expr`| 万能引用：左值→左值引用，右值→右值引用     |
| `const auto x`   | 手动加 const                               |

##### decltype 的推导规则
```cpp
int i = 42;
int* p = &i;
int& r = i;

decltype(i)   a;    // int（变量名 → 变量声明的类型）
decltype(r)   b = i; // int&（引用 → 保留引用）
decltype(*p)  c = i; // int&（解引用 → 左值引用）
decltype((i)) d = i; // int&（加括号的变量 → 左值引用！陷阱！）
decltype(i+0) e;     // int（表达式结果为右值 → 值类型）
```

**decltype 的两条规则**：
1. 操作数是**不带括号的变量名** → 返回该变量声明的类型
2. 操作数是**表达式**（含带括号的变量名） → 根据值类别：左值→`T&`，右值→`T`

##### C++14 decltype(auto)
结合两者：用 auto 的语法，用 decltype 的规则推导：
```cpp
int x = 42;
int& rx = x;

decltype(auto) a = x;    // int（decltype(x) = int）
decltype(auto) b = rx;   // int&（decltype(rx) = int&）
decltype(auto) c = (x);  // int&（decltype((x)) = int&）← 小心！
```

#### 3. 代码示例
```cpp
#include <vector>
#include <iostream>

int main() {
    std::vector<int> v = {1, 2, 3};
    
    // auto 简化迭代器
    for (auto it = v.begin(); it != v.end(); ++it) {
        *it *= 2;
    }
    
    // auto&& 万能引用（range-for 最佳实践）
    for (auto&& elem : v) {
        std::cout << elem << " ";  // 2 4 6
    }
    
    // decltype 用于返回值类型推导
    // 返回容器元素的引用
    auto get = [](auto& container, size_t i) -> decltype(auto) {
        return container[i];  // 返回 int&，而非 int
    };
    get(v, 0) = 100;  // 可以修改
}
```

#### 4. 优缺点与权衡

**优点**：
- 减少冗余类型声明：`auto it = container.begin()` 比 `std::unordered_map<std::string, std::vector<int>>::iterator it = ...` 简洁得多，尤其在模板代码和 lambda 场景下，手写类型几乎不可能
- 避免类型不匹配的隐式转换：手写类型可能写错（如用 `int` 接 `size_t`），auto 保证类型与初始化表达式精确匹配，消除窄化转换和符号不匹配的隐患
- 重构友好：auto 变量在被初始化函数的返回类型改变时自动适应，不需要在调用处逐一修改类型声明

**缺点/代价**：
- 可读性下降：`auto x = foo()` 在没有 IDE 辅助时，读者无法从这行代码判断 x 的类型，必须跳转到 foo() 的定义才能理解。在 code review 和调试时增加认知负担
- 意外丢失修饰符：auto 推导规则与模板参数推导一致，会剥离顶层 const 和引用。`auto x = const_ref` 得到值拷贝而非引用，这是最常见的 auto 陷阱，可能导致性能问题（拷贝大对象）或语义错误（修改副本而非原对象）
- decltype((x)) 的陷阱：`decltype(x)` 和 `decltype((x))` 语义完全不同（前者是声明类型，后者是左值引用），在 `decltype(auto)` 返回值场景下，`return (x)` 会意外返回局部变量的引用（悬空引用）

**权衡**：auto 适合类型明显（如 `auto p = std::make_unique<Widget>()`）或类型复杂（如迭代器、lambda）的场景。对于类型不明显且读者需要立即理解类型的场景（如 `auto result = compute()`），显式写出类型更利于代码可读性。团队应建立统一的 auto 使用规范——过度使用和完全不用都不是最优解。

#### 5. 陷阱与实战经验

**❌ 错误：auto 丢失引用导致意外拷贝**
```cpp
std::vector<std::string> names = {"alice", "bob"};
for (auto name : names) {   // auto → string（拷贝！）
    name[0] = toupper(name[0]);  // 修改的是副本，原容器不变
}
```

**✅ 正确：用 auto& 或 const auto&**
```cpp
for (auto& name : names) {       // 需要修改
for (const auto& name : names) { // 只读访问
```

**❌ 错误：decltype((x)) 意外产生引用**
```cpp
int x = 42;
decltype((x)) ref = x;  // int&！不是 int！
ref = 0;  // x 也变成 0
```

**✅ 记住**：`decltype(变量名)` 得到声明类型，`decltype((变量名))` 得到引用类型

> **面试速答**
>
> auto 让编译器从初始化表达式推导类型，推导规则跟模板参数推导一致；decltype 则从表达式推导类型，而且完整保留 const、引用、volatile 这些修饰符。两者的核心动机都是避免手写冗长类型（迭代器、lambda 这种），同时减少类型不一致导致的隐式转换 bug。
>
> 底层机制上有个关键区别：auto 推导会剥离顶层 const 和引用（和模板 T 推导一致），需要保留就得手动加 const 或 &；decltype 不求值表达式，只在编译期分析表达式的类型，完整保留所有修饰符。另外 auto 必须有初始化表达式才能用，decltype 不需要。
>
> 有个高频面试陷阱要记住：`decltype(变量名)` 返回声明类型，`decltype((变量名))` 返回左值引用，多一层括号语义完全不同。括号在这里让表达式变成了左值表达式，decltype 就推导出引用。
>
> 选型上 range-for 用 `auto&` 或 `const auto&` 避免拷贝；返回类型复杂时用 `decltype(auto)` 做精确转发；普通变量优先用 auto，简洁清爽。

### explicit 关键字

#### 1. 概念与本质
`explicit`禁止编译器对构造函数进行**隐式类型转换**，强制调用者显式构造对象，避免语义模糊的自动转换。

#### 一、explicit 关键字的核心作用
`explicit` 关键字仅用于修饰**单个参数**的类构造函数（或除第一个参数外其余参数均有默认值的多参数构造函数），作用是**禁止编译器对该构造函数进行隐式类型转换**。类构造函数默认声明为 `implicit`（隐式），会自动触发类型转换，而 `explicit` 则强制构造函数只能被显式调用。

#### 二、隐式转换的问题（无 explicit 场景）
##### 示例代码
```cpp
class CxString  // 未使用explicit，默认隐式声明
{  
public:  
    char *_pstr;  
    int _size;  

    // 单参数构造函数（int类型）
    CxString(int size)  
    {  
        _size = size;                // 预设字符串内存大小
        _pstr = (char*)malloc(size + 1);    // 分配内存
        memset(_pstr, 0, size + 1);
    }  

    // 单参数构造函数（const char*类型）
    CxString(const char *p)  
    {  
        int size = strlen(p);
        _pstr = (char*)malloc(size + 1);    // 分配内存
        strcpy(_pstr, p);            // 复制字符串
        _size = strlen(_pstr);
    }  

    // 析构函数省略（实际需释放_pstr）
};  

// 调用示例
int main() {
    CxString string1(24);     // OK：显式调用，分配24字节内存
    CxString string2 = 10;    // OK：隐式转换为 CxString string2(10)
    // CxString string3;      // 错误：无默认构造函数
    CxString string4("aaaa"); // OK：显式调用，初始化字符串
    CxString string5 = "bbb"; // OK：隐式转换为 CxString string5("bbb")
    CxString string6 = 'c';   // OK：隐式转换，'c'的ASCII码作为int参数传入
    string1 = 2;              // OK：隐式转换后赋值
    string2 = 3;              // OK：隐式转换后赋值
    // string3 = string1;     // 编译通过，但析构时可能因未重载=导致内存错误
    return 0;
}
```

##### 问题分析
`CxString string2 = 10;` 能编译通过的原因：  
C++ 中单个参数的构造函数会触发**隐式类型转换**，编译器自动将 `int` 类型的 `10` 转换为 `CxString` 对象，等效于：
```cpp
CxString temp(10);
CxString string2 = temp;
```
这种隐式转换会导致语义模糊（如 `CxString string6 = 'c';` 本意是传字符，却被转换成 `int` 类型的 ASCII 码），易引发逻辑错误。

#### 三、explicit 禁止隐式转换（有 explicit 场景）
##### 示例代码
```cpp
class CxString  // 使用explicit，强制显式转换
{  
public:  
    char *_pstr;  
    int _size;  

    // 单参数构造函数添加explicit
    explicit CxString(int size)  
    {  
        _size = size;  
        _pstr = (char*)malloc(size + 1);  
        memset(_pstr, 0, size + 1);
    }  

    CxString(const char *p)  
    {  
        int size = strlen(p);
        _pstr = (char*)malloc(size + 1);  
        strcpy(_pstr, p);  
        _size = strlen(_pstr);
    }  
};  

// 调用示例
int main() {
    CxString string1(24);     // OK：显式调用仍有效
    // CxString string2 = 10;  // 错误：explicit禁止隐式转换
    // CxString string3;       // 错误：无默认构造函数
    CxString string4("aaaa"); // OK：显式调用仍有效
    CxString string5 = "bbb"; // OK：const char*构造函数未加explicit，仍可隐式转换
    // CxString string6 = 'c'; // 错误：explicit禁止int参数的隐式转换
    // string1 = 2;            // 错误：禁止隐式转换赋值
    return 0;
}
```

#### 四、explicit 对多参数构造函数的影响
##### 1. 纯多参数（无默认值）：explicit 无效
```cpp
class CxString  
{  
public:  
    char *_pstr;  
    int _age;  
    int _size;  

    // 多参数且无默认值，explicit无意义
    explicit CxString(int age, int size)  
    {  
        _age = age;  
        _size = size;  
        // 内存分配逻辑省略
    }  
};  

// 有无explicit效果一致，均需显式调用
CxString s(18, 24); // OK
// CxString s = 18;  // 错误：无隐式转换可能
```

##### 2. 多参数（含默认值）：explicit 仍有效
```cpp
class CxString  
{  
public:  
    int _age;  
    int _size;  

    // 除第一个参数外其余有默认值，等效单参数构造函数
    explicit CxString(int age, int size = 0)  
    {  
        _age = age;  
        _size = size;  
    }  
};  

// 调用示例
int main() {
    CxString string1(24);     // OK：显式调用
    // CxString string2 = 10;  // 错误：explicit禁止隐式转换
    return 0;
}
```

#### 五、优缺点与权衡

**优点**：
- 消除隐式转换陷阱：`void process(MyString s)` 不加 explicit 时可以被 `process(42)` 调用（42 隐式构造 MyString），加 explicit 后编译器直接报错，避免语义模糊的 bug
- 提高代码可审查性：所有构造都是显式的，代码中不会出现"看不见的临时对象构造"，让 code review 者更容易理解实际的对象创建和转换行为
- 防止意外的性能开销：隐式转换会构造临时对象（可能涉及堆内存分配），explicit 强制调用者显式构造，让性能开销变得可见

**缺点/代价**：
- 牺牲了某些合法的隐式转换便利性：比如 `std::string s = "hello"` 依赖 `const char*` 到 `string` 的隐式转换，如果 `string` 的构造函数标了 explicit，这种自然的写法就不行了，必须写 `std::string s("hello")`
- 增加调用者的代码量：每次都要显式写构造函数名，在某些场景下让代码变得冗长，特别是链式调用或表达式中频繁需要类型转换时
- C++11 列表初始化的交互复杂：`explicit` 构造函数不能被 `{}` 拷贝列表初始化（`MyClass x = {args}`），但可以被直接列表初始化（`MyClass x{args}`），这种微妙区别容易让人困惑

**权衡**：工程实践中的共识（Google C++ Style Guide / Effective C++ Item 5）是**所有单参数构造函数默认标 explicit**，只有明确设计为"透明类型转换"的包装器类（如 `std::string_view` 从 `std::string` 构造）才省略。代价很小（调用者多写几个字符），但收益很大（防止整类隐式转换 bug）。

#### 六、总结
1. `explicit` 仅对**单参数构造函数**（或含默认值的多参数构造函数）有效，无参数/纯多参数构造函数加 `explicit` 无意义；
2. `explicit` 核心作用是**禁止隐式类型转换**，避免非预期的类型转换导致代码语义模糊；
3. 工程规范建议：所有单参数构造函数都声明为 `explicit`，仅极少数透明包装器类的拷贝构造函数可例外（参考 Google C++ 规范/Effective C++）。

> **面试速答**
>
> explicit 的作用是禁止编译器对单参数构造函数（C++11 起也包括转换运算符）做隐式类型转换，强制调用者显式构造。为什么要搞这个？没有 explicit 的时候，`void f(MyClass x)` 可以被 `f(42)` 直接调用，编译器偷偷帮你构造了一个 MyClass 对象，语义非常模糊，很容易引入逻辑 bug。加了 explicit 之后代码从 `f(42)` 变成 `f(MyClass(42))`，意图明确，重构时也不会被隐式转换坑到。
>
> 底层机制上，编译器在需要类型转换时会搜索可用的转换路径（构造函数 + 转换运算符），explicit 就是把这条路径标记为"仅限显式调用"，隐式场景下不考虑它。
>
> 要特别注意一个坑：多参数构造函数在 C++11 的列表初始化 `{}` 语法下也可能被隐式调用，所以 C++11 起 explicit 也可以用于多参数构造函数。
>
> 工程规范上 Google C++ Style 和 Effective C++ 都要求所有单参数构造函数默认加 explicit，只有极少数透明包装类的拷贝构造函数可以例外。

### new 与 delete

#### 1. 概念与本质
`new`/`delete` 是 C++ 的**类型安全内存管理运算符**，与 `malloc`/`free` 的本质区别是：`new` = 分配内存 + 调用构造函数，`delete` = 调用析构函数 + 释放内存。

#### 2. 底层原理

##### new 表达式的执行流程
```cpp
Widget* p = new Widget(42);
```
编译器将其分解为三步：
```cpp
// 1. 调用 operator new 分配原始内存（底层通常调用 malloc）
void* raw = operator new(sizeof(Widget));
// 2. 在 raw 指向的内存上调用构造函数（placement new 语法）
Widget* p = static_cast<Widget*>(raw);
new (p) Widget(42);  // placement new
// 3. 如果构造函数抛异常，自动调用 operator delete 释放内存
```

##### operator new 可以重载
```cpp
#include <cstdlib>
#include <cstdio>

class Pool {
public:
    // 类级别的 operator new 重载
    static void* operator new(size_t size) {
        printf(“自定义 new: 分配 %zu 字节\n”, size);
        void* p = std::malloc(size);
        if (!p) throw std::bad_alloc();
        return p;
    }
    
    static void operator delete(void* p) noexcept {
        printf(“自定义 delete: 释放内存\n”);
        std::free(p);
    }
    
    int data;
};

int main() {
    Pool* p = new Pool;       // 输出: 自定义 new: 分配 4 字节
    delete p;                  // 输出: 自定义 delete: 释放内存
}
```

##### placement new（定位 new）
在**已分配好的内存**上构造对象，不分配新内存：
```cpp
#include <new>      // 需要此头文件
#include <cstdlib>

int main() {
    // 1. 预分配一块内存（栈上或堆上都行）
    alignas(Widget) char buf[sizeof(Widget)];
    
    // 2. 在 buf 上构造 Widget 对象
    Widget* p = new (buf) Widget(42);  // placement new
    
    // 3. 必须手动调用析构函数（不能用 delete！）
    p->~Widget();
    
    // 内存池的核心原理就是 placement new
}
```

##### 简易内存池示例
```cpp
class MemPool {
    static constexpr size_t BLOCK_SIZE = 4096;
    char* pool_;
    size_t offset_ = 0;
public:
    MemPool() : pool_(static_cast<char*>(std::malloc(BLOCK_SIZE))) {}
    ~MemPool() { std::free(pool_); }
    
    void* allocate(size_t size) {
        if (offset_ + size > BLOCK_SIZE) throw std::bad_alloc();
        void* p = pool_ + offset_;
        offset_ += size;
        return p;
    }
    
    void reset() { offset_ = 0; }  // 整块重置，不逐个释放
};

// 用法
MemPool pool;
void* mem = pool.allocate(sizeof(Widget));
Widget* w = new (mem) Widget(42);  // placement new
w->~Widget();
pool.reset();
```

#### 3. 对比表：new/delete vs malloc/free

| 特性               | `new` / `delete`               | `malloc` / `free`              |
|--------------------|-------------------------------|-------------------------------|
| 本质               | C++ 运算符（可重载）          | C 库函数                      |
| 类型安全           | 返回正确类型指针              | 返回 `void*`，需手动转换     |
| 构造/析构          | **自动调用**                  | 不调用                        |
| 分配失败           | 抛 `std::bad_alloc`          | 返回 `NULL`                  |
| 大小指定           | 自动计算 `sizeof(T)`         | 需手动传入字节数              |
| 可重载             | 是（全局/类级别）             | 否                            |
| 数组版本           | `new[]` / `delete[]`         | 一次性分配，无数组概念       |

#### 4. delete vs delete[]
```cpp
Widget* single = new Widget;
Widget* array  = new Widget[10];

delete single;     // 调用 1 次析构 + 释放内存
delete[] array;    // 调用 10 次析构 + 释放内存

// ❌ 错误：用 delete 释放 new[] 分配的数组
delete array;      // 未定义行为！只调用了第一个元素的析构函数
```

##### delete[] 如何知道数组长度？
`new[]` 在分配的内存块**头部**额外存储数组元素个数（通常 8 字节）：
```
内存布局：
[元素个数(8字节)][对象0][对象1]...[对象N-1]
 ↑                ↑
 实际分配起点     new[] 返回的指针
```
`delete[]` 从返回指针向前偏移读取元素个数，然后逐个调用析构。

#### 5. malloc 底层原理
`malloc` 底层通过两种系统调用分配内存：

1. **brk 系统调用**（< 128KB）：移动堆顶指针，在堆区扩展连续内存。`free` 后不一定归还操作系统（放入空闲链表复用）。
2. **mmap 系统调用**（>= 128KB）：在进程地址空间中映射独立内存块。`free` 后直接 `munmap` 归还系统。

**共性**：初始分配的是**虚拟内存**，首次访问时触发缺页中断（page fault），内核才分配物理内存。

#### 6. 优缺点与权衡

**优点**：
- 类型安全：new 返回正确类型的指针，无需 void* 强制转换；自动计算 sizeof(T)，避免手动传错大小
- 自动调用构造/析构：new 在分配内存后调用构造函数初始化对象，delete 先调用析构释放资源再释放内存。malloc/free 做不到这一点，对 C++ 对象（有虚函数表、RAII 资源）来说至关重要
- 可重载定制分配策略：类级别 operator new/delete 可以实现对象池、内存池、对齐分配等定制策略，对特定类的所有分配透明生效

**缺点/代价**：
- 手动管理生命周期易出错：裸 new 后必须手动 delete，忘记 delete 导致泄漏，异常路径更容易遗漏。据统计，内存泄漏是 C++ 项目中最常见的 bug 类型之一
- new/delete 不配对导致 UB：new 配 free、malloc 配 delete、new[] 配 delete 都是未定义行为，且编译器不会报错，只在运行时出问题
- 分配器开销：每次 new 调用 malloc，malloc 需要遍历空闲链表、可能触发 brk/mmap 系统调用，频繁小对象分配时开销显著

**权衡**：现代 C++ 的共识是尽量不直接使用裸 new/delete。`std::make_unique`/`std::make_shared` 提供异常安全的 RAII 封装，是 90% 场景的首选。裸 new 仅在以下场景仍有价值：(1) 需要 placement new 在预分配内存上构造对象（内存池）；(2) 需要重载 operator new 实现定制分配策略；(3) 与不理解智能指针的 C API 交互时。如果必须用裸 new，务必保证 delete 在同一个作用域或同一个 RAII 包装中。

#### 7. 陷阱与实战经验

**❌ 错误：new/delete 与 malloc/free 混用**
```cpp
int* p = new int(42);
free(p);    // 未定义行为！不会调用析构函数（对int无所谓，但对类对象致命）

Widget* w = (Widget*)malloc(sizeof(Widget));
delete w;   // 未定义行为！w 未被构造，析构函数操作未初始化的成员
```

**✅ 正确：配对使用**
```
new    → delete
new[]  → delete[]
malloc → free
```

**❌ 错误：忘记 placement new 需要手动析构**
```cpp
char buf[sizeof(Widget)];
Widget* w = new (buf) Widget;
// 直接离开作用域 → Widget 析构函数不会被调用 → 资源泄漏
```

**✅ 正确：手动调用析构**
```cpp
w->~Widget();  // placement new 必须手动析构
// 不要 delete w！内存不是 new 分配的
```

> **现代C++最佳实践**：尽量用 `std::make_unique` / `std::make_shared` 代替裸 `new`，用 RAII 管理资源。

> **面试速答**
>
> new 表达式其实是两步：先用 operator new 分配内存，再调用构造函数；delete 也是两步：先调用析构函数，再用 operator delete 释放内存。这两步操作不可分割，这就是 new/delete 跟 malloc/free 的本质区别——C 的 malloc/free 只管字节不管对象，没法正确管理 C++ 对象的生命周期（构造和析构都跑不起来）。
>
> 底层上 operator new 通常走 malloc（可以全局重载自定义分配策略）；new[] 会在内存块头部额外存储元素个数，delete[] 就是根据这个信息逐个调用析构函数。
>
> 有几个坑要特别注意：new/delete 跟 malloc/free 混用是 UB，比如 new 分配的内存用 free 去释放；new[] 必须配 delete[]，用 delete 释放数组只会析构第一个元素，其他对象资源全泄漏；异常安全场景下裸 new 很容易泄漏，比如 `f(new A, new B)` 中间抛异常就完蛋。跟 make_unique 对比，make_unique 就是异常安全的 RAII 封装，把这些坑都兜住了。
>
> 现代 C++ 优先用 make_unique/make_shared 替代裸 new；只有需要自定义内存分配策略时才会去重载 operator new。

### mutable 关键字

#### 1. 概念与本质
`mutable` 是一个**存储类说明符**，作用是**允许在 const 成员函数中修改被标记的成员变量**。它表达的语义是：这个成员变量不属于对象的"逻辑状态"，而是实现细节（如缓存、互斥锁、访问计数），修改它不算破坏对象的 const 契约。

#### 2. 底层原理

##### 为什么需要 mutable
const 成员函数的约束是：`this` 指针类型变成 `const T*`，通过它访问的所有成员都带 const。但有些成员必须在 const 函数中修改：

```cpp
class TextBlock {
    std::string text_;
    mutable std::size_t cached_length_ = 0;  // 缓存：不影响逻辑状态
    mutable bool length_valid_ = false;
    mutable std::mutex mtx_;                  // 锁：必须在 const 方法中 lock
public:
    std::size_t length() const {              // const 方法
        std::lock_guard<std::mutex> lk(mtx_); // ✅ mtx_ 是 mutable，可以 lock
        if (!length_valid_) {
            cached_length_ = text_.size();     // ✅ 修改 mutable 成员
            length_valid_ = true;
        }
        return cached_length_;
    }
};
```

没有 mutable 的话，`length()` 作为 const 方法无法 lock mutex 也无法写缓存，你只能把 const 去掉——但 `length()` 逻辑上确实不改变对象状态，去掉 const 是语义错误。

##### 编译器怎么处理
mutable 成员在内存布局上跟普通成员完全一样，**不会改变 sizeof、对齐或偏移**。它只影响编译器的类型检查：编译器在 const 上下文中访问 mutable 成员时，不插入 const 限定。

##### mutable lambda（C++11）
lambda 默认以值捕获的变量是 const 的（因为 `operator()` 默认是 const）：
```cpp
int x = 0;
auto f = [x]() { x++; };            // ❌ 编译错误：x 是 const
auto g = [x]() mutable { x++; };    // ✅ operator() 变成非 const
```
mutable lambda 的本质：让编译器生成的闭包类的 `operator()` 去掉 const 修饰。

#### 3. 代码示例

##### 典型场景：线程安全的缓存
```cpp
class ExpensiveData {
    std::vector<int> raw_data_;
    mutable std::optional<double> cached_avg_;
    mutable std::mutex cache_mtx_;
public:
    double average() const {
        std::lock_guard lk(cache_mtx_);
        if (!cached_avg_) {
            double sum = std::accumulate(raw_data_.begin(), raw_data_.end(), 0.0);
            cached_avg_ = sum / raw_data_.size();
        }
        return *cached_avg_;
    }

    void update(int idx, int val) {
        raw_data_[idx] = val;
        std::lock_guard lk(cache_mtx_);
        cached_avg_.reset();  // 数据变了，清缓存
    }
};
```

##### 错误用法：滥用 mutable 绕过 const
```cpp
class BadDesign {
    mutable int balance_;  // ❌ balance 是核心状态，不该 mutable
public:
    void withdraw(int amount) const {  // const 方法居然能改余额？
        balance_ -= amount;            // 语义上是错的
    }
};
```

#### 4. 对比表

| 维度 | mutable 成员 | const_cast 绕过 |
|------|-------------|----------------|
| 语义 | 显式声明"不属于逻辑状态" | 强行去掉 const，语义不清 |
| 安全性 | 合法，定义行为 | 修改原本 const 对象是 UB |
| 可读性 | 一看就知道这个成员可变 | 隐蔽，容易被忽略 |
| 适用场景 | 缓存、锁、计数器 | 适配不 const-correct 的老 API |

#### 5. 优缺点与权衡

**优点**：
- 实现 bitwise const（编译器检查）和 logical const（语义约束）的分离
- 是唯一合法的"在 const 方法中修改成员"方式
- 让 mutex、缓存等实现细节不破坏接口的 const 正确性

**缺点/代价**：
- 滥用会让 const 承诺变成空话——如果把核心状态标 mutable，const 就失去了意义
- 多线程场景下 mutable 成员仍需加锁保护

> **面试速答**
>
> mutable 是一个存储类说明符，允许在 const 成员函数中修改被标记的成员变量。它表达的语义是"这个成员不属于对象的逻辑状态"——典型的就是缓存、互斥锁、访问计数器这些实现细节。
>
> 为什么需要它？因为 const 成员函数会把 this 变成 `const T*`，所有成员都不能改。但像 `length() const` 这样的方法可能需要加锁（lock mutex）或者写缓存，不用 mutable 的话你只能去掉 const，但语义上 length 确实不改变对象状态，去掉 const 反而是错的。
>
> 底层上 mutable 不影响内存布局，不改变 sizeof 和对齐，纯粹是编译器类型检查层面的放行。另外 C++11 的 mutable lambda 也是这个思路——让闭包类的 `operator()` 去掉 const，这样值捕获的变量就能在 lambda 体内修改了。
>
> 有个要特别注意的原则：只有不属于逻辑状态的成员才该标 mutable，如果把核心业务数据标 mutable，const 承诺就变成空话了。

#### 6. 自测问题

### thread_local 关键字

#### 1. 概念与本质
`thread_local` 是 C++11 引入的**存储期说明符**，声明变量为**线程局部存储（TLS）**——每个线程持有独立副本，互不影响，线程结束时自动析构。

#### 2. 底层原理

##### TLS 的实现机制
```
┌─ 线程 A ──────────────┐    ┌─ 线程 B ──────────────┐
│ TLS 块                │    │ TLS 块                │
│ ┌─────────────────┐   │    │ ┌─────────────────┐   │
│ │ thread_local x  │───┼──×─┼─│ thread_local x  │   │
│ │ = 独立副本 A    │   │    │ │ = 独立副本 B    │   │
│ └─────────────────┘   │    │ └─────────────────┘   │
└───────────────────────┘    └───────────────────────┘
```

**Linux 实现**：每个线程有自己的 TLS 段，通过 `fs`（x86-64）或 `tpidr_el0`（ARM64）段寄存器定位：
```asm
; x86-64 访问 thread_local 变量
mov rax, QWORD PTR fs:[tls_offset]   ; 从线程的 TLS 段读取
```

**编译器生成的代码**：首次访问 thread_local 变量时，编译器插入 `__tls_init` 调用确保变量被正确初始化（类似局部 static 的 magic statics 机制）。

##### 四种存储期对比

| 存储期 | 关键字 | 生命周期 | 典型用途 |
|--------|--------|---------|---------|
| 自动（automatic） | 无 / auto | 进入作用域→离开作用域 | 局部变量 |
| 静态（static） | static / 全局 | 程序启动→程序结束 | 全局状态 |
| **线程（thread）** | **thread_local** | **线程创建→线程结束** | **线程私有状态** |
| 动态（dynamic） | new / delete | malloc→free | 堆对象 |

#### 3. 代码示例

##### 典型场景：线程安全的错误码
```cpp
// 每线程独立的 errno（glibc 就是这么实现的）
thread_local int my_errno = 0;

void thread_func() {
    my_errno = 42;  // 只改本线程的副本，其他线程看不到
}

int main() {
    my_errno = 0;
    std::thread t(thread_func);
    t.join();
    std::cout << my_errno << std::endl;  // 输出 0，不受线程 t 影响
}
```

##### 线程级缓存 / 无锁计数器
```cpp
struct ThreadStats {
    uint64_t requests = 0;
    uint64_t errors = 0;
    ~ThreadStats() {
        // 线程退出时汇总到全局
        global_stats.merge(*this);
    }
};
thread_local ThreadStats tl_stats;  // 每线程独立，写时无需加锁

void handle_request() {
    tl_stats.requests++;  // 无锁，无原子操作，极快
}
```

##### 与 static 组合
```cpp
void foo() {
    thread_local static int count = 0;  // 每线程一份，函数作用域
    count++;
    // 线程 A 调 foo 10 次 → count = 10
    // 线程 B 调 foo 5 次  → count = 5（独立）
}
```

#### 4. 对比表：thread_local vs 其他线程隔离方案

| 方案 | 性能 | 析构保证 | 可移植性 | 适用 |
|------|------|---------|---------|------|
| `thread_local` | 极快（寄存器寻址） | ✅ 线程结束自动析构 | C++11 标准 | 首选 |
| `pthread_key_t` | 快（hash 查找） | ✅ 需注册析构回调 | POSIX | C 代码 / 需运行时创建 |
| `__thread`（GCC） | 同 thread_local | ❌ 不调用 C++ 析构 | GCC 扩展 | 仅 POD 类型 |
| 线程参数传递 | 无额外开销 | 手动管理 | 通用 | 简单场景 |

#### 5. 优缺点与权衡

**优点**：
- 线程私有，读写无需加锁，性能接近普通全局变量
- 自动初始化 + 线程结束自动析构（RAII 友好）
- 解决 errno、随机数种子、线程缓存等经典"线程安全全局变量"问题

**缺点/代价**：
- 每个线程都有一份副本，线程数多时内存消耗 × N
- 动态库（dlopen）中的 thread_local 初始化可能有性能开销（需走 `__tls_get_addr` 慢路径）
- 不能替代线程间通信——thread_local 本质是隔离不是共享

**陷阱**：
- thread_local 对象析构顺序在标准中未定义——不要在一个 thread_local 的析构函数中访问另一个 thread_local
- 线程池中 thread_local 不会在任务之间重置——线程是复用的，上一个任务的残留状态会泄漏到下一个任务

> **面试速答**
>
> thread_local 是 C++11 引入的存储期说明符，声明的变量每个线程持有独立副本，互不影响，线程结束自动析构。底层实现上，Linux 用 `fs` 段寄存器（x86-64）指向每线程的 TLS 段，访问 thread_local 变量就是一条寄存器偏移寻址指令，性能接近普通全局变量。
>
> 最典型的用途是线程安全的全局状态——比如 glibc 的 errno 就是 thread_local 实现的，还有线程级缓存、无锁计数器、随机数种子这些场景。它的核心优势是读写无需加锁，比 mutex 保护的共享变量快一个数量级。
>
> 有两个容易踩的坑：第一，线程池里 thread_local 变量不会在任务之间自动重置，因为线程是复用的，上一个任务的残留状态会泄漏到下一个任务。第二，多个 thread_local 对象的析构顺序在标准中未定义，不要在一个 thread_local 的析构函数里去访问另一个 thread_local。
>
> 跟 `pthread_key_t` 对比：thread_local 是编译期绑定更快，`pthread_key_t` 是运行时创建更灵活。跟 GCC 扩展 `__thread` 对比：`__thread` 只支持 POD 类型不调用 C++ 析构函数，thread_local 没这个限制。优先用 thread_local。

> 提示：线程池复用线程，thread_local 变量不会在任务之间重置。

> 提示：`__thread` 是 GCC 扩展，不调用 C++ 构造/析构函数。

### consteval 与 constinit（C++20）

#### 1. 概念与本质
C++20 引入两个关键字，补全了编译期计算的"最后一块拼图"：
- `consteval`：**立即函数**——调用必须产生编译期常量，否则编译报错（比 constexpr 更严格）
- `constinit`：**保证变量的初始化在编译期完成**，但变量本身运行期可以修改（解决 Static Initialization Order Fiasco）

```
const        → 运行期只读（可能编译期初始化也可能不是）
constexpr    → 尽量编译期（但允许运行期调用）
consteval    → 必须编译期（运行期调用直接报错）
constinit    → 初始化必须编译期（之后可以改）
```

#### 2. 底层原理

##### consteval 的编译器行为
```cpp
consteval int square(int n) { return n * n; }

int a = square(5);     // ✅ 编译期求值：a = 25
int x = 5;
int b = square(x);     // ❌ 编译错误！x 不是编译期常量
```
和 constexpr 函数的区别：constexpr 函数参数不是常量时会退化为运行期调用，consteval **绝不退化**。

编译器处理：consteval 函数的调用在编译期全部展开为常量值，**最终二进制中不存在这个函数的代码**——它只存在于编译期。

##### constinit 解决的问题：Static Initialization Order Fiasco
```cpp
// a.cpp
int compute() { return 42; }
int g_a = compute();            // 动态初始化，顺序不确定

// b.cpp
extern int g_a;
int g_b = g_a + 1;              // 如果 g_a 还没初始化 → g_b = 1（bug！）
```

constinit 强制变量必须用编译期常量初始化，从根源上杜绝这个问题：
```cpp
constinit int g_a = 42;         // ✅ 编译期初始化，zero-initialization 阶段就搞定
constinit int g_b = compute();  // ❌ 编译错误！compute() 不是 constexpr
```

**注意**：constinit 只约束初始化，变量之后可以修改：
```cpp
constinit int counter = 0;      // 编译期初始化为 0
void inc() { counter++; }       // ✅ 运行期可以改
```

#### 3. 代码示例

##### consteval：编译期查表 / 哈希
```cpp
consteval uint32_t compile_time_hash(const char* str) {
    uint32_t hash = 5381;
    while (*str) {
        hash = hash * 33 + static_cast<uint32_t>(*str++);
    }
    return hash;
}

// 使用：编译期计算字符串哈希，用于 switch
void dispatch(const char* cmd) {
    switch (compile_time_hash(cmd)) {  // ❌ cmd 不是编译期常量，报错
        // ...
    }
    // 正确用法：
    constexpr auto h = compile_time_hash("start");  // ✅ 字面量是编译期常量
}
```

##### constinit：安全的全局配置
```cpp
struct Config {
    int max_threads;
    int buffer_size;
};

constinit Config g_config = {8, 4096};  // 编译期确定初始值
// 运行期可以根据环境覆盖：
void init() { g_config.max_threads = std::thread::hardware_concurrency(); }
```

##### consteval + if consteval（C++23）
```cpp
constexpr int maybe_compile_time(int n) {
    if consteval {                     // C++23：检测当前是否在编译期
        return n * n;                  // 编译期走这条路
    } else {
        return slow_runtime_compute(n); // 运行期走这条路
    }
}
```

#### 4. 对比表

| 维度 | const | constexpr | consteval | constinit |
|------|-------|-----------|-----------|-----------|
| 标准版本 | C++98 | C++11 | C++20 | C++20 |
| 用于变量 | ✅ | ✅ | ❌ | ✅（全局/static） |
| 用于函数 | ❌ | ✅ | ✅ | ❌ |
| 编译期求值 | 可能 | 尽量 | **必须** | 初始化必须 |
| 运行期可改 | ❌ | ❌ | — | ✅ |
| 二进制中有代码 | — | 可能有 | **无** | — |
| 解决的问题 | 只读语义 | 编译期计算 | 强制编译期 | 初始化顺序 |

#### 5. 优缺点与权衡

**consteval 优点**：
- 保证编译期求值——如果误传了运行期变量，编译器直接报错而不是静默退化
- 函数不会出现在二进制中，零运行期开销
- 适合安全关键场景：编译期验证、编译期加密、编译期哈希

**consteval 缺点**：
- 调试困难——函数只在编译期存在，不能设断点
- 限制更严格——不能在运行期调用，灵活性不如 constexpr

**constinit 优点**：
- 从根源上消灭 Static Initialization Order Fiasco
- 比 constexpr 灵活——初始化后允许修改

**constinit 缺点**：
- 只能用于全局/static 变量
- 初始值必须是 constexpr 可求的，某些复杂初始化做不到

> **面试速答**
>
> consteval 和 constinit 是 C++20 引入的，补全了编译期计算的最后一块拼图。
>
> consteval 声明"立即函数"，调用必须产生编译期常量，否则直接编译报错。跟 constexpr 函数的区别是：constexpr 参数不是常量时会退化为运行期调用，consteval 绝不退化。它的好处是如果你误传了运行期变量，编译器会立刻拦你而不是静默退化。编译完之后 consteval 函数在二进制中不存在，零运行期开销。
>
> constinit 解决的是另一个问题——Static Initialization Order Fiasco。多个翻译单元的全局变量如果有动态初始化，执行顺序在标准中未定义，可能一个变量用到另一个还没初始化的变量。constinit 强制变量必须用编译期常量初始化，从根源上杜绝这个问题。但它和 constexpr 变量不同的是：constinit 变量初始化后运行期可以修改。
>
> 记忆口诀：const = 只读，constexpr = 尽量编译期，consteval = 必须编译期，constinit = 初始化必须编译期但之后随便改。

### final 与 override（C++11）

#### 1. 概念与本质
- `override`：显式标记"我在覆写基类的虚函数"——如果签名不匹配，编译器报错
- `final`：禁止进一步继承类或覆写虚函数——阻断继承链

它们不是保留关键字，而是**上下文关键字（contextual keywords）**——只在特定位置有特殊含义，其他位置可以当普通标识符用（可以定义变量名叫 override，虽然不建议）。

#### 2. 底层原理

##### override 防止的经典 bug
```cpp
class Base {
public:
    virtual void process(int x);
    virtual void on_event(const std::string& name);
};

class Derived : public Base {
public:
    void process(double x);                    // ❌ 参数类型不同，是隐藏不是覆写！
    void on_event(const std::string name);     // ❌ 少了 &，签名不同！
};
```

这两个函数**编译通过但行为完全不对**——通过基类指针调用时走不到派生类版本。加了 override 编译器就能检查：

```cpp
class Derived : public Base {
public:
    void process(double x) override;           // ❌ 编译错误：Base 没有 process(double)
    void on_event(const std::string name) override; // ❌ 编译错误：签名不匹配
};
```

##### final 的编译器优化：去虚拟化（Devirtualization）
```cpp
class Shape {
public:
    virtual void draw() const;
};

class Circle final : public Shape {  // final 类
public:
    void draw() const override;
};

void render(const Circle& c) {
    c.draw();  // 编译器知道 Circle 不可能有子类
               // → 虚调用可以优化为直接调用
               // → 进一步内联
}
```

普通虚调用：`mov rax, [vtable]; call [rax+offset]`（间接调用，不能内联）
final 优化后：`call Circle::draw`（直接调用，可以内联）

##### final 虚函数
```cpp
class Base {
public:
    virtual void foo();
};
class Middle : public Base {
public:
    void foo() override final;  // 到此为止，子类不能再覆写 foo
};
class Bottom : public Middle {
public:
    void foo() override;        // ❌ 编译错误：foo 在 Middle 中是 final
};
```

#### 3. 代码示例

##### 经典继承体系
```cpp
class Animal {
public:
    virtual ~Animal() = default;
    virtual void speak() const = 0;
    virtual int legs() const { return 4; }
};

class Dog final : public Animal {         // final 类：不能被继承
public:
    void speak() const override { std::cout << "Woof\n"; }
    // legs() 继承默认实现
};

class GuideDog : public Dog {};           // ❌ 编译错误：Dog 是 final

// 正确做法：如果需要扩展，去掉 Dog 的 final
```

##### override 与析构函数
```cpp
class Base {
public:
    virtual ~Base() = default;
};
class Derived : public Base {
public:
    ~Derived() override = default;  // ✅ 推荐：显式标记覆写析构
};
```

#### 4. 对比表

| 维度 | override | final（类） | final（函数） |
|------|----------|------------|-------------|
| 作用 | 确保覆写正确 | 禁止继承 | 禁止覆写 |
| 位置 | 虚函数声明后 | 类名后 | 虚函数声明后 |
| 编译期检查 | 签名匹配 | 继承尝试 | 覆写尝试 |
| 运行期开销 | 无 | 无（可能更快） | 无（可能更快） |
| 能否优化 | 无 | 去虚拟化 + 内联 | 去虚拟化 + 内联 |

#### 5. 优缺点与权衡

**override 优点**：
- 将"隐藏基类函数"的隐蔽 bug 变成编译错误
- 自文档化——一眼看出哪些函数是覆写
- 零开销，纯编译期检查

**final 优点**：
- 设计意图明确——"这个类/函数到此为止"
- 启用编译器去虚拟化优化——虚调用变直接调用再变内联
- 防止脆弱基类问题（Fragile Base Class Problem）

**final 缺点**：
- 过早 final 会限制扩展性——你认为不需要继承，但其他人可能需要 mock/测试
- 在需要扩展的库 API 上用 final 是设计错误

**陷阱**：
- override 和 final 不是互斥的，可以同时用：`void foo() override final;`
- 忘了在基类声明 virtual，派生类加 override 会报错——这其实是好事，说明你没有真的在覆写

> **面试速答**
>
> override 和 final 是 C++11 引入的上下文关键字。override 标记"我在覆写基类虚函数"，如果签名不匹配直接编译报错。这解决了一个非常经典的 C++ 隐蔽 bug：你以为覆写了基类虚函数，实际参数类型差了一点（比如 int 写成 double、少了个引用），变成了函数隐藏而不是覆写，通过基类指针调用时走不到你的版本。没有 override 的话编译器不报错，运行时才发现行为不对，排查起来很痛苦。
>
> final 有两个用法：放在类名后面禁止继承，放在虚函数后面禁止覆写。除了表达设计意图之外，final 还有个实际好处——编译器去虚拟化优化。如果编译器知道一个类是 final 的，那通过它的引用调用虚函数就不需要查虚表了，可以直接调用甚至内联，性能从间接调用变成零开销。
>
> 有个值得注意的点：它们是上下文关键字不是保留关键字，理论上你可以定义变量叫 override，虽然没人会这么干。实际工程建议：所有虚函数覆写都加 override，这应该是团队规范级别的要求。final 则要谨慎——在库的公开 API 上过早 final 会限制测试 mock 和用户扩展。

> 提示：不报错，变成函数隐藏而非覆写，基类指针调用走基类版本。
