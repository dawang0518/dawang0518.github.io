## 1. 左值与右值

> 理解左值和右值是后面所有内容（引用、移动语义、完美转发）的基础。这一节先把这个概念讲透。

### 1.1 什么是左值、什么是右值

**最直觉的判断方法：能不能取地址（`&`）？**

```cpp
int a = 10;
int b = 20;

// 左值：有明确内存位置，可以取地址
&a;          // ✅ a 是左值，存在栈上某个确定的地址
&b;          // ✅ b 是左值

// 右值：没有持久的内存位置，不能取地址
// &10;       // ❌ 编译错误！10 是右值，是个临时的值，不住在任何地方
// &(a + b);  // ❌ 编译错误！a + b 算出来一个临时结果 30，用完就没了
```

**生活类比**：左值是"有房子的人"——有固定住址，你可以写信给他（取地址）；右值是"路上的快递"——送到就签收消失了，你不能寄信给一个快递包裹。

### 1.2 常见的左值和右值

```cpp
int x = 42;
std::string name = "hello";
int* ptr = &x;

// ===== 左值（有名字、有地址、活着的对象）=====
x              // 变量名
name           // 变量名
name[0]        // 下标访问的结果
*ptr           // 解引用的结果
++x            // 前缀自增返回对象本身（还是 x）

// ===== 右值（临时的、即将消失的值）=====
42             // 字面量
x + 1          // 算术表达式的结果（临时值 43）
std::string("temp")  // 临时对象
x++            // 后缀自增返回的是旧值的副本（临时的）
```

**`++x` 是左值但 `x++` 是右值——为什么？**

```cpp
int x = 5;
++x;    // 先把 x 加 1，返回 x 本身 → x 还活着，有地址，是左值
x++;    // 先记住旧值 5，把 x 加 1，返回那个旧值 5 → 这个 5 是个临时副本，没地址，是右值

&(++x); // ✅ 合法，返回的就是 x 的地址
// &(x++); // ❌ 编译错误，返回的是一个临时的旧值
```

### 1.3 为什么这个区分重要

C++03 时代，左值右值的区分主要是编译器用来检查语法合法性（比如赋值号左边必须是左值）。**C++11 赋予了这个区分全新的意义**：

```
核心洞察：右值马上就要销毁了
         ↓
既然它要销毁，我能不能在销毁前"偷走"它的资源？
         ↓
不用傻傻地拷贝一份，直接接管它的内存指针
         ↓
这就是"移动语义"的出发点
```

> [!note]- "接管内存指针"是什么意思？（点击展开）
> 
> `std::string` 内部其实就是一个指向堆内存的指针 + 长度：
> 
> ```cpp
> // std::string 内部大致长这样（简化）：
> class string {
>     char* data_;   // 指向堆上的字符数组
>     size_t size_;  // 字符串长度
> };
> ```
> 
> 当你写 `std::string s = 临时string;` 时：
> 
> ```
> ===== C++03 拷贝方式 =====
> 
> 临时对象（即将销毁）              s（新创建）
> ┌──────────────────┐          ┌──────────────────┐
> │ data_ ──→ 堆内存A │          │ data_ ──→ 堆内存B │ ← 新分配的！
> │ size_ = 13       │          │ size_ = 13       │
> └──────────────────┘          └──────────────────┘
>       │                              │
>       ▼                              ▼
> 堆内存A: "Hello, World!\0"    堆内存B: "Hello, World!\0"  ← 逐字节拷贝过来的
>       │
>       └── 临时对象析构时 delete 掉
> 
> 代价：分配堆内存B + 拷贝 13 字节 + 释放堆内存A = 白白浪费
> 
> 
> ===== C++11 移动方式 =====
> 
> 临时对象（即将销毁）              s（新创建）
> ┌──────────────────┐          ┌──────────────────┐
> │ data_ = nullptr  │          │ data_ ──→ 堆内存A │ ← 直接拿了临时对象的指针！
> │ size_ = 0        │          │ size_ = 13       │
> └──────────────────┘          └──────────────────┘
>   ↑ 指针被"偷走"了                     │
>   临时对象析构时                        ▼
>   delete nullptr = 啥都不做     堆内存A: "Hello, World!\0"  ← 还是原来那块，没拷贝
> 
> 代价：拷贝一个指针（8字节）+ 置空源指针 = 几乎零开销
> ```
> 
> **关键**：数据还在原来的堆内存上，一个字节都没动。只是把"谁拥有这块内存"从临时对象换成了 `s`。临时对象的指针被置空，这样它析构时 `delete nullptr` 不会释放那块内存。

```cpp
std::string make_greeting() {
    return std::string("Hello, World!");  // 返回一个临时对象（右值）
}

// C++03：分配新堆内存 → 逐字节拷贝 → 释放旧堆内存。浪费！
// C++11：编译器知道临时对象是右值，直接把它的 data_ 指针偷给 s。零拷贝。
std::string s = make_greeting();
```

但要实现"偷资源"，需要一种新的引用类型来绑定临时对象——这就引出了下一章：右值引用。

### 1.4 C++11 值类别体系（完整版）

C++03 只有简单的左值/右值二分法。C++11 细化为 5 种值类别，因为需要多一个概念来描述"有名字但我明确说不要了"的状态（将亡值）：

```
                   表达式 (expression)
                   /              \
          广义左值 (glvalue)      右值 (rvalue)
           /          \          /         \
    左值 (lvalue)  将亡值 (xvalue)     纯右值 (prvalue)
```

用两个属性来区分这五种值类别：**有没有身份**（能不能取地址）和**能不能被移动**（资源能不能被偷走）。

| 值类别 | 中文名 | 有身份？ | 可移动？ | 一句话理解 | 典型示例 |
|--------|-------|:--:|:--:|----------|----------|
| lvalue | 左值 | Yes | No | 有名有姓，还在用，不能偷 | 变量名 `x`、`*p`、`str[0]`、函数返回 `T&` |
| xvalue | 将亡值 | Yes | Yes | 有名有姓，但主人说"我不要了，你拿走" | `std::move(x)`、函数返回 `T&&` |
| prvalue | 纯右值 | No | Yes | 无名临时工，用完就扔 | 字面量 `42`、`std::string("hi")`、函数返回 `T` |
| glvalue | 广义左值 | Yes | - | lvalue + xvalue 的统称 | 所有有身份的表达式 |
| rvalue | 右值 | - | Yes | xvalue + prvalue 的统称 | 所有可移动的表达式 |

**关键洞察**：
- **有身份** = 可以取地址（`&expr` 合法），说明内存中有确定的位置
- **可移动** = 资源可以被安全偷走（要么是临时对象，要么主人明确放弃了）
- `std::move(x)` 不移动任何东西，它只是把左值转成将亡值，告诉编译器"这个对象可以被偷了"
- 将亡值是连接左值和右值的桥梁——它**有身份**（像左值），但**可以被移动**（像右值）

> **面试速答**
>
> 左值就是有明确内存地址、可以取地址的表达式（变量名、解引用等），右值就是临时的、没有持久地址的表达式（字面量、算术结果、临时对象）。最直接的判断：能 `&expr` 就是左值，不能就是右值。
>
> C++11 把右值进一步拆成纯右值和将亡值。将亡值就是 `std::move(x)` 产生的——对象还在内存里（有身份），但主人明确说不要了（可移动）。这个区分的意义是让编译器能区分"这个对象还在用"和"这个对象可以偷资源"，从而支持移动语义。

## 2. 引用：左值引用与右值引用

> 理解了左值和右值之后，这一节讲 C++ 如何用两种引用来分别"抓住"它们。

### 2.1 为什么需要左值引用和右值引用？

#### 左值引用 `T&` 解决了什么问题？

C 语言传参只有两种方式：**传值**（拷贝一份）和**传指针**（传地址）。两种都有痛点：

```cpp
// 痛点 1：传值 → 大对象拷贝开销巨大
void process(std::string s);      // 每次调用都拷贝整个字符串
std::string data = "very long...";
process(data);                     // 拷贝！如果 data 有 1MB，这里就拷 1MB

// 痛点 2：传指针 → 语法繁琐，还要判空
void process(std::string* p);
process(&data);                    // 调用方要取地址
if (p != nullptr) { p->size(); }   // 函数内要判空、要用 -> 访问
```

**左值引用 `T&` 的作用**：用引用语法实现"不拷贝、直接操作原对象"，既避免了拷贝开销，又比指针更安全（不会为 null）、更干净（用 `.` 而不是 `->`）：

```cpp
void process(std::string& s);     // 不拷贝，直接操作原对象
process(data);                     // 调用方写法和传值一样自然
// 函数内直接用 s.size()，不用判空，不用 ->
```

**但左值引用有一个根本局限**：它只能绑定左值（活着的、有名字的对象），不能绑定临时对象（右值）。

#### 为什么还需要右值引用 `T&&`？

```cpp
std::string greet() { return "hello"; }

std::string& ref = greet();   // ❌ 编译错误：左值引用不能绑定临时对象
// 为什么？因为临时对象马上就要销毁了，给你一个引用指向即将销毁的内存，必然悬垂
```

C++03 的妥协是 `const T&` 可以绑定右值（临时对象的生命周期会延长到引用的生命周期）。但 `const` 意味着**不能修改**——你没法偷走临时对象的资源：

```cpp
const std::string& ref = greet();  // ✅ 能绑定，但 ref 是 const，不能修改
// 你明知这个临时对象马上就没人用了，却不能"偷"走它的内存来避免拷贝
```

**这就是 C++11 引入 `T&&`（右值引用）的动机**：专门绑定临时对象，**而且可以修改**——你可以把临时对象的内存"偷"过来，而不是傻傻地拷贝一份。这就是移动语义的基础。

```
C 语言：传值（拷贝）/ 传指针（繁琐、不安全）
  ↓ C++ 引入左值引用
T&：避免拷贝，安全地操作已有对象 → 但不能绑定临时对象
  ↓ C++11 引入右值引用  
T&&：绑定临时对象并接管其资源 → 移动语义、完美转发的基础
```

### 2.2 绑定规则

C++ 的引用分为四种：非常量左值引用 `T&`、常量左值引用 `const T&`、非常量右值引用 `T&&`、常量右值引用 `const T&&`。被引用的值也有四种类别：非常量左值、常量左值、非常量右值、常量右值。

#### 绑定规则总览

| 引用类型 | 非常量左值 | 常量左值 | 非常量右值 | 常量右值 | 引用本身是 |
|---------|:---------:|:-------:|:---------:|:-------:|-----------|
| `T&`（非常量左值引用） | ✅ | ❌ | ❌ | ❌ | 非常量左值 |
| `const T&`（常量左值引用） | ✅ | ✅ | ✅ | ✅ | 常量左值 |
| `T&&`（非常量右值引用） | ❌ | ❌ | ✅ | ❌ | 非常量左值 |
| `const T&&`（常量右值引用） | ❌ | ❌ | ✅ | ✅ | 常量左值 |

> 常量左值引用 `const T&` 是"万能"引用类型，可以绑定到所有类型的值。
> 所有引用变量本身都是左值（有名字、可取地址），即使右值引用变量也是左值——这是第 3 章移动语义的关键前置知识。

**设计逻辑**：
```
T&：    "我要借用一个活着的对象来修改"   → 只能绑定非常量左值
const T&："我只读，什么都行"            → 万能绑定
T&&：   "我要接管一个即将销毁的对象"     → 只能绑定非常量右值
const T&&："我只读临时对象"             → 绑定非常量/常量右值（实践中极少用）
```

> [!note]- 为什么加了 `const` 就扩大了绑定范围？（点击展开）
> 
> 编译器禁止某种绑定，本质上是在防两种 bug：
> 
> | 担心 | 具体场景 | 后果 |
> |------|---------|------|
> | **担心 1：修改了不该修改的东西** | 通过非 `const` 引用修改 `const` 对象 | 未定义行为 |
> | **担心 2：修改了但没有效果** | 通过引用修改临时对象，改完立刻销毁 | 代码看着改了，其实什么都没发生 |
> 
> `const` 的作用是承诺 **"我不会通过这个引用修改任何东西"**。这个承诺一做，两种担心都不存在了：
> 
> **左值引用加 `const`：`T&` → `const T&`**
> 
> ```
> T& 不能绑定 const 左值  → 因为怕你改 const 对象
> T& 不能绑定右值（临时对象）→ 因为改了也白改，而且临时对象马上销毁
> 
> 加了 const 后：
> const T& 能绑定 const 左值  → 你承诺不改，绑 const 对象完全安全
> const T& 能绑定右值         → 你承诺不改，临时对象"改了白改"的问题不存在了
>                               编译器还会延长临时对象的生命周期到引用的作用域结束
> ```
> 
> **右值引用加 `const`：`T&&` → `const T&&`**
> 
> ```
> T&& 不能绑定 const 右值  → 因为右值引用的目的是接管资源（移动），
>                           但 const 对象不能被移动（移动需要修改源对象）
> 
> 加了 const 后：
> const T&& 能绑定 const 右值  → 你承诺不改（也不打算移动），
>                                那绑定一个 const 临时对象就无害了
> ```
> 
> **一句话总结**：`const` = "我不修改"的承诺 → 编译器不需要再防你修改 → 限制放开。本质上，绑定规则的严格程度和你对对象的"修改权限"成反比——你要的权限越少，能绑定的范围越大。

#### 非常量左值引用 `T&` —— 只能绑定非常量左值

```cpp
int ncLeft1 = 2;
int& ref1 = ncLeft1;       // ✅ ncLeft1 是非常量左值
int& ref2 = ++ncLeft1;     // ✅ 前缀 ++ 返回非常量左值
int& ref3 = ref2;          // ✅ ref2 本身是非常量左值

const int cLeft1 = 2;
// int& ref4 = cLeft1;     // ❌ 常量左值：不能通过非 const 引用修改 const 对象
// int& ref5 = ncLeft1 + 1;// ❌ 非常量右值：临时对象即将销毁，修改无意义
// int& ref6 = 3;          // ❌ 常量右值：同上，且不可修改
```

#### 常量左值引用 `const T&` —— 万能绑定

```cpp
// ✅ 绑定非常量左值
int ncLeft1 = 2;
const int& cRef1 = ncLeft1;
const int& cRef2 = ++ncLeft1;     // 前缀 ++ 返回非常量左值

// ✅ 绑定常量左值
const int cLeft1 = 2;
const int& cRef3 = cLeft1;

// ✅ 绑定非常量右值
const int& cRef4 = ncLeft1 + 1;   // 算术表达式返回非常量右值
const int& cRef5 = ncLeft1++;     // 后缀 ++ 返回非常量右值（原值的拷贝）
const int& cRef6 = add(1, 2);     // 函数按值返回是非常量右值
const int& cRef7 = [] { return 5; }(); // lambda 调用返回非常量右值

// ✅ 绑定常量右值
const int& cRef8 = 3;             // 字面量是常量右值
```

> [!tip] 为什么 `const T&` 能绑定右值？
> C++11 之前没有右值引用，如果 `const T&` 不能绑定右值，很多接口就没法用了——比如 `void push_back(const T& val)` 就不能写 `vec.push_back(42)`。
> 本质上，`const T&` 承诺"我不修改"，所以指向一个临时对象也无害：你反正不会通过它改什么东西。

#### 非常量右值引用 `T&&` —— 只能绑定非常量右值

```cpp
int ncLeft1 = 1;
int&& rRef1 = ncLeft1 + 1;         // ✅ 算术表达式返回非常量右值
int&& rRef2 = ncLeft1++;           // ✅ 后缀 ++ 返回非常量右值
int&& rRef3 = add(1, 2);           // ✅ 函数按值返回是非常量右值
int&& rRef4 = [] { return 5; }();  // ✅ lambda 调用返回非常量右值

// int&& rRef5 = ncLeft1;          // ❌ ncLeft1 是左值，不能偷走还在用的对象
// int&& rRef6 = 3;                // ❌ 3 是常量右值，非 const 引用不能绑定 const 值
```

> 右值引用变量 `rRef1` 本身是**左值**（有名字、可取地址）。这是理解移动语义的关键：`int&& x = 42;` 之后，`x` 是左值，不能再直接传给另一个 `T&&` 参数。

#### 常量右值引用 `const T&&` —— 绑定非常量/常量右值

```cpp
int ncLeft1 = 1;
const int&& crRef1 = ncLeft1 + 1;          // ✅ 非常量右值
const int&& crRef2 = ncLeft1++;            // ✅ 非常量右值
const int&& crRef3 = add(1, 2);            // ✅ 非常量右值
const int&& crRef4 = [] { return 5; }();   // ✅ 非常量右值
const int&& crRef5 = 3;                    // ✅ 常量右值

// const int&& crRef6 = ncLeft1;           // ❌ 左值不能绑定到右值引用
```

> `const T&&` 实践中极少使用——既然是右值引用（目的是接管资源），加 `const` 又不让修改，那就没法移动了。知道它的存在即可。

#### 对比表

| 维度 | `T&` | `const T&` | `T&&` | `const T&&` |
|------|:----:|:----------:|:-----:|:-----------:|
| 绑定非常量左值 | ✅ | ✅ | ❌ | ❌ |
| 绑定常量左值 | ❌ | ✅ | ❌ | ❌ |
| 绑定非常量右值 | ❌ | ✅ | ✅ | ✅ |
| 绑定常量右值 | ❌ | ✅ | ❌ | ✅ |
| 延长临时对象生命周期 | ❌ | ✅ | ✅ | ✅ |
| 能修改绑定的对象 | ✅ | ❌ | ✅ | ❌ |
| 引用本身是什么 | 左值 | 左值 | **左值** | 左值 |
| 典型用途 | 函数传引用、输出参数 | 只读参数（万能接收） | 移动构造/赋值、完美转发 | 极少使用 |

### 2.3 为什么绑定规则必须这么严格

用反面例子理解——假如规则不是这样会怎样：

**假如 `T&&` 能绑定左值：别人还在用的对象被偷了**

```cpp
// ⚠️ 假想的错误世界：T&& 能绑定左值
void steal(std::string&& s) {
    std::string mine = std::move(s);  // 偷走 s 的内存
}

std::string name = "Alice";
steal(name);      // 假如允许，name 的内存就被偷走了！
std::cout << name; // name 变成了空壳，但调用者根本不知道
```

现实中编译器会阻止 `steal(name)`，强制你写 `steal(std::move(name))`——`std::move` 就是你的**签字确认**："我知道 name 之后不能用了"。

**假如 `T&` 能绑定右值：改了个寂寞**

```cpp
// ⚠️ 假想的错误世界：T& 能绑定右值
void addOne(int& x) { x += 1; }

addOne(42);        // 假如允许，42+1=43……然后呢？
                   // 43 存在哪？没有变量接收，临时对象马上销毁
                   // 你以为改了什么，其实什么都没改
```

这种 bug 极难发现——代码看着像做了什么，实际什么效果都没有。所以编译器直接禁止，逼你传一个真正的变量进去。

### 2.4 关键陷阱：右值引用变量本身是左值

这是最容易搞混的点。**"右值引用"描述的是变量的类型，不是变量本身的值类别**。

类比：一个写着"临时工"的工牌（类型是右值引用），但这个人已经坐在了工位上开始干活（有了名字、有了地址）——他就不再是"临时的"了。

```cpp
void receive_copy(const std::string& s) { std::cout << "拷贝版本\n"; }
void receive_move(std::string&& s)      { std::cout << "移动版本\n"; }

void foo(std::string&& x) {
    // x 的类型是 string&&（右值引用）—— 这是工牌
    // 但 x 这个表达式是左值 —— 因为它有名字、可以 &x 取地址

    receive_copy(x);            // ✅ 匹配 const string&，走拷贝（x 是左值）
    // receive_move(x);         // ❌ 编译错误！x 是左值，不能绑定到 string&&
    receive_move(std::move(x)); // ✅ std::move 把左值再次转为右值，走移动
}

foo(std::string("hello"));     // "hello" 是右值，能传给 foo 的 string&& 参数
```

**为什么这样设计？** 安全。如果右值引用变量自动当右值用，资源在第一次使用时就被偷走了，后续代码再用这个变量就是 use-after-move：

```cpp
void process(std::string&& s) {
    log(s);             // 如果 s 自动当右值，这里就把 s 的内存偷走了
    save(s);            // s 已经是空壳了！
    // 正确做法：只在最后一次使用时 move
    log(s);             // 左值，安全拷贝
    save(std::move(s)); // 最后一次使用，move 交出资源
}
```

### 2.5 重载决议：编译器如何根据左值/右值选函数

```cpp
void handle(const std::string& s) { std::cout << "拷贝版本\n"; }
void handle(std::string&& s)      { std::cout << "移动版本\n"; }

std::string name = "Alice";
handle(name);              // 调用 const string& 版本（name 是左值）
handle(std::string("Bob"));// 调用 string&& 版本（临时对象是右值）
handle(std::move(name));   // 调用 string&& 版本（move 转成右值）
```

编译器的选择逻辑：右值优先匹配 `T&&`，匹配不到才退而求其次选 `const T&`。这就是为什么写移动构造函数后，临时对象会自动走移动路径。

> **面试速答**
>
> C++11 引入右值引用 `T&&`，和原来的左值引用 `T&` 形成分工：`T&` 只能绑定左值（还在用的对象），`T&&` 只能绑定右值（临时对象或 `std::move` 标记的对象），`const T&` 两者都能绑。这个绑定限制是有意设计的——防止你不小心偷走还在用的对象的资源，也防止你修改一个即将销毁的临时对象白费功夫。
>
> 有个关键陷阱：右值引用变量本身是左值。`void foo(string&& x)` 里 x 的类型是 `string&&`，但 x 这个表达式有名字、能取地址，所以是左值。往下传还得 `std::move(x)` 一次。这是为了安全——同一个变量在函数体里可能被用多次，只有最后一次才应该 move。

## 3. 移动语义

> 有了左值/右值的区分和右值引用之后，就可以实现"偷资源而不是拷贝"——这就是移动语义。

### 3.1 问题：C++03 的拷贝代价

```
传统 C++，函数返回一个含 10 万元素的 vector：

    源对象          →  分配新内存（10万×sizeof）  →  逐元素拷贝  →  销毁源对象
    临时的，要销毁的      白白分配的                    白白拷贝的      这才是它的归宿

    全程代价: O(n)，而且源对象本来就要销毁，拷贝完全是浪费
```

**移动语义的解法**：既然源对象要销毁，直接偷走它的内部指针就行了：

```
移动路径: 源对象 → 偷走指针 → 源对象置空     代价: O(1)
```

### 3.2 移动构造与移动赋值：怎么"偷"

```cpp
#include <iostream>
#include <cstring>
#include <utility>

class DynArray {
private:
    int* data_;
    size_t size_;

public:
    explicit DynArray(size_t n) : data_(new int[n]()), size_(n) {
        std::cout << "  构造: 分配 " << n << " 个 int\n";
    }

    ~DynArray() {
        std::cout << "  析构: 释放 " << (data_ ? "有效" : "空") << " 指针\n";
        delete[] data_;
    }

    // ======= 拷贝语义：深拷贝，O(n) =======

    DynArray(const DynArray& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::memcpy(data_, other.data_, size_ * sizeof(int));
        std::cout << "  拷贝构造: 深拷贝 " << size_ << " 个 int\n";
    }

    DynArray& operator=(const DynArray& other) {
        std::cout << "  拷贝赋值\n";
        DynArray tmp(other);
        swap(*this, tmp);
        return *this;
    }

    // ======= 移动语义：偷资源，O(1) =======

    DynArray(DynArray&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;     // 关键！源对象必须置空，否则析构时 double free
        other.size_ = 0;
        std::cout << "  移动构造: 偷走 " << size_ << " 个 int 的指针\n";
    }

    DynArray& operator=(DynArray&& other) noexcept {
        std::cout << "  移动赋值\n";
        if (this != &other) {
            delete[] data_;        // 释放自己的旧资源
            data_ = other.data_;   // 偷走源对象的资源
            size_ = other.size_;
            other.data_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }

    friend void swap(DynArray& a, DynArray& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    size_t size() const { return size_; }
    int& operator[](size_t i) { return data_[i]; }
};

DynArray createArray(size_t n) {
    DynArray arr(n);
    arr[0] = 42;
    return arr;  // NRVO 或 移动构造
}

int main() {
    std::cout << "=== 1. 工厂函数返回值 ===\n";
    DynArray a = createArray(1000);  // NRVO 优化，可能零拷贝

    std::cout << "\n=== 2. 拷贝构造 ===\n";
    DynArray b = a;  // 拷贝构造: O(n)

    std::cout << "\n=== 3. 移动构造 ===\n";
    DynArray c = std::move(a);  // 移动构造: O(1)，a 之后不可再用

    std::cout << "\n=== 4. 移动赋值 ===\n";
    DynArray d(10);
    d = std::move(b);  // 移动赋值: O(1)

    std::cout << "\n=== 析构顺序（逆序）===\n";
    return 0;
}
```

### 3.3 std::move 的真相

`std::move` 的实现极其简单，它只是一个 `static_cast`：

```cpp
// 标准库的 std::move 本质（简化版）
template <typename T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}
```

**std::move 不移动任何东西！** 它只是把一个左值 cast 成右值引用（将亡值），相当于告诉编译器："我不再需要这个对象了，你可以偷它的资源。"

真正的移动发生在移动构造函数或移动赋值运算符里。

**底层关键：`remove_reference` 类型萃取**

`std::move` 的核心依赖 `remove_reference`，通过模板偏特化移除类型的引用属性：

```cpp
// 主模板：非引用类型直接返回自身
template <typename T>
struct remove_reference { using type = T; };

// 偏特化：移除左值引用
template <typename T>
struct remove_reference<T&> { using type = T; };

// 偏特化：移除右值引用
template <typename T>
struct remove_reference<T&&> { using type = T; };
```

无论输入是左值、左值引用、右值引用还是临时对象，`std::move` 最终都返回对应的右值引用（`T&&`）。参数 `T&& t` 是万能引用，既能接收左值也能接收右值，然后通过 `static_cast` 统一转换为右值引用。

### 3.4 std::move 之后，原对象还存在吗？

**存在，但被"掏空"了。** 对象本身没有被销毁，它的内存还在栈上，析构函数也还没调用。但它内部管理的资源（堆内存、文件句柄等）已经被新对象偷走了。

```cpp
std::string a = "hello world";   // a 拥有堆上的字符数据
std::string b = std::move(a);    // b 偷走了 a 的内部指针

// 此刻：
// a 还活着吗？→ 活着，还在栈上，可以调用方法
// a 的值是什么？→ 未指定（标准只保证"有效但未指定"状态）
// a 能用吗？   → 能，但只能做两件事：销毁 或 重新赋值
```

用内存布局看得更清楚：

```
move 之前：
a: [ptr] ──→ 堆: "hello world"    b: [未初始化]

move 之后（移动构造内部做了什么）：
a: [ptr=nullptr, size=0]           b: [ptr] ──→ 堆: "hello world"
    ↑ 资源被偷走，                      ↑ 资源转移到这里
      但对象本身还在栈上
```

**对不同类型，move 后的状态不同：**

| 类型 | move 后原对象的状态 | 原因 |
|------|-------------------|------|
| `std::string` | 通常变为空字符串 `""` | 内部指针被转移，size 置 0 |
| `std::vector` | 变为空容器 `{}` | 内部缓冲区指针被转移 |
| `std::unique_ptr` | 变为 `nullptr` | 独占指针，所有权转移 |
| `int`、`double` 等基本类型 | **值不变** | 基本类型没有"资源"可偷，move 等于 copy |
| 自定义类 | 取决于你的移动构造怎么写 | 标准只要求处于"可析构"的有效状态 |

> [!warning] "有效但未指定"是什么意思？
> 标准说 move 后的对象处于 **valid but unspecified state**：
> - **valid（有效）**：对象没坏，可以安全地析构、赋值
> - **unspecified（未指定）**：不保证具体值是什么，不要读它的内容来做业务逻辑判断
> 
> 所以 move 后只做两件事是安全的：**销毁**（让它正常走析构）或**重新赋值**（给它新值后又能正常使用）。

```cpp
std::string name = "Alice";
std::string other = std::move(name);

// ❌ 不要这样做
if (name == "Alice") { ... }   // name 的值未指定，结果不可靠
std::cout << name[0];          // 可能越界（如果 name 变成了空串）

// ✅ 可以这样做
name = "Bob";                  // 重新赋值，name 又是正常对象了
// 或者什么都不做，让 name 在作用域结束时自动析构
```

### 3.5 编译器何时自动生成移动操作

| 条件 | 自动生成移动构造? | 自动生成移动赋值? |
|------|:-:|:-:|
| 没有声明任何拷贝操作、移动操作、析构函数 | Yes | Yes |
| 声明了拷贝构造函数 | **No** | **No** |
| 声明了拷贝赋值运算符 | **No** | **No** |
| 声明了移动构造函数 | - | **No** |
| 声明了移动赋值运算符 | **No** | - |
| 声明了析构函数 | **No** | **No** |

**Rule of Five**：如果你声明了析构函数、拷贝构造、拷贝赋值、移动构造、移动赋值中的任何一个，你应该声明全部五个（或者用 `= default` / `= delete` 显式表态）。

> [!note]- 为什么声明了一个就要声明全部五个？（点击展开）
> 
> 因为声明其中任何一个，通常意味着你的类在**管理资源**（堆内存、文件句柄、网络连接等），而编译器默认生成的其他操作只做**逐成员浅拷贝/浅移动**，对管理资源的类来说几乎必然是错的：
> 
> ```cpp
> class BadString {
>     char* data_;
>     size_t size_;
> public:
>     BadString(const char* s) : size_(strlen(s)), data_(new char[size_ + 1]) {
>         memcpy(data_, s, size_ + 1);
>     }
>     ~BadString() { delete[] data_; }  // 你声明了析构 → 说明类管理了堆内存
> 
>     // 没声明其他四个 → 编译器自动生成的拷贝构造只做浅拷贝：
>     // BadString(const BadString& o) : data_(o.data_), size_(o.size_) {}
>     //                                       ↑ 两个对象指向同一块内存！
> };
> 
> BadString a("hello");
> BadString b = a;     // 浅拷贝：b.data_ == a.data_，指向同一块堆内存
> // a 析构 → delete[] data_
> // b 析构 → delete[] data_  → 同一块内存被 delete 两次！→ 崩溃
> ```
> 
> **逻辑链条**：
> 
> ```
> 你声明了析构函数
>   → 说明类管理了资源（否则不需要自定义析构）
>   → 编译器默认的浅拷贝会导致多个对象共享同一份资源
>   → 拷贝后多次释放 → double free / use-after-free
>   → 所以必须自定义拷贝构造和拷贝赋值（深拷贝）
>   → 同理，默认的移动也可能有问题，必须自定义移动构造和移动赋值
>   → 五个操作环环相扣，缺一个就可能踩坑
> ```
> 
> 反过来也成立：如果你声明了拷贝构造，说明默认的逐成员拷贝不对，那默认的析构、移动大概率也不对。
> 
> **五个操作的分工**：
> 
> | 操作 | 职责 | 不声明的后果 |
> |------|------|------------|
> | 析构函数 | 释放资源 | 资源泄漏 |
> | 拷贝构造 | 深拷贝资源 | 浅拷贝 → double free |
> | 拷贝赋值 | 释放旧资源 + 深拷贝新资源 | 旧资源泄漏 + 浅拷贝 |
> | 移动构造 | 偷走资源 + 置空源对象 | 退化为拷贝（性能差），或编译器不生成 |
> | 移动赋值 | 释放旧资源 + 偷走新资源 + 置空源对象 | 同上 |

```cpp
// ✅ 正确做法：五个全部显式声明
class Good {
    ~Good() = default;
    Good(const Good&) = default;
    Good& operator=(const Good&) = default;
    Good(Good&&) = default;
    Good& operator=(Good&&) = default;
};
```

> [!tip] Rule of Zero
> 如果你的类**不直接管理资源**（用 `std::string`、`std::vector`、`std::unique_ptr` 等 RAII 类型做成员），那五个操作都不需要声明——编译器默认生成的就是对的。这叫 **Rule of Zero**，是比 Rule of Five 更优先的选择。

### 3.6 拷贝构造为什么必须是 const T&

**1. 避免无限递归**：如果参数是值传递，调用拷贝构造时需要先拷贝实参，又触发拷贝构造，无限循环。

```cpp
// ❌ 值传递导致无限递归
MyClass(MyClass other) { }  // 传入实参时要拷贝 → 又调用拷贝构造 → 死循环

// ✅ 引用传递
MyClass(const MyClass& other) { }
```

**2. 支持 const 对象和临时对象**：`MyClass&` 无法绑定 const 对象或临时对象，`const MyClass&` 是万能接收者。

**3. 防止意外修改源对象**：拷贝的语义是"复制"而非"修改"。

### 3.7 赋值运算符为什么返回 T&

**1. 支持链式赋值**：`a = b = c;` 等价于 `a.operator=(b.operator=(c))`，`b = c` 必须返回 `b` 的引用。

**2. 与内置类型行为一致**：`int x, y, z; x = y = z = 10;` 返回的是引用。

**3. 避免不必要的拷贝**：返回值类型每次赋值都触发拷贝构造。

### 3.8 陷阱集锦

#### 陷阱 1: move 后继续使用对象

```cpp
// ❌ move 后对象处于"有效但未指定"状态，不要再读它的值
std::string name = "hello";
std::string other = std::move(name);
std::cout << name.size();   // 合法，但结果可能是 0 也可能不是

// ✅ move 后只能做两件事：销毁 或 重新赋值
name = "world";            // 重新赋值后又可以正常使用了
```

#### 陷阱 2: const 对象无法移动

```cpp
// ❌ const 对象 move 了个寂寞，实际调用的是拷贝构造
const std::string s = "big string";
std::string t = std::move(s);  // 调用的是拷贝构造！

// 原因：移动构造签名是 String(String&&)，不接受 const String&&
// const String&& 退而匹配 const String&（拷贝构造），所以退化为拷贝
```

#### 陷阱 3: 右值引用参数在函数体内是左值

```cpp
// ❌ 以为 arg 是右值引用就能自动传递右值性
void process(DynArray&& arg) {
    DynArray local = arg;  // 调用的是拷贝构造！arg 有名字，是左值
}

// ✅ 需要再次 std::move
void process(DynArray&& arg) {
    DynArray local = std::move(arg);  // 调用移动构造
}
```

#### 陷阱 4: 移动构造不加 noexcept 的后果

```cpp
class Bad {
public:
    Bad(Bad&& other) { /* 没有 noexcept */ }
};

// std::vector 扩容时，如果移动构造不是 noexcept，
// vector 会选择拷贝而不是移动！
// 原因：移动到一半抛异常，已移动的元素无法恢复
// 拷贝则安全——源对象不变，出错可以回滚

// ✅ 移动操作尽量标记 noexcept
class Good {
public:
    Good(Good&& other) noexcept { /* ... */ }
};
```

### 3.9 优缺点与权衡

**优点**：
- 将深拷贝 O(n) 降为指针转移 O(1)：对于持有堆资源的类型（`string`、`vector`、`unique_ptr`），移动构造只需要拷贝几个指针/size 字段并将源置空
- 编译器自动优化临时对象：函数返回值、容器插入临时对象等场景，编译器通过重载决议自动选择移动构造
- 支持"只移动"类型：`unique_ptr`、`std::thread`、`std::fstream` 等独占资源的类型禁止拷贝但允许移动，语义上更精确地表达所有权转移

**缺点/代价**：
- 移后状态需要自律：标准只保证移后对象处于"有效但未指定"状态，读取其值结果不确定
- 必须标记 `noexcept` 否则退化：`std::vector` 扩容时检查移动构造是否 `noexcept`，如果不是则退化为拷贝
- const 对象无法移动：`const T&&` 匹配不到移动构造，默默退化为拷贝

**权衡**：移动语义在含动态资源的类型上收益巨大，对于 POD 类型移动 == 拷贝没有额外代价。RVO/NRVO 这类 copy elision 是零开销（连移动都省了），能消除拷贝/移动就不要强行 `std::move`。

> **面试速答**
>
> 移动语义让 C++ 能把临时对象的深拷贝 O(n) 变成指针转移 O(1)。核心实现是移动构造函数：接管源对象的堆指针，然后把源对象置空。`std::move` 本身只是 `static_cast<T&&>` 的类型转换，真正的移动逻辑写在移动构造/赋值里。
>
> 有几个坑：第一，右值引用变量本身是左值，往下传还得 `std::move` 一次；第二，移动操作一定要标 `noexcept`，不然 vector 扩容会退化成拷贝；第三，const 对象 move 后实际调用的还是拷贝构造。
>
> `std::move` 是无条件转右值，`std::forward` 是有条件保持值类别（配合万能引用用，见完美转发章节）。RVO/NRVO 比移动更优（零开销），能消除拷贝/移动就不要强行 `std::move`。

## 4. 完美转发

> **一句话定义**：完美转发让模板函数在传递参数时，保持参数原来的值类别（左值还是右值）不变。

### 4.1 问题：为什么需要完美转发？

假设你写一个工厂函数，转发参数给构造函数：

```cpp
// 不用完美转发，你需要写 2^n 个重载（n 是参数个数）
template <typename T>
std::unique_ptr<T> make(const T& arg) { return std::make_unique<T>(arg); }     // 左值版本
template <typename T>
std::unique_ptr<T> make(T&& arg) { return std::make_unique<T>(std::move(arg)); } // 右值版本
// 两个参数就要 4 个重载，三个参数 8 个... 指数爆炸！
```

### 4.2 万能引用（转发引用）

当 `T&&` 出现在**模板参数推导**上下文中时，它不是右值引用，而是**万能引用**（又叫转发引用）：

```cpp
template <typename T>
void f(T&& arg);  // T&& 是万能引用：既能接收左值，也能接收右值

auto&& x = expr;  // auto&& 也是万能引用
```

**注意区分**：

```cpp
template <typename T>
void f(T&& arg);           // ✅ 万能引用（T 需要推导）

void g(int&& arg);         // ❌ 不是万能引用，就是普通右值引用

template <typename T>
void h(std::vector<T>&& v); // ❌ 不是万能引用，v 就是 vector 的右值引用
                             //    因为 T 不是直接修饰 &&

template <typename T>
class Foo {
    void bar(T&& arg);     // ❌ 不是万能引用！T 在类实例化时已确定，不参与推导
};
```

### 4.3 引用折叠规则

C++ 不允许"引用的引用"，但模板推导中会产生，编译器用**引用折叠**解决：

| T 推导结果 | T&& 折叠后 | 规则 |
|-----------|-----------|------|
| `int&` | `int& &&` → `int&` | 左值引用 + 任何引用 = 左值引用 |
| `int&&` | `int&& &&` → `int&&` | 右值引用 + 右值引用 = 右值引用 |
| `int` | `int&&` | 无折叠 |

**口诀**：只要有一个 `&`，结果就是 `&`。只有 `&& + &&` 才保持 `&&`。

```
& + &   = &     (左 + 左 = 左)
& + &&  = &     (左 + 右 = 左)
&& + &  = &     (右 + 左 = 左)
&& + && = &&    (右 + 右 = 右)
```

### 4.4 std::forward 的实现原理

```cpp
// 简化版 std::forward
template <typename T>
constexpr T&& forward(std::remove_reference_t<T>& arg) noexcept {
    return static_cast<T&&>(arg);
}
```

**推导过程**：

```cpp
template <typename T>
void wrapper(T&& arg) {
    target(std::forward<T>(arg));
}

// 情况 1：传入左值 int x = 42; wrapper(x);
//   T 推导为 int&
//   forward<int&>(arg) → static_cast<int& &&>(arg) → static_cast<int&>(arg)
//   结果：左值引用 → 传给 target 的是左值 ✅

// 情况 2：传入右值 wrapper(42);
//   T 推导为 int
//   forward<int>(arg) → static_cast<int&&>(arg)
//   结果：右值引用 → 传给 target 的是右值 ✅
```

### 4.5 工厂函数中的完美转发示例

```cpp
#include <memory>
#include <utility>
#include <iostream>
#include <string>

class Connection {
public:
    Connection(const std::string& host, int port)
        : host_(host), port_(port) {
        std::cout << "Connection(const string&, int): 拷贝 host\n";
    }
    Connection(std::string&& host, int port)
        : host_(std::move(host)), port_(port) {
        std::cout << "Connection(string&&, int): 移动 host\n";
    }
private:
    std::string host_;
    int port_;
};

// 完美转发工厂函数：参数原封不动转发给构造函数
template <typename T, typename... Args>
std::unique_ptr<T> make(Args&&... args) {
    return std::make_unique<T>(std::forward<Args>(args)...);
}

int main() {
    std::string host = "127.0.0.1";

    // 传左值 → 拷贝
    auto c1 = make<Connection>(host, 8080);

    // 传右值 → 移动
    auto c2 = make<Connection>(std::string("192.168.1.1"), 9090);

    // 移动已有变量
    auto c3 = make<Connection>(std::move(host), 3306);
}
```

输出：
```
Connection(const string&, int): 拷贝 host
Connection(string&&, int): 移动 host
Connection(string&&, int): 移动 host
```

完美转发保证了：左值进去出来还是左值（走拷贝），右值进去出来还是右值（走移动），零额外开销。

### 4.6 完整转发链路：从调用到落地

用一个完整例子把万能引用、引用折叠、`std::forward` 串起来，逐步跟踪类型变化：

```cpp
#include <iostream>
#include <utility>

// 落地函数 1：左值引用 + 右值引用
void bar(int& a, int&& b) {
    int c = a + b;
    std::cout << "bar: a=" << a << " b=" << b << " c=" << c << std::endl;
}

// 落地函数 2：值传递 + 右值引用
void func(int a, int&& b) {
    int c = a + b;
    std::cout << "func: a=" << a << " b=" << b << " c=" << c << std::endl;
}

// 中间转发层：T&& 是万能引用，std::forward 保持值类别不变
template <typename A, typename B>
void foo(A&& a, B&& b) {
    bar(std::forward<A>(a), std::forward<B>(b));
}

int main() {
    int a = 10;

    // 场景 1：通过模板完美转发
    foo(a, 20);

    // 场景 2：非模板场景直接使用 std::forward
    func(std::forward<int>(a), std::forward<int&&>(30));

    return 0;
}
```

#### 场景 1：模板完美转发 `foo(a, 20)`

`foo` 的模板参数 `A&&`、`B&&` 是万能引用，编译器根据传入的实参推导类型：

```
调用 foo(a, 20)：
                                    
  a (左值 int)                20 (右值 int)
       │                           │
  A 推导为 int&               B 推导为 int
       │                           │
  A&& = int& && → int&       B&& = int&& → int&&     ← 引用折叠
       │                           │
  forward<int&>(a)            forward<int>(b)
  = static_cast<int&>(a)     = static_cast<int&&>(b)  ← forward 的本质
       │                           │
     左值                        右值
       │                           │
  bar(int& a,                 int&& b)                 ← 精确匹配！
```

展开后 `foo` 等价于：

```cpp
void foo(int& a, int&& b) {                  // 引用折叠后的签名
    bar(static_cast<int&>(a), static_cast<int&&>(b));  // forward 展开
}
```

转发前后，`a` 是左值、`b` 是右值，**值类别完全不变**。

#### 场景 2：非模板场景 `std::forward<int>(a)`

`std::forward` 不是只能在模板里用——你也可以手动指定类型参数来直接调用：

```cpp
func(std::forward<int>(a), std::forward<int&&>(30));
```

逐个分析：

| 调用 | T 是什么 | `forward<T>` 展开 | 结果 |
|------|---------|-------------------|------|
| `std::forward<int>(a)` | `int` | `static_cast<int&&>(a)` | 右值 → 传给 `func` 的第一个参数（按值接收，触发拷贝/移动） |
| `std::forward<int&&>(30)` | `int&&` | `static_cast<int&& &&>(30)` → `static_cast<int&&>(30)` | 右值（引用折叠 `&& + && = &&`） |

> [!warning] 非模板场景下 `forward` 等价于手写 `static_cast`
> 在非模板代码中，`std::forward<int>(a)` 就是 `static_cast<int&&>(a)`，和 `std::move(a)` 效果相同。这种用法合法但不推荐——**`forward` 的设计意图是配合万能引用做转发**，非模板场景用 `std::move` 更清晰。

#### 对比：不用 forward 会怎样？

```cpp
template <typename A, typename B>
void foo_broken(A&& a, B&& b) {
    bar(a, b);  // ❌ a 和 b 都是左值（有名字）！
                // b 的类型是 int&&，但 b 这个表达式是左值
                // 传给 bar 的第二个参数是左值，匹配不了 int&&，编译错误！
}
```

这就是 `std::forward` 存在的意义——**恢复被"有了名字"而丢失的右值属性**。

#### 4.7 优缺点与权衡

**优点**：
- 消除指数级重载爆炸：没有完美转发，N 个参数的工厂函数需要 2^N 个重载（每个参数都有左值/右值两种版本）。一个万能引用 + `std::forward` 就能统一处理，`std::make_unique`、`emplace_back` 等标准库接口都依赖此机制
- 零开销值类别保持：`std::forward` 编译后只是一个 `static_cast`，没有任何运行时开销，左值进来出去还是左值（走拷贝），右值进来出去还是右值（走移动），透明转发不丢性能
- 泛型代码的基石：完美转发让库作者可以写出"透明包装层"（工厂函数、装饰器、线程池任务提交），调用方完全感知不到中间层的存在

**缺点/代价**：
- 理解成本极高：万能引用、引用折叠、`std::forward` 的推导过程涉及模板元编程的核心机制，是 C++ 中认知门槛最高的特性之一。代码审查时如果团队不熟悉，会成为维护负担
- 错误信息极其晦涩：万能引用匹配范围极广，一旦参数类型不对，编译器的模板实例化错误信息可能长达数十行且难以定位根因
- 容易和普通重载冲突：万能引用 `T&&` 比 `const T&` 更贪婪（精确匹配优于 const 转换），容易意外拦截本该走其他重载的调用，导致编译错误或运行时行为不符预期

**权衡**：完美转发的收益主要在库代码和基础设施层——工厂函数、emplace 接口、线程池任务提交等需要透传参数的场景。普通业务代码很少需要直接使用 `std::forward`。如果团队对模板元编程不熟悉，应将使用 `forward` 的代码封装在底层库中，上层业务代码通过简洁的接口调用，避免理解成本扩散。

> **面试速答**
>
> `std::forward<T>(arg)` 是一个有条件的类型转换，作用是在模板转发中保持参数原来的值类别（左值还是右值）不变。它解决的问题是：万能引用 `T&&` 接收参数之后，参数本身变成了一个左值（它有名字了），如果直接往下传就丢失了原来的右值属性，移动语义就触发不了。
>
> 它的实现原理是万能引用加引用折叠规则：`& + && = &`，`&& + && = &&`。当你传入左值时 T 被推导成 `T&`，forward 返回左值引用；传入右值时 T 被推导为 `T`，forward 返回右值引用。这样就把原始值类别精确地透传下去了。
>
> 有个容易搞错的点：forward 只能用在模板转发场景里，非模板函数用它没意义；而且 T 必须是模板推导出来的，手动指定类型会破坏转发语义。
>
> std::move 和 std::forward 的分工很清晰：前者是无条件转右值，用于你明确要移动的场景；后者是有条件保持原值类别，只用在模板转发层。实际使用时，写工厂函数、emplace 接口、包装层这种需要透传参数的模板代码就用 forward；明确要移动局部对象就用 move。

## 5. Lambda 表达式

> **一句话定义**：Lambda 是就地定义的匿名函数对象，编译器会为每个 lambda 生成一个独一无二的匿名类。

### 5.1 语法结构拆解

```
[捕获列表](参数列表) mutable 异常声明 -> 返回类型 { 函数体 }
  │           │         │        │           │         │
  │           │         │        │           │         └─ 函数逻辑
  │           │         │        │           └─ 可省略（编译器推导）
  │           │         │        └─ 可选，通常省略
  │           │         └─ 可选，允许修改值捕获的变量
  │           └─ 和普通函数参数一样
  └─ 核心！决定 lambda 能访问哪些外部变量
```

### 5.2 捕获方式详解

| 捕获方式 | 语法 | 含义 | 注意事项 |
|----------|------|------|----------|
| 值捕获 | `[x]` | 拷贝 x 的值到 lambda 内部 | lambda 内的 x 是 const（除非 mutable） |
| 引用捕获 | `[&x]` | 引用外部的 x | x 销毁后 lambda 悬空！ |
| 隐式值捕获 | `[=]` | 按值捕获所有用到的变量 | 容易意外捕获 this |
| 隐式引用捕获 | `[&]` | 按引用捕获所有用到的变量 | 生命周期风险大 |
| 混合 | `[=, &x]` | 默认值捕获，x 引用捕获 | |
| 初始化捕获 (C++14) | `[y = std::move(x)]` | 移动 x 到 lambda 的成员 y | 解决了 C++11 无法移动捕获的问题 |
| 捕获 this | `[this]` | 捕获当前对象的 this 指针 | 小心对象先于 lambda 销毁 |
| 捕获 *this (C++17) | `[*this]` | 拷贝整个对象 | 安全但有开销 |

### 5.3 Lambda 底层：编译器生成的匿名类

每个 lambda 表达式，编译器都会生成一个匿名类。以下展示等价代码：

```cpp
// 你写的 lambda
int x = 10;
auto f = [x](int y) -> int { return x + y; };
int result = f(5);

// 编译器实际生成的等价代码（示意）
class __lambda_unique_name {
private:
    int x;  // 值捕获的变量成为成员
public:
    __lambda_unique_name(int x) : x(x) {}

    // operator() 默认是 const（所以不能修改值捕获的变量）
    int operator()(int y) const {
        return x + y;
    }
};

int x = 10;
auto f = __lambda_unique_name(x);  // 构造匿名对象，拷贝 x
int result = f(5);
```

**引用捕获的等价代码**：

```cpp
// 你写的 lambda
int x = 10;
auto f = [&x](int y) { x += y; return x; };

// 编译器生成的等价代码
class __lambda_ref {
private:
    int& x;  // 引用捕获 → 引用成员
public:
    __lambda_ref(int& x) : x(x) {}
    int operator()(int y) const {
        x += y;  // 注意：const 限定的是 operator()
        return x; // 但引用本身不变，修改的是引用指向的对象，所以合法
    }
};
```

**mutable lambda 的等价代码**：

```cpp
// 你写的 lambda
int x = 10;
auto f = [x]() mutable { return ++x; };
// f() → 11, f() → 12, f() → 13... 每次调用都修改内部副本

// 编译器生成的等价代码
class __lambda_mutable {
private:
    int x;
public:
    __lambda_mutable(int x) : x(x) {}
    // mutable → operator() 不是 const
    int operator()() {
        return ++x;  // 可以修改成员
    }
};
```

### 5.4 泛型 Lambda (C++14)

```cpp
// C++14: auto 参数 → 编译器生成模板 operator()
auto add = [](auto a, auto b) { return a + b; };

// 等价于:
struct __generic_lambda {
    template <typename T1, typename T2>
    auto operator()(T1 a, T2 b) const {
        return a + b;
    }
};

add(1, 2);         // T1=int, T2=int
add(1.5, 2);       // T1=double, T2=int
add(std::string("a"), std::string("b")); // T1=string, T2=string
```

### 5.5 Lambda 的类型：std::function 与函数指针

每个 Lambda 的类型是编译器生成的**唯一匿名类型**，无法直接声明，但可通过以下方式间接使用：

**1. auto 推导**：直接推导为闭包类的类型（最高效，零开销）。

**2. std::function 包装**：通用多态函数包装器，可存储任意可调用对象（有类型擦除开销）：

```cpp
#include <functional>
std::function<int(int, int)> add = [](int a, int b) { return a + b; };
```

**3. 函数指针（仅无捕获 Lambda）**：无捕获的 Lambda 可隐式转换为函数指针（因为其 `operator()` 是静态可调用的）：

```cpp
auto add = [](int a, int b) { return a + b; };
int (*func_ptr)(int, int) = add;  // 合法（无捕获）

// 有捕获的 Lambda 无法转换为函数指针
// 因为闭包对象包含成员变量，需要实例状态
int x = 10;
auto with_capture = [x](int a) { return a + x; };
// int (*p)(int) = with_capture;  // ❌ 编译错误
```

**选择建议**：优先用 `auto`（零开销）；需要存储不同签名的可调用对象时用 `std::function`；需要兼容 C 回调接口时用函数指针（仅无捕获 Lambda 可用）。

### 5.6 陷阱集锦

#### 陷阱 1: 引用捕获悬空

```cpp
// ❌ 错误：lambda 引用捕获了局部变量，函数返回后悬空
std::function<int()> make_counter() {
    int count = 0;
    return [&count]() { return ++count; };  // count 已销毁！
}

// ✅ 正确：值捕获 + mutable
std::function<int()> make_counter() {
    int count = 0;
    return [count]() mutable { return ++count; };
}
```

#### 陷阱 2: 捕获 this 的生命周期问题

```cpp
class Server {
    int port_ = 8080;
public:
    // ❌ 错误：lambda 捕获 this，但 Server 对象可能先于 lambda 销毁
    auto get_handler() {
        return [this]() {
            return port_;  // this 可能已经是野指针
        };
    }

    // ✅ 正确方案 1 (C++17)：拷贝整个对象
    auto get_handler_safe() {
        return [*this]() {
            return port_;  // 使用的是拷贝的副本
        };
    }

    // ✅ 正确方案 2 (C++14)：初始化捕获需要的成员
    auto get_handler_safe2() {
        return [port = port_]() {
            return port;  // 值捕获，安全
        };
    }
};
```

#### 陷阱 3: [=] 隐式捕获 this

```cpp
class Foo {
    int val_ = 42;
public:
    auto make_lambda() {
        // ❌ 风险：[=] 看起来是值捕获，但 val_ 实际通过 this 指针访问
        return [=]() { return val_; };  // 等价于 [this]() { return this->val_; }
        // C++20 中 [=] 隐式捕获 this 会产生 deprecation 警告

        // ✅ 明确：要么 [*this] 拷贝对象，要么 [val = val_] 拷贝值
    }
};
```

#### 5.7 优缺点与权衡

**优点**：
- 就地定义闭包，消除样板代码：不需要像传统 functor 那样单独定义一个类，算法谓词、回调函数可以在调用点直接编写，代码局部性极好，可读性提升
- 比 `std::function` 零开销：`auto` 推导的 lambda 类型是编译器生成的唯一类型，编译器可以完全内联 `operator()`，没有类型擦除和堆分配开销。相比之下 `std::function` 需要类型擦除 + 可能的堆分配（小对象优化除外）
- 捕获上下文的能力：值捕获和引用捕获让 lambda 可以访问外层作用域的变量，这是普通函数指针做不到的。C++14 的初始化捕获更支持移动语义（`[p = std::move(ptr)]`）

**缺点/代价**：
- 捕获生命周期陷阱：引用捕获 `[&]` 最危险——如果 lambda 的生存期超过被捕获变量（异步回调、存入容器、跨线程传递），引用悬空导致 UB，且 bug 极难定位
- `[=]` 在成员函数中隐式捕获 this 指针：看起来是值捕获所有变量，实际上成员变量通过 `this->` 访问，`this` 指向的对象销毁后 lambda 仍持有野指针。C++20 已将此行为标记为 deprecated
- 每个 lambda 类型唯一：两个签名完全相同的 lambda 是不同类型，无法放入同一个 `std::vector<auto>` 中，需要用 `std::function` 做类型擦除，又引入了额外开销

**权衡**：lambda 在同步、短生命周期场景中（STL 算法谓词、局部回调）是完美工具——零开销、就地定义、可内联。但在异步/长生命周期场景中（存入容器、跨线程传递、延迟执行），必须格外注意捕获变量的生命周期：优先值捕获 `[=]`，成员函数中用 `[*this]`（C++17）或初始化捕获 `[val = member_]`，彻底避免引用悬空。

> **面试速答**
>
> Lambda 本质上是编译器生成的匿名类（闭包类型）：捕获的变量会变成类成员，函数体变成 `operator()` 重载。它的价值在于替代函数指针和手写 functor 的冗长写法，让回调、算法谓词可以就地定义并捕获上下文。
>
> 底层机制上，值捕获生成值成员（operator() 默认是 const 的，所以不能改捕获的值，除非加 mutable），引用捕获生成引用成员。每个 lambda 的类型都是唯一的匿名类型，所以变量只能用 auto 来接。
>
> 和其他可调用对象相比：lambda 比函数指针灵活（能捕获上下文），比 std::function 高效（没有类型擦除的开销，能被内联）；而且无捕获的 lambda 还能隐式转成函数指针，可以兼容 C API 回调。
>
> 有两个坑要特别注意。一是引用捕获悬空——lambda 活得比引用的对象久就 UB，异步回调场景特别容易踩；二是在成员函数里写 `[=]` 其实是隐式捕获 this 指针（而不是拷贝成员值），成员对象析构后 lambda 再调用就悬空了，C++17 引入的 `[*this]` 就是专门解决这个问题的。
>
> 三者选型：追求性能、本地使用就用 lambda（零开销、类型唯一）；需要把可调用对象存起来、作为参数传递就用 std::function（类型擦除、可存储但可能有堆分配开销）；对接 C API 不需要捕获上下文就用函数指针。

## 6. auto 与 decltype

> **一句话定义**：`auto` 让编译器从初始化表达式推导类型；`decltype` 让编译器从表达式推导类型但不求值。

### 6.1 auto 推导规则

`auto` 的推导规则和模板参数推导一致：

| 场景 | 代码 | 推导结果 | 说明 |
|------|------|----------|------|
| 基本 | `auto x = 42;` | `int` | |
| 去 const | `const int ci = 0; auto x = ci;` | `int` | 值拷贝丢 const |
| 去引用 | `int& r = i; auto x = r;` | `int` | 值拷贝丢引用 |
| 保 const | `const int ci = 0; const auto& x = ci;` | `const int&` | 显式声明 |
| 指针 | `int* p = &i; auto x = p;` | `int*` | 指针不丢 |
| 万能引用 | `auto&& x = lvalue;` | `T&`(左值引用) | 引用折叠 |
| 万能引用 | `auto&& x = rvalue;` | `T&&`(右值引用) | |
| initializer_list | `auto x = {1, 2, 3};` | `initializer_list<int>` | 注意这个特殊规则 |
| 花括号直接初始化 | `auto x{42};` | C++17 起是 `int` | C++11/14 是 `initializer_list<int>` |

### 6.2 decltype 推导规则

`decltype` 有两套不同的规则，取决于操作数是**变量名**还是**表达式**：

| 操作数 | 规则 | 示例 | 结果 |
|--------|------|------|------|
| 无括号的变量名 | 得到变量的**声明类型** | `int x; decltype(x)` | `int` |
| 有括号的变量名 | 当作表达式处理（见下方） | `int x; decltype((x))` | `int&`（左值表达式→引用） |
| 左值表达式 | `T&` | `decltype(*p)` | `int&` |
| 右值表达式 | `T` | `decltype(42)` | `int` |
| xvalue 表达式 | `T&&` | `decltype(std::move(x))` | `int&&` |

**关键陷阱**：`decltype(x)` 和 `decltype((x))` 结果不同！

```cpp
int x = 0;
decltype(x) a = x;     // int    （变量名 → 声明类型）
decltype((x)) b = x;   // int&   （加了括号 → 左值表达式 → 引用）
```

### 6.3 decltype(auto) (C++14)

`decltype(auto)` 结合了 `auto` 的便利和 `decltype` 的精确：

```cpp
// auto 会丢引用，decltype(auto) 不会
int x = 0;
int& rx = x;

auto a = rx;            // int    （auto 丢引用）
decltype(auto) b = rx;  // int&   （保持引用）

// 常用于返回值：保持原始返回类型
template <typename Container, typename Index>
decltype(auto) access(Container&& c, Index i) {
    return std::forward<Container>(c)[i];  // 如果 c[i] 返回引用，这里也返回引用
}
```

### 6.4 返回值类型后置

```cpp
// 问题：返回类型依赖参数类型，写在前面编译器还没看到参数
template <typename T, typename U>
auto add(T a, U b) -> decltype(a + b) {
    return a + b;
}

// C++14 起可以省略后置返回类型（编译器从 return 语句推导）
template <typename T, typename U>
auto add(T a, U b) {
    return a + b;
}
```

#### 6.5 优缺点与权衡

**优点**：
- 消除冗长类型声明：`auto it = container.begin()` 替代 `std::map<std::string, std::vector<int>>::const_iterator it = ...`，代码简洁性和可维护性大幅提升，尤其在模板密集的代码中
- 保证初始化：`auto x = expr;` 语法上强制要求有初始化表达式，不可能出现未初始化变量的 bug
- `decltype` 支持精确类型计算：`decltype(auto)` 可以在返回值中保留引用和 cv 限定符，是泛型库代码实现完美返回类型的核心工具

**缺点/代价**：
- 可读性降低：过度使用 `auto` 会导致阅读代码时无法直接看出变量类型，尤其在没有 IDE 辅助的代码审查场景中。`auto x = foo();` 必须跳转到 `foo()` 的定义才能知道 x 的类型
- `auto` 丢引用丢 const 可能引入隐蔽 bug：`auto x = const_ref;` 会拷贝一份而非绑定引用，意外拷贝大对象既影响性能又可能导致逻辑错误（修改了副本而非原对象）
- `decltype((x))` 的括号陷阱：`decltype(x)` 是 `int`，`decltype((x))` 多一对括号就变成 `int&`，在 `decltype(auto)` 返回值中写 `return (x)` 会返回局部变量的引用（悬空引用 UB）

**权衡**：`auto` 最适合用在类型显而易见或类型极其冗长的场景——迭代器、lambda、模板返回值。对于简单类型（`int`、`bool`、`std::string`），直接写明类型可读性更好。团队应制定使用规范：接口的函数签名和公开 API 显式声明类型，内部实现可以大方使用 `auto`。

> **面试速答**
>
> auto 和 decltype 都是 C++11 引入的编译期类型推导工具，但职责不同：auto 从初始化表达式推导类型，decltype 从表达式推导类型但不求值。它们解决的核心问题都是避免冗长类型声明——迭代器类型、模板返回类型这些写起来又长又容易错。
>
> 推导规则有区别。auto 的规则和模板参数推导一致，值拷贝的时候会丢掉 const 和引用（想保留就得写 `const auto&`）。decltype 有两套规则，这是考试重点：无括号的变量名得到声明类型（`decltype(x)` 是 int），加括号就当表达式处理（`decltype((x))` 是 int&）。
>
> 有两个容易踩的坑。一是 `decltype((x))` 多一对括号就变成引用类型，review 的时候很难发现；二是 `decltype(auto)` 搭配 `return (x)` 会返回悬空引用，因为括号让它变成表达式推导为引用了。
>
> 三者对比：auto 安全但不精确（丢引用丢 const）；decltype 精确但有括号陷阱；C++14 引入的 `decltype(auto)` 结合两者优点。选型建议：局部变量优先 auto；需要精确保留返回类型用 decltype(auto)；泛型库代码做类型计算用 decltype。

## 7. constexpr

> **一句话定义**：`constexpr` 告诉编译器"这个东西可以在编译期计算"，把运行时开销搬到编译期。

### 7.1 constexpr 变量 vs const

| 维度 | `const` | `constexpr` |
|------|---------|-------------|
| 含义 | "我承诺不修改" | "我能在编译期确定值" |
| 初始化时机 | 运行期也行 | **必须编译期** |
| 能用在哪 | 几乎任何变量 | 字面类型（整数、浮点、指针等） |
| 数组大小 | 不一定行（非整型常量） | 一定行 |

```cpp
const int a = get_value();      // OK：运行期初始化
constexpr int b = get_value();  // ❌ 错误：get_value() 不是 constexpr

constexpr int size = 100;
int arr[size];                  // OK：size 是编译期常量

const int size2 = 100;
int arr2[size2];                // OK（但只因为 size2 是整型常量表达式，换成其他类型不一定行）
```

### 7.2 constexpr 函数

**C++11 限制**：
- 函数体只能有一条 return 语句
- 不能有局部变量、循环、分支（条件运算符除外）

**C++14 放宽**：
- 允许局部变量、循环、if-else
- 可以修改局部变量

```cpp
// C++11: 只能用递归
constexpr int factorial_11(int n) {
    return n <= 1 ? 1 : n * factorial_11(n - 1);
}

// C++14: 可以写正常循环
constexpr int factorial_14(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i)
        result *= i;
    return result;
}

// 编译期计算
static_assert(factorial_14(5) == 120, "5! should be 120");

// constexpr 函数可以在运行期调用（参数不是编译期常量时）
int n;
std::cin >> n;
int result = factorial_14(n);  // 运行期调用，合法
```

### 7.3 编译期计算示例：编译期字符串哈希

```cpp
constexpr uint32_t fnv1a_hash(const char* str) {
    uint32_t hash = 2166136261u;
    while (*str) {
        hash ^= static_cast<uint32_t>(*str++);
        hash *= 16777619u;
    }
    return hash;
}

// 编译期计算哈希值，可用于 switch-case
void handle(const char* cmd) {
    switch (fnv1a_hash(cmd)) {
        case fnv1a_hash("start"):  // 编译期常量
            start(); break;
        case fnv1a_hash("stop"):
            stop(); break;
    }
}
```

#### 7.4 优缺点与权衡

**优点**：
- 将运行时开销搬到编译期：`constexpr` 函数在参数为编译期常量时，结果直接嵌入机器码（等价于字面量），运行时零开销。典型应用：编译期哈希、查找表生成、数学常量计算
- 比宏更安全：`constexpr` 有完整的类型检查、作用域规则和调试支持，替代 `#define MAX_SIZE 1024` 这类宏定义，消除宏的括号优先级陷阱和命名空间污染
- 编译期错误检测：配合 `static_assert` 可以在编译期验证业务约束（`static_assert(sizeof(Packet) == 64)`），把运行时才能发现的 bug 提前到编译期

**缺点/代价**：
- C++11 限制过严：只允许一条 return 语句，不能有局部变量、循环、if-else，强迫用递归写法，可读性差。C++14 大幅放宽但仍不支持动态内存分配（C++20 才支持编译期 new）
- 编译时间增长：复杂的 constexpr 计算（大型查找表、深递归）会显著增加编译时间，且编译器对 constexpr 递归深度有限制（通常 512 层，可通过编译选项调整）
- constexpr 函数不保证编译期执行：当参数不是编译期常量时，`constexpr` 函数退化为普通函数在运行期执行，程序员可能误以为已经编译期计算了实际没有。要强制编译期求值需用 `consteval`（C++20）

**权衡**：`constexpr` 的最大价值在于"免费的性能优化"——编译期能算的就不要留到运行期。适合常量定义、小型工具函数（哈希、位运算、数学公式）、查找表生成。但不应该为了 constexpr 把代码写得晦涩难懂，如果编译期计算让代码复杂度大幅上升，保留运行期计算并用注释说明理由更务实。

> **面试速答**
>
> constexpr 的核心意思是告诉编译器"这个值或函数可以在编译期计算"，把原本运行时的开销搬到编译期。它要替代的是两种老做法：宏定义常量（没有类型安全）和模板元编程中的编译期计算（写起来太复杂），constexpr 更安全、可调试、类型检查完备。
>
> 底层机制有个很巧妙的点：constexpr 函数在参数是编译期常量时就编译期执行，结果直接嵌到机器码里；参数是运行期值时自动退化为普通函数，一份代码两种用途。C++11 的 constexpr 限制比较严（只能单一 return），C++14 放宽了，允许循环和 if-else，实用性大增。
>
> 有个容易混淆的细节：constexpr 变量必须编译期初始化，const 变量可以运行期初始化，两者语义不同不要当成近义词用。横向看，const（不可修改，可运行期初始化）、constexpr（必须编译期确定）、C++17 的 `if constexpr`（编译期分支，优雅替代 SFINAE）是三个不同工具。
>
> 实际工程里，编译期常量就不要再用 #define 和 const 了，一律换成 constexpr；编译期哈希、查找表生成、static_assert 静态校验都是典型应用场景。

## 8. nullptr

> **一句话定义**：`nullptr` 是类型安全的空指针常量，类型为 `std::nullptr_t`。

### 为什么不用 NULL

```cpp
void f(int);
void f(int*);

f(NULL);     // ❌ 二义性！NULL 在大多数实现中是 #define NULL 0
             // 编译器不知道你想调 f(int) 还是 f(int*)

f(nullptr);  // ✅ 明确调用 f(int*)，因为 nullptr 是指针类型
f(0);        // 明确调用 f(int)
```

`nullptr` 的类型是 `std::nullptr_t`，可以隐式转换为任何指针类型，但不能转换为整数。

```cpp
int* p = nullptr;    // OK
int  n = nullptr;    // ❌ 编译错误
bool b = nullptr;    // OK: false（指针到 bool 的隐式转换）
```

> **面试速答**
>
> nullptr 是 C++11 引入的类型安全空指针常量，它的类型是 `std::nullptr_t`，可以隐式转换为任何指针类型，但不能转换为整数。它解决的问题是 NULL 留下的历史坑——NULL 本质上就是个宏 `#define NULL 0`，在函数重载时 `f(NULL)` 根本没法区分到底该匹配 `f(int)` 还是 `f(int*)`，nullptr 把这个二义性彻底消除了。
>
> 底层上 `std::nullptr_t` 是一个独立类型，编译器给它定义了到任意指针类型的隐式转换，同时禁止转换到整数类型，所以重载决议时能明确匹配指针版本。
>
> 有个模板相关的坑要特别注意：在模板代码中传 0 或 NULL 会被推导为 int 而不是指针类型，只有 nullptr 才能正确推导为指针。
>
> 选型没什么好犹豫的——所有空指针一律用 nullptr，杜绝 NULL 和 0，这是零成本的代码安全升级。

## 9. 初始化列表（initializer_list）

> **一句话定义**：`{}` 提供了统一的初始化语法，`std::initializer_list<T>` 是它在标准库层面的支撑。

### 9.1 统一初始化语法 {}

C++11 之前，初始化方式混乱：

```cpp
// C++03: 不同场景用不同语法
int a = 0;           // 赋值初始化
int b(0);            // 直接初始化
int c[3] = {1,2,3};  // 聚合初始化
std::vector<int> v;  // 默认初始化（然后 push_back）

// C++11: 统一用 {}
int a{0};
int b{0};
int c[]{1, 2, 3};
std::vector<int> v{1, 2, 3};
std::map<std::string, int> m{{"one", 1}, {"two", 2}};
```

### 9.2 std::initializer_list 的底层

`std::initializer_list<T>` 本质是一个**轻量级的视图对象**，内部只有两个指针（或一个指针加一个 size）：

```cpp
// 标准库实现（概念上）
template <typename T>
class initializer_list {
    const T* begin_;
    size_t   size_;
public:
    size_t size() const { return size_; }
    const T* begin() const { return begin_; }
    const T* end() const { return begin_ + size_; }
};
```

编译器在遇到 `{1, 2, 3}` 时，会在栈上创建一个 `const int[3]` 临时数组，然后构造一个 `initializer_list` 指向它。

```cpp
// 自定义类支持 initializer_list
class Matrix {
    std::vector<double> data_;
public:
    Matrix(std::initializer_list<double> list) : data_(list) {}
};

Matrix m = {1.0, 0.0, 0.0, 1.0};  // 4个元素的对角矩阵
```

#### 9.2.1 优缺点与权衡

**优点**：
- 统一初始化语法：`{}` 可以初始化任何类型——基本类型、数组、struct、容器、自定义类，告别 C++03 时代 `=`、`()`、`{}` 混用的混乱局面
- 禁止窄化转换（narrowing conversion）：`int x{3.14}` 编译报错，而 `int x = 3.14` 只是 warning，`{}` 在编译期就阻止了精度丢失的隐蔽 bug
- `initializer_list` 是轻量视图：内部只是一个指向栈上临时数组的指针 + size，拷贝 `initializer_list` 本身只拷贝指针，不拷贝元素，传递成本极低

**缺点/代价**：
- `{}` 优先匹配 `initializer_list` 构造函数导致语义歧义：`vector<int> v{10, 0}` 是两个元素（10 和 0），而 `vector<int> v(10, 0)` 是十个 0。这是 C++11 中最经典的"最令人惊讶"行为之一
- 窄化检测在某些场景过于严格：`char c{int_var}` 即使运行时值在 char 范围内也编译失败（因为编译器不知道运行时值），需要显式 `static_cast`，增加了代码冗余
- `initializer_list` 元素是 const 的：底层临时数组是 `const T[]`，无法移动其中的元素，对于 `initializer_list<string>` 等类型，每个 string 都会被拷贝而非移动

**权衡**：`{}` 初始化在大多数场景下比 `()` 更安全（禁止窄化），但在调用有 `initializer_list` 重载的构造函数时容易被"劫持"。实践中的策略是：普通变量和结构体用 `{}`，调用特定构造函数时用 `()` 避免歧义（如 `vector<int> v(10, 0)`）。

### 9.3 窄化转换的禁止

`{}` 初始化禁止窄化转换（narrowing conversion），这是比 `()` 初始化更安全的地方：

```cpp
int x = 3.14;    // OK（但丢精度，编译器可能只给 warning）
int y{3.14};     // ❌ 编译错误！double → int 是窄化转换

int a = 1000;
char b = a;      // OK（可能溢出，静默截断）
char c{a};       // ❌ 编译错误！int → char 可能窄化

// 安全的情况：
char d{65};      // OK: 65 在 char 范围内，编译期可确认
```

**注意 `{}` 和 `()` 的优先级陷阱**：

```cpp
std::vector<int> v1(10, 0);   // 10 个元素，都是 0
std::vector<int> v2{10, 0};   // 2 个元素: 10 和 0（优先匹配 initializer_list 构造函数！）
```

> **面试速答**
>
> `std::initializer_list<T>` 是 `{}` 统一初始化语法的底层支撑，本质上是个轻量级的视图对象，里面就一个指针加一个 size，指向编译器在栈上创建的 const T 临时数组。它解决的核心问题是统一初始化语法——让容器和自定义类型都能用 `{1, 2, 3}` 这种写法，而且 `{}` 还会禁止窄化转换提升安全性。
>
> 底层流程很直接：编译器遇到 `{...}` 时在栈上创建一个临时 const 数组，然后构造 initializer_list 视图传给构造函数。数组的生命周期和 initializer_list 绑定。`{}` 比 `()` 更安全，像 `int x{3.14}` 会直接编译报错禁止窄化；代价是增加了重载决议的复杂性。
>
> 有个非常经典的坑：`{}` 会优先匹配 initializer_list 构造函数。典型例子是 `vector<int> v(10, 0)` 是 10 个 0，而 `vector<int> v{10, 0}` 是两个元素 10 和 0，语义完全不同，容易写错。
>
> 实际建议：普通变量初始化优先用 `{}`（安全、禁止窄化）；调用特定构造函数时用 `()`，避免被 initializer_list 劫持。

### 10.2 enum class（强类型枚举）

```cpp
// C++03 的 enum 问题：枚举值泄漏到外层作用域，隐式转换为 int
enum Color { Red, Green, Blue };
enum Fruit { Apple, Orange };  // ❌ 如果有 Orange 和 Color 的 Orange 重名就冲突
int x = Red;  // 隐式转为 int，可能不是你想要的

// C++11 的 enum class：强类型、有作用域
enum class Color : uint8_t { Red, Green, Blue };  // 可指定底层类型
enum class Fruit { Apple, Orange };  // 不冲突

Color c = Color::Red;     // 必须带作用域
// int x = Color::Red;    // ❌ 不能隐式转换
int x = static_cast<int>(Color::Red);  // 需要显式转换
```

#### 10.2.1 优缺点与权衡

**优点**：
- 类型安全：不同 `enum class` 之间不能隐式比较或赋值（`Color::Red == Fruit::Apple` 编译报错），消除了传统 enum 隐式转 int 导致的逻辑 bug
- 作用域隔离：枚举值限定在枚举类型内（`Color::Red` 而非裸 `Red`），不会污染外层命名空间，不同枚举可以有同名成员（`Color::None` 和 `ErrorCode::None` 不冲突）
- 可控的底层类型：可以显式指定底层类型（`enum class Status : uint8_t`），保证跨平台 ABI 一致性和内存布局可预测

**缺点/代价**：
- 不能隐式转 int 增加了代码冗余：用于位运算、数组下标、日志输出等需要整数值的场景时，每次都要 `static_cast<int>(val)`，写法较啰嗦
- switch-case 需要完整作用域前缀：每个 case 都要写 `case Color::Red:` 而非 `case Red:`，当枚举值很多时代码变长（C++20 的 `using enum` 可以缓解）
- 不支持隐式布尔转换：传统 enum 可以直接 `if (error_code)` 判断非零，`enum class` 需要显式比较 `if (error_code != ErrorCode::None)`

**权衡**：`enum class` 用少量代码冗余（`static_cast`、作用域前缀）换取了类型安全和命名空间隔离。对于新项目，一律使用 `enum class` 替代传统 enum 是零争议的最佳实践。在需要频繁与整数交互的场景（位掩码），可以自定义 `operator|` / `operator&` 等运算符重载来减少 `static_cast` 的冗余。

> **面试速答**
>
> enum class 是 C++11 引入的强类型枚举：枚举值限定在作用域内，不会隐式转换为整数。它解决的是传统 enum 的三个历史问题——枚举值会泄漏到外层作用域导致命名冲突、隐式转 int 让类型安全形同虚设、底层类型由编译器决定导致跨平台 ABI 不一致。
>
> 用法上，枚举值必须带作用域访问（比如 `Color::Red`），不能隐式转 int，需要 int 值时得显式 static_cast，还可以显式指定底层类型（比如 `enum class Color : uint8_t`）来控制 ABI。
>
> 有个小代价：需要整数值时必须显式 static_cast，用于 switch 时每个 case 都要写完整作用域前缀，看起来略显冗长，但换来的是彻底的类型安全。
>
> 横向看：enum class 类型安全、作用域隔离；传统 enum 隐式转 int、污染命名空间；constexpr int 灵活但没有枚举语义。所有新代码一律用 enum class 替代传统 enum，零成本的类型安全升级。

## 11. 自测问题

**1. `std::move` 到底做了什么？它会触发移动构造吗？**

> [!note]- 答案
> `std::move` 不移动任何东西。它只是一个 `static_cast`，把左值转换为右值引用（xvalue）。真正的移动发生在移动构造函数或移动赋值运算符被调用时。如果目标类型没有移动构造函数，`std::move` 后的值会匹配到拷贝构造函数（`const T&` 能绑定右值），结果就是拷贝。

**2. 为什么右值引用变量本身是左值？这会导致什么问题？**

> [!note]- 答案
> 右值引用变量有名字、可以取地址，所以是左值。这意味着在函数 `void f(T&& x)` 内部，`x` 是左值。如果你把 `x` 传给另一个函数，它会被当作左值处理（调用拷贝而非移动）。要想保持右值性，必须再次 `std::move(x)` 或使用 `std::forward<T>(x)`（在模板中）。

**3. 移动构造函数不加 `noexcept` 会导致什么实际后果？**

> [!note]- 答案
> `std::vector` 在扩容（reallocate）时需要把旧元素搬到新内存。如果元素的移动构造函数不是 `noexcept`，vector 会退化为使用拷贝构造。原因：移动到一半如果抛异常，已经被移动的元素无法恢复原状；而拷贝不会修改源对象，出错可以安全回滚。因此移动操作不加 `noexcept` 可能导致严重的性能退化。

**4. `decltype(x)` 和 `decltype((x))` 有什么区别？为什么？**

> [!note]- 答案
> `decltype(x)` 中 x 是变量名，返回变量的声明类型（如 `int`）。`decltype((x))` 中 `(x)` 是表达式（加括号后不再是纯变量名），按表达式规则推导：`x` 是左值表达式，所以结果是 `int&`。这个区别在 `decltype(auto)` 返回值中尤其危险：`decltype(auto) f() { int x = 0; return (x); }` 返回 `int&`，是悬空引用！

**5. 万能引用和普通右值引用怎么区分？给出三个不是万能引用的 `&&` 例子。**

> [!note]- 答案
> 万能引用必须同时满足两个条件：(1) 形式是 `T&&`（T 直接修饰 `&&`）；(2) T 需要被推导。不是万能引用的例子：(a) `void f(int&&)` — 类型确定，不需要推导；(b) `template<typename T> void f(vector<T>&&)` — T 不直接修饰 `&&`；(c) 类模板成员 `template<typename T> class C { void f(T&&); }` — T 在类实例化时已确定，不参与推导。

**6. 引用折叠规则是什么？`T& &&` 和 `T&& &&` 分别折叠成什么？**

> [!note]- 答案
> 规则：只要有一个 `&`，结果就是 `&`。只有 `&& + &&` 才是 `&&`。所以 `T& &&` 折叠为 `T&`，`T&& &&` 折叠为 `T&&`。这就是完美转发的基础：传入左值时 T 推导为 `T&`，`T& &&` 折叠为 `T&`（保持左值）；传入右值时 T 推导为 `T`，`T&&` 就是右值引用（保持右值）。

**7. Lambda 的值捕获为什么默认是 const 的？如何修改？**

> [!note]- 答案
> Lambda 被编译器翻译成匿名类，值捕获的变量变成类成员，`operator()` 默认是 `const` 成员函数。const 成员函数不能修改成员变量，所以值捕获默认不可修改。使用 `mutable` 关键字可以去掉 `operator()` 的 const 限定，从而允许修改值捕获的副本。注意：修改的是 lambda 内部的副本，不影响外部原变量。

**8. `[=]` 捕获在类成员函数中有什么隐藏风险？**

> [!note]- 答案
> `[=]` 在类成员函数中会隐式捕获 `this` 指针（而不是拷贝成员变量）。虽然看起来像值捕获，实际上成员变量是通过 `this->member` 访问的。如果 lambda 的生命周期超过对象本身（比如存入异步回调），this 指针就会悬空。C++17 的 `[*this]` 可以拷贝整个对象来避免这个问题。C++20 中 `[=]` 隐式捕获 this 已被弃用。

**9. `const` 对象上调用 `std::move` 会发生什么？**

> [!note]- 答案
> `std::move(const_obj)` 返回 `const T&&` 类型。移动构造函数签名是 `T(T&&)`，不能接受 `const T&&`。但拷贝构造函数 `T(const T&)` 可以接受（`const T&` 能绑定任何值类别）。所以最终会调用拷贝构造函数，实际没有发生移动——move 了个寂寞。这是一个静默的性能陷阱，编译器不会报错。

**10. 为什么 `{}` 初始化比 `()` 初始化更安全？但什么时候会踩坑？**

> [!note]- 答案
> `{}` 禁止窄化转换（如 `int x{3.14}` 编译错误），而 `()` 允许静默截断。但踩坑点：如果类有接受 `initializer_list` 的构造函数，`{}` 会优先匹配它。经典例子：`vector<int> v1(10, 0)` 是 10 个 0，但 `vector<int> v2{10, 0}` 是两个元素 10 和 0。

**11. `constexpr` 函数一定在编译期执行吗？**

> [!note]- 答案
> 不一定。`constexpr` 函数在参数是编译期常量时可以在编译期执行，但如果参数是运行期值，它就退化为普通函数在运行期执行。只有用在 `constexpr` 变量初始化、`static_assert`、模板参数、数组大小等需要编译期常量的上下文中，编译器才会强制编译期求值。

**12. Rule of Five 是什么？什么时候可以不遵守？**

> [!note]- 答案
> 如果声明了析构函数、拷贝构造、拷贝赋值、移动构造、移动赋值中的任何一个，就应该声明全部五个。不遵守的唯一合理场景：类的所有成员都自己管理好了资源（如只包含智能指针、string、vector 等），此时全部用 `= default` 或者干脆一个都不写（Rule of Zero），让编译器自动生成。

**13. `std::forward` 和 `std::move` 的本质区别是什么？什么时候用哪个？**

> [!note]- 答案
> 两者都是 `static_cast`，但语义不同：`std::move` 无条件地转换为右值引用（"我不要了，随便偷"）；`std::forward<T>` 有条件地转换——保持参数原来的值类别（"它是左值就传左值，是右值就传右值"）。使用场景：`move` 用在你确定不再需要某个对象时；`forward` 只用在模板中转发万能引用参数时。在非模板代码中用 `forward` 没有意义。

**14. `enum class` 相比传统 `enum` 解决了哪些问题？**

> [!note]- 答案
> 三个问题：(1) 作用域污染——传统 enum 的枚举值泄漏到外层作用域，容易命名冲突；(2) 隐式转换——传统 enum 隐式转换为 int，`Color::Red == 0` 编译通过，类型不安全；(3) 底层类型不确定——传统 enum 的底层类型由编译器决定，可能导致 ABI 不一致。`enum class` 解决了全部三个：枚举值有作用域、不能隐式转换、可以指定底层类型。

**15. C++17 的 `if constexpr` 解决了什么问题？和普通 `if` 有什么本质区别？**

> [!note]- 答案
> `if constexpr` 是编译期分支：不走的分支直接被丢弃，不参与编译（不需要合法）。普通 `if` 的两个分支都必须能编译通过，即使条件是编译期可确定的。`if constexpr` 替代了大量 SFINAE、`enable_if`、标签分发（tag dispatch）等复杂的模板元编程技巧，让编译期分支逻辑变得直观可读。典型用途：根据类型特征选择不同实现路径。
