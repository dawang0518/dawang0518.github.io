### "现代 C++" 的核心理念变化

```
传统 C++:                          现代 C++:
├─ 手动 new/delete                 ├─ 智能指针，RAII 一切
├─ 原始指针传递所有权               ├─ unique_ptr 表达所有权，引用表达借用

```

**一句话总结**：现代 C++ 的核心是"零开销抽象" + "值语义优先" + "编译期计算最大化"。

## 1. 右值引用与移动语义（重点）

> **一句话定义**：右值引用让 C++ 能区分"即将销毁的临时对象"和"还要继续使用的对象"，从而安全地偷走临时对象的资源，而不是傻傻地拷贝一份。

### 1.1 值类别体系

C++11 把表达式分为 5 种值类别，形成一棵树：

```
              expression
              /        \
         glvalue      rvalue
         /    \       /    \
      lvalue  xvalue      prvalue
```

| 值类别 | 全称 | 有身份(identity)? | 可移动(movable)? | 典型示例 |
|--------|------|:--:|:--:|----------|
| lvalue | left value | Yes | No | 变量名 `x`、`*p`、`str[0]`、函数返回 `T&` |
| xvalue | expiring value | Yes | Yes | `std::move(x)`、函数返回 `T&&` |
| prvalue | pure rvalue | No | Yes | 字面量 `42`、`true`、`std::string("hi")`、函数返回 `T` |
| glvalue | generalized lvalue | Yes | - | lvalue + xvalue |
| rvalue | right value | - | Yes | xvalue + prvalue |

**关键洞察**：
- **有身份** = 可以取地址（`&expr` 合法）
- **可移动** = 资源可以被安全偷走
- `std::move(x)` 不移动任何东西，它只是把 lvalue 转成 xvalue，告诉编译器"这个对象可以被偷了"

### 1.2 左值引用 vs 右值引用 对比表

| 维度 | 左值引用 `T&` | 右值引用 `T&&` | `const T&` |
|------|--------------|---------------|------------|
| 绑定左值 | Yes | No | Yes |
| 绑定右值 | No | Yes | Yes |
| 延长临时对象生命周期 | No | Yes | Yes |
| 能修改绑定的对象 | Yes | Yes | No |
| 典型用途 | 函数参数传引用、输出参数 | 移动构造/赋值、完美转发 | 只读参数（万能接收） |
| 绑定后，引用本身是什么 | 左值 | **左值**（这是关键陷阱！） | 左值 |

**关键陷阱**：右值引用变量本身是左值！

```cpp
void foo(int&& x) {
    // x 的类型是 int&&（右值引用）
    // 但 x 这个表达式本身是左值！因为它有名字、可以取地址
    bar(x);            // 传给 bar 的是左值！
    bar(std::move(x)); // 需要再次 move 才能当右值传递
}
```

### 1.3 移动语义的本质：资源转移而非拷贝

传统 C++ 中，函数返回一个大对象时：

```
拷贝路径: 源对象 → 分配新内存 → 逐字节拷贝 → 销毁源对象    代价: O(n)
移动路径: 源对象 → 偷走指针 → 源对象置空                   代价: O(1)
```

### 1.4 完整示例：管理动态数组的类

```cpp
#include <iostream>
#include <cstring>
#include <algorithm>
#include <utility>

class DynArray {
private:
    int* data_;
    size_t size_;

public:
    // 构造函数
    explicit DynArray(size_t n) : data_(new int[n]()), size_(n) {
        std::cout << "  构造: 分配 " << n << " 个 int\n";
    }

    // 析构函数
    ~DynArray() {
        std::cout << "  析构: 释放 " << (data_ ? "有效" : "空") << " 指针\n";
        delete[] data_;
    }

    // ======= 拷贝语义 =======

    // 拷贝构造函数: 深拷贝，O(n)
    DynArray(const DynArray& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::memcpy(data_, other.data_, size_ * sizeof(int));
        std::cout << "  拷贝构造: 深拷贝 " << size_ << " 个 int\n";
    }

    // 拷贝赋值运算符: copy-and-swap 惯用法
    DynArray& operator=(const DynArray& other) {
        std::cout << "  拷贝赋值\n";
        DynArray tmp(other);       // 拷贝构造临时对象
        swap(*this, tmp);          // 交换资源
        return *this;              // tmp 析构时释放旧资源
    }

    // ======= 移动语义 =======

    // 移动构造函数: 偷资源，O(1)
    DynArray(DynArray&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;     // 关键！源对象必须置空
        other.size_ = 0;
        std::cout << "  移动构造: 偷走 " << size_ << " 个 int 的指针\n";
    }

    // 移动赋值运算符: O(1)
    DynArray& operator=(DynArray&& other) noexcept {
        std::cout << "  移动赋值\n";
        if (this != &other) {      // 自赋值检查
            delete[] data_;        // 释放自己的旧资源
            data_ = other.data_;   // 偷走源对象的资源
            size_ = other.size_;
            other.data_ = nullptr; // 源对象置空
            other.size_ = 0;
        }
        return *this;
    }

    // swap 函数（供 copy-and-swap 使用）
    friend void swap(DynArray& a, DynArray& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    size_t size() const { return size_; }
    int& operator[](size_t i) { return data_[i]; }
};

// 工厂函数：返回临时对象（触发移动）
DynArray createArray(size_t n) {
    DynArray arr(n);
    arr[0] = 42;
    return arr;  // NRVO 或 移动构造
}

int main() {
    std::cout << "=== 1. 工厂函数返回值 ===\n";
    DynArray a = createArray(1000);  // NRVO 优化，可能零拷贝

    std::cout << "\n=== 2. 拷贝构造 ===\n";
    DynArray b = a;  // 拷贝构造: O(n)

    std::cout << "\n=== 3. 移动构造 ===\n";
    DynArray c = std::move(a);  // 移动构造: O(1)，a 之后不可再用

    std::cout << "\n=== 4. 移动赋值 ===\n";
    DynArray d(10);
    d = std::move(b);  // 移动赋值: O(1)

    std::cout << "\n=== 析构顺序（逆序）===\n";
    return 0;
}
```

### 1.5 编译器何时自动生成移动操作

这是 C++11 最容易搞混的规则之一：

| 条件 | 自动生成移动构造? | 自动生成移动赋值? |
|------|:-:|:-:|
| 没有声明任何拷贝操作、移动操作、析构函数 | Yes | Yes |
| 声明了拷贝构造函数 | **No** | **No** |
| 声明了拷贝赋值运算符 | **No** | **No** |
| 声明了移动构造函数 | - | **No** |
| 声明了移动赋值运算符 | **No** | - |
| 声明了析构函数 | **No** | **No** |

**Rule of Five**：如果你声明了析构函数、拷贝构造、拷贝赋值、移动构造、移动赋值中的任何一个，你应该声明全部五个（或者用 `= default` / `= delete` 显式表态）。

```cpp
class Good {
    // Rule of Five: 要么全不写（编译器全自动生成），要么全写
    ~Good() = default;
    Good(const Good&) = default;
    Good& operator=(const Good&) = default;
    Good(Good&&) = default;
    Good& operator=(Good&&) = default;
};
```

### 1.6 拷贝构造为什么必须是 const T&

拷贝构造函数参数声明为 `const MyClass&` 的三个原因：

**1. 避免无限递归**：如果参数是值传递（非引用），调用拷贝构造时需要先拷贝实参，又触发拷贝构造，形成无限递归。引用传递直接绑定到原对象，不触发拷贝。

```cpp
// ❌ 错误：值传递导致无限递归
MyClass(MyClass other) { }  // 传入实参时要拷贝 → 又调用拷贝构造 → 死循环

// ✅ 正确：引用传递
MyClass(const MyClass& other) { }
```

**2. 支持 const 对象和临时对象**：非 const 引用（`MyClass&`）无法绑定到 const 对象或临时对象。`const MyClass&` 是万能接收者。

```cpp
const MyClass a;
MyClass b(a);  // 如果参数是 MyClass&，这里编译失败
```

**3. 防止意外修改源对象**：拷贝的语义是"复制"而非"修改"，`const` 在编译期保证源对象不被修改。

### 1.7 赋值运算符为什么返回 T&

赋值运算符返回 `MyClass&`（自身引用）的核心原因：

**1. 支持链式赋值**：`a = b = c;` 等价于 `a.operator=(b.operator=(c))`，`b = c` 必须返回 `b` 的引用才能继续赋值给 `a`。

```cpp
MyClass a, b, c;
a = b = c;  // 若返回值类型而非引用，会产生不必要的临时副本
```

**2. 与内置类型行为一致**：`int x, y, z; x = y = z = 10;` 返回的是引用。自定义类遵循同样的约定（最小惊讶原则）。

**3. 避免不必要的拷贝**：返回值类型每次赋值都触发拷贝构造，返回引用则零拷贝开销。

### 1.8 std::move 的真相

`std::move` 的实现极其简单，它只是一个 `static_cast`：

```cpp
// 标准库的 std::move 本质（简化版）
template <typename T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}
```

**std::move 不移动任何东西！** 它只是把一个左值 cast 成右值引用（xvalue），相当于告诉编译器："我不再需要这个对象了，你可以偷它的资源。"

真正的移动发生在移动构造函数或移动赋值运算符里。

**底层关键：`remove_reference` 类型萃取**

`std::move` 的核心依赖 `remove_reference`，通过模板偏特化移除类型的引用属性：

```cpp
// 主模板：非引用类型直接返回自身
template <typename T>
struct remove_reference { using type = T; };

// 偏特化：移除左值引用
template <typename T>
struct remove_reference<T&> { using type = T; };

// 偏特化：移除右值引用
template <typename T>
struct remove_reference<T&&> { using type = T; };
```

无论输入是左值、左值引用、右值引用还是临时对象，`std::move` 最终都返回对应的右值引用（`T&&`）。参数 `T&& t` 是万能引用，既能接收左值也能接收右值，然后通过 `static_cast` 统一转换为右值引用。

### 1.9 陷阱集锦

#### 陷阱 1: move 后继续使用对象

```cpp
// ❌ 错误：move 后对象处于"有效但未指定"状态，不要再读它的值
std::string name = "hello";
std::string other = std::move(name);
std::cout << name;          // 未定义行为? 不是——合法但值不确定
std::cout << name.size();   // 合法，但结果可能是 0 也可能不是

// ✅ 正确：move 后只能做两件事：销毁 或 重新赋值
std::string name2 = "hello";
std::string other2 = std::move(name2);
name2 = "world";            // 重新赋值，name2 又可以正常使用了
```

#### 陷阱 2: const 对象无法移动

```cpp
// ❌ 错误：const 对象 move 了个寂寞，实际调用的是拷贝构造
const std::string s = "big string";
std::string t = std::move(s);  // 调用的是拷贝构造！因为 const string&& 匹配不到移动构造

// 原因：移动构造函数签名是 String(String&&)，不接受 const String&&
// 而 const String&& 能匹配 const String&（拷贝构造函数），所以退化为拷贝
```

#### 陷阱 3: 右值引用参数在函数体内是左值

```cpp
// ❌ 错误：以为 arg 是右值引用就能自动传递右值性
void process(DynArray&& arg) {
    DynArray local = arg;  // 调用的是拷贝构造！arg 有名字，是左值
}

// ✅ 正确：需要再次 std::move
void process(DynArray&& arg) {
    DynArray local = std::move(arg);  // 调用移动构造
}
```

#### 陷阱 4: 移动构造/赋值不加 noexcept 的后果

```cpp
class Bad {
public:
    Bad(Bad&& other) { /* 没有 noexcept */ }
};

// std::vector 扩容时，如果移动构造不是 noexcept，
// vector 会选择拷贝而不是移动！
// 原因：如果移动到一半抛异常，已移动的元素无法恢复
// 拷贝则安全——源对象不变，出错可以回滚

// ✅ 正确：移动操作尽量标记 noexcept
class Good {
public:
    Good(Good&& other) noexcept { /* ... */ }
};
```

#### 1.10 优缺点与权衡

**右值引用 / 移动语义**：

**优点**：
- 将深拷贝 O(n) 降为指针转移 O(1)：对于持有堆资源的类型（`string`、`vector`、`unique_ptr`），移动构造只需要拷贝几个指针/size 字段并将源置空，避免了分配新内存 + 逐字节拷贝的开销
- 编译器自动优化临时对象：函数返回值、容器插入临时对象等场景，编译器通过重载决议自动选择移动构造，无需程序员干预即可获得性能提升
- 支持"只移动"类型：`unique_ptr`、`std::thread`、`std::fstream` 等独占资源的类型禁止拷贝但允许移动，语义上更精确地表达所有权转移

**缺点/代价**：
- 移后状态（moved-from state）需要程序员自律：标准只保证移后对象处于"有效但未指定"状态，读取其值是合法但结果不确定的，容易引入隐蔽 bug
- 必须标记 `noexcept` 否则退化：`std::vector` 扩容时检查移动构造是否 `noexcept`，如果不是则退化为拷贝（为了强异常安全保证），忘记标记 `noexcept` 会导致性能意外回退
- const 对象无法移动：`const T&&` 匹配不到 `T(T&&)` 移动构造，会默默退化为拷贝构造，代码看起来用了 `std::move` 但实际没有移动

**std::move 的核心权衡**：
- `std::move` 本身只是 `static_cast<T&&>` 的语法糖，不执行任何移动操作。真正的移动发生在移动构造/赋值函数中。调用 `std::move` 是程序员对编译器的承诺："我不再需要这个对象了"，如果承诺后继续使用该对象，责任在程序员

**权衡**：移动语义在含动态资源的类型上收益巨大（string、vector、map 等 STL 容器全部受益），且对于 POD 类型移动 == 拷贝，没有额外代价。主要的认知成本在于理解移后状态和 noexcept 的重要性。总体而言，这是 C++11 中投入产出比最高的特性。

> **面试速答**
> - **是什么/本质**：右值引用（T&&）是一种新的引用类型，能绑定到临时对象；std::move 只是 `static_cast<T&&>` 的无条件转换，本身不移动任何东西
> - **解决什么问题**：避免临时对象的深拷贝，将 O(n) 的拷贝变成 O(1) 的指针/资源"偷取"，大幅提升含动态资源的类（string、vector）的性能
> - **底层原理**：移动构造函数接管源对象的堆指针后将源置空，真正的移动逻辑在移动构造/移动赋值函数中实现，编译器通过重载决议选择
> - **坑点**：右值引用变量本身是左值（有名字可取地址），传递时需再次 std::move；移动操作必须标记 noexcept，否则 vector 扩容退化为拷贝；const 对象 move 后实际调用拷贝构造
> - **横向对比**：std::move（无条件转右值）vs std::forward（有条件保持值类别）；移动语义 vs copy elision（RVO/NRVO 比移动更优，零开销）

## 2. 完美转发

> **一句话定义**：完美转发让模板函数在传递参数时，保持参数原来的值类别（左值还是右值）不变。

### 2.1 问题：为什么需要完美转发？

假设你写一个工厂函数，转发参数给构造函数：

```cpp
// 不用完美转发，你需要写 2^n 个重载（n 是参数个数）
template <typename T>
std::unique_ptr<T> make(const T& arg) { return std::make_unique<T>(arg); }     // 左值版本
template <typename T>
std::unique_ptr<T> make(T&& arg) { return std::make_unique<T>(std::move(arg)); } // 右值版本
// 两个参数就要 4 个重载，三个参数 8 个... 指数爆炸！
```

### 2.2 万能引用（转发引用）

当 `T&&` 出现在**模板参数推导**上下文中时，它不是右值引用，而是**万能引用**（又叫转发引用）：

```cpp
template <typename T>
void f(T&& arg);  // T&& 是万能引用：既能接收左值，也能接收右值

auto&& x = expr;  // auto&& 也是万能引用
```

**注意区分**：

```cpp
template <typename T>
void f(T&& arg);           // ✅ 万能引用（T 需要推导）

void g(int&& arg);         // ❌ 不是万能引用，就是普通右值引用

template <typename T>
void h(std::vector<T>&& v); // ❌ 不是万能引用，v 就是 vector 的右值引用
                             //    因为 T 不是直接修饰 &&

template <typename T>
class Foo {
    void bar(T&& arg);     // ❌ 不是万能引用！T 在类实例化时已确定，不参与推导
};
```

### 2.3 引用折叠规则

C++ 不允许"引用的引用"，但模板推导中会产生，编译器用**引用折叠**解决：

| T 推导结果 | T&& 折叠后 | 规则 |
|-----------|-----------|------|
| `int&` | `int& &&` → `int&` | 左值引用 + 任何引用 = 左值引用 |
| `int&&` | `int&& &&` → `int&&` | 右值引用 + 右值引用 = 右值引用 |
| `int` | `int&&` | 无折叠 |

**口诀**：只要有一个 `&`，结果就是 `&`。只有 `&& + &&` 才保持 `&&`。

```
& + &   = &     (左 + 左 = 左)
& + &&  = &     (左 + 右 = 左)
&& + &  = &     (右 + 左 = 左)
&& + && = &&    (右 + 右 = 右)
```

### 2.4 std::forward 的实现原理

```cpp
// 简化版 std::forward
template <typename T>
constexpr T&& forward(std::remove_reference_t<T>& arg) noexcept {
    return static_cast<T&&>(arg);
}
```

**推导过程**：

```cpp
template <typename T>
void wrapper(T&& arg) {
    target(std::forward<T>(arg));
}

// 情况 1：传入左值 int x = 42; wrapper(x);
//   T 推导为 int&
//   forward<int&>(arg) → static_cast<int& &&>(arg) → static_cast<int&>(arg)
//   结果：左值引用 → 传给 target 的是左值 ✅

// 情况 2：传入右值 wrapper(42);
//   T 推导为 int
//   forward<int>(arg) → static_cast<int&&>(arg)
//   结果：右值引用 → 传给 target 的是右值 ✅
```

### 2.5 工厂函数中的完美转发示例

```cpp
#include <memory>
#include <utility>
#include <iostream>
#include <string>

class Connection {
public:
    Connection(const std::string& host, int port)
        : host_(host), port_(port) {
        std::cout << "Connection(const string&, int): 拷贝 host\n";
    }
    Connection(std::string&& host, int port)
        : host_(std::move(host)), port_(port) {
        std::cout << "Connection(string&&, int): 移动 host\n";
    }
private:
    std::string host_;
    int port_;
};

// 完美转发工厂函数：参数原封不动转发给构造函数
template <typename T, typename... Args>
std::unique_ptr<T> make(Args&&... args) {
    return std::make_unique<T>(std::forward<Args>(args)...);
}

int main() {
    std::string host = "127.0.0.1";

    // 传左值 → 拷贝
    auto c1 = make<Connection>(host, 8080);

    // 传右值 → 移动
    auto c2 = make<Connection>(std::string("192.168.1.1"), 9090);

    // 移动已有变量
    auto c3 = make<Connection>(std::move(host), 3306);
}
```

输出：
```
Connection(const string&, int): 拷贝 host
Connection(string&&, int): 移动 host
Connection(string&&, int): 移动 host
```

完美转发保证了：左值进去出来还是左值（走拷贝），右值进去出来还是右值（走移动），零额外开销。

#### 2.6 优缺点与权衡

**优点**：
- 消除指数级重载爆炸：没有完美转发，N 个参数的工厂函数需要 2^N 个重载（每个参数都有左值/右值两种版本）。一个万能引用 + `std::forward` 就能统一处理，`std::make_unique`、`emplace_back` 等标准库接口都依赖此机制
- 零开销值类别保持：`std::forward` 编译后只是一个 `static_cast`，没有任何运行时开销，左值进来出去还是左值（走拷贝），右值进来出去还是右值（走移动），透明转发不丢性能
- 泛型代码的基石：完美转发让库作者可以写出"透明包装层"（工厂函数、装饰器、线程池任务提交），调用方完全感知不到中间层的存在

**缺点/代价**：
- 理解成本极高：万能引用、引用折叠、`std::forward` 的推导过程涉及模板元编程的核心机制，是 C++ 中认知门槛最高的特性之一。代码审查时如果团队不熟悉，会成为维护负担
- 错误信息极其晦涩：万能引用匹配范围极广，一旦参数类型不对，编译器的模板实例化错误信息可能长达数十行且难以定位根因
- 容易和普通重载冲突：万能引用 `T&&` 比 `const T&` 更贪婪（精确匹配优于 const 转换），容易意外拦截本该走其他重载的调用，导致编译错误或运行时行为不符预期

**权衡**：完美转发的收益主要在库代码和基础设施层——工厂函数、emplace 接口、线程池任务提交等需要透传参数的场景。普通业务代码很少需要直接使用 `std::forward`。如果团队对模板元编程不熟悉，应将使用 `forward` 的代码封装在底层库中，上层业务代码通过简洁的接口调用，避免理解成本扩散。

> **面试速答**
> - **是什么/本质**：`std::forward<T>(arg)` 是有条件的类型转换，在模板转发中保持参数的原始值类别（左值/右值）不变
> - **解决什么问题**：万能引用 `T&&` 接收参数后，参数变成了左值（有名字），直接传递会丢失右值属性导致无法触发移动语义
> - **底层原理**：依赖万能引用 + 引用折叠规则（`& + && = &`，`&& + && = &&`）。传入左值时 T 推导为 `T&`，forward 返回左值引用；传入右值时 T 推导为 `T`，forward 返回右值引用
> - **坑点**：forward 只能用在模板转发场景中，非模板函数中使用无意义；T 必须来自模板推导，手动指定类型会破坏转发语义
> - **横向对比**：std::move 是无条件转右值（用于明确要移动的场景），std::forward 是有条件保持原值类别（只用于模板转发层）
> - **选型建议**：写工厂函数、emplace 接口、包装层等需要透传参数的模板代码时用 forward；明确要移动局部对象时用 move

## 3. Lambda 表达式

> **一句话定义**：Lambda 是就地定义的匿名函数对象，编译器会为每个 lambda 生成一个独一无二的匿名类。

### 3.1 语法结构拆解

```
[捕获列表](参数列表) mutable 异常声明 -> 返回类型 { 函数体 }
  │           │         │        │           │         │
  │           │         │        │           │         └─ 函数逻辑
  │           │         │        │           └─ 可省略（编译器推导）
  │           │         │        └─ 可选，通常省略
  │           │         └─ 可选，允许修改值捕获的变量
  │           └─ 和普通函数参数一样
  └─ 核心！决定 lambda 能访问哪些外部变量
```

### 3.2 捕获方式详解

| 捕获方式 | 语法 | 含义 | 注意事项 |
|----------|------|------|----------|
| 值捕获 | `[x]` | 拷贝 x 的值到 lambda 内部 | lambda 内的 x 是 const（除非 mutable） |
| 引用捕获 | `[&x]` | 引用外部的 x | x 销毁后 lambda 悬空！ |
| 隐式值捕获 | `[=]` | 按值捕获所有用到的变量 | 容易意外捕获 this |
| 隐式引用捕获 | `[&]` | 按引用捕获所有用到的变量 | 生命周期风险大 |
| 混合 | `[=, &x]` | 默认值捕获，x 引用捕获 | |
| 初始化捕获 (C++14) | `[y = std::move(x)]` | 移动 x 到 lambda 的成员 y | 解决了 C++11 无法移动捕获的问题 |
| 捕获 this | `[this]` | 捕获当前对象的 this 指针 | 小心对象先于 lambda 销毁 |
| 捕获 *this (C++17) | `[*this]` | 拷贝整个对象 | 安全但有开销 |

### 3.3 Lambda 底层：编译器生成的匿名类

每个 lambda 表达式，编译器都会生成一个匿名类。以下展示等价代码：

```cpp
// 你写的 lambda
int x = 10;
auto f = [x](int y) -> int { return x + y; };
int result = f(5);

// 编译器实际生成的等价代码（示意）
class __lambda_unique_name {
private:
    int x;  // 值捕获的变量成为成员
public:
    __lambda_unique_name(int x) : x(x) {}

    // operator() 默认是 const（所以不能修改值捕获的变量）
    int operator()(int y) const {
        return x + y;
    }
};

int x = 10;
auto f = __lambda_unique_name(x);  // 构造匿名对象，拷贝 x
int result = f(5);
```

**引用捕获的等价代码**：

```cpp
// 你写的 lambda
int x = 10;
auto f = [&x](int y) { x += y; return x; };

// 编译器生成的等价代码
class __lambda_ref {
private:
    int& x;  // 引用捕获 → 引用成员
public:
    __lambda_ref(int& x) : x(x) {}
    int operator()(int y) const {
        x += y;  // 注意：const 限定的是 operator()
        return x; // 但引用本身不变，修改的是引用指向的对象，所以合法
    }
};
```

**mutable lambda 的等价代码**：

```cpp
// 你写的 lambda
int x = 10;
auto f = [x]() mutable { return ++x; };
// f() → 11, f() → 12, f() → 13... 每次调用都修改内部副本

// 编译器生成的等价代码
class __lambda_mutable {
private:
    int x;
public:
    __lambda_mutable(int x) : x(x) {}
    // mutable → operator() 不是 const
    int operator()() {
        return ++x;  // 可以修改成员
    }
};
```

### 3.4 泛型 Lambda (C++14)

```cpp
// C++14: auto 参数 → 编译器生成模板 operator()
auto add = [](auto a, auto b) { return a + b; };

// 等价于:
struct __generic_lambda {
    template <typename T1, typename T2>
    auto operator()(T1 a, T2 b) const {
        return a + b;
    }
};

add(1, 2);         // T1=int, T2=int
add(1.5, 2);       // T1=double, T2=int
add(std::string("a"), std::string("b")); // T1=string, T2=string
```

### 3.5 Lambda 的类型：std::function 与函数指针

每个 Lambda 的类型是编译器生成的**唯一匿名类型**，无法直接声明，但可通过以下方式间接使用：

**1. auto 推导**：直接推导为闭包类的类型（最高效，零开销）。

**2. std::function 包装**：通用多态函数包装器，可存储任意可调用对象（有类型擦除开销）：

```cpp
#include <functional>
std::function<int(int, int)> add = [](int a, int b) { return a + b; };
```

**3. 函数指针（仅无捕获 Lambda）**：无捕获的 Lambda 可隐式转换为函数指针（因为其 `operator()` 是静态可调用的）：

```cpp
auto add = [](int a, int b) { return a + b; };
int (*func_ptr)(int, int) = add;  // 合法（无捕获）

// 有捕获的 Lambda 无法转换为函数指针
// 因为闭包对象包含成员变量，需要实例状态
int x = 10;
auto with_capture = [x](int a) { return a + x; };
// int (*p)(int) = with_capture;  // ❌ 编译错误
```

**选择建议**：优先用 `auto`（零开销）；需要存储不同签名的可调用对象时用 `std::function`；需要兼容 C 回调接口时用函数指针（仅无捕获 Lambda 可用）。

### 3.6 陷阱集锦

#### 陷阱 1: 引用捕获悬空

```cpp
// ❌ 错误：lambda 引用捕获了局部变量，函数返回后悬空
std::function<int()> make_counter() {
    int count = 0;
    return [&count]() { return ++count; };  // count 已销毁！
}

// ✅ 正确：值捕获 + mutable
std::function<int()> make_counter() {
    int count = 0;
    return [count]() mutable { return ++count; };
}
```

#### 陷阱 2: 捕获 this 的生命周期问题

```cpp
class Server {
    int port_ = 8080;
public:
    // ❌ 错误：lambda 捕获 this，但 Server 对象可能先于 lambda 销毁
    auto get_handler() {
        return [this]() {
            return port_;  // this 可能已经是野指针
        };
    }

    // ✅ 正确方案 1 (C++17)：拷贝整个对象
    auto get_handler_safe() {
        return [*this]() {
            return port_;  // 使用的是拷贝的副本
        };
    }

    // ✅ 正确方案 2 (C++14)：初始化捕获需要的成员
    auto get_handler_safe2() {
        return [port = port_]() {
            return port;  // 值捕获，安全
        };
    }
};
```

#### 陷阱 3: [=] 隐式捕获 this

```cpp
class Foo {
    int val_ = 42;
public:
    auto make_lambda() {
        // ❌ 风险：[=] 看起来是值捕获，但 val_ 实际通过 this 指针访问
        return [=]() { return val_; };  // 等价于 [this]() { return this->val_; }
        // C++20 中 [=] 隐式捕获 this 会产生 deprecation 警告

        // ✅ 明确：要么 [*this] 拷贝对象，要么 [val = val_] 拷贝值
    }
};
```

#### 3.7 优缺点与权衡

**优点**：
- 就地定义闭包，消除样板代码：不需要像传统 functor 那样单独定义一个类，算法谓词、回调函数可以在调用点直接编写，代码局部性极好，可读性提升
- 比 `std::function` 零开销：`auto` 推导的 lambda 类型是编译器生成的唯一类型，编译器可以完全内联 `operator()`，没有类型擦除和堆分配开销。相比之下 `std::function` 需要类型擦除 + 可能的堆分配（小对象优化除外）
- 捕获上下文的能力：值捕获和引用捕获让 lambda 可以访问外层作用域的变量，这是普通函数指针做不到的。C++14 的初始化捕获更支持移动语义（`[p = std::move(ptr)]`）

**缺点/代价**：
- 捕获生命周期陷阱：引用捕获 `[&]` 最危险——如果 lambda 的生存期超过被捕获变量（异步回调、存入容器、跨线程传递），引用悬空导致 UB，且 bug 极难定位
- `[=]` 在成员函数中隐式捕获 this 指针：看起来是值捕获所有变量，实际上成员变量通过 `this->` 访问，`this` 指向的对象销毁后 lambda 仍持有野指针。C++20 已将此行为标记为 deprecated
- 每个 lambda 类型唯一：两个签名完全相同的 lambda 是不同类型，无法放入同一个 `std::vector<auto>` 中，需要用 `std::function` 做类型擦除，又引入了额外开销

**权衡**：lambda 在同步、短生命周期场景中（STL 算法谓词、局部回调）是完美工具——零开销、就地定义、可内联。但在异步/长生命周期场景中（存入容器、跨线程传递、延迟执行），必须格外注意捕获变量的生命周期：优先值捕获 `[=]`，成员函数中用 `[*this]`（C++17）或初始化捕获 `[val = member_]`，彻底避免引用悬空。

> **面试速答**
> - **是什么/本质**：编译器生成的匿名类（闭包类型），捕获的变量变成类成员，函数体变成 `operator()` 重载
> - **解决什么问题**：替代函数指针和 functor 的冗长写法，让回调、算法谓词可以就地定义并捕获上下文
> - **底层原理**：值捕获生成值成员（operator() 默认 const 不可修改，加 mutable 才可改），引用捕获生成引用成员。每个 lambda 类型唯一，只能用 auto 推导
> - **优缺点**：比函数指针灵活（能捕获上下文），比 std::function 高效（无类型擦除开销，可内联）；无捕获 lambda 可隐式转函数指针兼容 C API
> - **坑点**：引用捕获悬空是最大陷阱；`[=]` 在成员函数中隐式捕获 this 指针而非拷贝成员值，C++17 用 `[*this]` 解决
> - **横向对比**：lambda（零开销、类型唯一）vs std::function（类型擦除、可存储但有堆分配开销）vs 函数指针（无法捕获上下文）

## 4. auto 与 decltype

> **一句话定义**：`auto` 让编译器从初始化表达式推导类型；`decltype` 让编译器从表达式推导类型但不求值。

### 4.1 auto 推导规则

`auto` 的推导规则和模板参数推导一致：

| 场景 | 代码 | 推导结果 | 说明 |
|------|------|----------|------|
| 基本 | `auto x = 42;` | `int` | |
| 去 const | `const int ci = 0; auto x = ci;` | `int` | 值拷贝丢 const |
| 去引用 | `int& r = i; auto x = r;` | `int` | 值拷贝丢引用 |
| 保 const | `const int ci = 0; const auto& x = ci;` | `const int&` | 显式声明 |
| 指针 | `int* p = &i; auto x = p;` | `int*` | 指针不丢 |
| 万能引用 | `auto&& x = lvalue;` | `T&`(左值引用) | 引用折叠 |
| 万能引用 | `auto&& x = rvalue;` | `T&&`(右值引用) | |
| initializer_list | `auto x = {1, 2, 3};` | `initializer_list<int>` | 注意这个特殊规则 |
| 花括号直接初始化 | `auto x{42};` | C++17 起是 `int` | C++11/14 是 `initializer_list<int>` |

### 4.2 decltype 推导规则

`decltype` 有两套不同的规则，取决于操作数是**变量名**还是**表达式**：

| 操作数 | 规则 | 示例 | 结果 |
|--------|------|------|------|
| 无括号的变量名 | 得到变量的**声明类型** | `int x; decltype(x)` | `int` |
| 有括号的变量名 | 当作表达式处理（见下方） | `int x; decltype((x))` | `int&`（左值表达式→引用） |
| 左值表达式 | `T&` | `decltype(*p)` | `int&` |
| 右值表达式 | `T` | `decltype(42)` | `int` |
| xvalue 表达式 | `T&&` | `decltype(std::move(x))` | `int&&` |

**关键陷阱**：`decltype(x)` 和 `decltype((x))` 结果不同！

```cpp
int x = 0;
decltype(x) a = x;     // int    （变量名 → 声明类型）
decltype((x)) b = x;   // int&   （加了括号 → 左值表达式 → 引用）
```

### 4.3 decltype(auto) (C++14)

`decltype(auto)` 结合了 `auto` 的便利和 `decltype` 的精确：

```cpp
// auto 会丢引用，decltype(auto) 不会
int x = 0;
int& rx = x;

auto a = rx;            // int    （auto 丢引用）
decltype(auto) b = rx;  // int&   （保持引用）

// 常用于返回值：保持原始返回类型
template <typename Container, typename Index>
decltype(auto) access(Container&& c, Index i) {
    return std::forward<Container>(c)[i];  // 如果 c[i] 返回引用，这里也返回引用
}
```

### 4.4 返回值类型后置

```cpp
// 问题：返回类型依赖参数类型，写在前面编译器还没看到参数
template <typename T, typename U>
auto add(T a, U b) -> decltype(a + b) {
    return a + b;
}

// C++14 起可以省略后置返回类型（编译器从 return 语句推导）
template <typename T, typename U>
auto add(T a, U b) {
    return a + b;
}
```

#### 4.5 优缺点与权衡

**优点**：
- 消除冗长类型声明：`auto it = container.begin()` 替代 `std::map<std::string, std::vector<int>>::const_iterator it = ...`，代码简洁性和可维护性大幅提升，尤其在模板密集的代码中
- 保证初始化：`auto x = expr;` 语法上强制要求有初始化表达式，不可能出现未初始化变量的 bug
- `decltype` 支持精确类型计算：`decltype(auto)` 可以在返回值中保留引用和 cv 限定符，是泛型库代码实现完美返回类型的核心工具

**缺点/代价**：
- 可读性降低：过度使用 `auto` 会导致阅读代码时无法直接看出变量类型，尤其在没有 IDE 辅助的代码审查场景中。`auto x = foo();` 必须跳转到 `foo()` 的定义才能知道 x 的类型
- `auto` 丢引用丢 const 可能引入隐蔽 bug：`auto x = const_ref;` 会拷贝一份而非绑定引用，意外拷贝大对象既影响性能又可能导致逻辑错误（修改了副本而非原对象）
- `decltype((x))` 的括号陷阱：`decltype(x)` 是 `int`，`decltype((x))` 多一对括号就变成 `int&`，在 `decltype(auto)` 返回值中写 `return (x)` 会返回局部变量的引用（悬空引用 UB）

**权衡**：`auto` 最适合用在类型显而易见或类型极其冗长的场景——迭代器、lambda、模板返回值。对于简单类型（`int`、`bool`、`std::string`），直接写明类型可读性更好。团队应制定使用规范：接口的函数签名和公开 API 显式声明类型，内部实现可以大方使用 `auto`。

> **面试速答**
> - **是什么/本质**：auto 是编译期类型推导，从初始化表达式推导类型；decltype 从表达式推导类型但不求值
> - **解决什么问题**：避免冗长的类型声明（迭代器、模板返回类型），提升泛型编程的可维护性
> - **底层原理**：auto 推导规则和模板参数推导一致——值拷贝时丢掉 const 和引用。decltype 有两套规则：无括号变量名得声明类型（`decltype(x)` 是 int），加括号当表达式处理（`decltype((x))` 是 int&）
> - **坑点**：`decltype((x))` 多一对括号就变成引用类型，极易踩坑；`decltype(auto)` 搭配 `return (x)` 会返回悬空引用
> - **横向对比**：auto（丢引用丢 const，安全但不精确）vs decltype（保留完整类型信息，精确但有括号陷阱）vs `decltype(auto)`（C++14，结合两者优点）
> - **选型建议**：局部变量优先用 auto；需要精确保留返回类型时用 decltype(auto)；泛型库代码中用 decltype 做类型计算

## 5. constexpr

> **一句话定义**：`constexpr` 告诉编译器"这个东西可以在编译期计算"，把运行时开销搬到编译期。

### 5.1 constexpr 变量 vs const

| 维度 | `const` | `constexpr` |
|------|---------|-------------|
| 含义 | "我承诺不修改" | "我能在编译期确定值" |
| 初始化时机 | 运行期也行 | **必须编译期** |
| 能用在哪 | 几乎任何变量 | 字面类型（整数、浮点、指针等） |
| 数组大小 | 不一定行（非整型常量） | 一定行 |

```cpp
const int a = get_value();      // OK：运行期初始化
constexpr int b = get_value();  // ❌ 错误：get_value() 不是 constexpr

constexpr int size = 100;
int arr[size];                  // OK：size 是编译期常量

const int size2 = 100;
int arr2[size2];                // OK（但只因为 size2 是整型常量表达式，换成其他类型不一定行）
```

### 5.2 constexpr 函数

**C++11 限制**：
- 函数体只能有一条 return 语句
- 不能有局部变量、循环、分支（条件运算符除外）

**C++14 放宽**：
- 允许局部变量、循环、if-else
- 可以修改局部变量

```cpp
// C++11: 只能用递归
constexpr int factorial_11(int n) {
    return n <= 1 ? 1 : n * factorial_11(n - 1);
}

// C++14: 可以写正常循环
constexpr int factorial_14(int n) {
    int result = 1;
    for (int i = 2; i <= n; ++i)
        result *= i;
    return result;
}

// 编译期计算
static_assert(factorial_14(5) == 120, "5! should be 120");

// constexpr 函数可以在运行期调用（参数不是编译期常量时）
int n;
std::cin >> n;
int result = factorial_14(n);  // 运行期调用，合法
```

### 5.3 编译期计算示例：编译期字符串哈希

```cpp
constexpr uint32_t fnv1a_hash(const char* str) {
    uint32_t hash = 2166136261u;
    while (*str) {
        hash ^= static_cast<uint32_t>(*str++);
        hash *= 16777619u;
    }
    return hash;
}

// 编译期计算哈希值，可用于 switch-case
void handle(const char* cmd) {
    switch (fnv1a_hash(cmd)) {
        case fnv1a_hash("start"):  // 编译期常量
            start(); break;
        case fnv1a_hash("stop"):
            stop(); break;
    }
}
```

#### 5.4 优缺点与权衡

**优点**：
- 将运行时开销搬到编译期：`constexpr` 函数在参数为编译期常量时，结果直接嵌入机器码（等价于字面量），运行时零开销。典型应用：编译期哈希、查找表生成、数学常量计算
- 比宏更安全：`constexpr` 有完整的类型检查、作用域规则和调试支持，替代 `#define MAX_SIZE 1024` 这类宏定义，消除宏的括号优先级陷阱和命名空间污染
- 编译期错误检测：配合 `static_assert` 可以在编译期验证业务约束（`static_assert(sizeof(Packet) == 64)`），把运行时才能发现的 bug 提前到编译期

**缺点/代价**：
- C++11 限制过严：只允许一条 return 语句，不能有局部变量、循环、if-else，强迫用递归写法，可读性差。C++14 大幅放宽但仍不支持动态内存分配（C++20 才支持编译期 new）
- 编译时间增长：复杂的 constexpr 计算（大型查找表、深递归）会显著增加编译时间，且编译器对 constexpr 递归深度有限制（通常 512 层，可通过编译选项调整）
- constexpr 函数不保证编译期执行：当参数不是编译期常量时，`constexpr` 函数退化为普通函数在运行期执行，程序员可能误以为已经编译期计算了实际没有。要强制编译期求值需用 `consteval`（C++20）

**权衡**：`constexpr` 的最大价值在于"免费的性能优化"——编译期能算的就不要留到运行期。适合常量定义、小型工具函数（哈希、位运算、数学公式）、查找表生成。但不应该为了 constexpr 把代码写得晦涩难懂，如果编译期计算让代码复杂度大幅上升，保留运行期计算并用注释说明理由更务实。

> **面试速答**
> - **是什么/本质**：告诉编译器"这个值/函数可以在编译期计算"，把运行时开销搬到编译期
> - **解决什么问题**：替代宏定义常量和模板元编程中的编译期计算，更安全、可调试、类型检查完备
> - **底层原理**：constexpr 函数在参数是编译期常量时编译期执行（结果嵌入机器码），参数是运行期值时退化为普通函数。C++14 放宽限制，允许循环和 if-else
> - **坑点**：constexpr 变量必须编译期初始化，const 变量可以运行期初始化——两者语义不同不要混淆
> - **横向对比**：const（不可修改，可运行期初始化）vs constexpr（必须编译期确定）vs `if constexpr`（C++17，编译期分支，替代 SFINAE）
> - **选型建议**：编译期常量一律用 constexpr 替代 #define 和 const；编译期哈希、查找表、static_assert 校验是典型应用场景

## 6. nullptr

> **一句话定义**：`nullptr` 是类型安全的空指针常量，类型为 `std::nullptr_t`。

### 为什么不用 NULL

```cpp
void f(int);
void f(int*);

f(NULL);     // ❌ 二义性！NULL 在大多数实现中是 #define NULL 0
             // 编译器不知道你想调 f(int) 还是 f(int*)

f(nullptr);  // ✅ 明确调用 f(int*)，因为 nullptr 是指针类型
f(0);        // 明确调用 f(int)
```

`nullptr` 的类型是 `std::nullptr_t`，可以隐式转换为任何指针类型，但不能转换为整数。

```cpp
int* p = nullptr;    // OK
int  n = nullptr;    // ❌ 编译错误
bool b = nullptr;    // OK: false（指针到 bool 的隐式转换）
```

> **面试速答**
> - **是什么/本质**：C++11 引入的类型安全空指针常量，类型为 `std::nullptr_t`，可隐式转换为任何指针类型但不能转换为整数
> - **解决什么问题**：NULL 本质是宏 `#define NULL 0`，在函数重载时 `f(NULL)` 无法区分 `f(int)` 和 `f(int*)`，nullptr 消除二义性
> - **底层原理**：`std::nullptr_t` 是独立类型，编译器为它定义了到任意指针类型的隐式转换，但禁止到整数类型的转换，重载决议时明确匹配指针版本
> - **坑点**：在模板代码中传 0 或 NULL 会被推导为 int 而非指针类型，只有 nullptr 能正确推导
> - **选型建议**：所有空指针一律用 nullptr，杜绝 NULL 和 0，这是零成本的代码安全升级

## 7. 初始化列表（initializer_list）

> **一句话定义**：`{}` 提供了统一的初始化语法，`std::initializer_list<T>` 是它在标准库层面的支撑。

### 7.1 统一初始化语法 {}

C++11 之前，初始化方式混乱：

```cpp
// C++03: 不同场景用不同语法
int a = 0;           // 赋值初始化
int b(0);            // 直接初始化
int c[3] = {1,2,3};  // 聚合初始化
std::vector<int> v;  // 默认初始化（然后 push_back）

// C++11: 统一用 {}
int a{0};
int b{0};
int c[]{1, 2, 3};
std::vector<int> v{1, 2, 3};
std::map<std::string, int> m{{"one", 1}, {"two", 2}};
```

### 7.2 std::initializer_list 的底层

`std::initializer_list<T>` 本质是一个**轻量级的视图对象**，内部只有两个指针（或一个指针加一个 size）：

```cpp
// 标准库实现（概念上）
template <typename T>
class initializer_list {
    const T* begin_;
    size_t   size_;
public:
    size_t size() const { return size_; }
    const T* begin() const { return begin_; }
    const T* end() const { return begin_ + size_; }
};
```

编译器在遇到 `{1, 2, 3}` 时，会在栈上创建一个 `const int[3]` 临时数组，然后构造一个 `initializer_list` 指向它。

```cpp
// 自定义类支持 initializer_list
class Matrix {
    std::vector<double> data_;
public:
    Matrix(std::initializer_list<double> list) : data_(list) {}
};

Matrix m = {1.0, 0.0, 0.0, 1.0};  // 4个元素的对角矩阵
```

#### 7.2.1 优缺点与权衡

**优点**：
- 统一初始化语法：`{}` 可以初始化任何类型——基本类型、数组、struct、容器、自定义类，告别 C++03 时代 `=`、`()`、`{}` 混用的混乱局面
- 禁止窄化转换（narrowing conversion）：`int x{3.14}` 编译报错，而 `int x = 3.14` 只是 warning，`{}` 在编译期就阻止了精度丢失的隐蔽 bug
- `initializer_list` 是轻量视图：内部只是一个指向栈上临时数组的指针 + size，拷贝 `initializer_list` 本身只拷贝指针，不拷贝元素，传递成本极低

**缺点/代价**：
- `{}` 优先匹配 `initializer_list` 构造函数导致语义歧义：`vector<int> v{10, 0}` 是两个元素（10 和 0），而 `vector<int> v(10, 0)` 是十个 0。这是 C++11 中最经典的"最令人惊讶"行为之一
- 窄化检测在某些场景过于严格：`char c{int_var}` 即使运行时值在 char 范围内也编译失败（因为编译器不知道运行时值），需要显式 `static_cast`，增加了代码冗余
- `initializer_list` 元素是 const 的：底层临时数组是 `const T[]`，无法移动其中的元素，对于 `initializer_list<string>` 等类型，每个 string 都会被拷贝而非移动

**权衡**：`{}` 初始化在大多数场景下比 `()` 更安全（禁止窄化），但在调用有 `initializer_list` 重载的构造函数时容易被"劫持"。实践中的策略是：普通变量和结构体用 `{}`，调用特定构造函数时用 `()` 避免歧义（如 `vector<int> v(10, 0)`）。

### 7.3 窄化转换的禁止

`{}` 初始化禁止窄化转换（narrowing conversion），这是比 `()` 初始化更安全的地方：

```cpp
int x = 3.14;    // OK（但丢精度，编译器可能只给 warning）
int y{3.14};     // ❌ 编译错误！double → int 是窄化转换

int a = 1000;
char b = a;      // OK（可能溢出，静默截断）
char c{a};       // ❌ 编译错误！int → char 可能窄化

// 安全的情况：
char d{65};      // OK: 65 在 char 范围内，编译期可确认
```

**注意 `{}` 和 `()` 的优先级陷阱**：

```cpp
std::vector<int> v1(10, 0);   // 10 个元素，都是 0
std::vector<int> v2{10, 0};   // 2 个元素: 10 和 0（优先匹配 initializer_list 构造函数！）
```

> **面试速答**
> - **是什么/本质**：`std::initializer_list<T>` 是 `{}` 语法的底层支撑，本质是轻量视图对象（一个指针 + size），指向编译器在栈上创建的 const T 临时数组
> - **解决什么问题**：统一初始化语法，让容器、自定义类型都能用 `{1, 2, 3}` 语法初始化，且禁止窄化转换提升安全性
> - **底层原理**：编译器遇到 `{...}` 时在栈上创建临时 const 数组，构造 initializer_list 视图传给构造函数。数组生命周期与 initializer_list 一致
> - **优缺点**：`{}` 比 `()` 更安全（`int x{3.14}` 编译报错禁止窄化）；但增加了重载决议的复杂性
> - **坑点**：`{}` 优先匹配 initializer_list 构造函数：`vector<int> v(10, 0)` 是 10 个 0，而 `vector<int> v{10, 0}` 是两个元素 10 和 0，语义完全不同
> - **选型建议**：普通变量初始化用 `{}`；调用特定构造函数时用 `()` 避免被 initializer_list 劫持

## 8. 其他重要特性速查

### 8.1 using 别名模板

```cpp
// C++03: typedef 无法模板化
typedef std::map<std::string, int> StringIntMap;  // OK，但不能带模板参数

// C++11: using 支持模板别名
template <typename T>
using StringMap = std::map<std::string, T>;

StringMap<int> m1;      // std::map<std::string, int>
StringMap<double> m2;   // std::map<std::string, double>

// 也可以简单替代 typedef
using Callback = std::function<void(int)>;  // 比 typedef 更直观
```

### 8.2 enum class（强类型枚举）

```cpp
// C++03 的 enum 问题：枚举值泄漏到外层作用域，隐式转换为 int
enum Color { Red, Green, Blue };
enum Fruit { Apple, Orange };  // ❌ 如果有 Orange 和 Color 的 Orange 重名就冲突
int x = Red;  // 隐式转为 int，可能不是你想要的

// C++11 的 enum class：强类型、有作用域
enum class Color : uint8_t { Red, Green, Blue };  // 可指定底层类型
enum class Fruit { Apple, Orange };  // 不冲突

Color c = Color::Red;     // 必须带作用域
// int x = Color::Red;    // ❌ 不能隐式转换
int x = static_cast<int>(Color::Red);  // 需要显式转换
```

#### 8.2.1 优缺点与权衡

**优点**：
- 类型安全：不同 `enum class` 之间不能隐式比较或赋值（`Color::Red == Fruit::Apple` 编译报错），消除了传统 enum 隐式转 int 导致的逻辑 bug
- 作用域隔离：枚举值限定在枚举类型内（`Color::Red` 而非裸 `Red`），不会污染外层命名空间，不同枚举可以有同名成员（`Color::None` 和 `ErrorCode::None` 不冲突）
- 可控的底层类型：可以显式指定底层类型（`enum class Status : uint8_t`），保证跨平台 ABI 一致性和内存布局可预测

**缺点/代价**：
- 不能隐式转 int 增加了代码冗余：用于位运算、数组下标、日志输出等需要整数值的场景时，每次都要 `static_cast<int>(val)`，写法较啰嗦
- switch-case 需要完整作用域前缀：每个 case 都要写 `case Color::Red:` 而非 `case Red:`，当枚举值很多时代码变长（C++20 的 `using enum` 可以缓解）
- 不支持隐式布尔转换：传统 enum 可以直接 `if (error_code)` 判断非零，`enum class` 需要显式比较 `if (error_code != ErrorCode::None)`

**权衡**：`enum class` 用少量代码冗余（`static_cast`、作用域前缀）换取了类型安全和命名空间隔离。对于新项目，一律使用 `enum class` 替代传统 enum 是零争议的最佳实践。在需要频繁与整数交互的场景（位掩码），可以自定义 `operator|` / `operator&` 等运算符重载来减少 `static_cast` 的冗余。

> **面试速答**
> - **是什么/本质**：强类型枚举，枚举值限定在作用域内，不隐式转换为整数
> - **解决什么问题**：传统 enum 的三个问题——枚举值泄漏到外层作用域导致命名冲突、隐式转 int 导致类型不安全、底层类型由编译器决定导致 ABI 不一致
> - **底层原理**：枚举值必须带作用域访问（`Color::Red`），不能隐式转 int（需 static_cast），可显式指定底层类型（如 `enum class Color : uint8_t`）
> - **坑点**：需要整数值时必须显式 static_cast，用于 switch 时每个 case 都要写完整作用域前缀，略显冗长
> - **横向对比**：enum class（类型安全、作用域隔离）vs 传统 enum（隐式转 int、污染命名空间）vs constexpr int（灵活但无枚举语义）
> - **选型建议**：所有新代码一律用 enum class 替代传统 enum，是零成本的类型安全升级

### 8.3 noexcept

```cpp
// 声明函数不会抛异常
void safe_func() noexcept;           // 承诺不抛异常
void maybe_throw() noexcept(false);  // 可能抛异常
void cond(T t) noexcept(noexcept(t.foo()));  // 条件式 noexcept

// 关键影响：
// 1. 移动构造/赋值标记 noexcept → vector 扩容时用移动而不是拷贝
// 2. 如果 noexcept 函数实际抛了异常 → 直接 std::terminate()
// 3. 编译器可以做更激进的优化
```

### 8.4 override / final

```cpp
class Base {
public:
    virtual void foo(int);
    virtual void bar() const;
    virtual void baz();
};

class Derived : public Base {
public:
    void foo(int) override;       // ✅ 明确重写
    void foo(double) override;    // ❌ 编译错误！Base 没有 foo(double)
    // 没有 override 时，上面这行只是定义了新函数，不报错，bug 难查

    void bar() override;          // ❌ 编译错误！const 不匹配
    void baz() final;             // 重写 + 禁止子类再重写
};

class Final final {               // 禁止继承
    // ...
};
```

## 9. 自测问题

**1. `std::move` 到底做了什么？它会触发移动构造吗？**

> [!note]- 答案
> `std::move` 不移动任何东西。它只是一个 `static_cast`，把左值转换为右值引用（xvalue）。真正的移动发生在移动构造函数或移动赋值运算符被调用时。如果目标类型没有移动构造函数，`std::move` 后的值会匹配到拷贝构造函数（`const T&` 能绑定右值），结果就是拷贝。

**2. 为什么右值引用变量本身是左值？这会导致什么问题？**

> [!note]- 答案
> 右值引用变量有名字、可以取地址，所以是左值。这意味着在函数 `void f(T&& x)` 内部，`x` 是左值。如果你把 `x` 传给另一个函数，它会被当作左值处理（调用拷贝而非移动）。要想保持右值性，必须再次 `std::move(x)` 或使用 `std::forward<T>(x)`（在模板中）。

**3. 移动构造函数不加 `noexcept` 会导致什么实际后果？**

> [!note]- 答案
> `std::vector` 在扩容（reallocate）时需要把旧元素搬到新内存。如果元素的移动构造函数不是 `noexcept`，vector 会退化为使用拷贝构造。原因：移动到一半如果抛异常，已经被移动的元素无法恢复原状；而拷贝不会修改源对象，出错可以安全回滚。因此移动操作不加 `noexcept` 可能导致严重的性能退化。

**4. `decltype(x)` 和 `decltype((x))` 有什么区别？为什么？**

> [!note]- 答案
> `decltype(x)` 中 x 是变量名，返回变量的声明类型（如 `int`）。`decltype((x))` 中 `(x)` 是表达式（加括号后不再是纯变量名），按表达式规则推导：`x` 是左值表达式，所以结果是 `int&`。这个区别在 `decltype(auto)` 返回值中尤其危险：`decltype(auto) f() { int x = 0; return (x); }` 返回 `int&`，是悬空引用！

**5. 万能引用和普通右值引用怎么区分？给出三个不是万能引用的 `&&` 例子。**

> [!note]- 答案
> 万能引用必须同时满足两个条件：(1) 形式是 `T&&`（T 直接修饰 `&&`）；(2) T 需要被推导。不是万能引用的例子：(a) `void f(int&&)` — 类型确定，不需要推导；(b) `template<typename T> void f(vector<T>&&)` — T 不直接修饰 `&&`；(c) 类模板成员 `template<typename T> class C { void f(T&&); }` — T 在类实例化时已确定，不参与推导。

**6. 引用折叠规则是什么？`T& &&` 和 `T&& &&` 分别折叠成什么？**

> [!note]- 答案
> 规则：只要有一个 `&`，结果就是 `&`。只有 `&& + &&` 才是 `&&`。所以 `T& &&` 折叠为 `T&`，`T&& &&` 折叠为 `T&&`。这就是完美转发的基础：传入左值时 T 推导为 `T&`，`T& &&` 折叠为 `T&`（保持左值）；传入右值时 T 推导为 `T`，`T&&` 就是右值引用（保持右值）。

**7. Lambda 的值捕获为什么默认是 const 的？如何修改？**

> [!note]- 答案
> Lambda 被编译器翻译成匿名类，值捕获的变量变成类成员，`operator()` 默认是 `const` 成员函数。const 成员函数不能修改成员变量，所以值捕获默认不可修改。使用 `mutable` 关键字可以去掉 `operator()` 的 const 限定，从而允许修改值捕获的副本。注意：修改的是 lambda 内部的副本，不影响外部原变量。

**8. `[=]` 捕获在类成员函数中有什么隐藏风险？**

> [!note]- 答案
> `[=]` 在类成员函数中会隐式捕获 `this` 指针（而不是拷贝成员变量）。虽然看起来像值捕获，实际上成员变量是通过 `this->member` 访问的。如果 lambda 的生命周期超过对象本身（比如存入异步回调），this 指针就会悬空。C++17 的 `[*this]` 可以拷贝整个对象来避免这个问题。C++20 中 `[=]` 隐式捕获 this 已被弃用。

**9. `const` 对象上调用 `std::move` 会发生什么？**

> [!note]- 答案
> `std::move(const_obj)` 返回 `const T&&` 类型。移动构造函数签名是 `T(T&&)`，不能接受 `const T&&`。但拷贝构造函数 `T(const T&)` 可以接受（`const T&` 能绑定任何值类别）。所以最终会调用拷贝构造函数，实际没有发生移动——move 了个寂寞。这是一个静默的性能陷阱，编译器不会报错。

**10. 为什么 `{}` 初始化比 `()` 初始化更安全？但什么时候会踩坑？**

> [!note]- 答案
> `{}` 禁止窄化转换（如 `int x{3.14}` 编译错误），而 `()` 允许静默截断。但踩坑点：如果类有接受 `initializer_list` 的构造函数，`{}` 会优先匹配它。经典例子：`vector<int> v1(10, 0)` 是 10 个 0，但 `vector<int> v2{10, 0}` 是两个元素 10 和 0。

**11. `constexpr` 函数一定在编译期执行吗？**

> [!note]- 答案
> 不一定。`constexpr` 函数在参数是编译期常量时可以在编译期执行，但如果参数是运行期值，它就退化为普通函数在运行期执行。只有用在 `constexpr` 变量初始化、`static_assert`、模板参数、数组大小等需要编译期常量的上下文中，编译器才会强制编译期求值。

**12. Rule of Five 是什么？什么时候可以不遵守？**

> [!note]- 答案
> 如果声明了析构函数、拷贝构造、拷贝赋值、移动构造、移动赋值中的任何一个，就应该声明全部五个。不遵守的唯一合理场景：类的所有成员都自己管理好了资源（如只包含智能指针、string、vector 等），此时全部用 `= default` 或者干脆一个都不写（Rule of Zero），让编译器自动生成。

**13. `std::forward` 和 `std::move` 的本质区别是什么？什么时候用哪个？**

> [!note]- 答案
> 两者都是 `static_cast`，但语义不同：`std::move` 无条件地转换为右值引用（"我不要了，随便偷"）；`std::forward<T>` 有条件地转换——保持参数原来的值类别（"它是左值就传左值，是右值就传右值"）。使用场景：`move` 用在你确定不再需要某个对象时；`forward` 只用在模板中转发万能引用参数时。在非模板代码中用 `forward` 没有意义。

**14. `enum class` 相比传统 `enum` 解决了哪些问题？**

> [!note]- 答案
> 三个问题：(1) 作用域污染——传统 enum 的枚举值泄漏到外层作用域，容易命名冲突；(2) 隐式转换——传统 enum 隐式转换为 int，`Color::Red == 0` 编译通过，类型不安全；(3) 底层类型不确定——传统 enum 的底层类型由编译器决定，可能导致 ABI 不一致。`enum class` 解决了全部三个：枚举值有作用域、不能隐式转换、可以指定底层类型。

**15. C++17 的 `if constexpr` 解决了什么问题？和普通 `if` 有什么本质区别？**

> [!note]- 答案
> `if constexpr` 是编译期分支：不走的分支直接被丢弃，不参与编译（不需要合法）。普通 `if` 的两个分支都必须能编译通过，即使条件是编译期可确定的。`if constexpr` 替代了大量 SFINAE、`enable_if`、标签分发（tag dispatch）等复杂的模板元编程技巧，让编译期分支逻辑变得直观可读。典型用途：根据类型特征选择不同实现路径。
